# 1. GPT Architecture

## What is GPT?

GPT stands for **Generative Pre-trained Transformer**. Each word in the name conveys a key property:

- **Generative**: It can _produce new data_ rather than only classifying or analyzing existing inputs.
- **Pre-trained**: It is trained on vast datasets to learn grammar, facts, and context _before_ fine-tuning for specific tasks.
- **Transformer**: Built on the Transformer architecture using _self-attention_ to process text efficiently.

## GPT vs. BERT — Key Contrast

|Property|GPT-2|BERT|
|---|---|---|
|Architecture|Decoder-only|Encoder-only|
|Direction|Left-to-right (causal/autoregressive)|Bidirectional (full context at once)|
|Context vector initialization|Zero-initialized for first word|Uses entire surrounding context|
|Next token prediction|Yes (one token at a time)|No (masked language modeling)|

**GPT-2 does NOT require an encoder.** It has no encoder attention blocks. It takes word vectors as input and outputs probabilities for the next word.

## Three Major Phases of GPT Architecture

### Phase 1: Input Processing

- **Input Embedding**: Tokens (words/subwords) are converted into dense numerical vectors.
- **Positional Encoding**: Since transformers have no inherent notion of order, position information is _added_ to the token embeddings. GPT uses **absolute positional embeddings** that are learned/optimized during training.

### Phase 2: Transformer Blocks (stacked N times)

Each Transformer Block contains:

- **Multi-head Self-Attention Mechanism**: Allows each token to attend to all previous tokens.
    - Operations: Matmul → Mask → Softmax for each attention head.
    - Linear projections for **Query (Q)**, **Key (K)**, **Value (V)** vectors.
- **Residual Connections** (the "+" operation): Skip connections that add the block's input to its output, preventing vanishing gradients.
- **Layer Normalization**: Applied at multiple points to stabilize training.
- **Feed-Forward Network (FFN)**: Sequence: `Linear → GELU → Linear`.
- **Dropout Layers**: Applied during training for regularization.

### Phase 3: Output Processing

- **Final Layer Normalization**
- **Linear Projection Layer**: Projects to vocabulary size.
- **Softmax**: Converts logits (raw scores) into a probability distribution over all possible next tokens.

## Autoregressive Nature

GPT works **one token at a time**. Each token in the sentence has context of all _previous_ words. This is called the **autoregressive** property:

> Given tokens $w_1, w_2, \ldots, w_n$, GPT predicts $w_{n+1}$, appends it, then predicts $w_{n+2}$, and so on until a stop condition is reached.

---

## Q&A — GPT Architecture

**Q1: Why is GPT called "decoder-only"?** A: GPT only uses the decoder part of the original Transformer architecture. It does not have an encoder or encoder-attention blocks.

**Q2: What is the role of positional encoding in GPT?** A: Token embeddings alone don't capture _where_ a token appears in a sequence. Positional encodings add position information. GPT uses absolute positional embeddings that are learned during training and added to token embeddings.

**Q3: What is the difference between GPT and BERT in terms of attention?** A: GPT uses causal (masked) attention — each token only sees previous tokens. BERT uses bidirectional attention — every token sees every other token. GPT is thus suitable for text generation; BERT is better for understanding tasks.

**Q4: Why does GPT use residual connections?** A: Residual connections help gradients flow during backpropagation in deep networks, preventing vanishing gradients and improving training stability.

**Q5: What does "auto-regressive" mean in the context of GPT?** A: Auto-regressive means the model generates text one token at a time, using previously generated tokens as context for generating the next one. The context vector is zero-initialized for the first token.

# 2. Coding a GPT-like LLM (Stage 1)

Building a GPT-like LLM involves multiple stages. The key focus is **Stage 1: Data Preparation and Core Modules**.

## Stage 1: Data Preparation

- LLMs cannot process raw text. Raw text must be converted into numerical vectors called **embeddings**.
- The pipeline:
    1. **Tokenization**: Raw text is broken into tokens (words or subword units).
    2. **Token IDs**: Tokens are mapped to integer IDs.
    3. **Special Tokens**: Added for special purposes:
        - `<|unk|>` — unknown words.
        - `<|endoftext|>` — boundary between unrelated texts.
    4. **Byte Pair Encoding (BPE)**: Used in GPT-2/GPT-3. Handles unknown words by breaking them into subword units or individual characters.

## Stage 1(a): Sliding Window & Embeddings

- A **sliding window approach** is used on tokenized data to generate input–target pairs for LLM training.
- **Embedding layers in PyTorch** function as lookup tables — they retrieve vectors corresponding to token IDs.
- Two types of positional embeddings:
    - **Absolute**: Fixed or learned position vector added to token embedding. Used by GPT.
    - **Relative**: Encodes relative distances between tokens. Used by other models.

## Stage 1(b): Attention Mechanism

- **Self-attention** (also called scaled dot-product attention) uses _trainable weight matrices_ to compute:
    - **Queries (Q)**: What the token is looking for.
    - **Keys (K)**: What the token offers.
    - **Values (V)**: The actual content.
- **Causal attention mask**: Prevents the model from attending to future tokens (crucial for left-to-right language modeling). Implemented by zeroing out attention weights for future positions.
- **Dropout mask**: Added to reduce overfitting.
- **Multi-head attention**: Multiple instances of causal attention run in parallel, capturing different types of relationships.

## Stage 1(c): Transformer Block Components

To build a GPT-like model, you need:

- **GPT Placeholder Code** (the main model class)
- **Transformer Block**
- **Layer Normalization**
- **GELU Activation** (see Section 4)
- **Feed-Forward Neural Network**
- **Shortcut (Residual) Connections**

---

## Q&A — Coding GPT

**Q1: Why can't LLMs process raw text directly?** A: LLMs perform mathematical operations on numbers. Raw text must first be converted into numerical vectors (embeddings) via tokenization and embedding layers.

**Q2: What is the purpose of the causal attention mask?** A: It prevents the model from "seeing" future tokens while predicting the current token. This enforces the left-to-right autoregressive constraint — the model should only predict the next token based on what came before.

**Q3: What is the difference between absolute and relative positional embeddings?** A: Absolute positional embeddings assign a unique vector to each position index (position 1, position 2, etc.), added to the token embedding. Relative positional embeddings encode the _distance_ between pairs of tokens. GPT uses absolute positional embeddings.

**Q4: Why is multi-head attention used instead of a single attention mechanism?** A: Multiple attention heads allow the model to jointly attend to information from different representation subspaces at different positions. Each head may learn different types of relationships (syntax, semantics, coreference, etc.).

### GPT-1

- First model to use a transformer-based **decoder-only** architecture for language generation.
- 117 million parameters.

### GPT-2

- **13x more parameters** than GPT-1 (1.5B vs. 117M).
- Trained on **WebText** — a large, high-quality dataset.
- Demonstrated strong zero-shot generalization.

### GPT-3

- **175B parameters** — over 100x GPT-2.
- Trained on **500B tokens** from a diverse dataset.
- Excels at **few-shot learning** without task-specific fine-tuning.

### GPT-NeoX-20B

- 20B parameters.
- Improvements over GPT-3:
    - **Rotary Positional Embeddings (RoPE)**: More effective than absolute positional embeddings for capturing relative position.
    - **Parallel Attention and Feed-Forward layers**: Reduces sequential computation.
    - Exclusive use of **dense layers**.

## Small Language Models (SLMs)

SLMs are scaled-down versions of LLMs, designed to be resource-efficient:

- **Gemini Nano** — 122M parameters from Google, designed for edge devices.
- **DistilBERT** — 40% smaller than BERT, 60% faster, retains 97% of BERT's language understanding.
- **GPT-Neo** — 125M and 1.3B parameter models by EleutherAI.

## Shortcomings of GPT-like Models

1. Need a specific loss/reward function to quantify model error.
2. Trained to mimic human text by predicting the next token — this does NOT perfectly correlate with higher-level objectives like "reasoning."
3. Perform best on tasks _similar_ to training data but fail on sufficiently novel tasks.

# 4. GPT Design Choices

## 4.1 Contextual Word Embeddings

Traditional embeddings like Word2Vec and fastText assign a **fixed vector** to each word regardless of context. Transformer models generate **contextual embeddings** — the embedding for a word changes based on the sentence it appears in.

**Mathematical Formulation:**

- Let $w_1, w_2, \ldots, w_n$ be tokens in a sentence.
- Each token has a high-dimensional vector $x_i \in \mathbb{R}^d$.
- The contextual embedding $C_k$ is computed as:

$$C_k = f(w_k \mid w_1, w_2, \ldots, w_n) \in \mathbb{R}^d$$

Here, $f$ maps token $w_k$ to its embedding _given the full context_. This is crucial for tasks like Neural Machine Translation where the meaning of a word depends heavily on context (e.g., "bank" — river bank or financial bank).

## 4.2 LLM as a Next-Token Predictor

The core operation of an LLM:

1. Start with $n$ tokens as context.
2. Generate token at position $n+1$.
3. Append the new token to expand the context.
4. Generate token at position $n+2$ using the expanded context.
5. Repeat until a stop condition (specific stop token or max length).

**Why include the prompt?**

- **Context and Coherence**: The prompt tells the model what topic and constraints are required.
- **Prompt Engineering**: Including instructions or few-shot examples guides the model for higher quality outputs.
- **Controlling Behavior**: Prompts allow the model to adopt roles, follow formats, etc.

## 4.3 Dropout and Zoneout

**Dropout**:

- Randomly _deactivates (drops)_ certain neurons during each training iteration.
- Those neurons do not contribute to that forward/backward pass.
- Introduces randomness → improves generalization over unseen data.

**Zoneout**:

- Instead of dropping neurons, it _stochastically retains_ a neuron's original value (multiplies input by 1 — identity masking).
- Ensures controlled randomness.
- Balances exploration and exploitation during training.

**Brain analogy for Zoneout**: The brain randomly stops processing external stimuli during boredom/fatigue, similar to how Zoneout randomly preserves hidden state values.

## 4.4 GELU (Gaussian Error Linear Unit)

**Conceptual View**: GELU = ReLU + Dropout + Zoneout (not a direct formula, but shares properties)

**Why GELU over ReLU?**

|Property|ReLU|GELU|
|---|---|---|
|Output for negative input|Exactly 0 (hard threshold)|Small non-zero output (smooth)|
|Optimization|Sharp corner at zero, can be harder|Smoother, better gradient properties|
|Dead neurons|Can occur (always zero gradient)|Less likely (still contributes)|

**GELU Definition:** $$\text{GELU}(x) = x \cdot \Phi(x)$$

Where $\Phi(x)$ is the **cumulative distribution function (CDF)** of the standard normal distribution, often approximated with the error function:

$$\text{GELU}(x) \approx 0.5 \cdot x \cdot \left(1 + \tanh\left[\sqrt{\frac{2}{\pi}}\left(x + 0.044715x^3\right)\right]\right)$$

**Why normal distribution?** Neuron inputs tend to follow a normal distribution (especially after Batch Normalization). GELU stochastically multiplies the input by a value between 0 and 1 based on the input's probability in a standard normal, rather than using a hard threshold.

**Key insight**: GELU allows neurons that receive negative input to still contribute to learning (unlike ReLU which zeros them out completely), enabling more nuanced gradient flow.

## 4.5 Tokenization

**Pipeline:**

1. Normalization (optional)
2. Segmentation (determines vocabulary)
3. Token ID assignment

**Trade-off in vocabulary size:**

- Larger vocabulary → model understands more things → larger model size.
- Too large vocabulary → slower (more computations), excessive memory.

### BPE (Byte Pair Encoding) Recap

- Used by GPT-2 and GPT-3.
- Algorithm:
    1. Start with character-level tokens.
    2. Find pairs of consecutive tokens whose merging most shrinks the vocabulary.
    3. Merge them into a single token.
    4. Repeat until vocabulary is "small enough."
- Almost no normalization in modern BPE implementations.
- **Expensive** to build: must read input data many times to find most frequent combinations.
- After BPE vocabulary is built, model authors manually add domain-specific tokens.

### Tokenization and LLM Capabilities

- Most LLM tokenizers support Unicode (characters from most world alphabets).
- However, tokenizers are typically trained on English-dominated data, so non-English languages are tokenized less efficiently.
- **Cost implications**: Tokenizer efficiency directly affects inference cost. German and Italian cost ~50% more than English on ChatGPT. Tumbuka and Bulgarian are more than twice the cost.
- Input tokens are cheaper than output tokens (model reads input once; each output token requires a full forward pass).
- **KV-caching** is used to cache attention states for input tokens, reducing repeated computation.

### Tokenization Challenges

- **Word games**: LLMs don't understand letters, only tokens. "Amoxicillin" and "Amoxicillan" share no common tokens in GPT-3.
- **Mathematics**: "123456" becomes two tokens: ["123", "456"]. This makes digit-level reasoning harder. Fix: insert spaces between digits ("1 2 3 4 5 6").

### Detokenization

- LLM output is in subword units → must be decoded back to full words.
- Process: Predict token ID → Map ID to subword → Add to output string → Repeat until end-of-sequence token.

## 4.6 Adam vs. AdamW Optimizers

**Adam (Adaptive Moment Estimation):**

- Combines **momentum** (weighted average of current gradient and previous momentum) and **RMSprop** (weighted average of squared gradient).
- Momentum: Accelerates optimization.
- RMSprop: Adapts per-parameter learning rate.
- Also applies bias correction and has a combined parameter update formula.
- **Downside**: In Adam, weight decay is added as an L2 regularization term to the _loss function_. This interacts with the adaptive learning rates and can hinder optimal convergence.
- **Memory cost**: Adam requires ~3x the memory of plain SGD because it stores momentum and velocity (squared gradient) for every parameter.

**AdamW:**

- A smarter Adam. Decouples weight decay from the gradient update step.
- Instead of adding weight decay to the loss, it applies weight decay _directly during the parameter update_.
- This leads to more consistent regularization and better generalization.
- **Use Adam** for quick prototyping or simpler tasks.
- **Use AdamW** for training complex, high-dimensional models like LLMs.

## Q&A — GPT Design Choices

**Q1: What is the mathematical formula for contextual embedding in a transformer?** A: $C_k = f(w_k \mid w_1, w_2, \ldots, w_n) \in \mathbb{R}^d$, where $f$ maps the token $w_k$ to a high-dimensional vector conditioned on all tokens in the sentence.

**Q2: Why can't GELU be simply defined as ReLU?** A: GELU allows small non-zero outputs for negative inputs (smooth probabilistic weighting), whereas ReLU hard-zeroes all negative inputs. GELU provides better optimization properties especially in deep models, and is less prone to the "dying neuron" problem.

**Q3: What is the key difference between dropout and zoneout?** A: Dropout randomly deactivates neurons entirely (sets output to 0). Zoneout randomly retains the previous hidden state (identity masking). Both introduce controlled randomness, but zoneout preserves more information from previous states.

**Q4: Why does AdamW outperform Adam for LLM training?** A: In Adam, weight decay is applied as part of the loss function, which interferes with the adaptive learning rate mechanism. AdamW decouples weight decay from gradient updates, applying it directly to parameters. This results in more consistent regularization without distorting the adaptive learning rate properties.

**Q5: Why are larger vocabularies a trade-off?** A: A larger vocabulary allows the model to represent more words/subwords natively (better understanding), but increases model size, memory usage, and computational cost for every forward pass (the final projection layer must be of size vocabulary × hidden dimension).

# 5. Decoding Strategies & Hyperparameters

## 5.1 Overview of Decoding Options

When using a GPT-like LLM, the following hyperparameters control generation:

- **Decoding Strategy** (Greedy, Beam Search, Top-K, Top-P)
- **Temperature**
- **Frequency Penalty**
- **Presence Penalty**
- **Context Width**
- **Max Response**

## 5.2 Greedy Decoding

- At every step, select the **highest-probability token**.
- Simple and fast.
- **Problem**: Local minima trap — the best token right now may not lead to the globally best sequence overall.

## 5.3 Beam Search

- Maintains `k` candidate sequences (beams) at each step.
- Explores multiple paths simultaneously.

**Advantages:**

- Balances exploration and exploitation.
- Often more coherent and diverse than greedy.
- Considers multiple paths → more globally optimal sequences.

**Limitations:**

- Can still suffer from repetition.
- Does NOT capture the full diversity of possible outputs.
- Computationally expensive for large `k`.

## 5.4 Random Sampling

- Instead of always picking the highest-probability token, _randomly sample_ from the full probability distribution.
- Produces more diverse and unpredictable outputs.

**Problem with pure random sampling**: The **long tail** — the model assigns low probabilities to many possible words. Randomly sampling from this long tail produces nonsensical, grammatically incorrect, or incoherent text.

## 5.5 Top-K Sampling

- At each step, restrict sampling to the **top-K most probable tokens**.
- Randomly sample from this restricted set.

**Advantages:**

- Introduces randomness and diversity.
- Allows creative exploration beyond the single highest-probability choice.
- `k=1` is equivalent to greedy search.
- `k=V` (full vocabulary) is equivalent to pure random sampling.

**Limitations:**

- Less coherent than beam search.
- Choice of `k` is critical — too small: not enough diversity; too large: too much randomness.

## 5.6 Top-P (Nucleus) Sampling

- At each step, consider only the **smallest set of top tokens whose cumulative probability is at least p**.
- Unlike Top-K, the number of tokens considered is **dynamically adjusted** based on the probability distribution.

**Benefits:**

1. Balanced diversity and coherence.
2. Dynamic adjustment → more natural, human-like text.
3. Adjustable trade-off via parameter `p`.

**Effect of `p`:**

- `p = 0.9`: Small set of likely words → more coherent, less creative.
- `p = 0.5`: Larger set → more diverse, potentially less coherent.
- `p = 1.0`: All tokens → equivalent to pure sampling → highly creative but potentially incoherent.

## 5.7 Temperature

Temperature $T$ scales the logits before softmax, controlling how "peaked" or "flat" the probability distribution is.

$$P_i = \frac{e^{z_i / T}}{\sum_j e^{z_j / T}}$$

Where $z_i$ are the logits and $T$ is temperature.

- **T = 1**: Standard distribution — balanced diversity and coherence.
- **T > 1**: Flatter distribution — less probable words are more likely → more creative and diverse output.
- **T < 1**: Sharper distribution — highest probability tokens dominate → more deterministic, conservative output.

## 5.8 Frequency Penalty

- Applies a **penalty proportional to how many times** a token has already appeared.
- Higher frequency penalty → less likely that a word appears again.
- Reduces repetition of the same words.

## 5.9 Presence Penalty

- Applies a **flat penalty** to ANY token that has appeared at least once (regardless of how many times).
- A token appearing twice and a token appearing ten times get the SAME penalty.
- Prevents repetitive _phrases_ more than repetitive _words_.
- Higher presence penalty → more diverse vocabulary.

**Frequency vs. Presence Penalty Comparison:**

|Aspect|Frequency Penalty|Presence Penalty|
|---|---|---|
|Penalty basis|Proportional to count of appearances|Flat, binary (appeared or not)|
|Effect|Reduces high-frequency word repetition|Prevents any reuse of previously used words|
|Use case|Prevent getting "stuck in a loop"|Encourage broader vocabulary|

## 5.10 Context Window

- The range of tokens an LLM considers when making predictions.
- Larger context window → captures long-range dependencies → better predictions for complex tasks.

**Pros of larger context window:**

- Improved comprehension of long-range relationships.
- Better performance at complex tasks (translation, summarization, sentiment analysis).

**Cons of larger context window:**

- Increased computational requirements.
- Potential for overfitting.
- Diminishing returns beyond a certain point.

## 5.11 Max Response

- Sets the maximum number of tokens the model will generate.
- Once the limit is hit, generation stops — possibly mid-word or mid-sentence.
- Larger max response → higher computational demand → increased latency and cost.

---

## Q&A — Decoding Strategies

**Q1: What is the key difference between Top-K and Top-P sampling?** A: Top-K always samples from the K highest-probability tokens (fixed number). Top-P dynamically selects the smallest set of tokens whose cumulative probability meets or exceeds `p` (variable number). Top-P adapts to the probability distribution shape.

**Q2: When would you set temperature < 1?** A: When you need consistent, precise, deterministic output — such as code generation, factual Q&A, or any task where accuracy and predictability matter more than creativity.

**Q3: Why does greedy decoding fall into local minima?** A: Greedy decoding picks the best token at each step independently. The locally best token might not be part of the globally best sequence. For example, choosing "Paris" now might make it harder to form a grammatically correct and meaningful full sentence.

**Q4: What is the difference between frequency penalty and presence penalty?** A: Frequency penalty penalizes tokens proportionally to how often they have appeared (repeated tokens get increasingly penalized). Presence penalty applies the same flat penalty to any token that appeared at all, regardless of frequency.

**Q5: A developer wants creative story generation with no repetition. Which settings would you recommend?** A: High temperature (e.g., 0.8–1.0) for creativity, high presence penalty to avoid reuse of words, Top-P sampling (e.g., p=0.9) for controlled diversity. Avoid greedy or low-K sampling.

# 6. Training GPT-like LLMs

## 6.1 Pre-trained Base LLM

- Trained on a **massive corpus** of text data using **self-supervised learning** (next token prediction).
- Also called **raw language models** or **foundational models**.
- Learn powerful general representations but lack specific task expertise.
- **Example**: GPT-3's DaVinci model.
- Versatile, handles many tasks, but may not be accurate enough for specific domains.

## 6.2 Fine-Tuned LLM

- Takes a base LLM and trains it further on **task-specific** or **domain-specific** data.
- **Examples**:
    - Medical literature → better medical reasoning.
    - Customer service interactions → better customer support chatbot.
- Requires additional labeled data and training time.
- **Risk**: Overfitting the training data, limiting generalization to new examples.

## 6.3 Classification Fine-Tuning

- Replace the output layer of the LLM with a **small classification layer**.
- Example: Spam vs. Not-spam → the new classification layer has only 2 output nodes.
- Instead of predicting the next token, the model outputs a class label.
- Input is still text converted to token IDs.

## 6.4 Limitations of Pre-training and Fine-tuning

- Text completion objectives may provide only superficial comprehension.
- Methods often lack detailed task descriptions → models fail to generalize well.
- Example: A model fine-tuned on sentiment analysis without detailed instructions struggles on new sentiment analysis tasks.

## 6.5 Multitask Learning in T5

- T5 (Text-to-Text Transfer Transformer) uses a **text-to-text format** with task-specific prefixes.
- For example: "translate English to French: ..." or "summarize: ..."
- Step in the right direction, but failed at **cross-task generalization** — models couldn't generalize to truly new, unseen tasks.

---

## Q&A — Training

**Q1: What is the difference between a base LLM and a fine-tuned LLM?** A: A base LLM is pre-trained on massive text data to learn general language patterns. A fine-tuned LLM takes a base model and further trains it on specific domain/task data to improve task-specific performance.

**Q2: What is classification fine-tuning in the context of GPT?** A: Classification fine-tuning replaces the LLM's output layer (normally a vocabulary-size projection) with a small classification head (e.g., 2 nodes for binary classification). The model is then trained to predict a class label instead of the next token.

**Q3: Why did T5's multitask approach still have limitations?** A: T5 only provided task names or brief prompts as prefixes. While this improved performance on training tasks, the model couldn't generalize to entirely new, unseen task types — it lacked the cross-task generalization needed for truly general intelligence.

# 7. In-Context Learning (ICL)

## Definition

ICL allows a frozen LLM to perform tasks by providing examples _within the prompt_ — without updating model parameters or performing backpropagation.

- Also known as **few-shot learning** or **few-shot prompting**.
- Knowledge accumulated via ICL is **transient** — the LLM does not persistently store it after inference.
- Useful when direct model access is unavailable (e.g., using a model through an API).

## Types

- **Zero-shot learning**: No examples provided; model performs the task based on instructions alone.
- **One-shot learning**: One example provided.
- **Few-shot learning**: A small number of examples provided.

## Why ICL Became Necessary

- In October 2018, BERT-Large with 350M parameters was the largest transformer. Hardware struggled to fine-tune it.
- By 2022, public models grew to 176B parameters — a 500x increase.
- However, single-GPU RAM increased less than 10x.
- **Model size scaled ~100x faster than computational resources** — making fine-tuning of the largest models infeasible for most.
- ICL thus became the standard way to pass task training data to billion-scale models.

## Key Property

Research shows that ICL ability _improves as model parameter count increases_.

---

## Q&A — ICL

**Q1: How is ICL different from supervised learning?** A: Supervised learning requires a training phase with backpropagation to update model parameters. ICL operates without updating parameters — it uses pre-trained knowledge and examples in the prompt at inference time.

**Q2: Why is ICL knowledge transient?** A: Because ICL does not update model weights. The model uses the examples in the context window during that inference session only. After the session, the model reverts to its original state.

**Q3: Why did ICL become standard practice with very large models?** A: Because fine-tuning a 100B+ parameter model requires GPU VRAM far beyond what most users have. ICL allows effective task adaptation without any weight updates, making it feasible for anyone with API access.

# 8. Hard Prompting

## Hard Prompt Tuning vs. ICL

- **Hard prompts**: Manually handcrafted text prompts with discrete (real text) input tokens.
- Require significant expertise to design effective prompts.
- Static or at best template-based.
- **Hard prompt tuning**: Modifying input prompts to improve outputs _without retraining the entire model_.
- **ICL**: Uses examples within the prompt to guide the model, also without retraining.

Both operate without modifying model weights; they differ in that hard prompt tuning optimizes the prompt template itself, while ICL provides examples dynamically.

## Indexing as an ICL Workaround

Indexing converts LLMs into **information retrieval systems** for external data:

1. An indexing module breaks a document/website into smaller segments.
2. Segments are converted into vectors stored in a **vector database**.
3. When a user submits a query, the indexing module computes vector similarity between the embedded query and database vectors.
4. The top-K most similar embeddings are retrieved and used to generate the response.

This is the foundation of **Retrieval-Augmented Generation (RAG)**.

# 9. Parameter-Efficient Fine-Tuning (PEFT)

## Motivation: GPU RAM Requirements

**Storage for model parameters:**

- 1 parameter in 32-bit float = 4 bytes.
- 1B parameters → 4 GB storage.

**Additional memory for training:**

- Gradients, optimizer states (Adam/AdamW), activations from forward/backward pass.
- Requires ~12–20 additional bytes per parameter.
- Total for 1B model: ~80 GB + 4 GB = **84 GB**.
- NVIDIA A100 has 80 GB VRAM → training a >1B parameter model in 32-bit precision requires **multiple A100 GPUs**.

## Nuance in Gradient Descent for LLMs

- Adam optimizer stores momentum and velocity (squared gradient) for each parameter → **3x the memory** compared to plain SGD.
- Memory is the most critical factor in building LLMs.
- Even though Adam data is discarded after training, the system must be large enough to hold everything _during_ training.

## Latency Issues from Large Models

- Large models are **memory-bound** (speed limited by data movement) rather than compute-bound.
- Billions of weights must be moved from slow DRAM to fast GPU cache for every token generated.
- GPU memory bandwidth becomes the bottleneck.
- Tokens are generated **one by one** — sequential, no parallelization across tokens → high latency.

## What is PEFT?

**Parameter-Efficient Fine-Tuning** fine-tunes only a _small subset_ of a model's parameters while **freezing** the original pre-trained weights.

**Key benefits:**

- Significantly reduced computational cost.
- Resolves **catastrophic forgetting** (when fully fine-tuned models forget previous knowledge).
- Performance similar to full fine-tuning with far fewer trainable parameters.

**PEFT Methods:** Adapter, LoRA, Prefix Tuning, (Soft) Prompt Tuning.

---

# 10. Prompt Tuning (Soft Prompts)

## How Soft Prompt Tuning Works

Instead of crafting text prompts manually (hard prompts), soft prompt tuning learns a set of **continuous embedding vectors** as the task-specific prompt.

**Step-by-step process:**

1. Start with a large pre-trained LLM (e.g., T5, GPT).
2. **Freeze all main model parameters**.
3. Add a small set of **soft prompt embeddings** (trainable vectors) to the input.
4. The soft prompt is _prepended_ to the tokenized input text.
5. During training on a specific task, **backpropagation updates only the soft prompt parameters** — everything else is frozen.
6. Once trained, the soft prompts are stored and appended at inference time.
7. The same frozen model can be used for multiple tasks by switching the soft prompt.

**Analogy**: Soft prompt tuning is like training a "context primer" in the model's native language (embeddings). Instead of writing "Translate to French:", you use learned embedding vectors that guide the model's behavior.

## Prompt Tuning vs. Model Tuning

|Aspect|Prompt Tuning|Model Tuning (Fine-tuning)|
|---|---|---|
|Parameters updated|Only soft prompt vectors|Entire model (or large subset)|
|Multiple tasks|Use same model, switch prompts|Different fine-tuned models per task|
|Batching|Easy (append correct prompt)|Harder (can't batch across models)|
|Compute|Very lightweight|Resource-intensive|

---

# 11. Prefix Tuning

## How Prefix Tuning Works

Prefix tuning goes further than prompt tuning. Instead of only adding learnable tokens at the _input layer_, it **prepends learned vectors to the attention keys and values at EACH transformer layer**.

**Process:**

1. Initialize prefix vectors randomly or from a predetermined starting point.
2. Freeze all core LLM weights.
3. Prepend the current prefix (sequence of learnable embeddings) to the **hidden states (K and V)** at each transformer layer.
4. Process combined input through the model.
5. Measure output quality with cross-entropy loss.
6. Backpropagate to update **only the prefix vectors**.
7. After training, the learned prefix is fixed. To use a specific task, prepend this trained prefix to the input.
8. Different prefixes can be trained for different tasks and interchanged on the same model.

**Analogy**: Giving each layer of the model a custom "lens" to interpret the task without changing the core model weights.

## Prefix Tuning vs. Prompt Tuning

|Feature|Prompt Tuning|Prefix Tuning|
|---|---|---|
|Where applied|Input embedding space only|Every transformer layer (K and V)|
|Depth of task conditioning|Shallow (input level)|Deep (all layers)|
|Compute|Faster|Slightly more expensive|
|Performance on complex tasks|Good|Better|
|Flexibility|High|High|

**When to use:**

- Fast adaptation, low compute → **Prompt Tuning**
- High performance on complex tasks → **Prefix Tuning**

---

## Q&A — Prompt Tuning & Prefix Tuning

**Q1: What is the fundamental difference between hard prompts and soft prompts?** A: Hard prompts are manually written discrete text tokens. Soft prompts are continuous learnable embedding vectors that are not human-readable but are optimized through gradient descent to guide the model's behavior.

**Q2: How is prefix tuning different from prompt tuning?** A: Prompt tuning only adds learnable embeddings at the input layer. Prefix tuning injects learnable vectors into the key and value matrices of the attention mechanism at every transformer layer, enabling deeper task conditioning.

**Q3: Why is soft prompt tuning computationally efficient?** A: Because the core LLM (billions of parameters) remains frozen. Only the small soft prompt vectors (a few hundred parameters) are trained and updated.

# 12. Adapters

## Definition

An Adapter is a foundational PEFT method that inserts small, trainable **feed-forward layers (adapters)** between existing transformer layers while keeping the original model weights frozen.

## Architecture

- **Position**: Typically added after the self-attention and feed-forward sub-layers of each Transformer block.
- **Bottleneck Design**:
    1. **Downproject**: Project input from high-dimensional space $d$ to low-dimensional space $r$ (where $r \ll d$).
    2. **Non-linear transformation**: Apply activation function.
    3. **Upproject**: Project back from $r$ to $d$.
    4. **Residual connection**: The original input is added back (preserves original information).
- Only the **adapter weights** are trained; base model weights are frozen.

**Benefits:**

- Drastically reduces the number of parameters that need updating.
- A single pre-trained model can be adapted to many different tasks by swapping adapter modules.
- Achieves performance comparable to full fine-tuning with significantly less cost.
- Enables **efficient task switching** — different adapters can be plugged in for different tasks on the same base model.

---

## Q&A — Adapters

**Q1: Where are adapters inserted in a Transformer?** A: After the self-attention sublayer and after the feed-forward sublayer of each Transformer block (so two adapters per block in the standard configuration).

**Q2: What is the bottleneck design in adapters?** A: The adapter first downprojects the input from the model's hidden dimension `d` to a smaller dimension `r` (the bottleneck), applies a non-linearity, then upprojects back to `d`. A residual connection preserves the original signal.

**Q3: How do adapters enable efficient task switching?** A: Because the base model is frozen, you can train separate small adapter modules for each task. To switch tasks, you simply swap the adapter modules without modifying or reloading the base model.

---

# 13. LoRA (Low-Rank Adaptation)

## Motivation

Standard fine-tuning updates all model weights: $$W_{\text{current}} = W_{\text{previous}} - \eta \cdot \nabla W$$

Where:

- $W_{\text{current}}$: current iteration weights
- $W_{\text{previous}}$: previous iteration weights
- $\eta$: learning rate
- $\nabla W$: gradient of weights

This is memory-intensive for large models.

## Core Idea: Low-Rank Matrix Decomposition

**Hypothesis**: The change in weights $\Delta W$ during fine-tuning has a **low intrinsic rank** — many columns are linearly dependent. We can approximate $\Delta W$ with two smaller matrices.

**Decomposition:** $$\Delta W = B \cdot A$$

Where:

- Original matrix $W_0$ has dimensions $d \times k$ (full rank).
- $A$ has dimensions $r \times k$.
- $B$ has dimensions $d \times r$.
- $r \ll d, k$ is the chosen rank (hyperparameter).

**During fine-tuning:** $$W = W_0 + \Delta W = W_0 + B \cdot A$$

Where $W_0$ (original pretrained weights) is **frozen**, and only $A$ and $B$ are trained.

**Memory savings**: Instead of updating $d \times k$ parameters, only update $r(d + k)$ parameters (much smaller when $r \ll d, k$).

**Choice of r**: If $r$ is too low → too much information lost. Typically, $r = 4$ applied to both Query and Value weight matrices gives the best results in practice (found in GPT-3 experiments).

## LoRA During Inference

No additional latency at inference time: $$W = W_0 + B \cdot A$$

The adapter matrices are merged directly into the model weights before inference. To switch tasks, subtract $BA$ and add the new task's $B'A'$: $$W_{\text{new task}} = W_0 + B'A'$$

## Where to Apply LoRA?

In GPT-3 experiments, LoRA was applied to **self-attention module** weight matrices:

- Query ($W_q$), Value ($W_v$), and Key ($W_k$) weight matrices.
- Best results: LoRA on both Query and Value with **rank r = 4**.

## Low-Rank Prompt Tuning (LoPT)

Combines LoRA ideas with prompt tuning:

- Standard soft prompt matrix $X$ of size $n \times d$ contains redundancy.
- Approximate $X$ with two smaller matrices (low-rank factorization).
- Only these low-rank matrices are updated during training.

---

## Q&A — LoRA

**Q1: What is the key hypothesis behind LoRA?** A: The weight changes $\Delta W$ during fine-tuning have a low intrinsic rank — removing linearly dependent dimensions loses little information. Thus, $\Delta W$ can be approximated by the product of two smaller matrices $B$ and $A$.

**Q2: How does LoRA reduce memory during fine-tuning?** A: Instead of updating all $d \times k$ parameters, LoRA only trains two small matrices ($d \times r$ and $r \times k$). When $r \ll d, k$, the number of trainable parameters is dramatically reduced.

**Q3: Does LoRA add latency during inference?** A: No. After training, the LoRA matrices $B$ and $A$ are merged with the original weights ($W = W_0 + BA$), so inference is identical to using the full model.

**Q4: What is the difference between LoRA and Pruning/Distillation?**

|Aspect|LoRA|Pruning|Distillation|
|---|---|---|---|
|Primary Phase|Fine-Tuning|Inference|Inference|
|Key Benefit|Reduces training memory and cost|Increases speed, reduces size|Smaller, faster student model|
|Mechanism|Trains small adapter matrices; base model frozen|Removes redundant parameters/neurons/heads|Transfers knowledge from large teacher to compact student|

# 14. Quantization

## What is Quantization?

Quantization reduces the **precision** (number of bits) used to store model parameters and activations, reducing memory requirements while minimally impacting accuracy.

**Example**: Convert weights from `float32` (4 bytes each) to `int8` (1 byte each) → 4x memory reduction.

$$Q = g(x)$$

Where $g$ is a quantization operator that maps floating-point value $x$ to a quantized value $Q$.

## Types of Quantization

**Uniform Quantization:**

- Equal-sized quantization intervals across the entire range.
- Simpler to implement on hardware → more efficient.

**Non-uniform Quantization:**

- Variable-sized intervals tailored to data distribution.
- Applied differently across weights or layers.
- Better minimizes quantization error where precision matters.

**Affine Quantization:**

- General form: includes both scaling and shift (zero-point). $$x_q = \text{round}\left(\frac{x - z}{s}\right)$$ Where $s$ is the scale factor and $z$ is the zero-point.

**Scale Quantization:**

- Special case of affine quantization.
- Only scaling, no shift/zero-point.

## Quantization Math: FLOAT32 to INT8

Converting FLOAT32 (range $[\alpha, \beta]$) to INT8 (range $[\alpha_q, \beta_q]$):

$$\text{scale} = \frac{\beta - \alpha}{\beta_q - \alpha_q}$$ $$x_q = \text{round}\left(\frac{x}{\text{scale}}\right) + \text{zero_point}$$

## Quantization-Aware Training (QAT)

- Retrains the model WITH quantized weights and activations.
- The model adapts to quantization-induced changes during training.
- Better performance retention even at 2-bit quantization.
- **Computationally intensive** due to additional training.
- **QLoRA** is a form of QAT — fine-tunes low-rank adapters while base model is quantized in NF4 format.

## Post-Training Quantization (PTQ)

- Applies quantization to a **pre-trained model without further training**.
- Less computationally demanding than QAT.
- Typically limited to 8-bit or 6-bit quantization.
- Faster deployment; useful when computational resources are limited.

## Quantization Issues

- Very small or very large numbers may fall outside the reduced precision range → **NaN values**.
- Trade-off between memory efficiency and numerical accuracy.
- Libraries like **bitsandbytes** use 4-bit quantization for pretrained weights while computations (training/fine-tuning) occur in higher precision (float16 or bfloat16).

---

## Q&A — Quantization

**Q1: What is the main goal of quantization in LLMs?** A: To reduce the memory footprint of model parameters and activations by reducing precision (e.g., float32 → int8), enabling deployment on fewer/smaller GPUs while maintaining acceptable accuracy.

**Q2: What is the difference between QAT and PTQ?** A: QAT retrains the model with quantized weights/activations so the model adapts to quantization. PTQ quantizes a pre-trained model without any further training. QAT achieves better accuracy but requires more compute; PTQ is faster but less accurate.

**Q3: What can go wrong with quantization?** A: Numbers that fall outside the quantized range produce NaN (Not a Number) values. Also, very precise values get "rounded" to the nearest quantized bin, leading to accuracy loss.

---

# 15. QLoRA (Quantized LoRA)

## Overview

QLoRA combines LoRA's parameter-efficient fine-tuning with quantization for maximum memory efficiency. Goal: fine-tune a portion of the model using the storage and memory of a **single GPU**.

## Three Key Innovations

### 1. 4-bit NormalFloat (NF4)

- A new data type specifically designed for neural network weights that follow a **normal distribution**.
- Information-theoretically optimal for normally distributed data.
- Outperforms standard 4-bit integers and 4-bit floats empirically.
- Reduces memory by up to 4x compared to BF16.
- Packs weight values into 4-bit format, assigning shorter codes to more common values (following normal distribution).

### 2. Double Quantization

- Quantizes the **quantization constants themselves**.
- Blockwise quantization requires storing one quantization constant per block.
- For a 65B model, double quantization saves ~0.37 bits per parameter ≈ 3 GB total savings.

### 3. Paged Optimizers

- Uses **NVIDIA unified memory** for automatic CPU-GPU page transfers.
- Prevents **out-of-memory errors** from gradient checkpointing memory spikes.
- When GPU runs out of memory, optimizer states are moved to CPU RAM and transferred back when needed.

## BFloat16 vs. NF4

|Property|BFloat16 (BF16)|NF4 (4-bit NormalFloat)|
|---|---|---|
|Bits|16|4|
|Purpose|Training, high-precision inference|Memory-efficient inference, QLoRA fine-tuning|
|Dynamic range|Same as FP32 (8-bit exponent)|Optimized for normal distribution|
|Memory reduction|2x vs FP32|8x vs FP32|
|Use case|Standard training|QLoRA 4-bit quantized fine-tuning|

## Dequantization in QLoRA

Dequantization is the inverse of quantization. During forward/backward passes, quantized weights (frozen, not trained) must be dequantized back to 32-bit for accurate gradient computation:

$$W_{\text{full precision}} = \text{dequantize}(W_{\text{quantized}}, \text{quantization_constant})$$

If quantized values were used directly for backpropagation, gradients would be less accurate and unstable.

## Blockwise Quantization

- Divides input tensors into smaller blocks.
- Each block is quantized independently with its own quantization constant.
- Handles outliers better — very large/important values in one block don't distort the quantization of other blocks.
- Quantization constant = (total bins of datatype) / (absolute max of tensor in block).

---

## Q&A — QLoRA

**Q1: What are the three key innovations in QLoRA?** A: (1) 4-bit NormalFloat (NF4) — information-theoretically optimal 4-bit format for normally distributed weights; (2) Double Quantization — quantizing the quantization constants to save additional memory; (3) Paged Optimizers — dynamic CPU-GPU memory management to prevent OOM errors.

**Q2: Why is NF4 better than standard 4-bit integer quantization?** A: Neural network weights are approximately normally distributed. NF4 is designed to represent this distribution optimally — more common values near the mean get finer representation, while rare outliers get coarser. Standard 4-bit integers divide the range uniformly, wasting precision.

**Q3: What is double quantization and why is it needed?** A: Blockwise quantization requires storing one quantization constant per block, which itself consumes memory. Double quantization applies quantization to these constants, saving additional memory (~0.37 bits/parameter, ~3 GB for a 65B model).

# 16. Knowledge Distillation

## What is Distillation?

Knowledge distillation compresses and transfers knowledge from a large **teacher model** to a smaller **student model**, maintaining reasonable accuracy while dramatically reducing model size and cost.

## Three-Step Process

**Step 1: Train Teacher Model**

- Teacher is a large, well-trained model (billions/millions of parameters).
- Trained on the full training data until convergence.

**Step 2: Student Model Learning**

- Train on a **transfer set** (different from teacher's training data).
- Both teacher and student process the transfer set.
- Student learns from teacher's **soft predictions** (probability distributions over all classes) rather than from hard labels (ground truth only).
- Soft targets contain richer information about inter-class relationships.

**Step 3: Loss Computation and Gradient Update** $$\text{Total Loss} = \alpha \cdot \mathcal{L}_{\text{student}} + (1 - \alpha) \cdot \mathcal{L}_{\text{distillation}}$$

Where:

- $\mathcal{L}_{\text{student}}$: Cross-entropy loss between student prediction (temperature $T=1$) and ground truth → **hard labels**.
- $\mathcal{L}_{\text{distillation}}$: KL-divergence between teacher's soft outputs (temperature $T>1$) and student's soft outputs (temperature $T>1$) → **soft labels**.
- $\alpha$: Weighting factor between the two losses.

**Temperature in Softmax:** $$P_i = \frac{e^{z_i / T}}{\sum_j e^{z_j / T}}$$

Setting $T > 1$ "softens" the probability distribution, revealing more information about the teacher's confidence across classes.

**Gradients** are updated only for the student model. The teacher's weights remain frozen.

## Why Distillation Works

1. Student learns _how_ to perform tasks (not just raw data labels).
2. Soft targets preserve reasoning patterns and inter-class relationships.
3. Student generalizes better even with fewer parameters.

## Knowledge Distillation in DistilBERT

**DistilBERT** (Hugging Face):

- 40% smaller than BERT.
- 60% faster.
- Retains 97% of BERT's language understanding.
- **Loss function** for DistilBERT (three-component):
    - **Distillation loss**: KL-divergence between teacher and student logits (softened with T>1).
    - **Masked Language Modelling (MLM) loss**: Standard MLM loss on ground truth.
    - **Cosine Embedding Loss**: Aligns direction of teacher and student hidden vectors.
- Dynamic masking (different masking patterns per epoch).
- No Next Sentence Prediction (NSP) objective.
- Trained on English Wikipedia + Toronto Book Corpus, on 8× 16GB V100 GPUs for ~90 hours.

## What Distillation CAN and CANNOT Do

**Can do:**

- Transfer functional capabilities (how to perform specific tasks).
- Preserve reasoning patterns.
- Improve computational efficiency (less memory, faster).

**Cannot do:**

- Expand knowledge — student doesn't gain new information beyond the teacher.
- Exceed teacher performance — student aims to _match_, not surpass, the teacher.
- Replace fine-tuning for new domain knowledge.

## Benefits

- Can run on edge devices, mobile phones, local machines.
- Can reduce inference costs by up to **75%** compared to full-size models.
- Faster responses for specialized tasks.

## Disadvantages

- Computationally expensive distillation process.
- Performance loss (especially if student is much smaller than teacher).
- Distillation requires a training phase (iterative).

## Combined Strategy: Distillation + Fine-Tuning

1. Fine-tune a large LLM on domain data → expert teacher model.
2. Distill the expert teacher into a smaller student model.
3. The student retains domain expertise but is efficient for deployment.

---

## Q&A — Distillation

**Q1: What is the formula for total loss in knowledge distillation?** A: $\text{Total Loss} = \alpha \cdot \mathcal{L}_{\text{student}} + (1-\alpha) \cdot \mathcal{L}_{\text{distillation}}$, where student loss uses hard labels with T=1, and distillation loss uses KL-divergence between teacher and student soft outputs with T>1.

**Q2: Why are soft labels better than hard labels for training the student model?** A: Soft labels (probability distributions from the teacher) carry richer information about inter-class relationships. For example, if an image looks like both a cat and a dog, soft labels preserve this ambiguity. Hard labels (0 or 1) only tell the student the final answer, losing this nuance.

**Q3: What is the role of temperature T > 1 in distillation?** A: Temperature T > 1 "softens" the teacher's output distribution, spreading probability more evenly across classes. This reveals the teacher's relative confidence across all categories (not just the top class), providing more information for the student to learn from.

**Q4: What are the three components of DistilBERT's loss function?** A: (1) Distillation loss — KL-divergence between teacher and student softened logits; (2) Masked Language Modelling loss — cross-entropy on ground truth masked tokens; (3) Cosine Embedding Loss — aligns the direction of teacher and student hidden state vectors.

# 17. Model Pruning

## What is Pruning?

Pruning removes **less important neurons or weights** from a model — like cutting unwanted branches from a tree. It reduces model size and improves inference speed.

## Advantages

- **Reduce Model Size**: Removes unnecessary parameters.
- **Improve Inference Speed**: Fewer parameters = fewer computations.
- **Lower Computational Cost**: Less memory and compute needed.

## Types of Pruning

### Magnitude Pruning

- Set parameters **close to zero** to exactly zero.
- Simplest method.
- **Requires retraining** to recover accuracy.

### WANDA (Pruning by Weights and Activations)

- Key insight: In LLMs, a small fraction of parameters have very large magnitudes (outliers) with high impact on predictions. These must NOT be pruned even if other large weights are pruned.
- WANDA considers both weight magnitude AND activation magnitude.

**Algorithm:**

- Compute importance score of each weight as: $|W_i| \cdot |X_i|$ (element-wise product of weight magnitude and input activation norm).
- Example: If $|W_1| < |W_2|$ but $|X_1| \gg |X_2|$, then magnitude pruning would remove $W_1$, but WANDA keeps it because $|W_1 \cdot X_1| > |W_2 \cdot X_2|$.

**Advantages of WANDA:**

- One-shot process (single forward pass).
- No retraining required.
- High sparsity with low accuracy loss.

## Disadvantages of Pruning

- **Loss of Accuracy**: Excessive pruning degrades model performance.
- **Complexity**: Choosing which parameters to prune requires careful analysis.
- **Post-pruning fine-tuning**: May be needed to recover accuracy.

## Limitations of Pruning

- Generally a "lossy" technique.
- Excessive pruning leads to rapid performance collapse, especially on complex reasoning tasks.
- Should be combined with knowledge distillation or small amounts of fine-tuning (e.g., LoRA) to recover accuracy.

## Pruning vs. LoRA vs. Distillation (Summary)

|Technique|Primary Phase|Key Benefit|Mechanism|
|---|---|---|---|
|LoRA|Fine-Tuning|Reduces training memory and cost|Trains small adapter matrices; base model frozen|
|Pruning|Inference|Increases speed and reduces size|Systematically removes redundant parameters/neurons|
|Distillation|Inference|Smaller, faster student model|Transfers knowledge from large teacher to compact student|

## How LLM Providers Use Pruning + Distillation

- Llama 3.1 family: 8B, 70B, 405B.
- Only the largest model is fully trained from scratch.
- Smaller models are created by gradually trimming down (pruning) the big one and using distillation to regain accuracy.

---

## Q&A — Pruning

**Q1: What is the key difference between magnitude pruning and WANDA?** A: Magnitude pruning only considers weight magnitudes — prunes smallest weights. WANDA considers the product of weight magnitude AND input activation magnitude, keeping weights that may be small but receive large inputs (outlier activations that have high predictive impact).

**Q2: Why does WANDA not require retraining?** A: WANDA is a one-shot method that only requires a single forward pass to compute importance scores. By accurately identifying which weights are truly important (via activation norms), it avoids the accuracy loss that typically requires retraining.

**Q3: Why is LoRA for fine-tuning and Pruning/Distillation for inference?** A: LoRA reduces the cost of the training/fine-tuning process (fewer parameters to update). Pruning and Distillation reduce the size of the _final model_ used during inference, making it faster and cheaper to deploy.

# 18. Supervised Fine-Tuning (SFT) & RLHF

## Supervised Fine-Tuning (SFT)

SFT updates a pre-trained model using **labeled input–output pairs** curated by humans (usually domain experts). It teaches the model to perform specific tasks or follow certain behaviors reliably.

**Instruction Fine-Tuning (IFT)** is a specific form of SFT where the data consists of **natural language instructions and expected responses** — teaching the model to follow user commands.

### Instruction Fine-Tuning Process

1. Prepare dataset: Download instruction-response dataset, format entries, split into train/validation/test.
2. Construct training batches: Pad sequences, create target token IDs, mask padding.
3. Load a pre-trained GPT-like model.
4. Fine-tune using a training loop similar to pretraining.
5. Evaluate using model responses on a test set (scored by human evaluators or another LLM).

### Evaluation in Instruction Fine-Tuning

Three types of evaluation methods:

1. **Multiple-choice Q&A benchmarks** — e.g., MMLU (Massive Multitask Language Understanding).
2. **Human evaluation** — humans rate response quality.
3. **Automated conversational metrics** — e.g., AlpacaEval (uses GPT-4 as evaluator).

### Generalization Example

- Training: Model sees translation and summarization tasks.
- Inference: New composite task — "Translate this text and then summarize the translation."
- Result: Because it has been trained with explicit instructions, it generalizes to this unseen composite task.

### Key Instruction-Tuned Models

- **T0** (BigScience, 2021): Based on T5 architecture, reformulates NLP tasks as natural language prompts. Strong zero-shot performance. Outperforms models up to 16x larger.
- **Google FLAN** (2022): Compiled 62 NLP datasets into 12 task clusters. Created 10 instruction templates per dataset. Major breakthrough in instruction-following.
- **Flan-PaLM** (Google, 2022): Scaled instruction tuning on PaLM (540B parameters). Explored: (1) scaling number of tasks, (2) scaling model size, (3) fine-tuning on Chain-of-Thought (CoT) data.

## Reinforcement Learning from Human Feedback (RLHF)

**RLHF** is a multi-stage training process that uses **human preferences** (not just correct answers) to guide model behavior.

**Key differences from SFT:**

- SFT optimizes for task accuracy using labeled data.
- RLHF optimizes for subjective alignment qualities: **helpful, harmless, honest**.

### RLHF Pipeline

1. **Collect human ranking data**: Human evaluators compare two or more model responses to the same prompt and rank which one is better.
2. **Train a reward model**: The preferences are used to train a model that scores future outputs.
3. **Optimize the base model using RL**: The base model (already SFT'd) is further optimized using RL algorithms (e.g., PPO) to maximize the reward model's score.

### SFT vs. RLHF Comparison

|Aspect|SFT|RLHF|
|---|---|---|
|Data type|Labeled input–output pairs|Human preference rankings|
|Goal|Task accuracy|Behavioral alignment (helpful, harmless, honest)|
|Implementation complexity|Simpler|More complex (human raters + reward model + RL)|
|Best for|Structured tasks (extraction, classification, QA)|User-facing systems requiring nuance, context-awareness|
|Example|Fine-tune to summarize legal documents|Chatbots, tutors, agents|

### ChatGPT vs GPT

- **GPT**: A language model (not an app).
- **ChatGPT**: A web app built on GPT, specifically optimized for dialogue using a combination of:
    - **Unsupervised/Self-supervised Learning**: Pre-trained on vast internet text (next word prediction).
    - **Supervised Learning**: Trained on text data with human-curated labels.
    - **RLHF**: Human evaluators rank responses; best responses reinforce training.

### The Pipeline

```
Pre-training → SFT → RLHF
(Broad language)  (Task competence)  (Behavioral alignment)
```

---

## Q&A — SFT & RLHF

**Q1: What is instruction fine-tuning and how does it differ from standard SFT?** A: Standard SFT trains on any labeled input-output pairs. Instruction fine-tuning specifically uses instruction-based data — explicit natural language commands and expected responses — to teach the model to follow user instructions reliably.

**Q2: Why is RLHF more suitable than SFT when there's no single "correct" answer?** A: RLHF uses human preference rankings rather than fixed labels. When multiple responses could be correct and the "best" one is subjective (e.g., more helpful, more polite), human rankings capture this nuance better than fixed labels.

**Q3: What are the three stages of ChatGPT training?** A: (1) Unsupervised pre-training on vast internet text; (2) Supervised fine-tuning on curated instruction-response pairs; (3) RLHF where human evaluators rank model responses and a reward model guides further optimization.

# 19. Reasoning LLMs — Chain-of-Thought & DeepSeek R1

## Chain-of-Thought (CoT) Prompting

- Instead of asking models for direct answers, prompt them to generate **intermediate reasoning steps** before the final answer.
- Discovered by Google researchers (2022, "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models").
- Tested on Google's PaLM model — significantly better results, especially on multi-step math problems.

**Few-Shot CoT**: Provide examples of problems with step-by-step solutions in the prompt. The model learns to mimic this reasoning style.

## OpenAI o1

- Does Chain-of-Thought **internally** — intermediate steps not shown to the end user.
- **Inference Time Scaling**: Gives the model more computational time/resources to "think" before answering.
    - Method 1: Chain-of-thought prompting (step-by-step reasoning).
    - Method 2: Generate multiple possible answers and choose the best via voting.
- Ranked in the **89th percentile** on competitive programming (Codeforces).
- Significantly outperformed GPT-4o on reasoning-heavy tasks.

## DeepSeek V3 Base

- Pre-trained on massive multilingual corpus (web data, code).
- No supervised fine-tuning (SFT) after pretraining.

## DeepSeek R1-Zero

- **Skipped SFT entirely** — jumped straight to Reinforcement Learning on the base model.
- No human-curated reasoning data used.
- RL algorithm: **GRPO (Group Relative Policy Optimization)**.
    - Rewards based on two rule-based criteria:
        - **Format**: LLM judge ensures responses follow expected format (reasoning in `<think>` tags).
        - **Accuracy**: LeetCode compiler for code; deterministic system for math.
    - GRPO compares a group of outputs for the same input; model is updated to favor above-average outputs.

### DeepSeek R1-Zero Findings

- AIME 2024 pass@1 score jumped from **15.6% → 71.0%**.
- Extended thinking: Model started spending more time generating outputs (better reasoning).
- **Self-reflection**: The model spontaneously learned to revisit and re-evaluate its previous steps — not explicitly programmed.
- **"Aha moment"**: At an intermediate training stage, the model spontaneously learned to rethink and refine its approach.
- Issues: Poor readability, language mixing (blending English and other languages), poor formatting.

## DeepSeek R1 (Final)

**To fix R1-Zero's issues**, the team started with SFT first:

**Checkpoint 1**: SFT on initial data to improve basic reasoning and readability.

**Checkpoint 1.1**: RL training with same rewards as R1-Zero plus **language consistency reward** (proportion of words in target language).

**Generating Training Data**:

- Rejection sampling from Checkpoint 1.1: Generate multiple outputs, keep correct/high-quality ones.
- Broadened beyond math/coding to science, logic puzzles.
- Used generative AI reward model for non-rule-based quality checks.
- Filtered out: chain-of-thought with mixed languages, long paragraphs, code blocks.
- ~600K reasoning samples + ~200K general/non-reasoning samples = **~800K total samples**.

**Final Training**:

1. **SFT** on the 800K curated dataset.
2. **RL** with combined rewards: rule-based for math/coding + generative AI reward model for general alignment, helpfulness, harmlessness.

**Result**: DeepSeek-R1 matched or edged out OpenAI o1 on reasoning benchmarks, at 671B parameters.

**Distilling from DeepSeek-R1**:

- Fine-tuned smaller open-source bases (Qwen2.5 series, Llama3.3) on the 800K samples.
- Much more efficient than running full-scale RL on small models.
- Surprisingly strong performance despite being much smaller.

## Q&A — Reasoning LLMs

**Q1: What is Chain-of-Thought prompting and why does it help?** A: CoT prompting instructs the model to generate intermediate reasoning steps before the final answer. It helps because multi-step problems require sequential logical deduction. By externalizing this reasoning process, the model can check each step and avoid errors.

**Q2: How did DeepSeek R1-Zero train without any supervised data?** A: It applied GRPO (RL) directly on the base model, using only rule-based rewards — format correctness (reasoning in `<think>` tags) and answer accuracy (verified by LeetCode compiler or math checker). No human-curated reasoning examples were used.

**Q3: What spontaneous behaviors emerged in DeepSeek R1-Zero?** A: (1) Self-reflection — the model learned to revisit and re-evaluate previous steps. (2) Extended thinking — the model started spending more time generating outputs before answering. (3) The "aha moment" — the model spontaneously learned to rethink and refine its approach mid-problem. These behaviors were not explicitly programmed.

**Q4: Why was SFT added before RL in DeepSeek R1 (vs. R1-Zero)?** A: R1-Zero suffered from poor readability, language mixing, and unstructured output. Adding SFT first (Checkpoint 1) gave the model basic instruction-following and readability, making the subsequent RL training more effective and the outputs more user-friendly.

# 20. Encoder-Decoder Architecture Debate

## Popular LLM Architectures

|Architecture|Examples|Attention|Best For|
|---|---|---|---|
|Encoder-only|BERT|Bidirectional|NLU tasks (classification, NER)|
|Decoder-only (Causal)|GPT-3, Claude, LLaMA|Unidirectional (left-to-right)|Text generation|
|Encoder-Decoder|T5, BART|Bidirectional encoder + Unidirectional decoder|Seq-to-seq tasks (translation, summarization)|
|Prefix Decoder|GLM-130B|Bidirectional input prefix + Unidirectional output|NLU + NLG combined|

## Encoder-Decoder: Pros & Cons

- **Pro**: Versatility for text-to-text tasks; unified approach.
- **Con**: Complex training; slower inference (two-step: encode then decode).

## Causal Decoder: Pros & Cons

- **Pro**: Excellent at text generation; few-shot learning via prompting.
- **Con**: May struggle with deep understanding; computationally resource-intensive.

## Prefix Decoder (Prefix LM)

- Combines strengths of encoder-decoder and decoder-only.
- Allows **bidirectional attention** over the input prefix (any token can attend to any other).
- Generates output **autoregressively** (causal).
- Used in GLM-130B.

**Architecture Comparison:**

|Model Type|Reading|Writing|
|---|---|---|
|GPT-3 (Causal Decoder)|Left-to-right only|Left-to-right|
|T5 (Encoder-Decoder)|Full bidirectional (encoder)|Left-to-right (decoder)|
|GLM (Prefix Decoder)|Bidirectional (prefix)|Left-to-right (output)|

## Weakness of Decoder-Only Models

- **Tunnel vision**: Process inputs sequentially (token-by-token), predicting the next word without seeing the full context before responding.
- Fundamental architectural flaw for tasks requiring deep input comprehension.
- Lack **bidirectional attention** — can't see all parts of input simultaneously.
- Struggle with: analyzing long documents, understanding image composition, reasoning about contradictions.

## DeepMind's T5 Gemma 2 (Dec 2025) — Encoder-Decoder Revival

**Key Innovation**: Separate the reading task from the writing task.

- **Encoder**: Processes entire input **bidirectionally** — every token attends to every other token simultaneously. For a 100,000-token document, every word sees all other words instantly.
- **Decoder**: Generates output **autoregressively** (next-token prediction), querying the encoder's compressed representation.

**Why Superior:**

- **Long context**: Decoder-only models' KV-cache grows unboundedly. T5Gemma2's encoder compresses context efficiently.
- **Multi-modality**: Routes image patches through the encoder (bidirectional). A 270M model gains complete visual understanding.
- **Efficiency**: Available in 270M, 1B, 4B parameter sizes. 1B runs on consumer GPUs (16-24GB VRAM).

**Merged Attention (Key Technical Detail):** Instead of applying self-attention then cross-attention sequentially, T5Gemma2 **unifies both into a single operation**:

- Concatenate encoder hidden states $H$ and decoder states $X$ as keys and values.
- Queries come from the decoder only.
- **Hybrid masking**:
    - Causal masking on decoder columns (token $t$ cannot see $t+1$).
    - Bidirectional visibility on encoder columns (all tokens visible).
- Result: In a single softmax, the model chooses between relying on its generation history or the input context.

## Which Architecture to Choose?

- Specific NLU (deep understanding of input) → **Encoder-only**
- Input text → Output text (seq-to-seq) → **Encoder-Decoder**
- Autonomous text generation → **Causal Decoder**
- Both understanding + generation → **Prefix Decoder**

**Note**: No hard evidence that one architecture is fundamentally superior given sufficient training and model size.

## How Code LLMs Differ from General LLMs

### Data and Tokenization

# 22. Mixture of Experts (MoE)

## Concept

Instead of one large dense model, MoE uses **multiple specialized sub-models (experts)**, each trained to handle specific parts of a problem. A **gating network (router)** decides which experts to activate for each input.

**Components:**

1. **Input**: The problem or data.
2. **Experts**: Smaller neural networks, each specialized for a different domain or task. Only a few are active at any time (sparse).
3. **Gating Network (Router)**: A neural network that learns to assign inputs to the best experts.
4. **Output**: Combined output from activated experts.

## Expert Network

- Each expert is an independent neural network trained on different data/tasks.
- Sparse: Only a few experts are active for any given input.
- Prevents the system from being overwhelmed; ensures relevance.

## Gating Network (Router)

- Analyzes input and determines which experts are best suited.
- Types of routing:
    - **Top-K routing**: Picks the top K experts with highest affinity scores.
    - **Expert Choice**: Experts decide which data they can handle best.
    - **Sparse Routing**: Only activates a few experts per input.

## Training Phases

### Expert Training

- Each expert is trained on a specific subset of data or tasks.
- Focuses on one particular aspect of the broader problem.
- Follows standard neural network training (minimize loss for its specific subset).

### Gating Network Training

- Trained alongside the experts.
- Learns to predict a probability distribution over experts for each input.
- Trained using optimization that includes both gating accuracy and expert performance.

### Joint Training

- The entire MoE system (all experts + gating network) is trained together.
- Combined loss = individual expert losses + gating network loss.
- Gradients propagate through both, improving overall performance.

## Inference Phase

1. **Input Routing**: Gating network assigns probabilities across experts.
2. **Expert Selection**: Most appropriate experts are selected.
3. **Combining Output**: Outputs from selected experts are merged (often by weighted averaging, where weights = gating network probabilities).

## Benefits

- **Performance**: Only relevant experts activate → avoids unnecessary computation.
- **Flexibility**: Diverse expert capabilities.
- **Fault Tolerance**: If one expert fails, the rest still function.
- **Scalability**: Handles increasingly complex inputs by decomposing problems.

## Challenges

- **Training Complexity**: More hyperparameters (gating + experts); complex joint optimization.
- **Inference Efficiency**: Gating overhead, expert selection, and parallelization of multiple experts slows inference.
- **Increased Model Size**: Storage, deployment, and memory challenges.

## MoE in LLMs — Replacing the FFNN

In standard transformers, the FFNN (Feed-Forward Neural Network) in each decoder block is large and always fully activated (dense). **MoE replaces the FFNN with a sparse layer** — only a subset of expert FFNNs are activated per token.

**Dense vs. Sparse MoE:**

||Dense MoE|Sparse MoE|
|---|---|---|
|Expert activation|All experts, different distributions|Only K experts per input|
|Computation|Higher|Lower|
|Use in modern LLMs|Rare|Standard ("MoE" usually means Sparse MoE)|

**What do experts learn?**

- In encoder models, experts learn fine-grained domain specializations.
- In decoder models, experts are consistently used for certain _types_ of tokens but don't seem to have strict domain specializations.
- Different tokens take different "paths" through the experts across decoder blocks.

## Routing in MoE

- A **router** (gate network) is added just before the expert layer.
- Router is also an FFNN.
- Outputs probabilities for expert selection.
- **The MoE Layer = Router + Selected Experts**.
- Expert layer returns: output of selected expert × gate value (selection probability).

## Mixtral 8x7B (Mistral AI)

- 32 sequential transformer blocks.
- Each MLP layer replaced with a sparse MoE block of **8 experts**, of which **2 are activated per token**.
- Remaining layers (self-attention, normalization) are shared by all tokens.
- Total parameters: **47B** (not 8×7B=56B, because of shared layers).
- Each token processed by 7B parameters at each layer.
- Due to two experts per layer and shared architecture, total per token = **12.9B parameters** (not 2×7B=14B).

## Is MoE a PEFT Method?

Yes, in a sense: MoE enables large-scale model adaptation by training only a sparse subset of parameters:

- **Sparse Active Parameters**: Only a few experts update per input (like LoRA).
- **Specialized Training**: Can freeze base weights and only update expert networks (like Adapters).
- **Reduced Compute**: Better performance-per-computation ratio vs. dense models of similar total size.
- **Modular**: New experts can be added for new tasks without retraining the entire model.

---

## Q&A — MoE

**Q1: How does the gating network in MoE work?** A: The gating network is a small neural network (FFNN) that takes the input token and outputs a probability distribution over all experts. The top-K experts (highest probabilities) are selected. The expert layer returns the weighted average of selected expert outputs, using the gate probabilities as weights.

**Q2: Why does Mixtral 8x7B have only 47B parameters instead of 56B (8×7B)?** A: Because the self-attention and normalization layers are shared across all tokens — they are not replicated per expert. Only the FFNN layers are replaced with sparse MoE. Thus total parameters = shared layers + 8 expert FFNNs, not 8 full 7B models.

**Q3: What is the difference between dense and sparse MoE?** A: Dense MoE activates all experts for each input, just with different weightings. Sparse MoE activates only K (typically 2) experts per input. Modern LLMs use Sparse MoE because it is computationally cheaper.

# 23. NLG Evaluation Metrics

## Reference-Free Metrics for Summarization

**SUPERT:**

- Measures similarity of a generated summary with a BERT-based pseudo-reference (selected salient sentences from source documents).
- Uses contextualized embeddings and soft token alignment.
- Good correlation with human evaluation for multi-document summarization.

**BLANC:**

- Measures the performance boost a pre-trained language model gains when it has access to the document summary while doing language understanding tasks.
- Good correlation with human evaluators.
- Does NOT require human reference summaries.

**ROUGE-C (ROUGE-Consensus):**

- Reference-free variant of ROUGE.
- Does NOT require human-written "gold standard" references.
- Purpose: Evaluate machine-generated summaries against source documents directly.
- Methodology: Treats source documents as a "consensus" of most relevant information. Calculates similarity between the candidate summary and the source.
- Specifically developed for **multi-document summarization** evaluation.
- Unlike standard ROUGE (ROUGE-1, ROUGE-L) which measures lexical overlap with human references.

**QAFactEval (for Factual Consistency):** Framework to evaluate factual consistency by QA-based verification:

1. Extract key entities (named entities, noun phrases) from the generated summary.
2. Generate questions about these entities using a QG model (e.g., BART-large).
3. A QA model answers these questions using ONLY the source document.
4. Compare source-document answers to summary-derived answers.
5. Higher semantic overlap → higher factual consistency.

**SummaC (for Hallucination Detection):**

- Detects "hallucinations" in abstractive summarization (fluent but factually incorrect summaries).
- Uses **Natural Language Inference (NLI)** to evaluate factual consistency:
    1. **Document Segmentation**: Break source document and summary into individual sentences.
    2. **NLI Pair Matrix**: Compare each summary sentence against each source sentence → entailment scores matrix.
    3. **Aggregation**: Convert matrix to a single consistency score.
- Achieves 74.4% accuracy.

## Prompt-Based Evaluators (LLM as Judge)

LLMs' remarkable abilities led to their use as **evaluators of text**:

**Types of LLM-based evaluation:**

1. Text alone (reference-free): Judge fluency and coherence.
2. Generated text + original text + optional topic (reference-free): Judge consistency and relevancy.
3. Generated text vs. ground truth (reference-based): Judge quality and similarity.

**GEMBA (for Translation Quality):**

- Positions LLMs as tools for _evaluating_ translations, not just performing them.
- Uses specially designed **zero-shot prompts**.
- Supports both reference-based and reference-less evaluation.
- Includes source segment, candidate translation, and optional reference translation.

**G-Eval (LLM as Judge with CoT):**

- Uses highly capable LLMs (like GPT-4) as judges for evaluating other LLM outputs.
- Reference-free: Does NOT require human-written ground truth.
- Uses **Chain-of-Thought (CoT)** to break down evaluation criteria and reason through ratings.
- Achieves high correlation with human judgment.
- **DeepEval** is an open-source Python framework that supports G-Eval for LLM application testing.

# 25. Observations about Generative AI

## Why LLMs Perform So Well

- Classic ML algorithms (LSTM, CNN, xGBoost) are **clever designs** optimized for specific tasks.
- LLMs use a relatively "naive" algorithm — simply capture statistical relationships between tokens.
- The difference: LLMs are **massively bigger**, trained on far more data, with far more parameters.
- This **brute-force approach** has outpaced classic ML in general performance.

## Cultural Bias in LLMs

**World Values Survey (Inglehart-Welzel Cultural Map)** categorizes societies on two axes:

- **Traditional ↔ Secular-Rational**: Importance of religion, family, authority.
- **Survival ↔ Self-Expression**: Economic security vs. tolerance, democracy, diversity.

**Findings about LLM Bias:**

- LLMs (including ChatGPT) are primarily trained on English-dominant, Western internet data.
- They tend to reflect values aligned with high-secular-rational + high-self-expression societies (Sweden, Germany, France).
- **ChatGPT's answers to cultural questions may change depending on the language of the question** — demonstrating cultural conditioning.
- Example: India scores high on Traditional + Survival values; USA scores high on Secular-Rational + Self-Expression values.

## Hallucination in LLMs

Types of hallucinations:

1. **Factual Incorrectness**: Blood sugar level reported as 150 mg/dL when the correct value is 120 mg/dL.
2. **Misinterpretation**:
    - **Corpus Misinterpretation**: Model fails to correctly understand training data context.
    - **Prompt Misinterpretation**: Model fails to correctly understand the user's query.
3. **Needle in a Haystack**: Challenge of retrieving specific correct information from a vast corpus.
4. **Fabrication**: Pure invention — statements with NO basis in training data (unlike factual inaccuracies where existing data is misrepresented).

**Important**: Hallucinations can occur even with the best training, fine-tuning, or RAG.

## Emergent Abilities in LLMs

**Definition (adapted from Anderson, 1972 — "More Is Different"):**

> "Emergence is when quantitative changes in a system result in qualitative changes in behavior."

**Applied to LLMs:**

> "An ability is emergent if it is not present in smaller models but is present in larger models." — Jason Wei et al.

**Example from nature**: A flock of birds. Each bird follows simple rules (stay close, avoid collisions, match direction). Collectively, they create **murmuration** — a complex, dynamic pattern that cannot be explained by any single bird's behavior.

**Emergent abilities in LLMs:**

- Reasoning capabilities
- Low-resource language translation
- Math problem solving
- **137 documented emergent abilities** of LLMs have been identified in research.

---

## Q&A — Observations

**Q1: Why are LLMs considered culturally biased?** A: LLMs are primarily trained on internet data dominated by English-speaking, Western sources. They implicitly learn and reproduce the cultural values, norms, and worldviews most represented in this data. ChatGPT's answers to cultural questions can even change depending on the language used to ask the question.

**Q2: What is the difference between factual incorrectness and fabrication in hallucinations?** A: Factual incorrectness misrepresents existing information (reporting wrong numbers or facts). Fabrication invents entirely false statements with no basis in the training data whatsoever.

**Q3: What does "emergent ability" mean in the context of LLMs?** A: An emergent ability is a capability that appears in larger models but is absent in smaller ones — arising not from explicit programming but from the quantitative increase in scale and data. Examples include in-context reasoning, math problem solving, and low-resource language translation.

# 26. Multimodal NLP & MLLMs

## What is Multimodal NLP?

Multimodal NLP combines **different types of information** (text, speech, images, videos) to enhance NLP tasks. This allows machines to better comprehend human communication by using additional contextual information beyond text.

**Why Multimodal?** Modern NLP has moved beyond text-only. Models like GPT-4o and Gemini integrate multiple data streams to understand nuanced context (emotion, sarcasm in video).

## Attributes of MLLMs

1. **Multimodal Input Integration**: Process inputs from multiple modalities simultaneously (e.g., interpret image content while contextualizing with text).
2. **Contextual Understanding Across Modalities**: Achieve comprehensive understanding by aligning data from different sources.
3. **Shared Representations**: Encode diverse modalities into a unified representation for cross-modal reasoning.
4. **Cross-Modal Generation**: Generate outputs in one modality from inputs in another (image captioning, text-to-image, video description).

## Examples of Multimodal NLP Tasks

- **Image Captioning**: Generate descriptive text captions for images.
- **Visual Question Answering (VQA)**: Answer questions about an image based on both visual content and text.
- **Machine Translation with Visual Context**: Use images to resolve word ambiguity (e.g., "rock" as stone vs. music genre).
- **Multimodal Sentiment Analysis**: Analyze sentiment from spoken words + tone of voice + facial expression.

## Application Domains

- **Healthcare**: Combine medical images, patient records, and clinical notes for better diagnostics.
- **Autonomous Driving**: Process camera visuals, audio signals, and road sign text.
- **Assistive Technology**: Describe visual scenes for the visually impaired.
- **Human-Computer Interaction**: More natural interfaces for virtual assistants.

## Challenges

1. **Representation of Diverse Data**: Images have spatial information; text has sequential semantics. Unifying these within one model is complex.
2. **Scalability and Computational Demand**: Processing multimodal inputs at scale places huge strain on compute resources.

## Fusion Strategies in Multimodal Deep Learning

**Early Fusion**: Combine raw inputs from different modalities before any processing.

**Intermediate Fusion**: Each modality is partially processed separately, then combined at an intermediate layer.

**Late Fusion**: Each modality is fully processed separately, embeddings are generated, and then fused. (Used in CLIP, ALIGN.)

**Co-Attention**: An important fusion strategy where image-attended text is concatenated with text-attended image.

## Figurative Language (Pure Text)

Figurative language uses words beyond their literal meaning:

- **Metaphor**: "Time is a thief."
- **Simile**: "She's as sharp as a tack."
- **Sarcasm**: "Oh great, another Monday!"
- **Irony**: "What a beautiful day" (during a thunderstorm).
- **Hyperbole**: "I've told you a million times."

**Challenges for LLMs:**

- Context dependency: "He's on fire" = praise or literal emergency?
- Cultural references: Idioms don't translate across cultures.
- Tone and emotion: Text doesn't always convey tone.
- Training data gaps: Sarcasm/idiom patterns are underrepresented.
- Ambiguity: Multiple interpretations possible.

## Multimodal Figurative Language

- **Garden Path**: Punchline forces reinterpretation of the setup.
- **Red Light**: Punchline gives a completely different meaning.
- **Cross Road**: Punchline resolves an incongruity.

# 27. Vision Transformer (ViT)

## What is ViT?

A **Vision Transformer** is an adaptation of the Transformer architecture (originally for NLP) re-engineered for image data. Unlike CNNs which use convolutional filters, ViTs use **self-attention mechanisms** to process images.

## Key Concepts

1. **Patch-Based Image Processing**: Image is divided into fixed-size patches (e.g., 16×16 pixels). Each patch is treated as a token — analogous to a word in NLP.
2. **Linear Embedding**: Each patch is flattened into a vector and embedded into a high-dimensional space (like word embeddings).
3. **Positional Embedding**: Since Transformers don't inherently capture position, positional encodings are added to maintain spatial structure.
4. **Self-Attention**: Computes relationships between ALL image patches — captures both local and global features.
5. **CLS Token**: A special classification token that attends to all patches and is used for final predictions.

## Classic ViT Setup (from "An Image is Worth 16×16 Words")

**Step-by-step:**

1. **Image Patching**: Split input image into fixed-size patches (e.g., 16×16 pixels). Flatten and treat as tokens.
2. **Linear Embedding**: Embed each patch into a fixed-size vector.
3. **Positional Embedding**: Add positional encodings to preserve spatial structure.
4. **Self-Attention**: Compute relationships between all patches (global + local features).
5. **CLS Token**: Introduced as the first token; attends to all patches; used for classification.
6. **Final Classification**: The final hidden state of the CLS token passes through a classification head.

## CNN vs. ViT

|Feature|CNN|ViT|
|---|---|---|
|Context|Local patterns (convolutions)|Global relationships (self-attention)|
|Bias|Biased toward local features (edges, textures)|No local bias — can capture long-range dependencies|
|Preprocessing|Extensive (convolution layers)|Simple (split into patches, flatten)|
|Scalability|Good|Better for very large datasets|

---

## Q&A — ViT

**Q1: How does ViT treat an image as a sequence?** A: The image is divided into fixed-size patches (e.g., 16×16 pixels). Each patch is flattened into a 1D vector, linearly embedded, and positional encodings are added. This creates a sequence of "patch tokens" analogous to word tokens, which the Transformer processes with self-attention.

**Q2: Why is the CLS token important in ViT?** A: The CLS (classification) token is a special learnable token prepended to the sequence of patch embeddings. It attends to all patch tokens through self-attention, aggregating global information from the entire image. Its final hidden state is used as the image representation for classification.

# 28. CLIP & Separate Encoder with Late Fusion

## Separate Encoder with Late Fusion Architecture

**Process:**

- Text → Text Encoder (Transformer) → Text Embedding.
- Image → Image Encoder (CNN or ViT) → Image Embedding.
- Both embeddings fused into a **shared latent space**.
- Downstream tasks (classification, retrieval, reasoning) operate in this shared space.

**Strengths:**

- Modular and scalable.
- Easy to optimize each modality encoder independently.

**Limitations:**

- Cross-modal interactions only after fusion (late-stage).
- Less dynamic inter-modal interaction compared to unified transformers or cross-attention.

## CLIP (Contrastive Language-Image Pre-training)

**Architecture:**

- **Image Encoder** + **Text Encoder**.
- Both encoders are jointly trained to produce a **shared embedding space**.

**Training Objective:**

- For matching image-text pairs: Embeddings are "close" (high cosine similarity).
- For non-matching pairs: Embeddings are "far" (low cosine similarity).
- Loss function: **Contrastive loss**.
- Distance metric: **Cosine Similarity**.

**Image Encoder:**

- ResNet-50, ResNet-101, RN50x4, RN50x16, RN50x64.
- Vision Transformers: ViT-B/32, ViT-B/16, ViT-L/14.

**Text Encoder:**

- Transformer-based (12 layers, 512 width, 8 attention heads, ~63M parameters).
- Uses **BPE** with vocabulary size 49,152.
- Maximum sequence length: 76 tokens.
- Special [SOS] and [EOS] tokens.
- Linear projection to map to shared multi-modal embedding space.

**What CLIP IS:**

- A multimodal foundational model.
- Bridges visual and text inputs via contrastive training.
- Used for zero-shot image classification and image-text retrieval.
- Can be a component in multimodal LLMs.

**What CLIP is NOT:**

- Not designed for language generation like LLMs.
- Not designed for complex reasoning or conversation.

**CLIP Learns a Joint Image-Text Embedding Space:**

- Diagram summary: A matrix of image-text pairs. On the diagonal are matching pairs (high similarity). Off-diagonal are non-matching pairs (low similarity). CLIP is trained to maximize diagonal scores and minimize off-diagonal scores.

## ALIGN (Google)

- Similar to CLIP but trained on noisily labeled image-text pairs at massive scale.
- Demonstrates that scale can overcome noisy data.

# 29. Multimodal Converters & Perceivers

## Why Are They Needed?

Since LLMs are primarily trained on text, a **semantic gap** exists when processing visual/audio data. **Perceivers and Converters** bridge this gap.

- **Converters/Adapters**: Act as mappers that _translate_ modality-specific features into a format the LLM understands. Goal: make diverse data types "speak the same language."
- **Perceivers**: Use attention-based bottlenecks to process and fuse diverse inputs efficiently.

# 30. Unified Transformer Approach

## Unified Decoder Approach — Method A: Single Sequence

**How it works:**

- Textual inputs → word tokens.
- Visual data → patches → patch embeddings.
- These embeddings are **combined into a single sequence** and processed by the transformer together.
- Modality-specific identifiers are added to the sequence (marking each input's source).
- All modalities are treated as part of the same computational process.

## Unified Decoder Approach — Method B: Cross-Modality Attention

**How it works:**

- Same image encoder setup.
- But instead of encoding patches as inputs to the LLM, they are **connected to the LLM via cross-attention**.
- Patches feed into the multi-head attention layer via cross-attention mechanism.

## Method A vs. Method B

|Aspect|Method A (Unified Embedding)|Method B (Cross-Modality Attention)|
|---|---|---|
|Implementation|Easier (no LLM architecture modification)|More complex (requires cross-attention layers)|
|Computational Efficiency|Lower (overloads input context with image tokens)|Higher (image tokens introduced later in cross-attention)|
|Preserving text-only performance|Harder|Easier (LLM parameters can stay frozen during training)|

# 31. BLIP-2 & Q-Former

## Workflow of a VQA System

Simple approach (may not work for hard questions):

1. Pass image through a caption generator.
2. Combine user question + generated caption into a prompt.
3. Send prompt to LLM → final output.

**Problem**: For difficult questions requiring complex reasoning, the intermediate caption loses too much visual detail.

## Q-Former (Querying Transformer)

The Q-Former bridges image encoders and LLMs:

- Transforms a user's query based on the image.
- Extracts the correct information from the image based on the prompt.
- Provides this to the LLM in a format it can use.

## BLIP-2 Components

1. **Image Encoder**: Pretrained model (often CLIP) — embeds images into abstract representations.
2. **Text Encoder**: Pretrained model — embeds text into abstract representations.
3. **LLM**: Large language model for general language tasks (e.g., OPT, Flan-T5).
4. **Q-Former**: Transformer that combines image and text embeddings into a format compatible with the LLM. Its main job: contextualize both inputs and feed them to the LLM in a way conducive to text generation.

## Q-Former Details

- Acts as a **perceiver** — lightweight transformer extracting visual features from frozen image encoder using **learnable query vectors**.
- Uses **shared self-attention layers** for efficient image-text interaction.
- Contains two submodules working in tandem: image transformer and text transformer.
- Interaction occurs via **cross-attention** for seamless image-text modality integration.

## Q-Former Training

### Stage 1 (Three objectives from BLIP-1)

1. **Image-text contrastive learning**: Similar to CLIP (align matched pairs, push apart unmatched).
2. **Image-grounded text generation**: Generate image captions.
3. **Image-text matching**: Binary classification — does this image match this text?

### Stage 2

- Full model constructed by inserting a **projection layer** between Q-Former and LLM.
- Projection layer transforms Q-Former embeddings to LLM-compatible lengths.
- Task: Describe input images.
- During this stage: Image encoder and LLM remain frozen; only Q-Former and projection layer are trained.
- Used CLIP-pretrained image encoder + OPT or Flan-T5 for LLM.
- Outperforms Flamingo on various VQA tasks with a fraction of trainable parameters.

## CLIP vs. BLIP-2

|Aspect|CLIP|BLIP-2|
|---|---|---|
|Architecture|Dual-encoder (image + text, contrastive)|Q-Former (perceiver) connecting frozen encoders + frozen LLM|
|Strength|Zero-shot classification, image-text retrieval|Complex VQA, conversational visual interaction, detailed captioning|
|Generation|Cannot generate detailed captions|Yes, via LLM|
|Efficiency|Must retrain both encoders|Freezes large components; only trains small Q-Former|

---

## Q&A — BLIP-2

**Q1: What is the Q-Former's role in BLIP-2?** A: The Q-Former acts as a lightweight bridge between the frozen image encoder and the frozen LLM. It uses learnable query vectors and cross-attention to extract the most task-relevant visual features from the image and repackages them in a format compatible with the LLM's text input space.

**Q2: Why does BLIP-2 keep the image encoder and LLM frozen?** A: Freezing the large pretrained components avoids the enormous cost of retraining them. Only the small Q-Former (and projection layer) are trained, dramatically reducing compute requirements while still achieving strong performance (outperforming much larger models like Flamingo).

# 35. Floating Point Representations

## Components of Floating-Point Numbers

A floating-point number has three components:

1. **Sign bit**: 0 = positive, 1 = negative.
2. **Exponent**: Indicates the power of the base (usually 2). The actual exponent = stored exponent − bias.
3. **Mantissa (Significand)**: The fractional part. The mantissa is multiplied by $2^{\text{exponent}}$.

$$\text{Value} = (-1)^{\text{sign}} \times 2^{\text{exponent} - \text{bias}} \times \text{mantissa}$$

## Bias

Bias = $2^{k-1} - 1$ where $k$ = number of exponent bits.

- 8-bit float: bias = $2^{3-1} - 1 = 3$
- 32-bit float: bias = $2^{8-1} - 1 = 127$

## Example (FP32): `01000001110100000000000000000000`

- **Sign**: `0` → Positive.
- **Exponent**: `10000011` = 131 (decimal). Actual exponent = 131 − 127 = 4. So $2^4 = 16$.
- **Mantissa**: Next 23 bits. Fractional part: $1 \times \frac{1}{2} + 0 \times \frac{1}{4} + 1 \times \frac{1}{8} + \ldots = 0.625$. Mantissa = $1 + 0.625 = 1.625$.
- **Result**: $(-1)^0 \times 16 \times 1.625 = 26$.

## Floating Point Data Types Summary

|Type|Bits|Sign|Exponent|Mantissa|Range|Precision|Memory|
|---|---|---|---|---|---|---|---|
|**FP32** (IEEE 754)|32|1|8|23|1.18e-38 to 3.40e38|6-9 sig. decimal digits|4 bytes|
|**FP16** (IEEE 754)|16|1|5|10|5.96e-8 to 65504|4 sig. decimal digits|2 bytes|
|**BF16** (Google Brain)|16|1|8|7|1.18e-38 to 3.40e38|3 sig. decimal digits|2 bytes|
|**TF32** (NVIDIA A100)|19|1|8|10|1.18e-38 to 3.40e38|4 sig. decimal digits|~2.4 bytes|
|**INT8**|8|—|—|—|-128 to 127|—|1 byte|

### Key Notes

**FP32 (FLOAT32)**:

- IEEE 754 standard single precision.
- Default in all deep learning frameworks.
- 4 bytes per value.

**FP16 (FLOAT16)**:

- IEEE 754 half precision.
- Faster and less memory than FP32.
- Narrow dynamic range can cause overflow/underflow.
- Used in post-training quantization.

**BF16 (BFLOAT16)**:

- Developed by Google Brain (for deep learning).
- Same exponent bits as FP32 → same dynamic range as FP32.
- Fewer fraction bits than FP16 → less precise, but same range.
- When converting FP32 → BF16: exponent bits preserved; fraction bits reduced by truncation.
- 2 bytes per value.

**TF32 (TensorFloat32)**:

- New datatype in NVIDIA A100 GPUs.
- Same 10-bit fraction as FP16 (less precise than FP32).
- Same 8-bit exponent as FP32 (same range).
- Technically 19 bits total.

**INT8**:

- 8-bit signed integer.
- Range: -128 to 127.
- 1 byte per value.
- Used for aggressive quantization.

## Memory Reduction Impact

|Conversion|Memory Reduction|
|---|---|
|FP32 → FP16/BF16|2x|
|FP32 → INT8|4x|
|FP32 → INT4 (NF4)|8x|

Reducing memory requirements improves:

- Model inference efficiency.
- Fine-tuning feasibility on smaller GPU configurations.

## Q&A — Floating Point

**Q1: What is the bias in FP32 and how is it used?** A: Bias for FP32 = 127 (calculated as $2^{8-1} - 1$). The stored exponent is the actual exponent + 127. To find the actual exponent, subtract 127 from the stored 8-bit exponent value.

**Q2: Why was BF16 developed, and what problem does it solve?** A: The original IEEE 754 FP16 was not designed for deep learning — its narrow dynamic range (max 65504) is prone to overflow and underflow during training. BF16 keeps the same 8-bit exponent as FP32 (giving the same wide dynamic range), but reduces the mantissa to 7 bits. This makes FP32 → BF16 conversion simple (just truncate mantissa bits).

**Q3: Why does INT8 offer 4x memory reduction over FP32?** A: FP32 uses 4 bytes (32 bits) per value. INT8 uses 1 byte (8 bits) per value. Thus 4 bytes / 1 byte = 4x reduction.

# MASTER Q&A — Cross-Topic Review

**Q1: Trace the full journey of a text prompt through a GPT model, from input to output.** A: (1) Tokenization: Text → tokens → token IDs. (2) Token Embedding: Token IDs → dense vectors via embedding lookup. (3) Positional Encoding: Add absolute positional embeddings to token vectors. (4) N × Transformer Blocks: Self-attention (Q, K, V projection → masked attention → residual connection) → Layer Norm → FFN (Linear → GELU → Linear) → Residual → Layer Norm. (5) Final Layer Norm. (6) Linear projection to vocabulary size. (7) Softmax → probability distribution. (8) Decoding strategy selects next token. (9) Append token, repeat from step 3 until stop condition.

**Q2: Compare LoRA, Prefix Tuning, Soft Prompt Tuning, and Adapters as PEFT methods.**

|Method|Where added|What's trained|Depth|
|---|---|---|---|
|Soft Prompt Tuning|Input embedding layer only|Soft prompt vectors|Shallow|
|Prefix Tuning|K and V in every transformer layer|Prefix vectors|Deep|
|Adapter|After attention and FFN in every block|Small bottleneck adapter weights|Deep|
|LoRA|Q and V weight matrices (attention)|Low-rank matrices A and B|Deep (parameter level)|

**Q3: A company wants to deploy a large LLM on a mobile device. Which combination of techniques would be most effective?** A: (1) **QLoRA** — quantize the model to 4-bit NF4 and use LoRA for efficient fine-tuning. (2) **Knowledge Distillation** — create a smaller student model from the larger teacher. (3) **Pruning** — remove redundant weights from the distilled model. (4) **PTQ (Post-Training Quantization)** — quantize the final deployed model to INT8. This pipeline: large LLM → fine-tune with QLoRA → distill → prune → quantize → edge deployment.

**Q4: Explain the complete DeepSeek R1 training pipeline.** A: (1) Pre-train DeepSeek-V3-Base on massive multilingual corpus. (2) Initial SFT on reasoning data (Checkpoint 1). (3) RL training with GRPO using format + accuracy + language consistency rewards (Checkpoint 1.1). (4) Rejection sampling from Checkpoint 1.1 to generate 600K reasoning samples. (5) Add 200K general/non-reasoning samples → 800K total. (6) Final SFT on all 800K samples. (7) Final RL with combined rule-based and generative AI reward model → DeepSeek-R1.

**Q5: Why is Top-P (Nucleus) sampling generally preferred over Top-K for open-ended generation?** A: Top-K always considers exactly K tokens regardless of the probability distribution shape. If the distribution is very peaked (one word has 99% probability), K=50 would still add 49 near-zero tokens unnecessarily. If the distribution is flat, K=50 might miss many plausible words. Top-P dynamically adjusts: for peaked distributions, it selects fewer tokens (just the top ones summing to p); for flat distributions, it selects more. This adaptive behavior produces more natural and contextually appropriate text.

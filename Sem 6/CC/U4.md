## 1. Cloud Computing Recap & Cloud Controller

### 1.1 What is Cloud Computing?

Cloud computing is a model for enabling **ubiquitous, convenient, on-demand** network access to a shared pool of configurable computing resources (networks, servers, storage, applications, and services). These can be rapidly provisioned and released with minimal management effort. The key phrase here is **"as a service"** — you don't own or manage the hardware, you consume it like a utility (just like electricity).

The physical infrastructure supporting cloud computing is housed in **distributed data centers** spread across the world, accessible over the Internet.

### 1.2 Service Models (Recap)

The course has already covered three fundamental service models:

- **IaaS (Infrastructure as a Service):** Lowest abstraction. You rent raw compute, storage, and networking (e.g., AWS EC2, Google Compute Engine). You manage OS and above.
- **PaaS (Platform as a Service):** Middle abstraction. You rent a development platform with runtime, database, tools (e.g., Google App Engine, Heroku). You manage your application code only.
- **SaaS (Software as a Service):** Highest abstraction. You rent a fully functional application (e.g., Gmail, Salesforce, Office 365). You just use it.

### 1.3 Deployment Models (Recap)

- **Public Cloud:** Resources shared among many customers on third-party infrastructure (AWS, Azure, GCP).
- **Private Cloud:** Dedicated infrastructure for a single organization, higher control, higher cost.
- **Hybrid Cloud:** Combination of both; critical workloads on private, overflow or less-sensitive workloads on public.

### 1.4 The Cloud Controller — Why Does It Matter?

Until now, the course focused on **data paths** — how data flows through cloud services. Unit 4 shifts focus to the **control path** — how cloud resources are managed, coordinated, and made reliable.

Think of it this way: the data path is the highway, but the control path is traffic management — signals, laws, routing decisions.

#### Eucalyptus as an Illustrative Example

**Eucalyptus** is an open-source software framework for building private/hybrid clouds compatible with Amazon's EC2 and S3 APIs. It shows concretely how cloud control is structured:

|Component|Role|
|---|---|
|**UFS (User Facing Services)**|Handles AWS-compatible API requests from CLI/GUI|
|**Cloud Controller (CLC)**|High-level resource tracking; only ONE can exist per cloud|
|**Cluster Controller (CC)**|Manages Node Controllers; deploys/manages VMs within a cluster|
|**Node Controller (NC)**|Runs on each physical machine; controls VM lifecycle via OS/hypervisor|
|**Storage Controller**|Interfaces with storage systems|

#### How the CLC Makes Decisions

The Cloud Controller maintains a **System Resource State (SRS)** — a continuously updated picture of what resources are available across all nodes. When a user request arrives:

1. The SRS is consulted to make an **admission control decision** (can we serve this?).
2. If yes, resources are **reserved** in the SRS.
3. The request is sent **downstream** to CC → NC.
4. On success, the resource is **committed** in the SRS.
5. On failure, the reservation is **rolled back**.

This is essentially a distributed transaction — a concept we'll revisit deeply.

### 📝 Q&A: Cloud Computing Recap

**Q1: What is the difference between IaaS, PaaS, and SaaS in terms of what the user manages?**

> In IaaS, the user manages the OS, middleware, runtime, and applications. In PaaS, the user only manages their application and data; the platform handles OS, middleware, and runtime. In SaaS, the user manages nothing — the provider handles everything, and the user simply uses the application.

**Q2: Why can only ONE Cloud Controller (CLC) exist in a Eucalyptus deployment?**

> The CLC holds the System Resource State (SRS) — the single authoritative source of truth about what resources are available. Having multiple CLCs would require constant synchronization, creating consistency challenges. This is the classic master-slave choice: simplicity and consistency at the cost of being a single point of failure.

**Q3: What happens if a VM creation request fails midway in Eucalyptus?**

> If the request cannot be satisfied by the NC, the reserved resources in the SRS are rolled back — similar to a database transaction rollback. This ensures the SRS remains consistent and doesn't show resources as committed that are not actually in use.

**Q4: What is the "control path" vs. the "data path" in a cloud environment?**

> The data path refers to how actual application data flows between users and cloud resources (e.g., uploading a file, reading a database). The control path refers to how management decisions are made — scheduling, provisioning, failure detection, leader election, consensus — i.e., how the cloud knows what to do with those resources.

## 2. Distributed Systems — Architectural Models

### 2.1 What is a Distributed System?

A **distributed system** is a collection of independent computers (nodes) that appears to its users as a single coherent system. The nodes communicate by passing messages. Key characteristics:

- Components are on **different machines**.
- They **communicate and coordinate** actions.
- They **appear as one** to the end user.

Cloud infrastructure is inherently a distributed system. A user hitting "upload" on AWS S3 might interact with dozens of servers across multiple data centers simultaneously.

### 2.2 Three Models of Distributed Systems

The classification helps us reason about how a distributed system works and can fail:

#### 1. Architectural Models

These describe **where** components are placed and **how** they relate:

- **System Architecture:** Physical placement of components across machines. Examples: P2P model (symmetric), Client-Server model (asymmetric).
- **Software Architecture:** Logical organization of software. Example: 3-Tier Architecture (Presentation → Business Logic → Data).

#### 2. Interaction Models

These describe **how** components communicate and coordinate in time:

- **Synchronous distributed systems:** Every operation has a known maximum time bound. Messages arrive within time $d$. Processes respond within time $r$. Predictable, but hard to guarantee in practice.
- **Asynchronous distributed systems:** No time bounds on message delivery or process execution. The Internet is a perfect example.

#### 3. Fault Models

These describe **what can go wrong** and **how badly**:

- **Omission faults:** A component fails to send or receive a message.
- **Arbitrary (Byzantine) faults:** A component behaves in any unpredictable way — even sending incorrect or malicious information.
- **Timing faults:** A component violates its timing constraints (e.g., responds too late).

---

### 📝 Q&A: Distributed Systems Models

**Q1: Why is the Internet considered an asynchronous distributed system?**

> Because there is no guaranteed upper bound on how long it takes for a message to travel from source to destination. A packet might arrive in 1 ms on a good day, or be delayed by 5 seconds due to network congestion, and there's no way to know in advance.

**Q2: What is the difference between system architecture and software architecture in a distributed system?**

> System architecture is about physical placement — which machine runs which component. Software architecture is about logical organization — how software modules interact with each other logically, regardless of where they physically run. A 3-tier architecture (presentation, logic, data) describes software organization; whether the database runs on the same server as the business logic is a system architecture question.

**Q3: Why are Arbitrary (Byzantine) faults the hardest to handle?**

> With crash faults or omission faults, a failed component is simply silent — it stops responding. With Byzantine faults, the component actively misbehaves — it may send wrong values, lie about its state, or selectively respond to some nodes but not others. Designing a system that continues to work correctly even when some nodes are actively malicious requires significantly more complex algorithms and redundancy.

## 3. Peer-to-Peer (P2P) Architecture

### 3.1 What Is P2P?

In a **Peer-to-Peer (P2P)** system, **every node (peer) acts as both a client and a server simultaneously**. There is no dedicated server, no hierarchy, and no central coordinator. The roles of all entities are **symmetric**.

Think of BitTorrent: when you download a file, you also simultaneously upload pieces of that file to others. There is no central server giving out the file.

### 3.2 Key Characteristics of P2P

- **All nodes are equal** — no hierarchy exists.
- **No Single-Point-of-Failure (SPOF)** — no central node whose failure brings down everything.
- **No central coordinator needed** — the system is self-organizing.
- **Self-organizing:** Nodes autonomously join or leave the network and form connections with other nodes based on interests, capabilities, or requirements. No node has a global view of the entire system.
- **Processing and communication loads are distributed** across many computers and access links.
- **Decision making becomes harder** — without a central authority, getting all nodes to agree on anything is complex.
- **Scales out indefinitely** in principle — adding more peers increases capacity.
- **No performance bottleneck** in theory — because no single node handles all requests.

### 3.3 Overlay Networks

A fascinating property of P2P systems: peers form a **virtual overlay network** on top of the physical network topology. This means:

- **Logical paths ≠ physical paths.** Two peers that are "neighbors" in the overlay may be geographically far apart, resulting in higher latency.
- **Each peer routes traffic** through the overlay — every peer participates in routing.

### 3.4 Data Availability in P2P

- **Popular data is highly available** — many peers hold copies, so it's easy to find.
- **Unpopular data may disappear** — if no peer is actively sharing a piece of data, it becomes unavailable. This is why obscure torrents "die."

### 3.5 Advantages and Limitations of P2P

|Advantage|Limitation|
|---|---|
|Highly scalable — add any number of peers|Difficult to manage centrally|
|Resilient to faults and attacks|Security is challenging — no central policy enforcement|
|No need for global state knowledge per node|Consistency and coordination are hard|
|Supports high degree of replication|Securing each peer individually is complex|

### 3.6 Real-World Use: Amazon Aurora (Multi-Master)

Amazon Aurora is a relational database built for the cloud with SQL compatibility. Its **multi-master clusters** consist of DB instances that all have both read/write capacity — this is a P2P-like property, where all nodes are peers with equal capability. However, this comes with "unconventional" availability characteristics and more complex monitoring needs.

---

### 📝 Q&A: Peer-to-Peer Architecture

**Q1: How does a P2P system remain functional when some nodes leave the network?**

> Because data is replicated across multiple peers, the departure of any single peer doesn't necessarily remove data from the system. Other peers holding copies continue to serve that data. The system is inherently resilient through redundancy and the absence of a single authoritative node.

**Q2: Why is "no global view" both a strength and a weakness in P2P?**

> It is a strength because it means no single node needs to hold or process information about the entire network, enabling massive scale. It is a weakness because without a global view, it's hard to ensure consistency, enforce security policies uniformly, or efficiently locate resources — nodes must search through the overlay network, which can be slow or unreliable.

**Q3: What is a "virtual overlay network" and why do logical paths not match physical paths?**

> An overlay network is a logical network built on top of an existing physical network. Peers establish connections with each other based on the overlay protocol (e.g., shared interests, DHT routing table), not based on physical proximity. So two peers that are overlay-neighbors might be on different continents — their logical connection exists at the application layer even though the physical path between them is long.

**Q4: Why do cloud systems ultimately prefer master-slave over pure P2P despite P2P's advantages?**

> Cloud services are hosted within a single administrative domain (e.g., AWS us-east-1). The administrators need centralized monitoring, consistent security enforcement, and management of consistency/linearizability. The P2P model's decentralization, while great for resilience and scale, makes it difficult to monitor, audit, or ensure data consistency — all of which are non-negotiable for enterprise cloud services.

## 4. Client-Server / Master-Slave Architecture

### 4.1 What Is the Client-Server Model?

The **Client-Server model** is an asymmetric architecture where:

- **Servers** are machines that offer services (they perform computation, hold data, etc.).
- **Clients** are machines that consume those services by sending requests.

The interaction follows a simple **request/reply protocol**, often implemented via:

- `send()` / `receive()` primitives, or
- **Remote Procedure Calls (RPC)** — where calling a function on a remote server feels like calling a local function.

A server can serve multiple clients, and a client can request services from multiple servers, regardless of physical location.

### 4.2 Master-Slave Architecture

Master-Slave is the hierarchical variant of Client-Server where:

- **Nodes are unequal** — there is a clear hierarchy (master is "above" slaves).
- **The master acts as a central coordinator** — it assigns tasks, monitors work, and makes decisions.
- **Decision making is easy** — because one node has authority.
- **Vulnerable to Single-Point-of-Failure (SPOF)** — if the master dies, the whole system potentially stops.
- **Cannot scale out indefinitely** — as the number of workers increases, the master may become a performance bottleneck.

### 4.3 Real-World Example: MapReduce in Hadoop

MapReduce is the canonical example of master-slave in cloud computing:

- **Single Master node (JobTracker/ResourceManager):**
    - Assigns map and reduce tasks to worker nodes.
    - Monitors the progress of each assigned job.
    - Redistributes tasks if a worker node fails.
- **Multiple Worker nodes (TaskTrackers/NodeManagers):**
    - Execute the assigned map or reduce tasks.
    - Report status back to the master.

This architecture is the "most predominantly undertaken" in cloud systems because:

- **Monitoring is easy** — master has a global view.
- **Consistency and linearizability** are easier to guarantee.
- **Shortcoming:** If the master node fails, the entire job fails — classic SPOF.

---

### 📝 Q&A: Client-Server / Master-Slave Architecture

**Q1: What is a Remote Procedure Call (RPC) and why is it useful in client-server systems?**

> An RPC allows a client to call a function that executes on a remote server as if it were a local function call. The networking details (serializing parameters, sending over the network, deserializing results) are abstracted away by the RPC framework. This makes distributed programming much simpler, as developers don't need to manually write socket communication code.

**Q2: How does MapReduce handle worker node failure in a master-slave setup?**

> The master constantly monitors workers through heartbeats. If a worker doesn't respond within a timeout, the master considers it failed and reassigns its incomplete tasks to other available workers. The completed work of failed workers may also need to be redone (for map tasks, since their output on local disk is inaccessible), but reduce task outputs in HDFS are retained.

**Q3: What is the fundamental trade-off when choosing between P2P and Master-Slave for a cloud application?**

> P2P offers better fault resilience, scalability, and no SPOF, but is harder to manage, secure, and keep consistent. Master-Slave offers easier management, monitoring, and consistency, but introduces a single point of failure and potential bottleneck at the master. For most enterprise cloud applications that need predictable behavior, master-slave is preferred, sometimes augmented with leader election mechanisms to mitigate the SPOF.

## 5. Unreliable Communication in Distributed Systems

### 5.1 Why Is Communication Unreliable?

In a cloud environment where businesses host services on third-party distributed infrastructure, reliability is paramount. But the reality is that distributed systems run on **commodity hardware**, and failures are expected, not exceptional. Hardware failures can cascade into avalanches of failures causing major outages.

When you send a request in a distributed system and wait for a response, many things can go wrong:

1. **Your request was lost** (a network cable was unplugged).
2. **Your request is waiting in a queue** (network or recipient is overloaded).
3. **The remote node failed** (it crashed or was powered down).
4. **The remote node temporarily stopped responding** (it will come back).
5. **Your request was processed, but the response was lost** (a switch was misconfigured).
6. **Your request was processed, but the response is delayed** (will arrive later).

The critical insight: **you cannot distinguish between cases 1 through 6 from the sender's perspective without additional information**. Did your message get through? You don't know.

### 5.2 What Is Reliability?

**Reliability** is:

- The **probability** that a system will meet performance standards and yield correct output for a specific time.
- The ability to remain **available** over a period of time under expected load and data volume.
- Performing the function the user expected **without errors, disruptions, or significant performance reduction**.

High reliability is critical for cloud applications where failure can mean loss of business or even loss of life (consider medical systems, financial transactions, or power grid controls).

### 5.3 Timeouts and Unbounded Delays

The fundamental question: **How long should a timeout be?**

In an ideal world:

- Every packet is delivered within time $d$ or lost (never delayed beyond $d$).
- Every non-failed node handles a request within time $r$.

Then you could guarantee: every successful request receives a response within time $2d + r$.

Where:

- $d$ = maximum one-way packet delivery time
- $r$ = maximum request processing time
- $2d$ = one trip for request + one trip for response

If you don't receive a response in $2d + r$, you know something is wrong.

**But reality is different:**

- **Asynchronous networks have unbounded delays** — there is no upper limit on packet delivery time.
- **Servers cannot guarantee** they handle requests within a maximum time.

So there is **no correct value for a timeout**. Best practices:

- Measure the distribution of network round-trip times over many machines and extended time periods.
- Factor in the **application's characteristics** (how bad is a false positive vs. a false negative?).
- Use **adaptive timeouts**: continually measure response times and their **jitter** (variance), and automatically adjust timeout thresholds.

### 5.4 Network Congestion and Queueing

When many packets arrive simultaneously at a network switch:

- **Packets queue** at the switch's input buffer — they wait for an output slot.
- If the **queue fills up**, packets are **dropped** and must be retransmitted (TCP handles this, but the application sees increased latency).

In **public clouds and multi-tenant data centers**, this is exacerbated:

- Resources (network links, switches, NIC, CPU) are **shared** among many tenants via virtualization.
- Virtualization adds overhead (processing time to handle network traffic from multiple VMs).
- A **"noisy neighbor"** — a tenant on the same physical machine using excessive resources — can degrade your performance dramatically.
- **Batch workloads** (like MapReduce jobs) can easily saturate network links.

### 📝 Q&A: Unreliable Communication

**Q1: Why is the non-determinism of partial failures what makes distributed systems "hard"?**

> Because you cannot reason about the system's state with certainty. In a single machine, if a function call fails, you get an exception. In a distributed system, if you don't hear back from a remote node, you don't know whether your request was received, processed, responded to (with the response lost), or whether the node is just slow. This uncertainty means you cannot make safe decisions without additional mechanisms like consensus protocols.

**Q2: What is network "jitter" and why does it matter for setting timeouts?**

> Jitter is the variation (standard deviation or variability) in packet delivery times. High jitter means that even though the average round-trip time is, say, 10 ms, individual trips might take anywhere from 2 ms to 200 ms unpredictably. Setting a timeout too low in a high-jitter environment means you'll often declare a node failed when it's actually just experiencing a slow response, causing unnecessary failovers (false positives).

**Q3: How does TCP handle packet loss, and why does the application still see the effect?**

> TCP has a built-in retransmission mechanism: if a segment is not acknowledged within a timeout, TCP automatically retransmits it. The application layer doesn't see the lost packet directly — TCP presents a continuous byte stream. However, the application does see the increased latency caused by the retransmission delay.

**Q4: What is the "noisy neighbor" problem in multi-tenant clouds?**

> In a multi-tenant environment, multiple customers (tenants) share the same physical infrastructure. If one tenant runs a CPU-intensive workload or generates heavy network traffic, it consumes shared resources and degrades the performance of neighboring tenants. This is called the "noisy neighbor" problem. It makes network delays highly variable and hard to predict for any individual tenant.

## 6. Fault Models & Failure Detection

### 6.1 Faults, Errors, and Failures — The Hierarchy

These three terms are distinct and important:

- **Fault:** The underlying cause (e.g., a hardware defect, a software bug, a cosmic ray flipping a bit).
- **Error:** The incorrect system state caused by the fault (e.g., a corrupted memory value).
- **Failure:** When the system can no longer meet its promises to users (e.g., a web server stops responding).

The chain: **Fault → Error → Failure**

### 6.2 Types of Faults

|Fault Type|Description|
|---|---|
|**Transient**|Appears once, then disappears. E.g., a temporary network glitch. Easiest to handle.|
|**Intermittent**|Occurs, vanishes, reappears, with no real pattern. E.g., a loose connection that sometimes works. **Worst kind** — very hard to diagnose.|
|**Permanent**|Persists until the faulty component is replaced or repaired. E.g., a failed hard disk.|

### 6.3 Measuring Reliability: MTBF

**Mean Time Between Failures (MTBF)** is the average operating time between consecutive failures:

$$\text{MTBF} = \frac{\text{Total Uptime}}{\text{Number of Breakdowns}}$$

**Example:** If a system operates for 24 hours and experiences 12 failures in that time:

$$\text{MTBF} = \frac{24 \text{ hours}}{12} = 2 \text{ hours}$$

This means on average, the system runs for 2 hours before something goes wrong.

### 6.4 Fault Models — Designer's Perspective

To build systems with **predictable behavior** in the presence of faults, we need to formally model what can go wrong:

|Fault Model|Description|Example|
|---|---|---|
|**Crash-stop**|Node suddenly stops forever.|A VM that is hard-killed.|
|**Crash-recovery**|Node crashes and may restart later. Volatile memory lost; disk state preserved.|A server that reboots after a crash.|
|**Byzantine (Arbitrary)**|Node does anything — sends wrong data, behaves maliciously.|A compromised node in a blockchain network.|
|**Omission**|Node fails to send/receive messages.|Dropped network packets.|
|**Timing**|Node violates its timing constraints.|A slow node that always responds too late.|

The reason we need these models: **you can't build a fault-tolerant system unless you specify which faults you're tolerating**. A system designed to tolerate crash-recovery faults may fail completely if Byzantine faults occur.

### 6.5 Failure Detection Mechanisms

Once we have a fault model, we need to **detect** when a fault has occurred.

#### Heartbeats (Push Model)

- Each application **periodically sends a signal** ("I am alive") to a monitoring service.
- If the monitor doesn't receive a heartbeat for a configured number of periods, the application is **declared failed**.
- **Advantage:** The monitored process actively proves it's alive.
- **Additional use case (GFS example):** In Google File System, chunkservers send heartbeats to the master node. These heartbeats also carry **additional configuration information** — a dual-purpose mechanism. If a chunkserver fails, the master begins **replicating its data** to other chunkservers.

#### Probing (Pull Model)

- The monitor **periodically sends a lightweight probe** (a minimal service request) to the application instance.
- If the application doesn't respond to a configured number of probes, it's declared failed.
- Less efficient than heartbeats for many-to-many monitoring because it generates more messages.

#### The Dual Scheme (Hybrid)

- Monitored processes initially use the push model (send "I am alive").
- After a period, the monitor assumes that any process that did NOT send a liveness message is using the pull model, and begins probing it.
- Combines the efficiency of heartbeats with the robustness of probing.

#### Trade-off: Speed vs. Accuracy

- **Low threshold** (few missed heartbeats before declaring failure): **Fast detection** but more **false positives** (declare a slow-but-alive node as failed).
- **High threshold** (many missed heartbeats before declaring failure): Fewer false positives but **slower detection**.

Since declaring an application failed has severe consequences (triggering failover, losing in-flight requests), systems generally **set a high threshold** for missed heartbeats/probes.

### 📝 Q&A: Fault Models & Failure Detection

**Q1: Why are intermittent faults considered the "worst form" of faults?**

> Because they are nearly impossible to reproduce and diagnose. A transient fault is gone before you can investigate. A permanent fault is consistently present — you can replicate the problem and fix it. An intermittent fault appears and disappears unpredictably, so by the time engineers investigate, the fault may have vanished. This makes root cause analysis extremely difficult and time-consuming.

**Q2: A system fails 3 times in one month with a total downtime of 6 hours. What is the MTTR?**

> MTTR (Mean Time To Repair) = Total downtime / Number of failures = 6 hours / 3 = **2 hours per failure**. This means on average, each failure takes 2 hours to fix and restore service.

**Q3: What is the difference between a crash-stop and crash-recovery fault model?**

> In a crash-stop model, once a node fails, it is gone forever — it never comes back. In crash-recovery, the node may crash but is expected to restart (recovering from stable storage). The key design implication: crash-stop allows other nodes to eventually stop worrying about a failed node, while crash-recovery requires the system to handle nodes rejoining with potentially stale or incomplete state.

**Q4: Why might detecting Byzantine faults be impossible with heartbeats alone?**

> Heartbeats only tell you whether a node is alive — they don't tell you whether the node is behaving correctly. A Byzantine node can happily send heartbeats (proving it's "alive") while simultaneously sending incorrect data, manipulating other nodes, or selectively dropping messages. Byzantine fault detection requires cryptographic mechanisms (digital signatures) and majority voting to identify malicious behavior.

**Q5: What is the role of GFS chunkserver heartbeats beyond simple failure detection?**

> In GFS, heartbeat messages from chunkservers to the master serve a dual purpose: they prove the chunkserver is alive (failure detection), and they also carry status information — what chunks the chunkserver holds, their version numbers, etc. This allows the master to maintain an up-to-date picture of the distributed file system without constantly polling every chunkserver.

## 7. Fault Tolerance

### 7.1 System Availability Concepts

**Availability** is how reliably and consistently a service can be accessed. It's the fundamental metric for cloud SLAs.

- **Uptime:** The period when a service is running and operational. Typically expressed as a percentage (e.g., 99.999% = "five nines").
- **Downtime:** The period when a service is unavailable — due to unplanned failures (software crash, hardware failure, network outage) or planned maintenance.
- **Slow/non-responsiveness:** The service is technically "up" but responds unacceptably slowly — this is also a form of degraded availability.

#### The "Five Nines" Formula

Availability % tells you how much downtime is acceptable per year:

|Availability|Downtime Per Year|
|---|---|
|99% ("two nines")|~3.65 days|
|99.9% ("three nines")|~8.76 hours|
|99.99% ("four nines")|~52.6 minutes|
|99.999% ("five nines")|~5.26 minutes|
|99.9999% ("six nines")|~31.5 seconds|

Cloud providers measure availability differently:

- **Microsoft:** Time when the overall service is up / total time.
- **Gmail:** Percentage of successful interactive user requests (plus consecutive minutes of downtime).
- **AWS:** Average percentage of requests that result in errors in any 5-minute interval.

#### Why Does Availability Decrease as System Size Grows?

As a distributed system increases in size (more nodes, more connections), the **probability of at least one component failing** increases. Additionally, isolating failures in a larger system is harder. **High Availability (HA)** design removes all single points of failure through redundancy and automatic failover.

### 7.2 What Is Fault Tolerance?

**Fault Tolerance** is the system's ability to **continue operating uninterrupted** despite the failure of one or more of its components. Two flavors:

1. **Fail-Safe Tolerance:** Safety state is preserved even if performance drops. Example: Traffic lights failing to all-red (both directions stop). Safe, but inefficient.
2. **Graceful Degradation:** Application continues but in a degraded mode. Example: A recommendation system that falls back to "most popular" recommendations when personalization data is unavailable.

### 7.3 Approaches for Building Fault Tolerance

#### 1. Redundancy

Eliminate single points of failure by duplicating critical components:

- **Hardware redundancy:** Dual power supplies, RAID disks, multiple NICs, redundant network paths.
- **Software redundancy:** Multiple replicas of a service, running on different machines.

#### 2. Reliability Engineering

Design components to be dependable. Measured using:

- **MTTF (Mean Time To Failure):** Average time until the first failure.

$$\text{MTTF} = \frac{t_1 + t_2 + t_3 + \ldots}{n}$$

Where $t_i$ is the time-to-failure for the $i$-th unit and $n$ is the number of units tested.

_Example:_ If 3 identical systems fail at hours 10, 12, and 14: $$\text{MTTF} = \frac{10 + 12 + 14}{3} = 12 \text{ hours}$$

- **MTTR (Mean Time To Repair):** Average time to restore service after a failure.

$$\text{MTTR} = \frac{\text{Total Downtime}}{\text{Number of Failures}}$$

_Example:_ 3 failures with 6 total hours of downtime → MTTR = 2 hours.

#### 3. Recoverability

The ability to overcome a momentary failure without user-visible impact:

- **Retries** with exponential backoff for transient failures.
- **Acknowledgement-based protocols** (like TCP) to ensure data delivery.
- **Error Correction Codes (ECC):** Parity bits, Hamming codes, CRC checksums — detect and correct bit errors with much less redundancy than full replication.
- **Checkpointing and Rollbacks:** Periodically save application state; on failure, restore from the most recent checkpoint.

#### 4. Additional Techniques for Distributed Systems

- **Retries:** Automatically retry failed requests (with exponential backoff to avoid overwhelming a recovering service).
- **Timeouts:** Always set deadlines on requests — never wait indefinitely.
- **Circuit Breaker Pattern:** Track failure rates. When failures exceed a threshold, "open the circuit" — stop sending requests to the failing service for a cooldown period. After the cooldown, "close the circuit" and try again.
- **Bulkheads (Isolate Failures):** Separate resource pools (threads, connections) for different functions. A failure in one pool doesn't cascade to others — like watertight compartments in a ship.
- **Caching:** Serve cached data from a local/remote cache when the upstream service is unavailable.
- **Queuing:** Buffer requests for a failing service until it recovers.

### 7.4 Failover Architecture

**Failover** is switching from a failed component to a backup.

#### Active-Active (Symmetric) Failover

- Multiple servers all **actively serve requests** simultaneously.
- Each server can also handle the load of its peers if one fails.
- Databases are **replicated** across servers.
- Called **"Continuous Availability"** — no service interruption during failover.

#### Active-Passive (Asymmetric) Failover

- One **primary (active)** server handles all requests.
- One or more **secondary (passive/standby)** servers wait idle, ready to take over.
- On primary failure: the passive server becomes active; clients must reconnect.
- **Failback:** After the primary recovers, operations can optionally be migrated back.

### 7.5 VMware FT (Fault Tolerance)

VMware FT demonstrates hardware-level fault tolerance through **virtual lock-step**:

- A **Primary VM** and a **Backup VM** run in lock-step — executing all the same instructions.
- Only the **Primary VM** sends outputs to clients.
- If the Primary fails, the Backup **takes over immediately** with no interruption.
- When the Primary recovers, it becomes the new Backup.

### 📝 Q&A: Fault Tolerance

**Q1: What is the difference between Fail-Safe tolerance and Graceful Degradation?**

> Fail-Safe tolerance prioritizes safety over performance — the system moves to a known safe state even if it stops doing useful work (e.g., all traffic lights turn red). Graceful Degradation prioritizes continued operation — the system keeps working but at reduced capacity or quality (e.g., showing generic recommendations instead of personalized ones when the personalization service is unavailable).

**Q2: Calculate the MTTF if three systems fail at 15, 20, and 25 hours respectively.**

> MTTF = (15 + 20 + 25) / 3 = 60 / 3 = **20 hours**. On average, a system of this type will run for 20 hours before failing.

**Q3: Explain the Circuit Breaker pattern with a real-world analogy.**

> Just like an electrical circuit breaker that "trips" when current exceeds a safe threshold (preventing fire or equipment damage), a software circuit breaker "trips" when a downstream service failure rate exceeds a threshold. Instead of letting all calls pile up waiting to fail, the circuit breaker immediately rejects calls to the failing service (fast-fail), reducing load on the failing service and freeing up resources in the calling service. After a timeout, the circuit "half-opens" and allows a test call through — if successful, it fully closes.

**Q4: What is the key difference between Active-Active and Active-Passive failover?**

> In Active-Active, all servers are serving requests simultaneously — no idle servers. When one fails, the others absorb its load. In Active-Passive, only the primary is serving; the passive server sits idle until needed. Active-Active is more resource-efficient and provides continuous availability, while Active-Passive is simpler but wastes the standby server's capacity.

**Q5: Why might checkpointing not work perfectly in a distributed system?**

> In a distributed system, multiple processes run on different nodes. If a failure occurs while messages are in-transit between processes, the checkpoint of each process captures a point in time — but those in-flight messages were never received by their destination. Restoring from such a checkpoint creates an inconsistent global state (the sender's checkpoint shows the message was sent; the receiver's checkpoint shows it was never received). This is called the "distributed checkpoint consistency problem."

## 8. Cluster Coordination — Consensus

### 8.1 Replicated State Machines — The Big Picture

A **replicated state machine** is the fundamental concept behind distributed coordination:

- Multiple computers all **start with the same initial state**.
- For each state transition, each computer processes the same input and moves to the same next state.
- The state machine is **replicated** across all computers for fault tolerance.
- All replicas must maintain a **consistent transaction log** — everyone agrees on the order and content of every operation.

**The challenge:** Some computers might fail, the network might be slow, and there's no global clock to synchronize event ordering. Despite all this, all replicas must agree on their state.

This is the **consensus problem**.

### 8.2 The Consensus Problem — Formally Defined

Given N nodes, each with:

- An **input variable** $x_p$ ∈ {0, 1} (initial proposal)
- An **output variable** $y_p$ that can be set only once

An algorithm achieves consensus if it satisfies:

|Property|Definition|
|---|---|
|**Agreement**|All non-faulty nodes decide on the **same** output value|
|**Termination (Liveness)**|All non-faulty nodes **eventually decide** on some value|
|**Validity**|Any decided value was **proposed** by some node|
|**Integrity**|Each node decides **at most once**|
|**Non-Triviality**|There is at least one initial state leading to all-0s AND one leading to all-1s|

### 8.3 The Three Steps of Consensus

A consensus algorithm typically works in three steps:

**Step 1: Elect**

- Processes elect a single process (a **leader**) to make decisions.
- The leader proposes the next valid output value.

**Step 2: Vote**

- Non-faulty processes listen to the leader's proposed value, **validate** it, and vote for it as the next valid value.

**Step 3: Decide**

- If a **threshold** number of identical votes is received (meeting some criteria), all processes decide on that value.
- Otherwise, the process starts over.

### 8.4 Challenges in Reaching Consensus

1. **Reliable Multicast:** All servers in the group need to receive the same updates in the **same order**. Guaranteeing this in the presence of failures is equivalent to solving consensus.
    
2. **Membership Failure Detection:** When a group member fails, all remaining members must consistently agree that it has failed and update their view of the group. Equivalent to consensus.
    
3. **Leader Election:** Electing a single leader that everyone agrees on is equivalent to consensus.
    
4. **Mutual Exclusion:** Ensuring only one process accesses a critical resource at a time requires all processes to agree on who holds the lock. Equivalent to consensus.
    

### 8.5 Consensus in Synchronous vs Asynchronous Systems

#### Synchronous Systems

- Every message is received within a **known time bound** $d$.
- Each process step takes between a known lower and upper bound.
- **Consensus IS possible** in synchronous systems — we can use time bounds to detect failures and elect leaders by taking turns proposing values with maximum time limits.
- Example: A cluster of processors connected by a shared communication bus (like a Cray supercomputer).
- **But:** Assuming synchrony is impractical for most real systems (like the Internet) where message latency is unpredictable.

#### Asynchronous Systems

- **No bound** on process execution speed or message delivery time.
- The Internet is a perfect example.
- **Consensus is IMPOSSIBLE to solve deterministically** in asynchronous systems where even one process can fail.
- This is the famous **Fischer-Lynch-Paterson (FLP) Impossibility Result** (1985): There is no deterministic consensus algorithm that always terminates in an asynchronous system that can have even one faulty process.
- **Practical workaround:** Use **probabilistic** consensus algorithms (like Raft, Paxos) that work correctly with overwhelming probability, even if they can't guarantee termination in every adversarial execution.

### 8.6 The Paxos Consensus Algorithm

**Paxos** is the most well-known family of distributed consensus algorithms. It has been used by Google (for Chubby, Spanner) and Amazon (for DynamoDB).

#### Roles in Paxos:

1. **Proposers (Leaders/Coordinators):** Propose values to achieve agreement on.
2. **Acceptors:** Listen to proposals and vote on whether to accept them.
3. **Learners:** Learn about the final decided value.

**Key facts:**

- Paxos nodes can take multiple roles simultaneously.
- Nodes must know the total number of acceptors to determine a majority.
- A single run of Paxos aims to reach consensus on **one** value.

#### Paxos Algorithm Variables:

- $n$ = proposal number (monotonically increasing, unique)
- $v$ = proposed value
- $n_v$ = number of the highest-numbered proposal accepted earlier by an acceptor
- $v'$ = value of the highest-numbered proposal previously accepted

#### Paxos Algorithm Steps:

**Phase 1 (Prepare):**

1. The proposer sends `prepare(n, v)` to all acceptors (or a majority).
2. Each acceptor compares $n$ to the highest proposal number it has seen. If $n > n_v$, it responds with `ack(n_v, v')` — reporting its previously accepted proposal number and value. It also promises not to accept any proposal with a lower number.

**Phase 2 (Accept):** 3. The proposer waits for `ack` from a **majority** of acceptors. It then sends `accept(n, max(v, v'))` — where the value is either the proposer's own value or the highest-numbered previously accepted value (whichever is higher-numbered). 4. When an acceptor receives `accept(n, v)`, it accepts $v$ **unless** it has received a `prepare(n')` with $n' > n$ in the meantime. If a majority of acceptors accept a value, that value becomes the **decision**.

#### Paxos Example (from slides):

- Proposer A sends `prepare(n=2, v=8)` and Proposer B sends `prepare(n=4, v=5)`.
- Acceptors X and Y see A's proposal first; Acceptor Z sees B's proposal first.
- Later, when Z receives A's proposal (n=2), it ignores it because n=2 < n=4 (it already saw a higher-numbered proposal).
- When X and Y receive B's proposal (n=4), they respond with the highest proposal they've seen.
- Proposer B receives a majority of acks. It sees the value 8 from A's accepted proposal (which had n=2). Since it must use the value from the highest-numbered previously accepted proposal, B sends `accept(n=4, v=8)`.
- All acceptors accept value 8. Learners are notified.
- **Final decision: value 8.**

### 📝 Q&A: Consensus

**Q1: Why is consensus important for distributed systems beyond just "agreeing on a value"?**

> Consensus is a foundational primitive. Solving consensus means you can also solve leader election, reliable multicast, mutual exclusion, and failure detection — all of which are critical for building reliable distributed services. If you can build consensus, you can build correct distributed systems.

**Q2: State and explain the FLP Impossibility Result.**

> The Fischer-Lynch-Paterson (FLP) Impossibility Result (1985) states: In an asynchronous distributed system where even a single process can fail, there is no deterministic algorithm that always terminates and achieves consensus. This means it's impossible to build a perfectly correct and always-terminating consensus algorithm for asynchronous systems — you must either relax the safety guarantee or the liveness guarantee.

**Q3: Why must the proposer in Paxos use the value from the highest-numbered previously accepted proposal?**

> This is the key safety property of Paxos. If an acceptor has already accepted a value in a previous round, it means a quorum may have already "decided" on that value (or is close to deciding). If a new proposer could override this with a completely different value, two different majorities might decide on two different values — violating the Agreement property. By adopting the highest-numbered previously accepted value, Paxos ensures that once a value is "in the pipeline" toward being decided, it remains the candidate value.

**Q4: What is the difference between the Agreement and Termination properties of consensus?**

> Agreement is a safety property: all non-faulty nodes that make a decision must decide on the same value — no two nodes can decide differently. Termination (Liveness) is a progress property: every non-faulty node must eventually make a decision — no node can be stuck undecided forever. FLP Impossibility shows that in asynchronous systems with failures, you can't guarantee both simultaneously.

**Q5: What does "non-triviality" mean in consensus, and why is it required?**

> Non-triviality ensures that the consensus algorithm doesn't have a trivially correct but useless solution — like "always output 0 no matter what." It requires that for any value in the decision domain (0 or 1), there must exist some execution of the algorithm where that value is decided. This ensures the algorithm's output actually depends on the nodes' input values.

## 9. Leader Election — Ring Algorithm

### 9.1 Why Do We Need Leader Election?

Many distributed algorithms require a **coordinator** or **leader**:

- A Paxos proposer needs to be a single leader.
- MapReduce needs a single JobTracker master.
- GFS needs a single namespace master.

But leaders can fail. When the current leader fails, the system either stops or needs to **automatically elect a new leader**. This is the **leader election problem**.

**Goal of a leader election algorithm:**

1. Elect exactly **one** leader among all non-faulty processes.
2. All non-faulty processes **agree** on who the leader is.

**Formal Safety and Liveness conditions:**

- **Safety:** For all non-faulty processes $p$: their `elected` variable = the non-faulty process $q$ with the best attribute value (e.g., highest ID), or Null.
- **Liveness:** Every election terminates, and every non-faulty process's `elected` value is not Null after the election.

The "best attribute" that decides who becomes leader can be: fastest CPU, most disk space, highest process ID, highest priority number, etc.

### 9.2 The Ring Election Algorithm

**Setup:**

- $N$ processes/nodes are arranged in a **logical ring**.
- Each process communicates only with its **logical neighbor** (clockwise).
- All messages travel **clockwise** around the ring.

**Algorithm Steps:**

1. Any process $p_i$ that discovers the old coordinator has failed **initiates an election**. It creates an "election" message containing its own `id:attr` (where `attr` is the election attribute, e.g., process ID).
    
2. When any process $p_i$ receives an election message, it **compares** the received `attr` with its own:
    
    - **If received attr > own attr:** Forward the message unchanged (the candidate is better than you).
    - **If received attr < own attr AND you haven't sent your own election message yet:** Overwrite the message with your own `id:attr` and forward it (you're a better candidate).
    - **If received `id:attr` matches your own:** Your `attr` is the **greatest** seen so far — you are the new coordinator! Send an "elected" message with your ID clockwise.
3. When a process receives an "elected" message:
    
    - Set your `elected` variable to the ID in the message.
    - Forward the message, unless **you** are the new coordinator (in which case the ring circulation is complete).

**Message Complexity:** In the worst case (the counter-clockwise neighbor of the initiator has the highest attribute):

- $N-1$ election messages to reach the new coordinator-to-be.
- $N$ election messages for the coordinator-to-be to confirm it's the winner.
- $N$ "elected" messages to circulate the result.

$$\text{Total Message Complexity} = 3(N-1)$$

### 9.3 Modified Ring Election (Handles Failures During Election)

The basic ring algorithm doesn't handle failures **during** the election itself (a node might fail while the election message is circulating). The modified version:

1. Instead of **overwriting**, each process **appends** its `id:attr` to the message as it passes.
2. Once the message returns to the **initiator** (who now has a list of all responding nodes), it elects the process with the best attribute.
3. It sends a "coordinator" message with the winner's ID. Each process appends its ID to this message too.
4. When the "coordinator" message returns to the initiator:
    - If the coordinator's ID is in the list → election is complete.
    - Else → the would-be coordinator failed during the election → restart the algorithm.
5. Bypasses failed processes (skips them in the ring).

### 📝 Q&A: Ring Election Algorithm

**Q1: In the Ring Algorithm, why does a process become the new coordinator when it receives a message with its own ID?**

> Because the election message containing a process's ID has traveled all the way around the ring and returned. For this to happen, the message's attribute value must have been greater than every process it passed through (otherwise it would have been overwritten). So if your own ID comes back to you unchanged, it means no other process in the ring has a higher attribute value than you — making you the rightful leader.

**Q2: What is the worst-case scenario for the Ring Algorithm in terms of message count?**

> The worst case occurs when the process immediately counter-clockwise from the initiator has the highest attribute. In this case, the election message travels the entire ring (N-1 hops) before reaching the winner. Then the winner needs the message to circle once more to confirm it's the highest (N hops), and then sends N coordinator messages. Total = 3(N-1) messages.

**Q3: How does the Modified Ring Algorithm handle the case where a node fails during an election?**

> Instead of overwriting the candidate with the locally best node, every node appends its ID to the message. When the message returns to the initiator, it has a list of everyone who participated. If the elected coordinator's ID is in that list, they're still alive and the election succeeds. If not, the algorithm restarts — effectively detecting and recovering from failures during the election itself.

**Q4: Practice Problem: Processes 1-7 are in a ring. Process 7 fails. Process 3 initiates the election. Process 6 is announced as leader, then also fails. Describe the sequence of events.**

> First Election: Process 3 initiates with ID 3. Message passes 4 (4>3, overwrite), 5 (5>4, overwrite), 6 (6>5, overwrite). Since 7 is failed, 6's message is forwarded by whoever comes after 7 (process 1). The message 6 reaches 2 (2<6, forward), 3 (3<6, forward). But wait — 3 receives a message with attr 6 ≠ 3, so it forwards. Eventually 6 receives its own message back and sends "elected" message. All nodes set elected=6.

> Second Election: Process 3 (or another) notices 6 doesn't respond. Initiates a new election. Now processes available are 1, 2, 3, 4, 5. The highest is 5, so Process 5 becomes the new coordinator after a new election round.

## 10. Leader Election — Bully Algorithm

### 10.1 Why a Different Algorithm?

The Ring Algorithm has issues:

- In an asynchronous system, a **slow process might not be elected** even if it's the best candidate (it appears failed due to timeout).
- **Ring reorganization** is needed when nodes fail — the ring structure must be maintained.

The **Bully Algorithm** works in a system where **every process can send messages to every other process** (not just ring neighbors), and every process knows every other process's ID.

### 10.2 The Bully Algorithm

**Why "Bully"?** Because the process with the next highest ID after the failed leader simply **bullies** lower-ID processes into submission — it declares itself leader without waiting for them to agree.

#### Algorithm Logic:

**Case 1: A process knows it has the highest ID.**

- It immediately declares itself coordinator and sends a `Coordinator` message to all processes with lower IDs. Election is complete.

**Case 2: A process initiates an election and doesn't know if it's the highest.**

1. Send an `Election` message to all processes with **higher IDs only**.
2. **If no response within timeout:** No higher process is alive → declare yourself leader, send `Coordinator` to all lower-ID processes. Done.
3. **If a response is received:** A higher-ID process is alive → wait for a `Coordinator` message from that process. If none received within another timeout, start a new election.

**Case 3: A process receives an Election message.**

- Reply with a `Disagree`/`OK` message (signaling "I'm taking over from here").
- Start your own leader election protocol (if not already doing so).

#### Timeout Values:

The one-way message transmission time $T$ is assumed known.

- **First timeout** (waiting for response to your Election message): Must be approximately $2T$ + processing time ≈ $2T$.
    - Too small → a lower-ID process incorrectly declares itself leader while a higher-ID process is still alive.
    - Too large → unnecessary delay in election completion.

### 10.3 Bully Algorithm — Step-by-Step Example

Starting with processes 1–5, where **Process 5 is the leader**:

1. Process 5 fails.
2. **Process 2** notices Process 5 doesn't respond → initiates election → sends `Election` to processes 3 and 4.
3. Processes 3 and 4 respond to Process 2 ("I'll take over").
4. **Process 3** sends `Election` to Process 4 and Process 5.
5. Only Process 4 responds to Process 3.
6. **Process 4** sends `Election` to Process 5 only.
7. Process 5 doesn't respond → **Process 4 declares itself winner** → sends `Coordinator` to all.

### 10.4 Analysis of Bully Algorithm

**Worst Case:** The process with the **lowest ID** detects the failure.

- $N-1$ processes all begin elections simultaneously.
- The $i$-th highest ID process sends $i-1$ election messages.
- **Message overhead: O(N²)**
- **Turnaround time:** ≈ 5 message transmission times if no failures during election:
    1. Election from lowest-ID process
    2. Answer to lowest-ID from 2nd-highest process
    3. Election from 2nd-highest process
    4. Timeout for 2nd-highest process
    5. Coordinator message from 2nd-highest process

**Best Case:** The process with the **second-highest ID** detects the failure.

- Just $N-2$ Coordinator messages.
- **Turnaround time:** 1 message transmission time.

### 10.5 Practice Problem Solution

**Problem:** Out of 7 processes, P7 fails and P4 detects this first. How many election and response messages?

- P4 sends `Election` to P5 and P6 → **2 Election messages from P4**
- P5 and P6 respond to P4 → **2 Response messages**
- P5 sends `Election` to P6 → **1 Election message from P5**
- P6 responds to P5 → **1 Response message**
- P6 sends `Election` to P7 (no response) → **1 Election message from P6**
- P6 declares itself winner

**Total: 4 Election messages, 3 Response messages**

(The slide solution states 6 election messages and 3 response messages — accounting for P4 also sending to P6.)

### 📝 Q&A: Bully Algorithm

**Q1: Why is the Bully Algorithm called "bully"?**

> Because when a higher-priority (higher-ID) process hears of an election, it immediately overrides the lower-priority process's candidacy — it "bullies" the lower-ID process out of contention by saying "I'll take over." The process with the second-highest ID after the failed leader simply dominates and forces its election without requiring complex negotiation.

**Q2: What is the first timeout value in the Bully Algorithm, and why must it be set carefully?**

> The first timeout is the time a process waits for a response to its Election message. It should be approximately 2T (where T is one-way transmission time) plus processing time. If set too small, a lower-ID process might declare itself leader before it has actually heard from all higher-ID processes that are alive, resulting in an incorrect election. If set too large, elections take unnecessarily long.

**Q3: How does the Bully Algorithm handle a situation where a new failure occurs during an ongoing election?**

> If Process 4 fails after sending an Election message but before sending a Coordinator message, the process waiting for P4's Coordinator message will time out on its second timeout and start a new election round. This new election will naturally elect the process with the next highest available ID.

**Q4: Compare the Ring Algorithm and the Bully Algorithm in terms of message complexity.**

> The Ring Algorithm has a message complexity of O(N) — specifically 3(N-1) in the worst case — because messages only travel in one direction around a ring. The Bully Algorithm has a worst-case message complexity of O(N²) because all N-1 processes may start elections simultaneously, each sending messages to all higher-ID processes. However, the Bully Algorithm handles arbitrary network topologies (any-to-any), while the Ring Algorithm requires a specific ring structure.

## 11. Resource Management and Scheduling

### 11.1 What Is Scheduling?

**Scheduling** is the process of deciding how to allocate limited system resources — CPU cycles, memory, storage, I/O, network bandwidth — among competing users and tasks.

Scheduling involves **policies** (the principles guiding decisions) and **mechanisms** (the means to implement those policies).

### 11.2 Why Is Cloud Resource Management Complex?

- **Multi-objective optimization:** Optimize simultaneously for performance, cost, availability, energy — these often conflict.
- **No accurate global state:** In a distributed system, you can never know the exact state of all nodes in real-time.
- **Unpredictability:** Hardware failures, software bugs, and security attacks make the system non-deterministic.
- **Fluctuating loads:** Cloud providers must handle massive spikes (Black Friday, New Year) without over-provisioning for average loads.
- **Different strategies for different models:** IaaS, PaaS, and SaaS have fundamentally different resource management needs.

### 11.3 Cloud Resource Management (CRM) Policies

1. **Admission Control:** Prevent accepting workloads that violate high-level policies (e.g., don't accept a VM request if it would exceed SLA-guaranteed capacity).
2. **Capacity Allocation:** Allocate resources based on need — fixed or variable sizes (elasticity).
3. **Optimization and Load Balancing:** Distribute workload evenly across servers.
4. **Energy Optimization:** Minimize power consumption through intelligent component choice and operating mode selection.
5. **Quality of Service (QoS) Guarantees:** Meet conditions specified in Service Level Agreements (SLAs) — average response time, uptime percentage, etc.
6. **Locale-Based:** Consider geographic location for legal compliance, latency, and cost.
7. **Cost-Based:** Based on perceived value, auction mechanisms, or negotiated rates.

### 11.4 Mechanisms for Resource Management

1. **Control Theory:** Uses feedback (utilization metrics, latency measurements) to control resource allocation dynamically — ensuring stability and predictable transient behavior.
2. **Utility-Based:** Models user-level performance vs. cost. Allocations are made to maximize "utility" — rewards for meeting SLAs minus penalties for violations.
3. **Machine Learning:** Uses historical data to predict future resource requirements and pre-provision accordingly.
4. **Market-Oriented / Economic:** Uses auction mechanisms (e.g., combinatorial auctions) where users bid for resources.

### 11.5 Cloud Scheduling Algorithms

A scheduler runs at multiple levels:

- A physical server shared among several VMs.
- A VM supporting several applications.
- An application running multiple threads.

A good scheduler is: **efficient**, **fair**, and **starvation-free** (no task waits indefinitely).

#### Scheduling by System Type:

- **Batch systems:** Maximize throughput, minimize turnaround time.
- **Real-time systems:** Meet deadlines, be predictable.
- **Best-effort:** Use round-robin, FCFS, SJF, priority algorithms.

#### Deadline-Based Scheduling:

- **Hard deadlines:** Strict — missing a deadline causes penalties or system failure (e.g., number of cores, memory, VM type must be allocated within X seconds).
- **Soft deadlines:** Guidelines — can be missed slightly without severe consequences (e.g., data processing tasks that are "preferred" to complete by a certain time but have flexibility).

### 11.6 Workload Partition Rules

When distributing a workload across worker nodes:

- **Optimal Partitioning Rule (OPR):** Partitions workload so all workers **finish at the same time** — achieving the earliest possible completion. The head node sequentially distributes portions to workers.
- **Equal Partitioning Rule (EPR):** Assigns an **equal portion** of the workload to every worker regardless of their speed. Simpler but may not be optimal if workers have different capabilities.

### 11.7 Nova Scheduler (OpenStack Example)

When a user requests a VM in OpenStack, the **nova-scheduler** service decides where to launch it:

1. The request arrives at the Nova API.
2. Nova-scheduler queries available hosts using **filters** (e.g., sufficient RAM, CPU, disk).
3. After filtering, it uses **weighing** to rank the remaining hosts.
4. The VM is launched on the highest-ranked host.

A "host" in this context means a physical node running the nova-compute service.

### 📝 Q&A: Resource Management and Scheduling

**Q1: What is the difference between scheduling policy and scheduling mechanism?**

> A policy is the rule or principle governing a decision — for example, "always prioritize requests from customers with Gold-tier SLAs." A mechanism is the technical means to implement that policy — for example, a priority queue data structure and preemption logic in the scheduler. Policies answer "what to do"; mechanisms answer "how to do it."

**Q2: Why is OPR (Optimal Partitioning Rule) better than EPR in heterogeneous environments?**

> OPR ensures all workers finish at the same time, which minimizes the total job completion time (no worker is idle while another is still working). EPR gives equal work to all workers regardless of speed — faster workers finish early and sit idle while slower workers are still working. In a heterogeneous environment (workers with different speeds), OPR is significantly more efficient.

**Q3: What is the "energy optimization" consideration in cloud resource management?**

> Cloud data centers are massive power consumers. Energy optimization involves choosing hardware components with better power efficiency, running servers at lower clock speeds (which reduces both frequency-related performance and voltage-related energy consumption), consolidating workloads onto fewer machines (and powering down idle machines), and scheduling tasks on servers with lower cooling overhead. The key trade-off is between energy savings and performance/availability.

## 12. Distributed Locking

### 12.1 The Need for Distributed Locking

In a multi-threaded program on a single machine, **mutex locks** ensure only one thread accesses a critical resource at a time. In a distributed system, the equivalent is a **distributed lock** — but now different clients run on different machines across a network.

**Why locking in distributed systems?**

1. **Efficiency:** Prevent duplicate work — avoid expensive computations (like triggering a timer) being done twice simultaneously.
2. **Correctness:** Prevent data corruption from concurrent conflicting writes — critical for database consistency.
3. **Mutual Exclusion:** Ensure only one node performs a critical operation at any time.

### 12.2 Features of Distributed Locks

- **Mutual Exclusion:** Only one client globally can hold a lock at any moment.
- **Deadlock-Free:** Use **lease-based locking** — if a client acquires a lock and crashes (or takes too long), the lock automatically expires after a timeout. This prevents a crashed client from holding a lock forever.
- **Consistency:** After failovers (triggered by hardware or software failures), the lock's state must remain consistent — the same client must still hold the lock even if the server handling the lock failed and a replica took over.

### 12.3 Types of Distributed Locks

#### Optimistic Locking

"Instead of blocking when something potentially dangerous happens, we continue anyway, in the hope that everything will be ok."

How it works:

1. Read the data.
2. Make changes in memory.
3. **Before committing**, check if the data was updated by someone else while you were working.
4. If yes → abort and retry. If no → commit.

Best for: Low-contention scenarios where conflicts are rare.

#### Pessimistic Locking

"Block access to the resource before operating on it, and release the lock at the end."

How it works:

1. Acquire an **exclusive lock** on the data.
2. Read, modify, write.
3. Release the lock.

Best for: High-contention scenarios where conflicts are common.

### 12.4 The Problem with Distributed Locks: GC Pause Issue

A subtle but critical problem illustrated in the slides:

1. Client 1 acquires a lock with a lease timeout of, say, 30 seconds.
2. Client 1 is **paused by its garbage collector** (GC) for 40 seconds — longer than the lease.
3. The lock **expires** because Client 1 hasn't renewed it.
4. Client 2 acquires the lock (now legitimately).
5. Client 1's GC pause ends. Client 1 doesn't realize its lease has expired.
6. Client 1 proceeds to write to the shared resource — now **two clients think they own the lock simultaneously** → data corruption.

### 12.5 Fencing Tokens — The Solution

**Fencing** solves the GC-pause problem:

- Every time a lock is granted, the lock server issues a **fencing token** — a monotonically increasing number.
- Every write request to the storage service must **include the current fencing token**.
- The storage service **rejects any write with a token number lower than one it has already seen**.

**Example:**

1. Client 1 gets lock with token 33. GC pause begins.
2. Lease expires. Client 2 gets lock with token 34. Client 2 writes with token 34.
3. Client 1's GC pause ends. Client 1 tries to write with token 33.
4. Storage sees that it already processed a write with token 34 > 33. **Rejects Client 1's write.**

Fencing ensures that even if a client believes it holds the lock, stale writes cannot cause data corruption.

### 12.6 Distributed Lock Manager (DLM)

A **Distributed Lock Manager** runs on every node with an identical copy of a cluster-wide lock database. It:

- Provides software applications a means to synchronize access to shared resources.
- Treats every resource as an entity (file, record, memory area) whose shared access must be controlled.

Popular DLM implementations:

- **Google's Chubby:** A lock service for loosely coupled distributed systems (used by GFS, Bigtable).
- **Apache ZooKeeper:** Open-source distributed coordination service (discussed next).
- **Redis (Redlock Algorithm):** Redis can be used for distributed lock management.

### 📝 Q&A: Distributed Locking

**Q1: Why is distributed locking harder than mutex locking in a single-machine multi-threaded environment?**

> In a single machine, all threads share the same memory. A mutex lock is just a memory flag — thread A sets it, thread B sees it instantly. In a distributed system, there is no shared memory. Different clients run on different machines. A "lock" must be stored in a shared external service, and every lock acquisition/release requires network communication, which can fail, be delayed, or be intercepted. Additionally, individual network and node failures can occur independently — a client holding a lock might crash, leaving the lock permanently held without network-level mechanisms to detect and release it.

**Q2: Explain with an example why lease-based locking is essential in distributed systems.**

> Consider Client A acquires a lock and then crashes (power failure) before releasing it. Without a lease timeout, the lock would be held forever — no other client could ever acquire it. With a lease timeout, the lock server automatically expires Client A's lock after, say, 30 seconds. Client B can now acquire the lock. This is called "deadlock freedom through leases."

**Q3: What is the fencing token mechanism and why does it prevent stale write corruption?**

> A fencing token is a monotonically increasing number issued by the lock server with each lock grant. Every write to the shared resource must include the client's fencing token. The storage system rejects any write with a token number lower than or equal to the highest token it has already processed. This ensures that a client with an expired lock (and therefore an old, low token) cannot overwrite data written by a client with a newer lock (and higher token).

**Q4: What is the difference between Optimistic and Pessimistic locking, and when would you choose each?**

> Optimistic locking allows concurrent reads and only checks for conflicts at commit time — if a conflict is detected, the transaction is aborted and retried. It is best for low-contention scenarios where conflicts are rare, as it avoids lock overhead. Pessimistic locking prevents conflicts by exclusively locking the resource before any reads/writes — guaranteed no conflicts but reduces concurrency. It is best for high-contention scenarios where conflicts are frequent and retrying is expensive.

## 13. Apache ZooKeeper

### 13.1 What Is ZooKeeper and Why Is It Needed?

In a large Hadoop cluster spanning 500+ servers, you need centralized management for:

- Name and group management
- Synchronization services
- Configuration management
- Cross-cluster services

Building these from scratch for every distributed application is hard, error-prone, and repetitive. **Apache ZooKeeper** provides these as a ready-to-use service.

**ZooKeeper is:** An open-source, high-performance coordination service for distributed applications. It automates management of distributed host clusters, freeing developers to focus on business logic.

**ZooKeeper provides:**

- **Naming service:** Map names to information (which services are alive, where are they?).
- **Configuration management:** Central store for distributed system configuration.
- **Locks & synchronization:** Distributed mutex locks.
- **Group services:** Track membership and status of groups of nodes.
- **Leader election:** Off-the-shelf support for automatic failover.

**Design goals:**

- **Simple, Robust, Good Performance**
- **High Availability, High Throughput, Low Latency**
- **Wait-Free:** A slow or failed client will not interfere with fast clients.
- **Tuned for Read-dominant workloads** (reads vastly outnumber writes in most coordination scenarios).

### 13.2 ZooKeeper Architecture

#### Components:

- **Server:** Stores data and responds to clients. A leader is elected at startup. All servers store the full data tree in memory.
- **Client:** A node in the distributed application that accesses ZooKeeper. Sends periodic heartbeat messages to the server to signal it's alive. If the connected server dies, the client automatically redirects to another server.
- **Leader:** One designated server. Handles all write requests. Performs automatic recovery for failed nodes.
- **Follower:** A server that follows the leader's instructions. Handles client read requests locally.
- **Ensemble/Cluster:** The group of ZooKeeper servers. Typically 3, 5, or 7 servers (odd number to ensure a majority quorum for decisions).

#### Request Handling:

- **Reads:** Handled **locally** by the ZooKeeper server the client is connected to (fast, no coordination needed).
- **Writes:** Forwarded to the **leader** → leader goes through **majority consensus (Zab protocol)** among followers → response sent back.
- **Watches:** Between client and single server. A watch on a znode means "notify me when this changes."

### 13.3 ZooKeeper Data Model — znodes

ZooKeeper uses a **hierarchical namespace** (like a file system, but not a file system). Each node in this hierarchy is called a **znode**.

- Every znode is identified by an absolute, canonical path (e.g., `/cluster/node1/status`).
- A znode cannot be deleted if it has children.
- Every znode stores data (limited to a few kilobytes — not for big datasets, only coordination metadata like configuration, status, location info).
- Each znode has a **stat structure** with version numbers (for data, ACL, and timestamps).

#### Types of Znodes:

|Type|Persistence|Children?|Use Case|
|---|---|---|---|
|**Persistent**|Exists until explicitly deleted|Yes|Long-lived configuration, service registries|
|**Ephemeral**|Exists only while the creating session is active; auto-deleted on session end|No|Detecting client failures, leadership tokens|
|**Sequential**|Appends a monotonically increasing counter to the path|Both types|Distributed locks, leader election ordering|

**Key insight on Ephemeral znodes:** Since they're deleted when a client's session ends (i.e., the client disconnects or crashes), they are used to detect client termination. You can set a **watch** on an ephemeral znode — when the znode disappears (client died), all watchers are notified.

### 13.4 ZooKeeper Implementation

- **Request Processor:** Handles incoming client requests.
- **Atomic Broadcast (Zab Protocol):** Ensures writes are applied in the same order on all servers (total-order broadcast). This is ZooKeeper's consensus mechanism.
- **Replicated Database:** All znodes are stored in memory, replicated across all servers. Only transaction logs and snapshots persist to disk.

### 13.5 ZooKeeper Operations API

```
create(path, data, flags)     → Create a znode
delete(path, version)          → Delete a znode
exists(path, watch)            → Check if a znode exists (optionally set a watch)
getData(path, watch)           → Read data from a znode
setData(path, data, version)   → Write data to a znode
getChildren(path, watch)       → List children of a znode
sync(path)                     → Wait for data propagation to catch up
```

**Version control:** `setData` and `delete` require the current version number. If the version you supply doesn't match the server's version, the operation fails. This is **optimistic concurrency control** for znodes.

**Transaction IDs:** Every update gets a unique **zxid** (ZooKeeper Transaction ID). All updates are totally ordered — a higher zxid means a later update. This ensures a consistent global ordering of all events.

### 13.6 ZooKeeper Recipes (Practical Use Cases)

#### Naming Service

ZooKeeper can tell you which servers are running and provide their status by name. Extended to **group membership service** — look up all members of a group and their statuses.

#### Distributed Locks

With $N$ clients trying to acquire a lock:

1. Each client creates an **ephemeral, sequential** znode under `/Cluster/_locknode`.
2. The client with the **smallest ID** in the children list acquires the lock.
3. Other clients set a **watch** on the znode just before theirs (not on all children — this avoids "thundering herd").
4. When the lock holder releases (deletes its znode or disconnects), the next-smallest-ID client is notified and acquires the lock.

#### Leader Election

Similar to distributed locks — the client with the smallest sequential znode ID becomes the leader.

#### Configuration Management

New nodes joining a cluster automatically read the centralized configuration from ZooKeeper.

#### Message Queue

A znode designated as the queue root. Producers create child znodes (queue items). Consumers read and delete child znodes.

### 13.7 Advantages and Disadvantages

**Advantages:**

- Simple distributed coordination with a familiar interface.
- Synchronization via mutual exclusion.
- Ordered messages.
- Serialization (atomic transactions).
- Reliability via ensemble replication.
- Atomicity — transactions either fully succeed or fail completely.

**Disadvantages:**

- Lacks features like service discovery with health checks (Consul has this built-in).
- Ephemeral znode lifecycle tied to TCP connection — a process could be alive at the TCP level but functionally crashed (OOM-killed), keeping a "ghost" lock.
- Complex to operate correctly.

**Used by:** Yahoo!, Zynga, Netflix, LinkedIn, Rackspace, Apache Kafka, Apache HBase, Apache Storm, MapReduce (YARN), Neo4j.

### 📝 Q&A: Apache ZooKeeper

**Q1: Why does ZooKeeper store the entire data tree in main memory rather than on disk?**

> ZooKeeper is designed for low-latency, high-throughput coordination — disk I/O would be too slow. Since coordination data (configuration, status, location info) is small (znodes are limited to kilobytes), it fits comfortably in RAM. Only transaction logs and periodic snapshots are written to disk for durability — so the system can recover after a crash by replaying the log from the last snapshot.

**Q2: Why should ZooKeeper be deployed with an odd number of servers (3, 5, 7)?**

> ZooKeeper uses a majority quorum to agree on writes (via the Zab protocol). With an odd number N servers, a majority is (N+1)/2. With 3 servers, you can tolerate 1 failure; with 5 servers, you can tolerate 2 failures. An even number is wasteful — with 4 servers, you can still only tolerate 1 failure (same as 3 servers, but with more overhead). Odd numbers maximize fault tolerance relative to cluster size.

**Q3: How does ZooKeeper use ephemeral znodes and watches to detect node failures?**

> When a distributed service instance starts, it creates an ephemeral znode representing its presence. Other nodes set watches on this znode (or on the parent node listing all instances). When the service instance crashes or disconnects, its ZooKeeper session expires and the ephemeral znode is automatically deleted. ZooKeeper notifies all watchers of this deletion — effectively broadcasting "this service instance has died" to all interested parties without any polling.

**Q4: Why is using watches on the "next smallest" znode (not all children) important for distributed locks?**

> If every client waiting for a lock watches ALL children of the lock node, then when the current lock holder releases the lock, ALL waiting clients are simultaneously notified. They all try to re-evaluate who's next, generating a burst of ZooKeeper requests — the "thundering herd problem." By watching only the znode immediately before yours, only the next-in-line client is notified, eliminating the stampede.

**Q5: How does ZooKeeper ensure consistency between leaders and followers?**

> ZooKeeper uses the Zab (ZooKeeper Atomic Broadcast) protocol — a consensus algorithm similar to Paxos. When a write request arrives at any follower, it is forwarded to the leader. The leader broadcasts the write to all followers using Zab. Only when a majority of servers acknowledge the write does the leader commit it and respond to the client. This ensures all servers see writes in the same total order (identified by zxid).

## 14. Reverse Proxies & Forward Proxies (Nginx)

### 14.1 What Is a Reverse Proxy?

A **reverse proxy** is a server that sits **in front of web servers** and forwards client requests to those web servers. Crucially:

- The client doesn't know which backend server it's actually talking to.
- The resources appear to originate from the reverse proxy itself.
- Reverse proxies are owned/managed by the web **service** (not the client).

**How it works:**

1. Client sends a request to the reverse proxy (e.g., `www.example.com`).
2. The reverse proxy validates the request. If invalid, returns an error.
3. The reverse proxy checks its cache. If the requested resource is cached, return it immediately.
4. If not cached, forward the request to the appropriate backend content server.
5. Receive the response from the backend, cache it for future requests, and return it to the client.

### 14.2 What Is a Forward Proxy?

A **forward proxy** (commonly called just "a proxy") sits in front of **client machines** and forwards their requests to the internet on their behalf. The web server sees the proxy's IP, not the client's.

Reasons to use a forward proxy:

1. **Circumvent restrictions:** Get around government, school, or corporate firewalls.
2. **Content filtering:** Block certain content categories for a group of users (e.g., school network blocks social media).
3. **Anonymity:** Hide the client's real IP address for privacy or to avoid government surveillance.

### 14.3 Reverse Proxy vs. Forward Proxy — Key Difference

|Aspect|Reverse Proxy|Forward Proxy|
|---|---|---|
|**Sits in front of**|Backend web servers|Client machines|
|**Protects**|Servers from direct client access|Clients' identities from servers|
|**Owned by**|The web service provider|The client organization or user|
|**Client knows about it?**|No (transparent to client)|Yes (client is configured to use it)|

**Forward proxies are good for:** Content filtering, email security, NAT, compliance reporting.

**Reverse proxies are good for:** Load balancing, SSL offload/acceleration, caching, compression, content switching, application firewall, authentication, single sign-on.

### 14.4 Benefits of Reverse Proxies

#### 1. Security

- **Zero-trust architecture:** Only forwards approved requests; backend server IP addresses are hidden from the internet.
- **DDoS protection:** Can reject/blacklist traffic from specific IPs or rate-limit connections.
- **Web Application Firewall:** Inspect incoming requests for attack patterns.

#### 2. Scalability and Flexibility

- **Load balancing:** Distribute traffic across multiple backend servers, preventing any one from being overwhelmed.
- **No single point of failure:** If one backend server fails, the reverse proxy routes traffic to healthy ones.
- **Backend changes transparent:** The public-facing IP is the proxy's; backend servers can be added/removed/changed without clients knowing.

#### 3. Web Acceleration (Performance)

- **Caching:** Store commonly requested static content (images, videos) so backends don't need to serve them repeatedly.
- **Compression:** Compress responses before sending to clients.
- **SSL offloading:** Decrypt HTTPS at the proxy (saving backend servers from crypto overhead).

#### 4. Identity Branding

- Hide the fact that a payment processor or CMS is hosted externally — to users, everything appears to come from your domain.

### 14.5 Nginx — A Powerful Reverse Proxy

**Nginx** (pronounced "engine-x") is an open-source web server and reverse proxy known for extreme performance. It was originally designed for maximum performance and stability when handling massive concurrent connections.

#### Nginx Architecture: Master-Slave, Event-Driven, Asynchronous

- Nginx uses an **event-driven, non-blocking, asynchronous** connection model.
- A **single worker thread** can handle **thousands of connections** without spawning a new thread per connection.
- Nginx can handle **10,000+ simultaneous connections** with low CPU and memory usage.

**Components:**

- **Master Process:** Allocates jobs to workers based on incoming client requests. After delegating, immediately moves on to the next request (non-blocking — doesn't wait for worker responses).
- **Worker Processes:** Handle the actual request processing. Each worker handles 1,000+ requests simultaneously in a single thread (event-driven). Single-threaded workers share memory space efficiently.
- **Cache:** First request for a page → compute it and store in cache. Subsequent requests → serve from cache, bypassing backend servers entirely.

**Benefits of Nginx:**

- **Load balancing:** Evenly distributes client requests across multiple upstream servers.
- **Security:** Hides backend server identities; single access point regardless of topology.
- **Caching:** Serves static content directly, reducing backend server load.
- **Centralized logging:** Single place to audit all request/response logs.
- **TLS/SSL support:** Secure HTTPS with encrypted client-server communication.
- **Protocol support:** HTTP, HTTPS, HTTP/1.1, HTTP/2, gRPC, TCP, UDP with IPv4 and IPv6.

### 📝 Q&A: Reverse Proxies & Nginx

**Q1: A user at a university wants to access a blocked social media site. Should they use a forward proxy or a reverse proxy?**

> A forward proxy. The user would configure their browser to route requests through a forward proxy server (outside the university's network). The university's firewall sees traffic going to the proxy (allowed), not to the social media site (blocked). The proxy fetches the social media content and returns it to the user.

**Q2: How does a reverse proxy prevent DDoS attacks?**

> A reverse proxy can rate-limit incoming connections, blacklist IP addresses that send excessive requests, filter malformed or suspicious requests before they reach backend servers, and absorb large volumes of traffic using caching. Since only the proxy's IP is exposed publicly, the actual backend servers are hidden and cannot be directly targeted.

**Q3: What is SSL offloading and why is it beneficial?**

> SSL offloading means the reverse proxy handles all TLS/SSL encryption and decryption, communicating with backend servers over unencrypted connections (within the secure internal network). This removes the computational overhead of crypto operations from backend servers, allowing them to focus on application logic. It also simplifies certificate management — only one SSL certificate needs to be maintained (on the proxy).

**Q4: How does Nginx achieve high concurrency with a single-threaded event loop?**

> Traditional web servers (like Apache in prefork mode) create a new thread or process per connection — threads are expensive in memory and context-switching. Nginx uses an event-driven model: the worker registers many connections with the OS (via epoll/kqueue), then the OS notifies the worker when any connection has data ready. The worker processes that data in a tight event loop without blocking. This means one thread can "juggle" thousands of connections simultaneously, waiting only where I/O would block.

## 15. Scalability, Hybrid Cloud, and Cloud Bursting

### 15.1 Cloud Scalability

**Cloud scalability** is the ability to increase or decrease IT resources (compute, storage, networking) as needed to meet changing demand, quickly and easily, with little or no disruption.

Cloud scalability is much faster than traditional on-premises scaling — which could take weeks or months to procure and install physical hardware.

**Scalability vs. Elasticity:**

- **Scalability:** Long-term, planned growth. Adding capacity in a pre-planned manner to support growing user base over months. A scalable system handles increased workload with existing or incrementally added hardware.
- **Elasticity:** Short-term, real-time, automated response to sudden demand spikes. An elastic system automatically adds/removes resources in real-time to match current demand closely.

**Benefits of Cloud Scalability:**

- **Cost savings:** Pay only for what you use; no upfront hardware investment; hardware doesn't become obsolete since you don't own it.
- **Disaster recovery:** Scale backup/recovery infrastructure only when needed, without maintaining a separate full-scale DR data center.
- **Convenience:** Add VMs in minutes via dashboard.
- **Flexibility and speed:** Respond to unexpected demand spikes rapidly.

### 15.2 Scaling Strategies

#### Vertical Scaling (Scale Up)

Add **more CPU, RAM, or storage** to an **existing server**, or replace it with a more powerful one.

- AWS: Change EC2 instance type from `t3.medium` to `t3.xlarge`.
- Limited by the maximum specs of a single machine.
- Generally requires downtime to resize.

#### Horizontal Scaling (Scale Out)

Add **more servers** and distribute workload across them.

- AWS: Add more EC2 instances; put them behind a load balancer.
- Theoretically unlimited — add as many instances as needed.
- Requires the application to be designed for distribution (stateless services, distributed databases).
- More complex but more resilient — no single server failure brings everything down.

### 15.3 Scalability Through Reverse Proxy / Load Balancer

A reverse proxy (like Nginx) acts as a load balancer:

- To the outside world, there is only one server address.
- The load balancer distributes incoming requests across multiple backend servers.
- If one backend fails, the load balancer routes around it.
- New backend servers can be added to the pool without changing anything on the client side.

### 15.4 Hybrid Cloud

**Hybrid cloud** combines private and public clouds, enabling an organization to expand its local (private) infrastructure by adding public cloud resources on demand.

**Benefits:**

- **Dynamic workloads:** Use the public cloud for unpredictable spikes; keep stable workloads on private cloud.
- **Workload separation:** Sensitive data (financial records, customer PII) on private cloud; non-sensitive workloads on public cloud.
- **Big data processing:** Run occasional large analytics jobs on scalable public cloud; keep raw data behind private firewall.
- **Incremental migration:** Move workloads to cloud gradually, at your own pace.
- **Cost efficiency:** Best utilization of existing private infrastructure while using public cloud elasticity.

### 15.5 Cloud Bursting

**Cloud bursting** is the hybrid cloud technique of **automatically overflowing** workloads from a private cloud to a public cloud when demand exceeds private cloud capacity.

- Normally, the organization uses its own private infrastructure.
- When private resources hit 100% utilization → overflow traffic is automatically directed to the public cloud.
- The user pays for public cloud resources **only when they're actually used** (pay-per-use).

**When are cloud bursts triggered?**

1. **Software development/CI-CD:** DevOps teams need multiple VMs for testing, only temporarily. CI/CD tasks need one-off compute bursts per commit.
2. **Marketing campaigns:** Product launch traffic surges (Hollywood movie release, game launch) — temporary huge spikes that subside after the news cycle.
3. **Big data modeling:** One-time queries or model training jobs that exceed private cloud capacity. Examples: high-fidelity 3D rendering, AI/ML model training, autonomous vehicle simulation.
4. **Seasonal businesses:** Holiday shopping surges (Black Friday), financial quarter-end processing, election season campaign traffic.

### 15.6 OpenNebula for Cloud Bursting

**OpenNebula** is an open-source cloud middleware that manages heterogeneous distributed data center infrastructures. It:

- Manages both private (OpenNebula) and public (AWS EC2) VM instances.
- When combined with Nginx as a reverse proxy: OpenNebula controls server allocation across both clouds, while Nginx distributes load across web servers in both clouds simultaneously.
- Supports cloud bursting natively by treating EC2 as an extension of the local cloud.

### 📝 Q&A: Scalability and Cloud Bursting

**Q1: What is the difference between scalability and elasticity in cloud computing?**

> Scalability refers to the system's ability to handle more work by adding resources — it's about planned, longer-term growth. Elasticity refers to the system's ability to automatically and dynamically adjust resources in real-time based on instantaneous demand — growing and shrinking rapidly. A scalable system might handle 10x its current load if you add more servers (but that takes planning). An elastic system automatically handles a sudden 10x spike and scales back down when the spike ends.

**Q2: When should you use vertical scaling vs. horizontal scaling?**

> Use vertical scaling (scale up) when: your application isn't designed for distributed operation, you need quick capacity increase without architectural changes, or the workload doesn't have unpredictable spikes. Use horizontal scaling (scale out) when: you need high availability (no single server is a SPOF), you need to handle very large or unpredictable spikes, your application is stateless or can be made stateless, and you want to distribute load geographically.

**Q3: What is the key economic advantage of cloud bursting over simply buying more private hardware?**

> Cloud bursting means you only pay for public cloud resources when demand actually exceeds your private capacity. If you buy additional hardware for peak loads (e.g., Black Friday), that hardware sits idle for 95% of the year — you've paid for capacity you rarely use. With cloud bursting, you pay for extra capacity only during the actual peaks, and the cost is proportional to actual usage.

## 16. Multitenancy and Multitenant Databases

### 16.1 What Is Multitenancy?

**Multitenancy** means multiple customers (tenants) of a cloud vendor **share the same computing resources**. A single instance of a software application serves multiple customers.

**Tenant:** A group of users who share common access with specific privileges to a software instance. (In SaaS, your company is a "tenant" of Salesforce.)

**Why multitenancy?**

- **Better resource utilization:** One machine dedicated to one tenant wastes compute power. Sharing maximizes utilization.
- **Lower costs:** The cloud vendor spreads infrastructure costs across many tenants, making services cheaper per customer.

### 16.2 Multi-Tenancy Architecture Types

#### Type 1: Single Application Instance, Single Multi-Tenant Database

- One app instance, one database, all tenants share it.
- **Highest scalability** — adding tenants is easy.
- **Lowest cost** — maximum resource sharing.
- **High operational complexity** at design time — must carefully partition tenant data within the database.
- Data is separated by tenant ID columns, row-level security, etc.

#### Type 2: Single Application Instance, One Database Per Tenant

- One app instance, but each tenant has its own database.
- **Lower scalability** than Type 1 — each database adds overhead.
- **Higher cost** — maintaining N databases for N tenants.
- **Lower operational complexity** — each tenant's database is independent; easier to manage and isolate.
- Adding database nodes can compensate for lower per-tenant scalability.

#### Type 3: Separate Application Instance + Database Per Tenant

- Complete isolation — each tenant gets their own app instance and database.
- **Highest data isolation and security.**
- **Highest cost** — completely separate stacks per tenant.
- Equivalent to what traditional on-premises enterprise software does.
- Not truly "multi-tenant" — more of a "single-tenant replicated" model.

### 16.3 Multi-Tenancy Requirements

As sharing granularity increases (from Type 3 → Type 1), scalability increases, but so do:

- **Security requirements:** Each row of a shared table needs access control.
- **Fine-grained resource sharing** mechanisms.
- **Customer isolation guarantees** — one tenant's data must not be visible to another.
- **Customization support** — each tenant may need a slightly different data schema.

### 16.4 Four Levels of Multi-Tenancy

From lowest to highest maturity:

1. **Ad Hoc/Custom Instances:** Each customer has a different version of the software. (This is what most on-premises enterprise software currently looks like.) Extremely difficult to manage.
2. **Configurable Instances:** One software version, but with configuration options for customization (logo, workflows). Major management savings — only one codebase.
3. **Configurable, Multi-Tenant Efficient Instances:** One software version AND one running instance shared among all customers. Additional efficiency from single process.
4. **Scalable, Configurable, Multi-Tenant Efficient Instances:** All of Level 3, but hosted on a cluster so it scales limitlessly. The gold standard for SaaS.

### 16.5 Authentication in Multi-Tenant Systems

**Key challenge:** Securely sharing resources among multiple tenants requires strong authentication.

Two approaches:

- **Centralized authentication:** A central user database handles all authentication. The cloud admin gives each tenant's admin rights to manage their users. All logins verified against the central database.
- **Decentralized authentication:** Each tenant maintains its own user database. A **federation service** bridges the tenant's authentication framework to the cloud's authentication service. Enables **Single Sign-On (SSO)** — login once to access multiple services.

### 16.6 Implementing Resource Sharing

**Access control** is the key technology for secure multi-tenant resource sharing. Two approaches:

- **Roles:** A set of permissions bundled together. A "storage admin" role includes permission to define storage devices. Who can assign roles is itself a permission (Role-Based Access Control — RBAC).
- **Business Rules:** Fine-grained policies beyond what roles provide. Example: "A manager can approve transfers only up to $10,000, and only during business hours." Implemented using policy engines like Drools.

**Access Control Models:**

- **ACL (Access Control Lists):** Each resource has a list of permissions per role. ("FriendlyTable → FriendlyGarage: SELECT, INSERT, UPDATE")
- **Capability-Based:** A user's ability to access a resource is determined by whether they hold a reference (capability/key) to it. Like having a physical key to a room — holding the key grants access.

### 16.7 Data Sharing in Multi-Tenant Databases

**Two methods for sharing a single database among tenants:**

#### 1. Dedicated Tables Per Tenant

- Each tenant has its own set of tables.
- Access rights set via SQL GRANT statements.
- **High isolation** — tenants can't access each other's tables at the DB level.
- **More storage space** (empty tables waste space for smaller tenants).

#### 2. Shared Tables

- All tenants' data in the same table, with a `tenant_id` column differentiating rows.
- **Space-efficient** — no wasted table space.
- **Requires multiple `SELECT` statements** to retrieve tenant-specific data.
- An **auxiliary metadata table** tracks tenant-specific column metadata.
- **Security enforced at application level** — the app must ensure `WHERE tenant_id = current_tenant` in every query.

### 16.8 Customization in Shared Tables

When tenants need custom columns (different tenants need different data fields), three approaches:

1. **Pre-allocated columns:** Reserve extra columns (ext_col1, ext_col2...) in the shared table. Tenants use these columns for their custom fields. **Wastes space** for tenants who don't use them; limits flexibility.
2. **Name-Value Pairs:** An extra column pointing to a separate name-value pair table. Space-efficient, but requires table joins to reconstruct complete records.
3. **XML Column:** One column storing an arbitrary XML document. Completely flexible — any structure can be stored. Requires parsing the XML to access specific fields.

### 📝 Q&A: Multitenancy

**Q1: What is the key security challenge of shared tables in a multi-tenant database?**

> In shared tables, all tenants' data sits in the same table rows. Unlike dedicated tables (where OS or DB-level access controls can enforce isolation), shared tables rely entirely on the application to add `WHERE tenant_id = X` to every query. If the application has a bug or a SQL injection vulnerability, one tenant's query could retrieve another tenant's data — a serious security breach.

**Q2: Why do PostGreSQL measurements show the "shared process" approach is far more scalable than the "shared machine" approach?**

> In the "shared machine" approach, each customer gets their own database process — when you have 10,000 customers, you have 10,000 database processes, each consuming OS resources (file descriptors, memory for its process state, etc.). In the "shared process" approach, one database process handles all customers — memory consumption is far lower (80 MB vs. 55 MB × 10,000 = 550 GB). The disk usage grows (4,488 MB for 10,000 customers vs. 4 MB for one), but memory is the critical resource.

**Q3: What is federated identity in multi-tenant systems and why is it important for enterprises?**

> Federated identity allows a user authenticated by one organization's system (e.g., their company's Active Directory) to seamlessly access another organization's services without re-authenticating. Using standards like SAML or OAuth/OIDC, the cloud service trusts the enterprise's identity provider. This is essential for enterprises because: employees already have corporate credentials; maintaining separate cloud credentials for thousands of employees is impractical; and SSO improves security by reducing password proliferation.

## 17. Failure Detection — Checkpointing & Application Recovery

### 17.1 Failure Detection in Cloud Platforms

Cloud providers like AWS Elastic Beanstalk detect when an application instance fails and route new requests away from it. Two techniques support this:

1. **Failure Detection:** Detect failed instances and stop routing requests to them.
2. **Application Recovery:** Restart failed instances.

### 17.2 Failure Detection Mechanisms (Detailed)

#### Push Model (Heartbeats)

- Process $q$ sends "I am alive" messages to monitor process $p$ every $t$ seconds.
- Monitor $p$ expects a heartbeat every $t$ seconds.
- If after timeout $T$ no heartbeat arrives, $p$ **suspects** $q$ has failed.

#### Pull Model (Probing)

- Monitor $p$ sends "are you alive?" messages to process $q$ every $t$ seconds.
- Waits for "yes, I am alive" response from $q$.
- If no response after timeout $T$, $p$ suspects $q$ has failed.
- **Less efficient** than push — more messages generated.

#### Dual Scheme (Hybrid)

- Initially: all monitored processes use **push model** (send "I am alive").
- After a period: $p$ assumes any process that hasn't sent a liveness message is using **pull model** and begins probing it.
- Combines efficiency of push with robustness of pull.

**Speed-Accuracy Trade-off:**

- Low threshold → faster detection, more false positives.
- High threshold → fewer false positives, slower detection.
- Since false positives have severe consequences (unnecessary failover, lost in-flight requests), systems set **high thresholds**.

### 17.3 Redirection After Failure

After identifying failed instances, requests must be rerouted. A common mechanism for HTTP-based protocols: **HTTP Redirection**

- The web server returns a `3xx` code (e.g., `302 Found`) with a new URL.
- Example: User requests `http://www.pustakportal.com/`. The load balancer returns `302` with URL `http://pps5.pustakportal.com` — a server known to be up. The browser follows the redirect.

### 17.4 Application Recovery — Checkpointing

**Checkpointing** is the technique of **periodically saving the complete state of a running application** to stable storage. If the application fails, it can be restored from the most recent checkpoint rather than starting from scratch.

Benefits: Reduces recovery time significantly for long-running jobs.

**In a distributed environment:**

- All processes in the distributed application must be checkpointed **simultaneously** (or at a logically consistent point).
- All instances are **restarted from a common checkpoint** if any one fails.

**Challenges:**

- **Scalability:** Checkpointing all processes at once is expensive for large distributed jobs.
- **Consistency:** If a message is in-transit between processes when the checkpoint is taken, the checkpoint is inconsistent (one side shows the message was sent; the other shows it wasn't received). Recovery from such a checkpoint requires careful handling of in-transit messages.

---

### 📝 Q&A: Failure Detection & Recovery

**Q1: What is the difference between the push model and pull model for failure detection, and which is more efficient?**

> In the push model, the monitored process actively sends heartbeats to the monitor. In the pull model, the monitor actively queries the monitored process. The push model is more efficient for many-to-one monitoring because the monitored processes initiate communication — the monitor doesn't need to send $N$ probe messages for $N$ processes every $t$ seconds. With push, the monitor simply waits and flags processes that go silent.

**Q2: Why is distributed checkpointing challenging compared to single-process checkpointing?**

> In a single process, checkpointing captures the process's complete memory state at one moment — simple and consistent. In a distributed system, multiple processes run simultaneously and communicate via messages. A globally consistent checkpoint requires either stopping all processes simultaneously (impossible in practice for large systems) or using a sophisticated "consistent snapshot" algorithm (like Chandy-Lamport) that accounts for messages in-flight at the checkpoint time.

## 18. Cloud Security Requirements

### 18.1 What Is Cloud Security?

**Cloud security** is a set of control-based safeguards and technology protections designed to protect resources stored online from leakage, theft, or data loss. Protection covers cloud infrastructure, applications, and data from threats.

**Why is cloud security more demanding?**

- Cloud users leave applications entirely in the hands of third-party cloud providers.
- Strong resistance from users over lack of privacy protection, security assurance, and copyright protection.
- Multi-tenant environments mean a breach in one customer's data could expose others.

**Basic objectives:**

- **Confidentiality:** Data accessible only to authorized parties.
- **Integrity:** Data remains unaltered by unauthorized parties.
- **Availability:** Cloud services are accessible when needed.

**Additional objectives:** Cost-effectiveness, Reliability, Performance.

### 18.2 Physical Security

The cloud infrastructure must be secure against **physical threats**:

- Unauthorized physical entry by intruders.
- Natural hazards: floods, earthquakes, fire.
- Human error: accidentally powering down systems, switching off air conditioning.

**Multi-layered physical security includes:**

1. Central monitoring and control center with dedicated 24/7 staff.
2. Monitoring for each possible physical threat (intrusion sensors, flood sensors, smoke detectors).
3. Staff training in responding to threat situations.
4. Manual or automated backup systems (emergency power, flood pumps).
5. Secure access to the facility with layered entry controls (badge, biometric, mantrap).

### 18.3 Virtual Security Best Practices

#### Cloud Time Service

- Synchronize all systems to the same clock via **Network Time Protocol (NTP)**.
- Critical for: correct distributed system operation, correlating logs across geographically distributed systems.
- To protect against false time sources: NTP protocol messages can be encrypted.

#### Identity Management (IAM)

- Foundation for achieving CIA (Confidentiality, Integrity, Availability).
- Must scale to cloud's massive user count.
- In heterogeneous systems: **federated identity management** for single sign-on across different system types.
- Must satisfy legal/policy requirements (e.g., ability to delete all user data within X days per GDPR).
- Maintain historical records for investigation.

#### Access Management

- Allow cloud resources **only to authorized users**.
- Best practices:
    - Multi-factor authentication (MFA) for sensitive operations.
    - No shared accounts (like `admin`).
    - Whitelist IP addresses for remote administrative access.
    - Do not allow unrestricted access even for cloud management personnel.

#### Break-Glass Procedures

- Bypass normal security controls in emergency situations.
- Analogy: Breaking the glass to set off a fire alarm.
- Must be protected — can only be executed under controlled conditions and must trigger an alarm/audit event.

#### Key Management

- Encryption is critical for data isolation in shared-storage environments.
- The cloud must provide: secure key generation, assignment, revocation, archiving.
- Procedures for recovering from compromised keys.

#### Auditing

- Capture all security-related events with: timestamp, system, user ID.
- Centrally maintained, secure audit log.
- Must be sanitizable for sharing with cloud customers (remove sensitive internal details).

#### Security Monitoring

- Infrastructure for generating alerts on critical security events.
- **Cloud-wide intrusion detection systems (IDS)** — both network-level and host-level.
- Allow cloud users to implement their own intrusion/anomaly detection.

#### Security Testing

- Test all software in an isolated test bed before deployment.
- Test patches before production release.
- Ongoing security testing (penetration testing, vulnerability scanning).
- Third-party testing for high-risk areas.
- Remediation process for identified vulnerabilities.

### 18.4 Security Challenges Specific to Virtualization

In cloud environments with VMs:

- **Hypervisor malware:** Attacks targeting the hypervisor layer.
- **Guest hopping:** A compromised VM trying to escape its isolation to access the host or other VMs.
- **VM rootkits:** Malicious software embedded at the VM level.
- **Man-in-the-middle during VM migration:** Live migration of VMs can be intercepted.

**Defenses:** Program shepherding, VMware's vSafe/vShield, hypervisor security compliance, Intel vPro technology.

---

### 📝 Q&A: Cloud Security Requirements

**Q1: Why is "security only as strong as its weakest link" especially important in cloud environments?**

> A cloud data center might have the most sophisticated cybersecurity — firewalls, intrusion detection, encryption — but if physical security is weak (e.g., an attacker can walk in and plug a USB drive into a server), all virtual security measures are bypassed. Similarly, the most secure application in the world is compromised if an employee's laptop with admin credentials is stolen. Every layer of the security stack must be strong, because attackers will find and exploit the weakest point.

**Q2: What is the purpose of break-glass procedures and why must they be strictly controlled?**

> Break-glass procedures allow authorized personnel to bypass normal security controls in genuine emergencies (e.g., a system administrator needing to access a locked-down critical system during an outage when normal authentication is unavailable). They must be strictly controlled because: they circumvent security, so misuse could cause catastrophic breaches. Controls include: requiring multiple authorizations, triggering automated alerts, recording the event in a tamper-proof audit log, and immediately notifying security personnel.

## 19. Risk Management & Security Design Patterns

### 19.1 Why Use a Risk Approach?

Cloud outages are real and financially devastating:

- **March 2009:** Microsoft Azure down for 22 hours.
- **April 2011:** Amazon EC2 major outage.
- **August 2013:** AWS US-East-1 down for 59 minutes — Amazon.com itself rejected customers.

Organizations must evaluate **whether the cloud is appropriate** for their specific data and workloads — especially for mission-critical systems like accounting information systems.

### 19.2 Risk Management Process

**Risk management** = evaluating risks, deciding how to control them, and monitoring their effectiveness. Steps:

1. **Information Resource Categorization:** Assess each resource for:
    
    - **Criticality:** What's the business impact of a security failure on this resource?
    - **Sensitivity:** How confidential is this data?
2. **Select Security Controls:** Choose controls appropriate to the resource's criticality and sensitivity. A control appropriate for admin networks (IP whitelisting) may be too restrictive for user-facing services.
    
3. **Risk Assessment:** Determine if the chosen controls provide adequate protection against anticipated threats. Augment if needed.
    
4. **Implement Security Controls:** Deploy the chosen controls — administrative (policies, procedures), technical (firewalls, encryption), or physical (locks, cameras).
    
5. **Operational Monitoring:** Continuously monitor the effectiveness of implemented controls.
    
6. **Periodic Review:** Security controls need regular review because:
    
    - **New threats emerge** (new attack techniques, new malware).
    - **Operational changes** (new software, new infrastructure) may require updated security design.

### 19.3 Risk Management Concepts

**NIST Security Control Categories:**

- **Technical controls:** Firewalls, encryption, access control software.
- **Operational controls:** Configuration management, incident response procedures.
- **Management controls:** Risk assessments, security policies.

**FIPS 200 System Impact Levels:**

- **Low-impact:** Security breach causes limited degradation but primary functions remain.
- **Moderate-impact:** Primary functions degraded significantly.
- **High-impact:** System incapable of performing some primary functions.

Higher impact → more stringent security controls required.

### 19.4 Security Design Patterns

#### 1. Defense in Depth

Defenses are **layered** — an attacker must overcome multiple independent barriers:

**Example for cloud admin access:**

- Layer 1: VPN required for remote access.
- Layer 2: Only whitelisted IP addresses can connect through the VPN.
- Layer 3: One-time password (OTP/MFA) required even after VPN + IP check.

Even if an attacker compromises one layer, the remaining layers provide protection.

#### 2. Honeypots

A **honeypot** is a decoy system designed to appear attractive to attackers:

- Mimics a valuable target (database server, admin console).
- When an attacker attacks it, security personnel can **observe their techniques** and **trace their identity**.
- Can be deployed by cloud providers (to catch attacks against the infrastructure) or by customers (to catch attacks against their specific services).

In the cloud: Deploy a honeypot VM that monitors access attempts to itself — any access is suspicious since it's a decoy.

#### 3. Sandboxes

Execute software in a **restricted environment** within the OS:

- Software in the sandbox has limited access to OS resources.
- Even if the software is compromised, the attacker cannot access the full OS.
- Limits blast radius of an exploit.
- Also provides Defense in Depth (one more layer to overcome).
- Examples: Browser sandboxes for web content, container isolation.

#### 4. VM Isolation (Network Pattern)

VMs sharing the same physical hardware must isolate their traffic using:

- **Encryption of traffic between VMs** (since their traffic doesn't go through external switches).
- **Tightened security controls** — restrict which ports accept traffic.

#### 5. Subnet Isolation

- Physically separate networks for: admin traffic, customer-facing services, and storage systems.
- Physically separate (not just VLAN-separated) because VLANs can be misconfigured.
- Use firewalls to control routing between subnets.

#### 6. Common Management Database (CMDB)

- A central database containing an inventory of all IT components and their current configuration/status.
- Simplifies security management — all administrators see the same consistent view.
- Essential for incident response — quickly identify what changed and when.

---

### 📝 Q&A: Risk Management & Security Patterns

**Q1: Why is Defense in Depth more effective than a single strong security measure?**

> Defense in Depth recognizes that no single security measure is perfect. Any individual control can be bypassed — an attacker might have a zero-day exploit for your firewall, or might trick an employee into giving away credentials. With multiple independent layers of defense, compromising one layer doesn't immediately compromise everything — the attacker must defeat each additional layer, significantly increasing the cost and difficulty of a successful attack.

**Q2: What is the risk of false security from VLANs vs. physically separate networks?**

> VLANs (Virtual Local Area Networks) segment traffic logically using the same physical hardware. However, VLAN misconfiguration can allow traffic to "leak" between VLANs (VLAN hopping attacks). A determined attacker on one VLAN may exploit misconfigurations to access another VLAN. Physically separate networks provide true isolation at the hardware level — traffic literally cannot flow between them without going through a firewall/router, making cross-network attacks much harder.

**Q3: Describe a scenario where a Honeypot would provide valuable security intelligence.**

> An organization deploys a honeypot VM with a fake database containing fake "customer credit card data." No legitimate traffic should ever reach this VM. When an attacker (perhaps coming from an APT or insider threat) discovers this VM and begins probing it, the honeypot logs every action — what commands were run, what data was accessed, where the attacker's traffic originated. This intelligence helps the security team understand the attacker's techniques (TTPs — Tactics, Techniques, Procedures), identify compromised internal accounts, and trace the attack to its source.

## 20. Authentication in Cloud — OpenStack Keystone

### 20.1 What Is Keystone?

**Keystone** is the OpenStack service that provides:

- **API client authentication** — verifying identity of users and services.
- **Service discovery** — telling clients where to find other OpenStack services.
- **Distributed multi-tenant authorization** — managing permissions across projects and domains.

Keystone implements OpenStack's Identity API and is the **central authentication hub** for all OpenStack services.

### 20.2 Key Keystone Concepts

#### Projects

A **Project** is an abstraction used to group and isolate resources (servers, storage, networks). Projects are the fundamental unit of resource organization in OpenStack. Users are not directly owned by projects — instead, they are given **role assignments** that grant access.

#### Domains

A **Domain** isolates visibility of a set of Projects, Users, and Groups for a specific organization. A domain can represent:

- A logical division within an enterprise (different departments).
- Completely separate organizations (for multi-tenant cloud providers).

#### Users and User Groups (Actors)

**Users** are individuals using the cloud. **Groups** are collections of users. Together, Users and Groups are called **Actors** — the entities to which roles are "assigned."

**Relationship summary:** Domains contain Users, Groups, and Projects. Groups contain Users. Users can belong to many Groups. Roles are globally unique.

#### Roles

**Roles** convey authorization. A role is "assigned to" an Actor (User or Group) and "assigned on" a Target (Project or Domain). Example: User "Bob" has role "admin" on project "development."

#### Assignments

A role **assignment** is the triple: **(Actor, Target, Role)**. Assignments can be inherited — a role assigned at the Domain level cascades to all Projects within that Domain.

#### Targets

**Projects** and **Domains** are collectively called **Targets** — the entities on which roles are assigned.

#### Tokens

When a user successfully authenticates with Keystone, they receive a **token**:

- The token's **ID** is unique per cloud.
- The token's **payload** contains: when issued, when expires, which user, which project (scope), which roles, and a service catalog.
- All subsequent OpenStack API calls include this token in the HTTP header.

Sample token payload (simplified):

```json
{
  "token": {
    "issued_at": "2014-06-10T20:55:16",
    "expires_at": "2014-06-10T22:55:16",
    "roles": [{"id": "c703057b", "name": "admin"}],
    ...
  }
}
```

#### Catalog

The **service catalog** tells users and applications the URLs/endpoints for all cloud services:

- Admin URL, Internal URL, Public URL for each service.
- Without the catalog, no client would know where to send requests to create VMs, store objects, etc.

### 20.3 Identity Backends

Where does Keystone get user information from?

#### SQL Backend

- Keystone stores users in a local MySQL/PostgreSQL/DB2 database.
- **Pros:** Easy to set up; manageable via OpenStack APIs.
- **Cons:** Keystone shouldn't be an identity provider itself; weak password support; another username/password to remember; enterprises already have LDAP.

#### LDAP Backend

- Keystone reads user information from an enterprise LDAP server (like Active Directory).
- Keystone uses LDAP for **read operations** (user lookup, authentication via bind). Minimal privilege required.
- **Pros:** Leverages existing enterprise directory; Keystone doesn't need to maintain user accounts.
- **Cons:** Service accounts (for OpenStack internal services) can't go in LDAP; Keystone still sees user passwords.

#### Multiple Backends Per Domain

- One identity source per Keystone **domain**.
- Domain A → LDAP for department A.
- Domain B → LDAP for department B.
- Service accounts → SQL backend.
- **Pros:** Multiple LDAPs (e.g., after company mergers); can leverage existing infrastructure without impacting it.
- **Cons:** Authentication must be domain-scoped; slightly more complex.

#### Identity Providers (Federated Identity)

- Keystone consumes federated authentication (SAML 2.0, OpenID Connect) from external Identity Providers (IdPs) like Okta, Active Directory, Google, Facebook.
- Federated users are **ephemeral** — not stored in Keystone.
- **Pros:** Keystone never sees passwords; opens door for SSO and hybrid cloud; leverages existing infrastructure.
- **Cons:** Most complex setup.

### 20.4 Authentication Methods

#### Password Authentication

- Most common. User provides username, password, and optionally a scope (project/domain).
- Keystone verifies the password against the identity backend and issues a token if successful.
- The scope determines which service catalog is returned.

#### Token Authentication

- User provides an existing (valid) token to get a new token.
- Used to: refresh an expiring token, change scope from unscoped to scoped.

### 20.5 Access Management — RBAC via policy.json

Keystone enforces Role-Based Access Control (RBAC) on all API endpoints using a `policy.json` file. Format: Target → Rule.

```json
{
  "admin_required": "role:admin or is_admin:1",
  "owner": "user_id:%(user_id)s",
  "admin_or_owner": "rule:admin_required or rule:owner",
  "identity:list_projects": "rule:admin_required",
  "identity:delete_project": "rule:admin_required",
  "identity:list_user_projects": "rule:admin_or_owner"
}
```

- **Targets** (left side): The API endpoint or resource (e.g., `identity:delete_project`).
- **Rules** (right side): Who can access it (e.g., `rule:admin_required`).

---

### 📝 Q&A: Keystone Authentication

**Q1: Why is "Keystone shouldn't be an Identity Provider" a concern with the SQL backend?**

> In enterprise environments, identity management (user accounts, password policies, group management) is typically handled by dedicated IAM systems like Active Directory or LDAP. These systems enforce password complexity requirements, account lockout policies, password expiration, and audit trails. Maintaining a separate user database in Keystone duplicates this work, creates an "identity silo" (another password for users to remember), and doesn't benefit from existing enterprise security policies.

**Q2: What is the purpose of token scoping in Keystone?**

> A scoped token is associated with a specific project — it grants the user their roles on that project. An unscoped token only proves identity (who you are) without specifying which project's resources you want to access. Without a scoped token, the service catalog returned is empty or unusable. Scoping is like specifying context — "I'm Alice, and I want to work on the 'production' project" gives you the right permissions and service endpoints for that project.

**Q3: How does federated identity enable hybrid cloud scenarios?**

> With federated identity, a user authenticated by their organization's Identity Provider (IdP) can seamlessly access resources across both private OpenStack deployments and public clouds that trust the same IdP. For example, an employee authenticates with their company's Active Directory once, and receives a SAML assertion. This assertion is accepted by both their company's private OpenStack cloud AND by a public cloud partner's Keystone that has been configured to trust their company's IdP — achieving SSO across private and public clouds.

## 21. Cloud Threats — DoS, DDoS, and EDoS

### 21.1 Denial of Service (DoS) Attacks

A **Denial-of-Service (DoS) attack** is a malicious attempt to make a computer or service unavailable by overwhelming it with traffic or exploiting vulnerabilities.

**Characteristics:**

- Launched from a **single computer** (distinguishes from DDoS).
- Goal: Overwhelm the target's capacity so legitimate users can't be served.

**Two main categories:**

1. **Buffer Overflow Attacks:** Exploit a vulnerability that causes the target to consume all available memory, disk space, or CPU time — causing crashes or severe slowdowns.
    
2. **Flood Attacks:** Send such a massive volume of packets that the target server's capacity is completely saturated. The attacker needs **more bandwidth than the target** for flood attacks to work.
    

**Indicators of a DoS attack:**

- Unusually slow network performance (long load times).
- Inability to load a specific website.
- Sudden loss of connectivity across multiple devices on the same network.

### 21.2 Distributed Denial of Service (DDoS) Attacks

A **DDoS attack** is like a DoS attack, but the flood of traffic comes from **many distributed machines simultaneously** — making it far more powerful and harder to defend against.

**How DDoS works:**

1. Attacker infects many devices with **malware**, turning them into **bots** (zombies).
2. A collection of bots is called a **botnet**.
3. Attacker sends remote commands to the botnet.
4. All bots simultaneously flood the target with requests.
5. Because each bot is a **legitimate internet device**, it's hard to distinguish attack traffic from real user traffic.

**The highway analogy:** A DDoS attack is like an unexpected traffic jam clogging a highway, preventing regular traffic from reaching its destination.

**Protection challenges:** Since attack traffic comes from many legitimate-looking IP addresses (all the compromised devices), simply blocking one IP doesn't help. The attacker controls thousands.

### 21.3 Economic Denial of Service (EDoS) Attacks — The Cloud-Specific Threat

**EDoS** is an attack specifically designed for cloud computing. It exploits the cloud's **pay-per-use pricing model**:

**How EDoS works:**

- Attacker doesn't try to knock the service offline.
- Instead, the attacker **gradually sends illegitimate traffic** to the victim's cloud service.
- This triggers the cloud's **auto-scaling feature** — the victim's cloud resources scale up to handle the (fake) load.
- The victim is billed for all these additional resources.
- This causes **massive financial damage** to the victim — not service disruption but bill shock.

**Key differences between DDoS and EDoS:**

|Aspect|DDoS|EDoS|
|---|---|---|
|**Goal**|Cease all services|Maximize financial costs|
|**Method**|Maximum traffic in short time|Gradual, sustained traffic over long period|
|**Effect on service**|Service goes down|Service stays up but costs skyrocket|
|**Sophistication**|Relatively straightforward|More intelligent, targeted|
|**Who is affected**|All users of the service|Primarily the billing account|
|**Energy impact**|Server energy to handle attacks|May also inflate energy bills|

**Example:** An attacker slowly and steadily sends requests to a victim's auto-scaling web service 24/7 for a month. The service stays running (no one complains — it seems to be handling load fine), but the victim receives a monthly cloud bill that's 100× normal.

### 21.4 Cloud Defense Strategies for Security

**Basic cloud security enforcements:**

1. **Facility security:** Biometric readers, CCTV, motion detection, man traps in data centers (year-round, 24/7).
2. **Network security:** Fault-tolerant external firewalls, Intrusion Detection Systems (IDS), third-party vulnerability assessment.
3. **Platform security:** SSL/TLS encryption, strict password policies, system trust certification.

**Security architecture (from diagram in slides):**

- A secured gateway plus external firewalls.
- Public cloud access filtered by the gateway.
- Private cloud resources protected behind additional layers.

**Overall cloud security objectives:**

- Secure cloud infrastructure (physical and virtual).
- Delivering secure cloud services.
- Implementing current and future cloud standards and certifications.
- Automation of auditing and security.

### 📝 Q&A: DoS, DDoS, EDoS

**Q1: Why is a DDoS attack harder to defend against than a DoS attack?**

> A DoS attack comes from a single IP address — you can simply block that IP. A DDoS attack comes from thousands or millions of different IP addresses (all the compromised bots), each sending only moderate traffic. You can't block all of them without also blocking legitimate users from similar IP ranges. Additionally, each individual bot's traffic looks like normal user traffic — it's hard to distinguish the attack traffic from real requests.

**Q2: What makes EDoS particularly insidious in cloud environments compared to traditional environments?**

> In traditional (non-cloud) environments, an organization owns its physical hardware. An attack that generates extra load just burns CPU cycles — the organization doesn't get a bigger bill. In cloud environments, resources are billed per use. More traffic → more auto-scaled VMs → higher monthly bill. A prolonged, subtle EDoS attack can generate enormous financial losses before the victim even realizes they're being attacked — because the service appears to be functioning normally (just serving lots of "requests").

**Q3: How might you detect an EDoS attack before it bankrupts your cloud account?**

> Several approaches: (1) Set **budget alerts** in your cloud account to notify you when spending exceeds thresholds. (2) Implement **rate limiting** — cap the number of requests per IP per minute. (3) Analyze traffic patterns — look for unusual spikes or anomalous patterns (e.g., many requests from similar IP ranges, identical user agents, or requests with no real user behavior like clicking and navigating). (4) Use a **Web Application Firewall (WAF)** that can distinguish bots from humans. (5) Use **CAPTCHA** for expensive operations. (6) Monitor your **autoscaling events** — unusual scaling activity is a red flag.

## 22. Edge Computing and Fog Computing

### 22.1 What Is Edge Computing?

**Definition (Cao & Shi, 2018):**

> "Edge computing places storage and compute resources closer to where data is produced — on devices, gateways, cloudlets, or micro-Data Centers — so results return in milliseconds rather than travelling to a distant hyperscale cloud."

**The key idea:** Instead of sending all data to a distant cloud data center for processing, process data **at or near the source** (the "edge" of the network).

**Why is this needed now?**

- **Explosion of IoT devices** generating enormous data volumes.
- **Success of rich cloud services** creates demand for even more responsive experiences.
- Data generated at the edge needs **immediate processing** (autonomous vehicles can't wait 50 ms for a cloud decision).

### 22.2 Limitations of Cloud for IoT

|Limitation|Description|
|---|---|
|**Latency**|Round-trip to cloud ≥ 50ms — unacceptable for real-time apps (telesurgery, autonomous vehicles)|
|**Bandwidth**|Raw IoT data streams are enormous; uploading everything creates network congestion and high egress costs|
|**Privacy**|Sensitive data (medical, biometric) must leave the device and traverse multiple networks to reach the cloud|

**Data scale reality (Cisco Global Cloud Index):**

- IoT devices globally produce ~500 ZB (zettabytes) of data annually.
- Only ~10.4 ZB is expected to reach data centers.
- **45% of IoT-created data is processed at or near the edge** — not in the cloud.

### 22.3 Why We Need Edge Computing — Five Reasons

#### 1. Ultra-Low Latency

- Cloud round-trips typically ≥ 50 ms. Edge reduces this to single-digit milliseconds.
- **Last-mile proximity:** Data travels device → edge node → device (no WAN hops).
- **Hop reduction:** Fewer routers = less queuing delay.
- **Jitter control:** Local processing gives consistent response times.
- **Critical for:** Autonomous vehicles (must react in <10 ms), telesurgery (surgeon must feel real-time feedback), augmented reality.

#### 2. Bandwidth Optimization

- IoT and video devices generate massive data volumes.
- Edge nodes can **filter and aggregate** — send only anomalies or summaries to the cloud (not raw sensor streams).
- **Compress and deduplicate** before cloud transmission.
- **Prioritize traffic** — mission-critical flows aren't delayed by bulk IoT data.
- **Cost savings:** Cloud data egress is charged by the gigabyte — preprocessing at the edge dramatically reduces egress costs.

#### 3. Enhanced Privacy and Security

- Sensitive data (medical records, biometric scans) stays on local edge servers — less exposure to interception.
- Simplifies compliance with regulations like **GDPR** (data doesn't leave the EU) and **HIPAA** (health data sovereignty).
- **Key practices:**
    - Local encryption and anonymization before any cloud transfer.
    - Zero-trust edge security — authentication and threat detection at each node.
    - Isolated processing enclaves — confidential computing techniques protect code and data in use.

#### 4. Reliability and Resilience

- Edge nodes continue operating **autonomously** when connectivity to the cloud is lost.
- **Critical for:**
    - Industrial control systems (power grids, manufacturing lines must not stop if internet is down).
    - Remote monitoring (oil rigs, maritime vessels in connectivity dead zones).
    - Emergency response (disaster zones with intermittent networks).

#### 5. Cost Efficiency

- Reduce expensive cloud egress charges.
- Use heterogeneous edge hardware (CPUs, TPUs, FPGAs) optimized for specific workloads.
- Defer cloud scaling during peak bursts — edge handles local peaks.
- Lower bandwidth bills by only sending meaningful data to the cloud.

### 22.4 Key Terms

|Term|Definition|
|---|---|
|**Cloud**|Large-scale, centralized servers operated by providers (AWS, Azure) delivering resources over the internet|
|**Cloudlet**|Mobility-enhanced, small-scale cloud positioned at the network edge; intermediate tier between mobile devices and cloud|
|**Datacenter**|Dedicated facility housing large server racks, storage, networking with specialized cooling/power|
|**Micro Datacenter**|Low-scale, self-contained modular unit (single rack) with compute, storage, networking, power, and cooling|

### 22.5 Edge Computing Challenges

#### Resource Heterogeneity & Management

- Edge devices range from smartphones and microcontrollers to gateways and micro-servers — each with different ISA, OS, memory, energy budget.
- **Computing Stream** solution: Split applications into ordered function blocks, and dynamically assign each block to the most appropriate node (device, gateway, or cloud) based on current latency, energy, bandwidth, and cost constraints.

#### Security, Trust, and Privacy

- Edge data is sensitive: raw electricity/water consumption data can reveal home occupancy.
- ~49% of household Wi-Fi networks are unsecured.
- Traditional crypto stacks overwhelm tiny sensors.
- **Safeguards:** On-device data redaction (e.g., face blurring before transmission), hardware roots-of-trust, explicit user consent before data leaves premises.

#### Programming Models

- Developers must target heterogeneous, intermittently-connected nodes.
- Sensors emit diverse, noisy data formats.
- Gateway should normalize sensor data into a unified schema (id, time, name, data).
- Tools needed: cross-compilation for multiple ISAs, edge simulators, safe migration/rollback of stateful functions.

### 📝 Q&A: Edge Computing

**Q1: Why is autonomous vehicle control an ideal use case for edge computing rather than cloud computing?**

> An autonomous vehicle must make real-time safety decisions (braking, steering) in milliseconds. Even a 50 ms round-trip to a cloud data center is far too slow — a car traveling at 100 km/h moves ~1.4 meters in 50 ms. The vehicle needs local edge processing that can detect obstacles and make braking decisions in <10 ms. Cloud processing is fine for non-real-time tasks (map updates, route optimization, fleet analytics) but edge processing is essential for safety-critical real-time control.

**Q2: How does edge computing help with GDPR compliance for European IoT deployments?**

> GDPR restricts transferring personal data of EU citizens outside the EU. If a European factory's production line sensors are processed in a US cloud data center, every data transfer technically requires GDPR compliance measures. With edge computing, raw sensor data is processed locally (within the EU, on-premises), and only anonymized aggregates or alerts are sent to the cloud. This keeps personal or sensitive data within the EU, simplifying GDPR compliance significantly.

**Q3: What is the "Computing Stream" model and how does it address edge heterogeneity?**

> The Computing Stream model breaks an application into an ordered sequence of computational function blocks, each with associated latency, energy, and bandwidth requirements (SLAs). A scheduler dynamically maps each block to the most appropriate computing tier — device, gateway, or cloud — based on current resource availability and SLA constraints. If a gateway is overloaded, a block might be migrated to the cloud. If cloud connectivity is lost, a block might run on the device. This adaptive placement enables efficient use of heterogeneous edge infrastructure.

## 23. Container Security

### 23.1 Introduction to Container Security

**Containerization** (Docker, Kubernetes) packages applications with their dependencies for lightweight, portable deployment. Unlike virtual machines, containers **share the host OS kernel** — this is why they're lightweight, but it's also why they present unique security challenges.

**Container security** focuses on protecting containerized environments from:

- Unauthorized access
- Container escapes (breaking out of the container to access the host)
- Malware injection through images
- Resource misuse (one container starving others)

Since containers are **ephemeral** (short-lived and dynamically created/destroyed), traditional security approaches are often insufficient.

### 23.2 The Container Security Paradigm

The architectural difference from VMs:

|VM|Container|
|---|---|
|Each VM has its own **Guest OS**|Containers share the **Host OS kernel**|
|Strong isolation (hypervisor boundary)|Weaker isolation (OS-level namespaces)|
|Higher overhead (full OS per VM)|Lower overhead (just process isolation)|
|Kernel vulnerability affects only that VM|Kernel vulnerability affects ALL containers|

**Key risk:** A single kernel exploit (like "Dirty Pipe" CVE-2022-0847) can compromise ALL containers on a host simultaneously, because they all share the same kernel.

### 23.3 Threat Landscape

The course references a taxonomy of **47 exploit types [E]** across **11 attack vectors [A]** and **5 layers [L]**:

- **Orchestration (L₁):** API Server, Scheduler, Load Balancer exploits.
- **Application Code (L₂):** Code injection, supply chain attacks.
- **Container Engine (L₃):** Docker engine exploits (200+ CVEs between 2013-2024).
- **Kernel Space (L₄):** Namespace, cgroup, and syscall vulnerabilities.
- **Hardware (L₅):** Spectre, Meltdown side-channel attacks.

**The Shared Responsibility Model:**

|Layer|Responsible Party|
|---|---|
|Physical infrastructure, hardware (L₅)|Cloud Provider|
|Host OS (L₄), Container Engine (L₃), Kubernetes API Server (L₁)|Infrastructure/Platform Team|
|Application Code (L₂), image scanning, Pod Security Context|Developer/DevSecOps|

**Zero-Trust Integration:** Treat every layer as a potential entry point — assume lower layers may already be compromised.

### 23.4 Linux Security Foundations: Namespaces and cgroups

These are the two kernel primitives that make containers possible:

#### Namespaces (Isolation)

Namespaces give containers their "fiction" of being an isolated machine:

- **PID namespace:** Container sees only its own processes, not host processes.
- **Network namespace:** Container has its own network stack (interfaces, routing table).
- **Mount namespace:** Container sees only its own filesystem mounts, not host mounts.
- **User namespace:** Maps container root (UID 0) to a high non-root UID on the host — so "root" inside the container is NOT root on the host.
- **IPC namespace:** Isolates inter-process communication.

**Attack: Namespace Bypass (E₂₃):** If a container is misconfigured to share the host's PID or Network namespace, the isolation boundary is broken — the container can signal host processes or sniff host network traffic.

#### Control Groups / cgroups (Resource Limits)

cgroups control how much of each resource a container can use:

- **Memory hard limits:** Prevent a single container from exhausting host RAM (triggering Linux OOM Killer for all processes).
- **CPU shares and pinning:** Ensure fair CPU distribution.
- **PID limiting:** Mitigate fork bomb attacks (container can't spawn unlimited processes).

**Attack: Resource Starvation (E₃₄):** Without cgroup limits, one container can consume 100% CPU/RAM, causing all other containers (and the host) to fail — a "noisy neighbor" DoS attack from within.

### 23.5 Docker Security Basics

**Key risks in Docker:**

- **Docker Daemon Socket:** `/var/run/docker.sock` is a root-level gateway. Mounting this socket inside a container gives that container **full root access to the host**. This is a critical misconfiguration.
- **Daemon Disruption (E₁₉):** Maliciously crafted Docker images can crash the daemon itself, causing host-wide denial of service.
- **Remote API Exploitation:** Unprotected Docker REST APIs are the #1 target for crypto-jacking botnets — attackers find exposed Docker APIs and run their mining containers.
- **Debug Mode:** Can leak sensitive environment variables and auth tokens into logs.
- **Rootless Docker:** Running Docker daemon and containers without root reduces engine-level attack risks significantly.

### 23.6 Union File System Risks

Docker images use a **Union File System (layered)**:

- Base image → additional layers → writable top layer (copy-on-write).

**Security risks:**

- **Secret Persistence:** Secrets deleted in a later layer still exist in the image history (in an earlier layer). Anyone with access to the image can recover them.
- **Vulnerability Inheritance (E₂₃):** A vulnerability in a base layer propagates to ALL child images built from it.
- **Attack Surface from Extra Tools:** Bloated images containing `curl`, `netcat`, `bash` give attackers useful tools after breaching the container.

### 23.7 Kubernetes Security

#### Security Context (Pod Configuration)

Kubernetes **Security Contexts** configure security for individual Pods:

```yaml
securityContext:
  runAsNonRoot: true                  # Cannot run as root (UID 0)
  allowPrivilegeEscalation: false     # Cannot gain higher privileges than parent
  readOnlyRootFilesystem: true        # Root filesystem is immutable
  capabilities:
    drop: ["NET_RAW"]                 # Remove specific root powers
```

#### RBAC (Role-Based Access Control)

Defines: **Who** (User/ServiceAccount) can do **what** (verb: get, list, create, delete) on **which resource** (Pod, Secret, Node).

Best practices:

- Avoid `cluster-admin` role for users or CI/CD systems — use minimal, scoped roles.
- Do not auto-mount ServiceAccount tokens unless required.
- Integrate with OIDC providers (Okta, Active Directory) to enforce MFA.

#### Pod Security Standards (PSS)

Three levels:

1. **Privileged:** Unrestricted — for infrastructure components that need full access.
2. **Baseline:** Default — prevents known privilege escalation.
3. **Restricted:** Hardened — follows security best practices (non-root, no privilege escalation, read-only filesystem).

#### Admission Controllers

Intercept API requests **before** they are persisted. OPA/Gatekeeper can enforce custom policies (e.g., "all pods must have memory limits defined").

### 23.8 Image Security

**Vulnerability Scanning:**

- Tools like **Trivy** or **Clair** scan image layers for known CVEs.
- **Shift-Left strategy:** Integrate scanning in CI/CD pipeline — block deployment of images with critical vulnerabilities.
- Scan **continuously** — new CVEs are discovered daily; an image clean today may be vulnerable tomorrow.

**Image Signing (Cosign/Notary):**

- Cryptographically sign images. Kubernetes can be configured to only run signed images.
- Provides provenance — verifiable chain of custody for all production code.
- Prevents image spoofing (typosquatting attacks with look-alike malicious images).

**Trusted Registries:**

- Use private registries (Harbor, Azure Container Registry) with RBAC and built-in scanning.
- Avoid pulling from unverified public registries.

### 23.9 Runtime Security

**Seccomp Profiles:**

- Restricts which **syscalls** a container can make to the kernel.
- Docker's default profile disables ~44 dangerous syscalls (e.g., `reboot()`, `mount()`), reducing kernel exploit exposure.
- Custom profiles for high-security pods: allow only the specific syscalls the application actually needs.

**AppArmor / SELinux (Mandatory Access Control):**

- AppArmor (path-based): Restricts container access to specific filesystem paths.
- SELinux (label-based): Uses security labels for deep multi-tenant isolation.
- Even if a container gains root, MAC policies prevent it from damaging the host OS.

**Secrets Management:**

- Never hardcode secrets in Dockerfiles, code, or git repositories.
- Don't use environment variables — they can leak via `/proc/self/environ` or logs.
- Use external vaults (HashiCorp Vault, Azure Key Vault) to inject secrets at runtime.
- Kubernetes secrets in etcd are Base64-encoded by default (NOT encrypted) — must manually enable etcd encryption.

### 23.10 Monitoring and Security Tools

|Tool|Purpose|
|---|---|
|**Trivy**|Container image vulnerability scanning|
|**Kube-bench**|Kubernetes CIS Benchmark compliance checking|
|**Falco**|Runtime security monitoring using eBPF — detects suspicious activity in real-time|
|**Prometheus**|Resource monitoring — detects anomalies like CPU spikes indicating crypto-jacking|
|**Grafana**|Visualization of security metrics (failed logins, blocked traffic, alerts)|

**Falco detection scenarios:** Shell access into running container, unauthorized writes to `/etc`, connections to malicious IP addresses, process spawning unexpected children.

### 23.11 The Golden Rules of Container Security

1. **Containers ≠ Virtual Machines:** Containers share the host kernel — so security is critical.
2. **Defense in Depth:** Secure image → runtime → network → Kubernetes layers.
3. **Use Tools:** Trivy, Kube-bench, Falco, Prometheus automate detection at scale.
4. **Shared Responsibility:** Developers secure code/images; admins secure cluster/host.
5. **Principle of Least Privilege:** Grant only the minimum permissions necessary — at every layer.

### 📝 Q&A: Container Security

**Q1: Why does sharing the host OS kernel make containers less secure than VMs, despite being more efficient?**

> In a VM, each guest runs its own OS kernel — a vulnerability in one VM's kernel doesn't directly affect other VMs or the host (only the hypervisor boundary could be crossed). In containers, all containers share ONE host kernel. A kernel exploit (like Dirty Pipe) that allows privilege escalation affects ALL containers simultaneously — one compromised container can potentially escape to the host and affect all other containers. The efficiency gain (no duplicate OS per container) comes at the cost of a larger shared attack surface.

**Q2: What is a "container escape" and what security mechanisms prevent it?**

> A container escape is when malicious code running inside a container successfully breaks out of its isolation boundary and gains access to the host OS or other containers. Prevention mechanisms include: User namespaces (container root maps to unprivileged host UID), Seccomp profiles (restrict dangerous syscalls), AppArmor/SELinux (MAC prevents access outside allowed paths), avoiding privileged containers, dropping unnecessary Linux capabilities, and keeping the container runtime up-to-date (to patch escape vulnerabilities like runc CVEs).

**Q3: Why are hardcoded secrets in Docker images a severe security risk?**

> Docker images are layered — even if you add a layer that "deletes" a secret file, the earlier layer containing the secret still exists in the image history. Anyone with access to the image (e.g., if it's accidentally pushed to a public registry) can extract the secret by examining intermediate layers with `docker history`. Additionally, if the image is stored in a registry, it's permanently compromised. The fix is to never put secrets in images at all — use runtime secret injection from external vaults.

**Q4: Explain the "Principle of Least Privilege" in the context of Kubernetes RBAC.**

> The Principle of Least Privilege means: every subject (user, service account, pod) should have only the minimum permissions needed to perform its legitimate functions, and no more. In Kubernetes RBAC: a pod running a web application shouldn't have a ServiceAccount with `cluster-admin` rights. It should have a Role that only allows it to, say, `get` its own ConfigMap. A CI/CD pipeline shouldn't have permissions to delete production Pods. Restricting permissions limits the damage an attacker can do if they compromise a component — they inherit only that component's limited permissions.

**Q5: What is the difference between Seccomp and AppArmor as security mechanisms in containers?**

> Seccomp (Secure Computing Mode) works at the **syscall level** — it filters which system calls a process can make to the kernel. It cannot distinguish between different files or network addresses — just which syscall number is being invoked. AppArmor works at the **resource access level** — it can specify exactly which files, network ports, and capabilities a process can access, using file paths and labels. They are complementary: Seccomp restricts what kernel operations can be invoked; AppArmor restricts what resources can be accessed through those operations. Together they provide deep, multi-layer protection.

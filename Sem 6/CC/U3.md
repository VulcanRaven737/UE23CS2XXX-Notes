# 1. Introduction to Storage

## 1.1 What is Data?

**Data** is a collection of raw, unprocessed facts. Before we can talk about storage systems, we must understand what we are storing.

### Types of Data

|Type|Description|Storage Approach|
|---|---|---|
|**Structured Data**|Organized in rows and columns (tables)|DBMS (relational databases)|
|**Unstructured Data**|Cannot be organized in rows/columns; hard to uniquely identify and retrieve|Object stores|

**Why this distinction matters:** Structured data can be queried precisely with SQL-like languages. Unstructured data — think videos, images, sensor readings, social media posts — demands a different model entirely.

## 1.4 Three Fundamental Storage Types

The slides introduce three ways to store data, differing in **how data is held, organized, and presented**.

---

### 🗂️ 1.4.1 File Storage

**Concept:** Organizes data as a hierarchy of files inside folders — the classic folder/file tree you use on your laptop or server.

- Data is stored as a **single piece of information inside a folder**, like a physical paper document in a manila folder.
- Your computer must know the **path** to retrieve data (e.g., `/home/user/documents/resume.pdf`).
- Files are retrieved using **metadata** (file name, location path, creation date, etc.) that tells the OS where the file is stored.
- **Examples:** NAS (Network Attached Storage), DAS (Direct Attached Storage).
- **Scaling:** Typically **scale-out** — you add more storage units because each system has a limit.

**Analogy:** A filing cabinet in an office — every document has a drawer, a folder, and a name.

---

### 🧱 1.4.2 Block Storage

**Concept:** Breaks data into **evenly-sized, arbitrarily organized chunks called blocks**. Each block gets a **unique identifier**.

- The storage system can place blocks wherever convenient — they don't have to be contiguous.
- When you request data, the underlying storage software **reassembles the blocks** and presents them.
- **Usually used with SAN (Storage Area Networks).**
- Blocks can be retrieved quickly and accessed by any computing environment.
- Best for **enterprises with big transactions and large databases**.
- **Limitation:** Block storage is expensive and has **limited metadata handling** — metadata logic must be in the application or database layer.

**Analogy:** Splitting a large book into individual pages, distributing pages across filing cabinets, and reassembling when needed.

---

### 📦 1.4.3 Object Storage

**Concept:** Manages data as **discrete units called "objects"** stored in a **flat (non-hierarchical) repository** — no nested folders.

Each object is a self-contained unit with three parts:

```
┌─────────────────────────────────────┐
│  OBJECT                             │
│  ├── DATA     (the actual content)  │
│  ├── UNIQUE ID (for locating it)    │
│  └── METADATA (describing the data) │
└─────────────────────────────────────┘
```

- **Metadata is extensive** — it can include age, privacy settings, access permissions, and custom descriptors about the content itself.
- Retrieval uses **metadata + unique identifiers** for distributed load and robust search.
- Best for unstructured data at massive scale (images, videos, backups).
- **Example:** Amazon S3, Azure Blob Storage.

**Analogy:** A warehouse where items are tracked by barcode (unique ID), with a detailed label (metadata) attached to each item — but there's no "shelf A, row 3" concept.

---

### Summary Comparison Table

|Feature|File Storage|Block Storage|Object Storage|
|---|---|---|---|
|Structure|Hierarchical (files/folders)|Evenly-sized blocks|Flat namespace (objects)|
|Metadata|Limited (path, name)|Very limited|Rich and extensible|
|Performance|Moderate|High|Moderate|
|Scalability|Scale-out|Scale-up/out|Massive scale-out|
|Use Case|Home drives, NAS|Databases, SAN|Cloud, backup, media|
|Example|NFS, NTFS|SAN volumes|Amazon S3, Swift|

> **A:** Cost, Speed/Performance, Reliability, Availability, Scalability, Management.

> **A:** In block storage, blocks are placed wherever convenient in the storage subsystem and reassembled on read. # 2. Storage Architectures: DAS, NAS, SAN

## 2.1 Overview

Different ways of **connecting storage devices to servers over a network** lead to different **storage architectures**. There are three main architectures:

```
DAS (Direct Attached Storage)
NAS (Network Attached Storage)
SAN (Storage Area Network)
```

The slides show a diagram comparing all three side by side, illustrating how storage is connected to servers in each case.

---

## 2.2 Directly Attached Storage (DAS)

**Definition:** Storage that is physically and directly connected to a single server or workstation — not shared over a network.

- Think of it as the **hard drive inside your laptop**.
- No network involvement — the server communicates directly with the storage device.
- Very **low latency** and **high performance** because there's no network overhead.
- **Not shareable** — only the directly-attached host can access it (unless the host shares it over the network itself).
- **Scalability is limited** — adding more storage requires physical access to the server.

**Examples:** Internal HDDs/SSDs in a server, external drives connected via USB or eSATA.

**Best suited for:** Single-server applications where performance is critical and sharing is not required.

---

## 2.3 Network Attached Storage (NAS)

**Definition:** Storage accessed over a **standard TCP/IP network** (Ethernet/LAN), appearing to users as a shared network drive.

- The NAS device is a **dedicated file server** with its own OS, IP address, and file system.
- Multiple clients can access the same NAS simultaneously.
- Communicates at the **file level** — clients request files, not blocks.
- **Protocols:** NFS (Unix/Linux), CIFS/SMB (Windows), AFP (Mac).
- **Advantages:** Easy to set up, good for file sharing, works over existing Ethernet.
- **Limitations:** Performance is bound by the network bandwidth (typically GbE or 10GbE). Not ideal for high-throughput database workloads.

**Analogy:** A shared network folder in an office that everyone on the LAN can access.

**Examples:** Synology NAS, QNAP, enterprise NAS from NetApp.

---

## 2.4 Storage Area Network (SAN)

**Definition:** A **dedicated high-speed network** that connects servers to a pool of storage devices. The storage appears as **locally attached block storage** to each server.

- Uses **Fibre Channel (FC) or iSCSI** protocols, not standard Ethernet (though iSCSI does run on Ethernet).
- Operates at the **block level** — servers see raw storage blocks, not files.
- A **SAN switch** (similar to an Ethernet switch but for Fibre Channel) connects servers and storage.
- The slide shows that a SAN has **additional SAN ports, a network switch, and a dedicated storage fabric**.
- **Advantages:** Very high performance, low latency, excellent scalability, supports RAID and LUN (Logical Unit Number) management.
- **Limitations:** Expensive to set up and manage; requires specialized hardware.

**Best suited for:** Enterprise databases, virtualization environments (VMware, Hyper-V), mission-critical applications.

---

## 2.5 Architecture Evolution (Slide Summary)

The slide titled "Evolution of the Storage Systems Influenced by Architecture" shows a timeline of how storage architectures evolved:

- Early computing: **DAS** only — each server had its own storage.
- Growth of networking: **NAS** became popular for file sharing.
- Enterprise requirements: **SAN** was developed for high-performance, shared block storage.
- Modern cloud era: **Storage Virtualization** and **Object Storage** pools emerged to support cloud-scale applications.

## 🧠 Q&A Bank — Section 2

**Q1. What is the key difference between NAS and SAN in terms of the level at which storage is accessed?**

> **A:** NAS operates at the **file level** — clients request files. SAN operates at the **block level** — servers see raw storage blocks, similar to locally attached disks.

**Q2. Why does DAS have lower latency than NAS or SAN?**

> **A:** DAS is directly connected to the server with no network in between. NAS and SAN introduce network hops (even if dedicated), which add some latency.

**Q3. What protocol does NAS typically use for Linux/Unix clients?**

> **A:** NFS (Network File System).

**Q4. What networking technology does SAN traditionally use?**

> **A:** Fibre Channel (FC). It can also use iSCSI, which runs over standard Ethernet.

**Q5. Can multiple servers access a DAS simultaneously? Why or why not?**

> **A:** Not directly. DAS is attached to one server. Other servers can access it only if the attached server shares it over the network — but this adds a bottleneck at the sharing server.

**Q6. Why is SAN preferred over NAS for databases?**

> **A:** Databases require high-throughput block I/O with low latency. SAN provides raw block access with dedicated high-speed fabric (Fibre Channel), giving the database OS full control over I/O — something NAS's file-level access cannot match efficiently.

**Q7. What is a SAN switch and what is its role?**

> **A:** A SAN switch is a dedicated Fibre Channel (or iSCSI) switch that connects servers and storage devices in a SAN fabric, routing block-level I/O traffic between them.

- A proper enterprise disk subsystem has **redundant controllers** so a single controller failure doesn't bring down the storage.

## 3.2 RAID — Redundant Array of Independent Disks

### Definition

> **RAID** is a data storage virtualization technology that **combines multiple physical disk drives into one or more logical units** for the purposes of **data redundancy** and **performance improvement**.

Data is distributed across drives in different ways, called **RAID levels**, depending on requirements.

### How RAID Works Conceptually

Instead of having one big, expensive, fault-prone disk, RAID lets you combine many cheaper commodity disks and get something that is:

- More reliable (if one disk fails, data is not lost)
- Faster (multiple disks can be read/written in parallel)

### RAID Levels

The slides specifically mention RAID 0, RAID 1, and RAID 4. Here is a full explanation:

---

#### RAID 0 — Striping (No Redundancy)

```
Data: [A][B][C][D]
Disk 1:  [A][C]
Disk 2:  [B][D]
```

- Data is **striped** (split) across multiple disks.
- **Performance:** Excellent — reads and writes happen in parallel across all disks.
- **Redundancy:** Zero. If **any one disk fails**, all data is lost.
- **Use case:** High-performance temporary storage where data loss is acceptable (e.g., video editing scratch disks).

**Formula for capacity:** Total capacity = N × size of smallest disk (no overhead).

---

#### RAID 1 — Mirroring

```
Data: [A][B]
Disk 1:  [A][B]   ← Original
Disk 2:  [A][B]   ← Exact mirror
```

- Every bit of data is written to **two disks simultaneously**.
- **Redundancy:** Excellent — can survive a single disk failure.
- **Performance:** Good for reads (can read from either disk), write speed same as a single disk.
- **Overhead:** 50% — for every 2 TB of raw storage, you get only 1 TB usable.
- **Use case:** Operating system drives, critical databases.

---

#### RAID 4 — Striping with Dedicated Parity

```
Disk 1: [A1][B1][C1]
Disk 2: [A2][B2][C2]
Disk 3: [A3][B3][C3]
Disk 4: [P_A][P_B][P_C]  ← Dedicated Parity disk
```

- Data is striped across multiple disks, and a **dedicated parity disk** stores error-correction information.
- If one disk fails, the parity disk + remaining disks can **reconstruct** the lost data.
- **Bottleneck:** All writes update the parity disk → the parity disk becomes a write bottleneck.
- **Use case:** Mostly superseded by RAID 5.

---

#### RAID 5 — Striping with Distributed Parity (not explicitly in slides, but closely related)

Like RAID 4, but parity is **distributed across all disks**, eliminating the bottleneck. Most common RAID level in enterprises.

---

### RAID Summary Table

|RAID Level|Technique|Min Disks|Fault Tolerance|Read Perf|Write Perf|Overhead|
|---|---|---|---|---|---|---|
|RAID 0|Striping|2|None|Excellent|Excellent|0%|
|RAID 1|Mirroring|2|1 disk|Good|Same as 1 disk|50%|
|RAID 4|Striped + Parity (dedicated)|3|1 disk|Good|Parity bottleneck|1/N|
|RAID 5|Striped + Parity (distributed)|3|1 disk|Good|Good|1/N|

## 3.3 Magnetic Disk Structure and Disk Latency

The slide presents the **physical structure** of an HDD and introduces the equation for disk access time.

### Physical Structure of a Magnetic Disk

- A disk contains **platters** (circular magnetic plates).
- Each platter has concentric circular tracks.
- Tracks are divided into **sectors** (the smallest unit of storage).
- A **cylinder** is the set of tracks at the same position across all platters.
- The **read/write head** assembly moves radially over the platters.

### The Three Steps to Read/Write a Block

Reading or writing a block involves three sequential steps:

#### Step 1: Seek

The disk controller **positions the head assembly** at the **cylinder** containing the track where the block resides.  
→ Time taken = **Seek Time** ($T_{seek}$)

#### Step 2: Rotational Latency

The disk controller **waits** while the disk rotates until the **first sector of the block** is positioned under the read/write head.  
→ Time taken = **Rotational Latency** ($T_{rot}$)  
On average, the head waits for half a rotation: $$T_{rot(avg)} = \frac{1}{2} \times \frac{60}{RPM} \text{ seconds}$$

#### Step 3: Transfer

All sectors (and gaps between them) of the block **pass under the head** while data is read/written.  
→ Time taken = **Transfer Time** ($T_{transfer}$)

### Disk Latency Formula

$$\boxed{T_{disk} = T_{seek} + T_{rot} + T_{transfer}}$$

|Variable|Meaning|
|---|---|
|$T_{disk}$|Total disk access latency|
|$T_{seek}$|Time to move the head to the correct cylinder (track)|
|$T_{rot}$|Time for the correct sector to rotate under the head|
|$T_{transfer}$|Time for all sectors of the block to pass under the head|

**Typical values for a modern HDD:**

- $T_{seek}$: 3–15 ms
- $T_{rot}$: 2–6 ms (for 7200 RPM)
- $T_{transfer}$: depends on block size and RPM (sub-ms to a few ms)

**Why this matters for cloud storage:** SSDs eliminate seek time and rotational latency entirely (they have no moving parts), which is why cloud storage systems increasingly use NVMe SSDs. For HDDs, minimizing seek time (sequential access) drastically improves throughput — this is why databases prefer sequential I/O.

## 🧠 Q&A Bank — Section 3

**Q1. What is RAID and what are its two main goals?**

> **A:** RAID (Redundant Array of Independent Disks) is a technology that combines multiple physical disks into one or more logical units. Its two main goals are **data redundancy** (protection against disk failure) and **performance improvement** (parallel I/O).

**Q2. Which RAID level provides no redundancy at all? When would you use it?**

> **A:** RAID 0. You'd use it in high-performance, non-critical scenarios like video editing scratch storage or temporary processing buffers where speed matters and data loss is acceptable.

**Q3. RAID 1 requires 2 disks of 2 TB each. How much usable storage do you get?**

> **A:** 2 TB — because RAID 1 mirrors all data, 50% of raw capacity is overhead.

**Q4. Why does RAID 4 have a write performance bottleneck?**

> **A:** RAID 4 uses a **single dedicated parity disk**. Every write operation must update the parity disk, making it a bottleneck because all writes must access that one disk.

**Q5. What are the three components of disk latency? Which is typically the largest?**

> **A:** Seek time, rotational latency, and transfer time. **Seek time** is typically the largest for random access patterns.

**Q6. Why do SSDs have dramatically lower latency than HDDs?**

> **A:** SSDs have no moving mechanical parts — there is no seek time and no rotational latency. Data is accessed electronically at near-instantaneous speeds.

**Q7. What is a "cylinder" in a magnetic disk?**

> **A:** A cylinder is the set of tracks at the same radial position across all platters in a disk. For multi-platter disks, reading all tracks in a cylinder doesn't require head movement, saving seek time.

**Q8. A disk spins at 7200 RPM. What is the average rotational latency?**

> **A:**  
> Time for one full rotation = 60/7200 = 8.33 ms  
> Average rotational latency = 8.33/2 ≈ **4.17 ms**

**Q9. What is the role of cache in a storage controller?**

> **A:** The cache buffers both reads and writes between the host and the physical disks. For reads, frequently accessed data is served from cache (much faster). For writes, data is acknowledged once in cache and written to disk later, improving write speed.

**Q10. What role does the controller play in a storage subsystem?**

> **A:** The controller sits between the physical disks and external ports. It manages RAID operations for redundancy and performance, maintains cache, presents logical volumes (LUNs) to hosts, and handles I/O scheduling.

# 4. Cloud Storage & Its Enablers

## 4.1 The OS Analogy for Understanding Cloud Storage

The slides use a brilliant two-level analogy:

**Level 1 — Local Computer:**

- A PC user sees a rich interface: hierarchical directories, the ability to rename/move/compress files.
- But the actual HDD only understands **fixed-size block reads and writes**.
- The **Operating System** does the translation between rich file operations and raw block I/O.

**Level 2 — Cloud/Web Service:**

- A web user sees a rich interface: shopping carts, friend lists, product search.
- The actual backend storage is a **key/value store** — just read/write/delete operations.
- The **Web Service** does the translation between user-facing features and raw storage primitives.

This analogy teaches us: **cloud storage is a key-value abstraction layer over raw block/object storage.**

---

## 4.2 What is Cloud Storage?

> **Cloud Storage** is a data storage service model in which data is **maintained, managed, and backed up remotely**, and made available to users over a network (typically the internet).

### Key Characteristics

- The **cloud storage provider** is responsible for:
    - Keeping data available and accessible.
    - Running and protecting the physical environment.
- Users/organizations **buy or lease capacity** rather than owning hardware.
- Cloud storage can be visualized as a **virtual storage pool**.
- Access is via **web services APIs**.

### Cloud Storage Architecture (4-Layer Model from slides)

```
┌──────────────────────────────────────┐
│         User Access Layer            │  ← Authorized users log in from anywhere
├──────────────────────────────────────┤
│         Data Service Layer           │  ← Storage, space leasing, sharing, backup
├──────────────────────────────────────┤
│         Data Management Layer        │  ← User mgmt, security, replica, strategy mgmt
├──────────────────────────────────────┤
│         Data Storage Layer           │  ← Massive data pool organized for access
└──────────────────────────────────────┘
```

**Examples of cloud storage systems:** Amazon S3, Microsoft Azure Storage, OpenStack Swift, EMC Atmos, EMC ECS, Hitachi Content Platform.

## 4.5 What Constitutes Cloud Storage — The 9 Properties

The slides list 9 defining properties:

### 1. Dramatic Reduction in TCO (Total Cost of Ownership)

- Cuts storage cost by **more than 10× compared to block or file storage**.
- Built using commodity hardware and distributed software.

### 2. Unlimited Scalability

- Built on distributed technologies — seamlessly add or remove storage systems from the pool.

### 3. Elasticity

- Storage virtualization **decouples the storage pool from physical implementation**.
- The pool can grow and shrink **as required** — you're not locked into a fixed size.

### 4. On-Demand

- Uses a **pay-as-you-go** model — you pay only for data stored and accessed.
- For private cloud: minimal cluster to start, then add on demand.

### 5. Universal Access

- Traditional block storage requires the server to be on the same SAN.
- Cloud storage is accessible from anywhere, by any number of users.

### 6. Multitenancy

- Cloud storage is multi-tenant: **multiple users share the same infrastructure** but with logical isolation.
- This leads to centralized management, higher utilization, and lower costs.

### 7. Data Durability and Availability

- Runs on commodity hardware, but the **software layer provides high availability** even with partial hardware failures (via replication, erasure coding, etc.).

### 8. Usability

- Simple APIs make it easy for developers to integrate storage.

### 9. Disaster Recovery

- Built-in geographic replication makes cloud storage inherently resilient to regional disasters.

### 4.6.1 File Systems

> A **file system** is the structure and logic used to manage groups of information (files) and their names.

Without a file system, storage would be a single undifferentiated blob of data with no way to distinguish one piece from another.

**Key points:**

- The **block** is the basic unit of storage for I/O (read/write) operations.
- File systems control how data is stored and retrieved.
- Can be used on many different media types.
- **Network File Systems (NFS):** Make local files and directories available over a LAN. Multiple end users can work on common files.
    - NFS runs a daemon process called `nfsd` on the server.
    - Server administrator **exports directories** (makes them available).
    - Clients mount exported directories — access becomes transparent.

---

### 4.6.2 Logical Volume Manager (LVM)

LVM is an **independent layer between the file system and physical disk drives**.

**The LVM Stack:**

```
┌─────────────────┐
│   File System   │  (ext4, XFS, NTFS, etc.)
└────────┬────────┘
         │
┌────────▼────────┐
│  Logical Volume │  (e.g., /dev/vg0/lv_data)
└────────┬────────┘
         │
┌────────▼────────┐
│  Volume Group   │  (pool of physical volumes)
└────────┬────────┘
         │
┌────────▼────────┐
│ Physical Volume │  (disk partition, e.g., /dev/sda1)
└────────┬────────┘
         │
┌────────▼────────┐
│  Physical Disk  │
└─────────────────┘
```

**LVM Steps:**

1. Create **Physical Volumes (PVs)** on disk partitions.
2. Group PVs into a **Volume Group (VG)** — a storage pool.
3. Carve the VG into **Logical Volumes (LVs)** of any size.
4. Create a **file system** on each LV and mount it.

**Power of LVM:**

- **Dynamic resizing:** Grow or shrink logical volumes without reformatting.
- **Snapshots:** Take point-in-time copies.
- **Spanning:** One logical volume can span multiple physical disks.

---

### 4.6.3 Thin Provisioning (Virtual Provisioning)

**Concept:** Logical units presented to hosts are **larger than the physically allocated storage**.

```
Host sees:     [     1 TB Logical Volume     ]
Actually:      [  200 GB physically allocated  ]
```

**How it works:**

- Physical storage is allocated **only when the host actually writes data**.
- Provisioning decisions are not bound by currently available physical storage.
- A shared storage pool serves multiple hosts, each believing they have more space than exists.

**Analogy:** Like an airline that sells more seats than the plane has, betting not all passengers will show up. If everyone shows up → oversubscription problem.

**Risk:** If all tenants write more data than physically available → **out-of-space condition** → must add storage or thin provisioning policies kick in.

**Benefits:** Better utilization of physical hardware, reduced upfront purchase requirements.

# 5. Storage Virtualization

## 5.1 What is Storage Virtualization?

> **Storage virtualization** is the process of **abstracting physical storage subsystems** (disks, tapes, etc.) from user applications and presenting them as **logical entities**, hiding the complexity of the underlying storage.

**Key capabilities:**

- **Aggregation:** Combine capacity of multiple storage devices into unified storage pools.
- **Division:** Divide a large physical resource into multiple smaller logical resources.
- **Location independence:** The virtualization layer hides the physical location of data.
- **Mapping:** Software maintains a consistent mapping from logical to physical locations — this mapping is part of the **metadata**.

```
┌────────────────────────────────┐
│          Servers               │
└──────────────┬─────────────────┘
               │
┌──────────────▼─────────────────┐
│    Virtual Storage Layer       │  ← Logical entities (what servers see)
└──────────────┬─────────────────┘
               │ (Virtualization)
┌──────────────▼─────────────────┐
│    Physical Storage            │  ← Actual disks, tapes, arrays
└────────────────────────────────┘
```

**Examples:**

- Single virtual large disk created from multiple small disks.
- Many smaller virtual disks created from one large physical disk.

---

## 5.2 Categories of Storage Virtualization

There are **two major categories**:

### 5.2.1 Category 1: File Level Virtualization

**Definition:** Provides an abstraction of a **file system** to the application (via NFS or CIFS) and manages changes to distributed storage underneath.

**Benefits:**

- Eliminates dependency between **data accessed at file level** and the **physical location** where files are stored.
- Allows administrators to present users with a **single logical file system** across multiple servers.
- Provides opportunities for storage optimization, server consolidation, and **non-disruptive file migrations** (move files without downtime).

**Protocols used:** NFS, CIFS, HTTP, WebDAV.

---

#### Distributed File Systems (DFS)

A **Distributed File System (DFS)** is a network file system where the file system is **distributed across multiple servers**.

**Key features of DFS:**

- **Location transparency:** Users don't need to know where data is physically stored.
- **File directory replication:** Directories can be replicated across servers.
- **Fault tolerance:** The system continues to work even if some servers fail.
- **Caching:** Recently accessed disk blocks can be cached for performance.

**Two key approaches for managing metadata in DFS:**

---

##### Approach A: Centralized Metadata Server (used in Lustre)

- A **dedicated metadata server (MDS)** handles all metadata operations from clients.
- **Lock-based synchronization** is used for every read/write operation.
- **Advantage:** Scalable for **large file workloads** — if files are large, metadata operations are infrequent.
- **Disadvantage:** Metadata server can become a **bottleneck** if there are too many metadata operations (many small files).

**Lustre — Centralized Metadata DFS:**

Lustre is a **massively parallel, scalable distributed file system** for Linux, capable of scaling to thousands of clients with petabytes of storage.

**Lustre Architecture Components:**

```
┌─────────────────────────────────────────────────┐
│  1. Object Storage Servers (OSSes)              │
│     Store actual file data on Object Storage    │
│     Targets (OSTs)                              │
├─────────────────────────────────────────────────┤
│  2. Metadata Target (MDT)                       │
│     Stores metadata on Metadata Servers (MDS)  │
│     (Single MDT — centralized)                 │
├─────────────────────────────────────────────────┤
│  3. Lustre Clients                              │
│     Access data over network via POSIX interface│
└─────────────────────────────────────────────────┘
```

**Lustre Read/Write Flow:**

1. Client performs a **filename lookup** on the MDS.
2. MDS creates a metadata file or returns the layout of an existing file.
3. Client passes layout to **Logical Object Volume (LOV)** for read/write.
4. LOV maps the **offset and size** to objects on one or more OSTs.
5. Client **locks the file range** and executes **parallel read/write** directly to OSTs.

---

##### Approach B: Distributed Metadata (used in GlusterFS)

- Metadata is **distributed across all nodes** — no single metadata server.
- Greater complexity, but eliminates the single-metadata-server bottleneck.
- Better for workloads with many small files.

**GlusterFS — Distributed Metadata DFS:**

GlusterFS is an **open-source, distributed cluster file system** without a centralized metadata server.

**Key features:**

- Scales to thousands of clients and petabytes of capacity.
- Uses **modular, stackable user-space design**.
- Aggregates multiple **storage bricks** (server + directly attached storage) over InfiniBand RDMA or TCP/IP.
- Delivers as a **global namespace network file system**.

**GlusterFS Components:**

- **Gluster Server:** Clusters all physical storage servers; exports combined disk space as one GlusterFS.
- **Gluster Client (optional):** Provides highly available, massively parallel access to every storage node; handles node failures transparently.

**Translators in GlusterFS:**

- Translators are programs (like **filters**) inserted between file content and user access.
- Each translator implements a specific feature (e.g., load balancing, replication).
- Translators can be loaded on both client and server sides.
- **AFR (Automatic File Replication) translator:** Keeps identical copies of files on all subvolumes.
- **I/O Scheduler translators:** Provide load balancing.

---

### 5.2.2 Category 2: Block Level Virtualization

**Definition:** Virtualizes multiple physical disks and presents them as a **single logical disk**. Block addresses may reside on multiple physical storage subsystems but appear as one storage device.

**Three levels of block virtualization:**

---

#### Level A: Host-Based Block Virtualization

- Uses **Logical Volume Manager (LVM)** on the host.
- Creates a storage pool combining multiple disks.
- Supports **dynamic shrinking/growing** of physical volumes.
- **Where it lives:** In the server's OS software.

---

#### Level B: Storage-Device Level Virtualization

- Creates **Virtual Volumes** (LUNs) over the physical storage space of a specific storage subsystem.
- **RAID arrays** use this: Array controllers create LUNs spanning multiple disks in RAID Groups.
- Some disk arrays also virtualize **third-party external storage devices** attached to the array.
- **Advantages:**
    - Host-agnostic (doesn't depend on host OS).
    - **Low latency** — virtualization is in device firmware.

---

#### Level C: Network-Based Block Virtualization

> This is the **most commonly implemented** form of scalable virtualization.

- Virtualization is implemented **within the network** connecting hosts and storage (e.g., in a Fibre Channel SAN).
- Two sub-categories based on where virtualization happens:

##### 1. Switch-Based Network Virtualization

- Virtualization occurs **inside an intelligent switch** in the fabric.
- Works in conjunction with a **metadata manager in the network**.
- **Example:** HP StorageWorks SAN Virtualization Services Platform.

##### 2. Appliance-Based Network Virtualization

- An **appliance (server)** controls the virtualization layer.
- All I/O flows through the appliance.
- **Example:** IBM SAN Volume Controller.

**Two variants of appliance-based:**

|Variant|Description|
|---|---|
|**In-Band**|All I/O requests AND data pass through the appliance. Clients never interact directly with storage.|
|**Out-of-Band**|Appliance only handles metadata/control path. Actual data I/O goes directly from client to storage (with agents on each client).|

---

# 6. Object Storage, Amazon S3, OpenStack Swift & DynamoDB

## 6.1 Object Storage Deep Dive

**Why object storage dominates cloud storage:**

- Eliminates the hierarchical namespace constraints of file storage.
- Scales to billions of objects without name collision via a **flat namespace**.
- Uses **REST HTTP APIs** (GET, PUT, DELETE, UPDATE) — simple and web-native.

### REST API for Object Storage

Object storage uses **REpresentational State Transfer (REST)** APIs:

- **GET** — retrieve an object
- **PUT** — create or replace an object
- **DELETE** — remove an object
- **HEAD** — retrieve metadata only
- **POST** — partially update

**REST Principles:**

- Resources identified by **URIs** (Uniform Resource Identifiers)
- **Stateless** messages (each request is self-contained; server holds no client session state)
- Messages are typically in **XML or JSON**

**Object structure:**

```
┌────────────────────────────────────────────┐
│  OBJECT                                    │
│  ┌──────────────────────────────────────┐  │
│  │  DATA (actual content)               │  │
│  ├──────────────────────────────────────┤  │
│  │  UNIQUE IDENTIFIER (URI/key)         │  │
│  ├──────────────────────────────────────┤  │
│  │  METADATA                            │  │
│  │  - System metadata (age, size, ACL)  │  │
│  │  - Custom metadata (content type,    │  │
│  │    business tags, etc.)              │  │
│  └──────────────────────────────────────┘  │
└────────────────────────────────────────────┘
```

### Object Storage Architecture

- Built using **scale-out distributed systems** — no specialized expensive hardware.
- Each node typically runs on a **local filesystem**.
- Critical tasks:
    1. **Data placement** — deciding where to store each object.
    2. **Automating management** — ensuring durability and availability.

**Client request flow:**

1. Client sends HTTP request (GET/PUT) to **any node** in the cluster.
2. Object storage software translates to the **physical node** holding the object.
3. The software manages **durability** via:
    - Multiple copies (replication)
    - **Erasure coding (EC):** Data broken into fragments, expanded with redundant pieces, distributed across locations. If some fragments are lost, the original can be reconstructed.
    - Or a combination of both.

**Management features:**

- Periodic **health checks** and **self-healing**.
- **Data migration** handled automatically.
- **Single flat namespace** makes cluster management simple.

## 6.2 Amazon S3 (Simple Storage Service)

Amazon S3 is one of the foundational cloud services, a **highly reliable, scalable, and fast object store**.

### Organizing Data in S3: Buckets, Objects, and Keys

**Three-level hierarchy:**

```
S3 Account
└── Bucket (e.g., "johndoe")
    └── Object (e.g., "resume.doc")
        └── Accessed via Key (e.g., "resume.doc")
```

|Concept|Description|
|---|---|
|**Object**|The actual data being stored (up to **5 TB** per object)|
|**Bucket**|Container for objects; provides namespace isolation|
|**Key**|Unique identifier for an object within a bucket|

**Key limits:**

- Up to **100 buckets per account**.
- **Unlimited number of objects** per bucket.
- Objects replicated across **multiple geographic locations** for resilience.

**URL format:**

```
http://s3.amazonaws.com/johndoe/resume.doc
OR
http://johndoe.s3.amazonaws.com/resume.doc
```

By using slash-separated keys, S3 simulates a directory-like structure (though it's actually flat):

```
project/reports/q1.pdf
project/reports/q2.pdf
project/code/app.py
```

**Three ways to access S3:**

1. **AWS Console** — GUI for common operations.
2. **REST API** — programmatic access via HTTP (GET, PUT, DELETE, HEAD).
3. **S3 Browsers/SDKs** — libraries for Python, Java, Go, etc., plus desktop S3 browsers.

### S3 Security

Two security mechanisms:

|Mechanism|Description|
|---|---|
|**Access Control**|Set permissions on objects/buckets via AWS Management Console (who can read/write/delete)|
|**Audit Logs**|Enable logging for a bucket; S3 stores access logs in a separate bucket, recording: AWS account that accessed, time, IP address, operation performed|

---

### S3 Data Protection

**1. Replication:**

- By default, S3 replicates data across **multiple storage devices** and is designed to survive **two replica failures**.
- **Reduced Redundancy Storage (RRS):** For non-critical data — replicated twice, survives **one** replica failure.
- S3 provides **strong consistency** and guarantees consistency among replicas.

**2. Regions:**

- S3 data can be stored in specific geographic regions for performance, legal compliance, or availability reasons.
- Region is set at the **bucket level** during bucket creation.

**3. Versioning:**

- When enabled, S3 stores the **full history of all objects** in the bucket.
- You can restore any prior version.
- Even **deletes can be undone** — no data is inadvertently lost.

**4. Multi-part Uploads:**

- For large objects: API allows **splitting** a large object into parts and uploading each part **independently**.
- Parts can be uploaded in **parallel** to maximize network utilization.
- If one part fails: only that part needs to be re-tried (not the whole object).

## 6.3 OpenStack Swift

**Swift** is an open-source object storage system, part of the OpenStack cloud platform. The slides provide a conceptual overview.

### Key Concepts

|Concept|Description|
|---|---|
|**Partition**|Swift breaks storage into locations (not Linux partitions) where data will reside — like a moving bin in a warehouse. Core of the replication system.|
|**Account**|A user in the storage system; unlike file system volumes, accounts enable multiple users and applications to access storage simultaneously.|
|**Container**|Where accounts store data; conceptually like directories, used to group objects (name spaces).|
|**Object**|The actual data stored on disk (photos, videos, files, etc.).|
|**Ring**|Maps the partition space to physical disk locations — like an encyclopedia, except Swift uses **hash values** instead of letters to look up locations.|

**Swift Write flow:**

1. Client writes an object → Swift computes a hash of the object's path.
2. The **Ring** is consulted to determine which **partitions and physical nodes** should hold the data.
3. Data is written to multiple nodes for redundancy.

**Adding a node** → Leads to **reassignment of partitions** to rebalance data.

## 6.4 DynamoDB — NoSQL Cloud Database

DynamoDB is **Amazon's fully managed, cloud-based NoSQL key-value and document database**.

### Core Characteristics

- **Fully managed:** AWS handles provisioning, scaling, patching, backups.
- **Schema-less flavor:** Users define only the structure of **keys** and **local secondary indexes** upfront; attribute names and types for items can vary.
- **No JOIN support:** DynamoDB has no concept of joins. Any join-like logic must be implemented at the application level.
- **Item-level consistency:** Consistent operations only within a single item (analogous to row-level consistency in RDBMS).

### Core Data Model

```
DynamoDB Table
└── Items (= rows in RDBMS)
    └── Attributes (= columns in RDBMS, but can vary per item)
```

|DynamoDB Term|RDBMS Equivalent|
|---|---|
|Table|Table|
|Item|Row|
|Attribute|Column|

### Primary Keys

**Two kinds of primary keys:**

#### 1. Partition Key (Simple Primary Key)

- Single attribute as primary key.
- DynamoDB feeds the partition key value through an **internal hash function**.
- Output of hash → determines which **partition** (physical storage shard) the item goes to.
- All items with the same partition key go to the **same partition**.

#### 2. Partition Key + Sort Key (Composite Primary Key)

- Two attributes: partition key + sort key.
- All items with the same partition key are stored **together**, sorted by sort key value.
- Example: `(UserID, Timestamp)` — all activities of a user stored together, ordered by time.

### Secondary Indexes

- **Local Secondary Index (LSI):** Same partition key, different sort key. Must be defined at table creation.
- **Global Secondary Index (GSI):** Different partition key (and optionally sort key). Can be added anytime.
- Secondary indexes allow querying by non-primary-key attributes efficiently.

### Partitions and Data Distribution

- DynamoDB stores data in **partitions** (SSDs backed storage units).
- Each partition is **automatically replicated across multiple Availability Zones** (AZs) within an AWS Region.
- **Partition management is entirely automatic** — users never interact with partitions directly.

**DynamoDB Architecture (from the slide):**

- The architecture shows a distributed system with request routing, auto-sharding across partitions, and multi-AZ replication — all hidden from the user.

# 7. Partitioning and Consistent Hashing

## 7.1 Why Partitioning?

**Problem:** Cloud applications scale elastically, but if all data stays in a single data store, that store becomes a **bottleneck** no matter how many application servers you add.

> **Partitioning** is the deliberate practice of breaking large data into smaller subsets (**partitions/shards**), each of which can be placed on different nodes in a cluster.

**Benefits:**

- Query/IO operations execute **across multiple nodes in parallel**.
- Adding more nodes → more partitions → more throughput.
- Each node independently executes operations on its own partition.

**Rule:** Each piece of data (record, row, document) belongs to **exactly one partition**.

**Partitioning + Replication:** Partitioning is always combined with replication — each partition is stored on multiple nodes for fault tolerance. A record belongs to one partition but may be stored on several nodes.

---

## 7.2 Skew and Hot Spots

|Term|Definition|
|---|---|
|**Skew**|Some partitions have much more data than others|
|**Hot Spot**|A partition with disproportionately high load (too many reads/writes)|

The **goal of partitioning** is to spread data and query load **evenly** across all nodes.

---

## 7.3 Four Approaches to Partitioning

### Approach 1: Vertical Partitioning (Column Partitioning)

- Data is partitioned by **columns** — different sets of columns stored on different nodes.
- No two critical columns are stored together, improving performance of specific queries.
- Example: User profile table split: personal info on one store, purchase history on another.

```
Original Table:    [UserID | Name | Email | OrderHistory | PaymentInfo]
After splitting:
Node 1:            [UserID | Name | Email]
Node 2:            [UserID | OrderHistory | PaymentInfo]
```

---

### Approach 2: Workload-Driven Partitioning

- **Analyse data access patterns** from the application.
- Partition data based on how it is accessed, not just its structure.
- Improves scalability of transactions in throughput and response time.

---

### Approach 3: Random Assignment

- Randomly assign records to nodes.
- Distributes data **evenly** — naturally avoids hot spots.
- **Problem:** When reading a specific item, you don't know which node it's on → must **query all nodes in parallel** → expensive scatter/gather.

---

### Approach 4: Horizontal Partitioning (Most Important)

Rows are distributed across nodes. Four techniques:

---

#### 4.1 Partitioning by Key Range

- Assign a **continuous range of keys** to each partition.
- Key values are adjacent but non-overlapping within a partition.
- If we know the range boundaries, we can determine which partition contains a key → route request directly.
- Within each partition, keys can be kept in sorted order (enabling range queries).

**Example:**

```
Partition 1: keys A–F
Partition 2: keys G–M
Partition 3: keys N–Z
```

**Advantage:** Efficient range scans (e.g., all sensor readings for January).

**Disadvantage:** Can cause **hot spots**. If the key is a timestamp, all writes go to "today's" partition, overloading it while others are idle.

**Fix:** Use a different attribute (e.g., sensor ID + timestamp) as the first key element.

---

#### 4.2 Schema-Based Partitioning

- Partition the DB schema so that **related rows stay in the same partition**.
- Minimizes **distributed transactions** (transactions needing data from multiple nodes).
- Designed to keep data that is frequently accessed together in the same partition.

---

#### 4.3 Graph Partitioning

- Partition based on **analysing data access patterns** (workload-driven, static).
- Once partitioned, does not change unless explicitly repartitioned.
- Can also be dynamic (discussed under partition rebalancing).

---

#### 4.4 Partitioning by Hash of Key

- Apply a **hash function** to the key → get a hash value → assign to a partition based on the hash value's range.
- The hash function distributes keys **evenly** across partitions → avoids skew and hot spots.
- Works regardless of whether data is historical or not.

**How it works:**

```
hash("Alice")     → 0x3F2A → Partition 3
hash("Bob")       → 0x8B14 → Partition 7
hash("Charlie")   → 0x1C09 → Partition 1
```

**Trade-off:** You **lose the ability to do efficient range queries**. The hash destroys natural ordering.

**Concatenated index workaround:** For one-to-many relationships, use a **composite key** like `(user_id, update_timestamp)`. Now you can efficiently query all updates by a specific user in a time range — using `user_id` to route to the right partition, then `timestamp` for range scan within the partition.

---

### SQL Example from Slides (Range Partition + Hash):

```sql
-- Create a range partition function
CREATE PARTITION FUNCTION myRangePF1 (int)
  AS RANGE LEFT FOR VALUES (1, 100, 1000);

-- Create a partition scheme (mapping partitions to filegroups)
CREATE PARTITION SCHEME myRangePS1
  AS PARTITION myRangePF1
  TO (test1fg, test2fg, test3fg, test4fg);

-- Create a table on the partition scheme
CREATE TABLE PartitionTable (col1 int, col2 char(10))
  ON myRangePS1 (col1);
```

**Result:**

|Partition|Range|
|---|---|
|test1fg|col1 ≤ 1|
|test2fg|1 < col1 ≤ 100|
|test3fg|100 < col1 ≤ 1000|
|test4fg|col1 > 1000|

---

## 7.4 Distributed Hashing

**Problem:** For very large datasets, the hash table itself is too large for one machine.

**Solution:** Distribute the hash table across multiple servers.

**Mechanism:**

- For a key `k`, compute `hash(k)`.
- Determine server: `server = hash(k) mod N` where N = number of servers.

**Cache integration:**

1. Client needs partition for key "Name".
2. Check **cache** — if found, get server location from cache.
3. If cache miss: compute `hash("Name") mod N` → find server → get partition → populate cache.

---

## 7.5 The Rehashing Problem

**Problem:** When servers are added or removed, `hash(k) mod N` changes for most keys.

**Example:**

- 3 servers (0, 1, 2). Key "Alice" → `hash("Alice") mod 3 = 2` → Server 2.
- Server 2 fails → now 2 servers (0, 1). Key "Alice" → `hash("Alice") mod 2 = 0` → Server 0.
- **But all other keys also get remapped**, not just those on Server 2!
- With caches: **all cached entries become invalid** → massive cache miss storm → origin overloaded.

---

## 7.6 Consistent Hashing — The Solution

> **Consistent Hashing** is a distributed hashing scheme that **minimizes key redistribution** when servers are added or removed, by placing both objects and servers on an **abstract hash ring (circle)**.

### How the Hash Ring Works

**Step 1:** Take the hash output range (say, 0 to $2^{32}$) and map it onto a **circle** (0° to 360°).

**Step 2:** Hash each **server** using its name/IP:

```
hash("Server A") → placed at position 50°
hash("Server B") → placed at position 150°
hash("Server C") → placed at position 270°
```

**Step 3:** Hash each **key** and place it on the same circle:

```
hash("Alice") → placed at position 80°
hash("Bob")   → placed at position 200°
hash("Charlie") → placed at position 310°
```

**Step 4 — Assignment Rule:** Each key is assigned to the **first server encountered when moving clockwise** from the key's position.

```
Circle:  0°                                              360°
         ──────────────────────────────────────────────
          Server A @ 50°
                    Alice @ 80°     → assigned to Server B (next clockwise)
                              Server B @ 150°
                                       Bob @ 200°        → assigned to Server C
                                                  Server C @ 270°
                                                           Charlie @ 310° → wraps to Server A
```

### What Happens When a Server Is Added or Removed?

**Server removed (fails):**

- Only keys that were assigned to that server need to move to the next server clockwise.
- All other keys remain where they are.
- **Minimal disruption**: only ~1/N of all keys are affected.

**Server added:**

- New server is placed on the ring.
- Only keys between the new server and its predecessor on the ring need to move.
- Again, ~1/N of keys affected.

**Comparison:**

|Scheme|Keys moved when server changes|
|---|---|
|Simple hash mod N|Almost **all** keys|
|Consistent Hashing|Only **~1/N** keys|

---

## 7.7 Partitioning and Secondary Indexes

When data is partitioned, secondary indexes create complications.

**Two main approaches:**

### 7.7.1 Document-Based Partitioning (Local Index)

- Each partition maintains its own secondary index covering **only documents in that partition**.
- Writes only touch one partition (the one containing the document).
- **Reads are expensive:** To query by secondary index, you must **scatter/gather** — query all partitions and combine results.
- This approach is called **scatter/gather**.
- **Problem:** Prone to **tail latency amplification** (the overall response time is determined by the slowest partition).

### 7.7.2 Term-Based Partitioning (Global Index)

- One **global index** covers data in **all partitions**.
- The global index is itself partitioned (by term value or hash of term).
- **Reads are efficient:** A query on a term only needs to contact the partition holding that term.
- **Writes are expensive and complex:** Writing one document may update index entries in multiple partitions (every term in the document).
- Global indexes are usually **asynchronously updated** — there may be a lag between writing a document and the index reflecting it.

---

## 🧠 Q&A Bank — Section 7

**Q1. What is the goal of partitioning, and what happens when it fails?**

> **A:** The goal is to spread data and query load evenly across nodes. When it fails, **skew** occurs — some partitions get more data. In the extreme, one partition becomes a **hot spot**, receiving most of the load, defeating the purpose of partitioning.

**Q2. What is the disadvantage of partitioning by random assignment?**

> **A:** You lose the ability to route a read request to a specific node. Since you don't know which node has the item, you must query all nodes in parallel (scatter/gather), which is expensive.

**Q3. Why can timestamp-based range partitioning cause a hot spot?**

> **A:** All new writes go to the "current" partition (the range containing today's timestamp), while older partitions sit idle. This creates a hot write spot.

**Q4. What is the trade-off of hash partitioning versus key-range partitioning?**

> **A:** Hash partitioning distributes load evenly (avoids skew) but destroys ordering, making range queries inefficient. Key-range partitioning supports efficient range scans but is susceptible to skew/hot spots if keys cluster in a range.

**Q5. Explain the rehashing problem in distributed hashing.**

> **A:** When the number of servers changes (add or remove), `hash(key) mod N` changes for almost all keys — not just those on the changed server. This forces a massive redistribution of data and invalidates caches, causing a storm of cache misses.

**Q6. How does consistent hashing solve the rehashing problem?**

> **A:** Consistent hashing places both keys and servers on a hash ring. When a server is added or removed, only the keys between the changed server and its neighbour on the ring need to move — approximately 1/N of all keys, rather than almost all keys.

**Q7. What is scatter/gather in the context of secondary indexes?**

> **A:** In document-based (local) secondary index partitioning, a query on a secondary index must be sent to all partitions, each of which returns its matching results. The coordinator gathers and combines results. This is scatter/gather — expensive but necessary with local indexes.

**Q8. What is the advantage of term-based (global) index partitioning?**

> **A:** Reads are more efficient — instead of querying all partitions, you only contact the partition of the index that holds the search term. The trade-off is that writes become slow and complex.

**Q9. In consistent hashing, what is the assignment rule for mapping a key to a server?**

> **A:** Move clockwise (or counterclockwise, depending on convention) from the key's position on the hash ring. The first server encountered in that direction is assigned to the key.

**Q10. What is a composite key in hash partitioning and when is it useful?**

> **A:** A composite key combines two fields (e.g., `user_id + timestamp`). Hash partitioning routes to the right partition using `user_id`, while range queries within the partition use `timestamp`. This preserves locality for related records while distributing load across partitions.

# 8. Rebalancing Partitions and Request Routing

## 8.1 Why Rebalancing?

Clusters change over time:

- **Query throughput increases** → need more CPUs.
- **Dataset grows** → need more storage nodes.
- **Machine fails** → others must take over.
- **Machine recovers/joins** → redistribute load.

> **Rebalancing** = moving data and requests from one node to another as the cluster changes.

**Minimum requirements for rebalancing:**

1. **Fairness:** After rebalancing, load should be shared fairly among all nodes.
2. **Availability:** Database must continue accepting reads/writes during rebalancing.
3. **Efficiency:** Move as little data as possible — minimize network and disk I/O.

---

## 8.2 Hot Spots and Skew Relief

When a single key is extremely hot (all reads/writes go to it):

- **Simple technique:** Add a random number prefix/suffix to the key.
    - E.g., instead of `celebrity_post_123`, use `1_celebrity_post_123`, `2_celebrity_post_123`, ..., `100_celebrity_post_123`.
    - Writes spread across 100 keys (and thus 100 nodes).
    - **Trade-off:** Reads must now query all 100 keys and combine results.

---

## 8.3 Strategies for Rebalancing

### Strategy 1: Hash Mod N (Avoid!)

- Problem: Adding/removing one node causes almost all keys to be rehashed to different servers.
- **Excessively expensive** → not used in practice.

---

### Strategy 2: Fixed Number of Partitions

- Create **many more partitions than nodes** (e.g., 1000 partitions for 10 nodes → 100 partitions/node).
- When a new node joins: the new node **steals a few partitions from every existing node** until balanced.

```
4 nodes, 20 partitions (5 each):
Node 1: P1, P2, P3, P4, P5
Node 2: P6, P7, P8, P9, P10
Node 3: P11, P12, P13, P14, P15
Node 4: P16, P17, P18, P19, P20

Add Node 5: steals 1 partition from each:
Node 5: P5, P10, P15, P20
```

**Key properties:**

- The **number of partitions stays fixed** — only partition-to-node assignments change.
- While transfer is in progress, the **old assignment is used** for reads/writes.
- Partition boundaries also stay fixed — the assignment of keys to partitions doesn't change.

**Challenge:** Choosing the right number of partitions upfront is hard if total dataset size is variable.

---

### Strategy 3: Dynamic Partitioning

- Partitions **split** when they exceed a configured size → two partitions each with half the data.
- Partitions **merge** when they shrink below a threshold.

**Benefits:**

- Number of partitions adapts to data volume.
- No over/under-provisioning.

**Drawback:**

- **Cold start:** A new (empty) database starts with one partition → all writes go to one node until split happens.

Works for both key-range and hash-partitioned data.

---

### Strategy 4: Proportional to Number of Nodes

- Fixed number of **partitions per node** (e.g., 10 partitions per node).
- When a node is added: it **randomly selects existing partitions to split** and takes half of each.

**Behaviour:**

- More nodes → more (but smaller) partitions.
- Same nodes → partition size grows proportionally to dataset size.
- Averaging over many partitions, new node takes a **fair share of load**.

**Requirement:** Must use hash-based partitioning (to allow random partition boundary selection).

---

### Rebalancing Strategies Comparison

|Strategy|Partitions|Boundary Assignment|Best For|
|---|---|---|---|
|Hash Mod N|Dynamic|Automatic|Avoid — too disruptive|
|Fixed Count|Fixed|Fixed boundaries|Predictable datasets|
|Dynamic|Dynamic|Dynamic boundaries|Variable datasets|
|Proportional|Per-node|Random (hash)|Elastic scaling|

---

## 8.4 Request Routing

**Problem:** With partitions distributed across changing nodes, how does a client know **which node to contact** for a given key?

This is the **service discovery problem**.

### Approach 1: Contact Any Node (Round-Robin Load Balancer)

- Client contacts any node via a round-robin load balancer.
- If that node owns the partition → it handles the request directly.
- If not → it **forwards** the request to the correct node and passes the reply back.
- Simple for clients, but adds a hop on misses.

### Approach 2: Routing Tier

- All requests go through a **dedicated routing tier** first.
- The routing tier knows which partition is on which node and forwards accordingly.
- The routing tier is a **partition-aware load balancer** — it handles no actual data requests.

### Approach 3: Smart Clients

- Clients are **aware of the partitioning scheme** and partition-to-node mapping.
- Clients connect directly to the correct node — no intermediary.
- More complex client logic; requires clients to stay updated on cluster changes.

### Approach 4: ZooKeeper (Coordination Service)

ZooKeeper is a **distributed coordination service** used by many real-world systems.

**How ZooKeeper is used for routing:**

1. **Each node registers itself** in ZooKeeper with its partition assignment.
2. ZooKeeper maintains the **authoritative partition-to-node mapping**.
3. The routing tier or smart clients **subscribe to ZooKeeper** for updates.
4. When partitions are reassigned (rebalancing, node join/leave), ZooKeeper **notifies** subscribers.

**Real-world usage:**

- **HBase, SolrCloud, Kafka:** Use ZooKeeper for partition tracking.
    - Note: Newer Kafka versions replaced ZooKeeper with **KRaft** (Kafka Raft).
- **LinkedIn's Espresso:** Uses Helix (which uses ZooKeeper) for cluster management.
- **Cassandra and Riak:** Use a **gossip protocol** (peer-to-peer state dissemination) instead of ZooKeeper.

---

## 🧠 Q&A Bank — Section 8

**Q1. What are the three minimum requirements for any rebalancing operation?**

> **A:** (1) Load should be fairly distributed after rebalancing, (2) the database must continue to accept reads/writes during rebalancing, and (3) no more data than necessary should be moved.

**Q2. Why is "hash mod N" a bad rebalancing strategy?**

> **A:** When N changes by even 1 (adding or removing one server), almost all keys get a different `hash(key) mod N` result, requiring almost all data to be remapped and moved. This is excessively disruptive and expensive.

**Q3. In fixed-number-of-partitions rebalancing, what changes when a node joins?**

> **A:** Only the assignment of partitions to nodes changes — the new node takes some partitions from existing nodes. The number of partitions, partition boundaries, and key-to-partition mappings all remain unchanged.

**Q4. What is the cold start problem in dynamic partitioning?**

> **A:** A new empty database starts with a single partition. All writes go to the one node that owns that partition until the partition grows large enough to be split. This means the other nodes are idle at startup.

**Q5. In proportional-to-nodes rebalancing, what happens when a new node is added?**

> **A:** The new node randomly selects a fixed number of existing partitions to split and takes ownership of one half of each split partition, while the other half stays in place.

**Q6. What is ZooKeeper and how does it help with request routing?**

> **A:** ZooKeeper is a distributed coordination service. It maintains the authoritative mapping of partitions to nodes. Routing tiers and smart clients subscribe to ZooKeeper; when the cluster changes, ZooKeeper notifies them to update their routing tables.

**Q7. What is the "gossip protocol" used by Cassandra instead of ZooKeeper?**

> **A:** In a gossip protocol, nodes periodically exchange cluster state information with random other nodes (like gossip spreading through a group). Over time, all nodes converge to the same view of cluster topology without a centralized coordinator.

**Q8. What is the difference between Approach 2 (Routing Tier) and Approach 3 (Smart Client)?**

> **A:** With a Routing Tier, a dedicated middleware layer knows partition assignments and forwards requests; clients are dumb. With Smart Clients, the intelligence is in the client itself — clients know the partition map and connect directly to the right node, eliminating the routing tier.

# 9. Replication: Leader-Based, Multi-Leader, Leaderless

## 9.1 What is Replication and Why?

> **Replication** = keeping a copy of the same data on multiple machines connected via a network.

**Three reasons to replicate:**

1. **Reduce latency:** Keep data geographically close to users.
2. **Increase availability:** System continues working even if some parts fail.
3. **Increase read throughput:** Scale out read queries across multiple machines.

**Key challenge:** Every write must be processed by every replica. Managing changes in a replicated system is the core difficulty.

**Design decisions:**

- How many leaders? (0, 1, or multiple)
- Synchronous or asynchronous propagation?
- How to handle failed replicas?
- How to handle leader failure?

---

## 9.2 Leader-Based (Single-Leader) Replication

Also called **master-slave** or **leader-follower** replication.

### Architecture

```
                    Client (Write)
                         │
                  ┌──────▼──────┐
                  │   LEADER    │  ← Accepts all writes
                  │  (Primary)  │
                  └──────┬──────┘
                         │ Replication Log
               ┌─────────┴──────────┐
               │                    │
        ┌──────▼──────┐    ┌────────▼──────┐
        │  FOLLOWER 1 │    │  FOLLOWER 2   │
        │ (Read Only) │    │ (Read Only)   │
        └─────────────┘    └───────────────┘
              ↑                    ↑
         Client (Read)       Client (Read)
```

### Leader (Primary)

- Accepts **both reads and writes**.
- Writes new data to local storage.
- Sends **replication log** to all followers.

### Followers (Read Replicas)

- Accept **reads only**.
- Receive replication log from leader.
- Apply writes in the **same order** as the leader.

**Used in:** PostgreSQL, MySQL, MongoDB, Kafka, RabbitMQ.

---

### 9.2.1 Synchronous vs. Asynchronous Replication

#### Synchronous Replication

```
Client  →  Write to Leader  →  Leader writes locally
                            →  Leader sends log to Follower
                            →  Wait for Follower ACK
                            →  Report success to Client
```

- **Guarantee:** Follower is guaranteed to have the latest data.
- **Disadvantage:** If follower is slow/unavailable → write is **blocked** until follower responds.
- If leader fails with sync follower → follower has up-to-date data for failover.

#### Asynchronous Replication

```
Client  →  Write to Leader  →  Leader writes locally
                            →  Report success to Client immediately
                            →  (Then sends log to Follower)
```

- **Advantage:** Leader doesn't wait → better write performance and availability.
- **Disadvantage:** If leader fails before replication completes → data can be **lost**.

**Semi-synchronous:** In practice, one follower is synchronous and others asynchronous. If the synchronous follower becomes slow, switch another follower to synchronous (ensuring at least one up-to-date follower always exists).

---

### 9.2.2 Implementation of Replication Logs

Four methods:

#### 1. Statement-Based Replication

- Leader logs every write **SQL statement** and sends to followers.
- Followers execute the same SQL.
- **Problem:** Non-deterministic functions (`NOW()`, `RAND()`) produce different results on each replica.

#### 2. Write-Ahead Log (WAL) Shipping

- The WAL is an **append-only sequence of bytes** containing all writes.
- Leader writes WAL to disk and sends it to followers.
- Followers rebuild the exact same data structures from the WAL.
- **Problem:** WAL is tightly coupled to storage engine internals — upgrading storage engine can break replication.

#### 3. Logical Log (Change Data Capture — CDC)

- Log is a **sequence of records describing writes at row granularity** (not storage engine internals).
- Decoupled from storage engine — replicas can run different versions or engines.
- Easier to parse for external applications (event streaming, data pipelines).
- This is also called a **logical log**.

#### 4. Trigger-Based Replication

- A **trigger** fires when a write transaction occurs.
- Custom application code logs the change.
- An external process reads the log and applies changes to another system.
- **Advantage:** Very flexible — can replicate a subset of data, or apply transformations.
- **Disadvantage:** Higher overhead and more room for bugs.

---

### 9.2.3 Handling Failures

#### Follower Failure: Catch-Up Recovery

- Followers keep a **local log** of all changes received from the leader.
- If a follower crashes: on restart, it reads its log to find the **last processed transaction**.
- It reconnects to the leader and requests **all changes that occurred while it was down**.
- Once caught up → resumes normal replication.

#### Leader Failure: Failover

- One follower must be **promoted to new leader**.
- Clients must be reconfigured to send writes to the new leader.
- Other followers must start replicating from the new leader.

**Automatic Failover Steps:**

1. **Detecting leader failure:** Often done by timeout — if leader hasn't responded in N seconds, assume it's dead.
2. **Choosing a new leader:** Typically the follower with the **most up-to-date data** (fewest missing writes). Requires consensus among remaining nodes.
3. **Reconfiguring the system:** Updates routing so clients and followers connect to new leader.

**Failover risks:**

- If using async replication: the new leader may be **missing some writes** from the old leader → old leader's writes are discarded.
- **Split brain:** If the old leader comes back, two nodes might both believe they are the leader — can cause data corruption.

---

### 9.2.4 Replication Lag

- With async followers, reads from followers may return **outdated (stale) information**.
- Running the same query on the leader and a follower at the same time may give different results.

**Eventual Consistency:** If no new writes occur, followers will **eventually catch up** to the leader. The delay is called **replication lag**.

**Replication Lag Problems and Solutions:**

|Problem|Description|Solution|
|---|---|---|
|**Reading your own writes**|User writes data, then reads it back but gets old value|Read your own writes from the leader; for others, use followers|
|**Monotonic reads**|User sees data disappear (reads from different replicas out of order)|Always route same user to same replica|
|**Consistent prefix reads**|Reads see responses before questions|Ensure causally related writes go to same partition|
|**Replication lag too large**|Followers fall far behind|Monitor lag; prevent reads from lagging replicas|

---

## 9.3 Multi-Leader Replication

**Problem with single-leader:** The leader is a bottleneck. If the network between a user and the leader fails → no writes possible.

**Solution:** Allow **multiple nodes to accept writes** simultaneously.

Also known as **master/master** or **active/active** replication. Each leader simultaneously acts as a **follower to the other leaders**.

### Use Cases

|Use Case|How Multi-Leader Helps|
|---|---|
|**Multi-datacenter**|One leader per datacenter. Intra-datacenter: normal leader-follower. Inter-datacenter: leaders sync with each other asynchronously.|
|**Clients with offline operation**|Each device has its own local DB (acts as a leader). Syncs when online. (e.g., Google Docs offline, phone contacts).|
|**Collaborative editing**|Multiple users edit same document simultaneously — each edit is locally committed and propagated to others.|

### Multi-Leader vs. Single-Leader

|Aspect|Single-Leader|Multi-Leader|
|---|---|---|
|**Write latency**|High (writes must go over internet to central leader)|Low (write to local datacenter leader)|
|**Datacenter outage**|Failover needed to promote new leader|Other datacenters continue independently|
|**Network outage**|Writes blocked (synchronous link goes down)|Writes processed locally; catch up when network restores|
|**Conflict risk**|None|High — concurrent writes to different leaders can conflict|

### Multi-Leader Topologies

Three ways leaders connect to each other:

```
Circular: A → B → C → A
Star:     A → B, A → C, A → D (A is root)
All-to-All: A ↔ B ↔ C, A ↔ C
```

|Topology|Description|Fault Tolerance|
|---|---|---|
|**Circular**|Each node receives from one, sends to one|Poor — one broken node stops propagation|
|**Star**|One root node forwards to all others|Poor — root is a single point of failure|
|**All-to-all**|Every node connects to every other node|Best — multiple paths, no single point of failure|

### Handling Write Conflicts

When two leaders accept different writes to the same data simultaneously → **write conflict**.

**Solutions:**

1. **Conflict Avoidance:** Route all writes for a given record to the same leader (closest to the user). Works well but limits geographic distribution.
    
2. **Converge to Consistent State (Last Write Wins / LWW):** Assign each write a unique ID (timestamp, UUID, etc.). The write with the **highest ID wins**; others are discarded. Risk: data loss.
    
3. **Custom Conflict Resolution Logic:** Write application code that runs either on write or on read to resolve conflicts using business logic.
    

---

## 9.4 Leaderless Replication

**Philosophy:** No node is "special" — **all nodes are peers**. All nodes accept both reads and writes.

**Used by:** Amazon Dynamo, Apache Cassandra, Riak.

### Write Mechanics

```
Client ──→ Write request to ALL replicas in parallel
        ←── ACK from Replica 1  ✓
        ←── ACK from Replica 2  ✓
        ←── (Replica 3 down — no ACK) ✗

Write considered SUCCESS if ≥ k ACKs received (quorum)
```

**Write quorum:** The minimum number of replicas that must acknowledge a write for it to be considered successful.

### Read Mechanics

```
Client ──→ Read request to ALL replicas in parallel
        ←── Response from Replica 1: value v7
        ←── Response from Replica 2: value v7
        ←── Response from Replica 3: value v6 (stale)

Read considered SUCCESS when ≥ k reads agree (quorum)
Latest version determined by version number
```

### Quorum Formula

For a system with **n replicas**, if:

- **w** = write quorum (minimum ACKs for write success)
- **r** = read quorum (minimum matching reads for read success)

Then for consistency: $w + r > n$

This guarantees that any read will overlap with at least one write, ensuring you always read the latest value.

**Common choice:** $n$ is odd (e.g., 3 or 5), and:

$$\boxed{w = r = \frac{n + 1}{2}}$$

**Example:** n = 3, w = 2, r = 2. Any read of 2 will overlap with any write of 2 (since 2 + 2 = 4 > 3).

|Variable|Meaning|
|---|---|
|$n$|Total number of replicas|
|$w$|Write quorum — min ACKs needed for write success|
|$r$|Read quorum — min agreeing reads for read success|

### Handling Stale Values

**Problem:** A node that was down missed writes. When it comes back, it has stale data.

**Two repair mechanisms:**

#### Read Repair

- During a read, the client sees different versions from different replicas.
- It detects the stale replica (using version numbers) and **writes the latest value back** to it.
- Works well for data that is frequently read.

#### Anti-Entropy Process

- A **background process** constantly compares data between replicas.
- Finds differences and copies missing data between replicas proactively.
- Often uses **gossip-like** protocols where replicas periodically exchange state.

### Detecting Concurrent Writes

**Problem:** Multiple clients write to the same key simultaneously. Different replicas receive writes in different orders → permanent inconsistency.

**Version-based algorithm:**

1. Server maintains a **version number** for every key, incremented on every write.
2. **Read before write:** Client reads a key → gets all values + latest version number.
3. **Write with version:** Client writes with the version number from its prior read + merges all values received.
4. **Server conflict resolution:** Server overwrites all values with version ≤ incoming write's version number; keeps values with higher version numbers (those are concurrent).

**Last Write Wins (LWW):** Use a timestamp as the version — highest timestamp wins. Risk: clock drift across nodes can cause the wrong write to win.

**Version Vectors:** For multi-replica systems, maintain a **version number per replica** (version vector) to track which values are concurrent and which supersede others.

### Multi-Datacenter Leaderless Replication

- Leaderless replication is well-suited for multi-datacenter operation.
- `n` includes nodes from all datacenters.
- Client waits only for quorum from **local datacenter** nodes.
- Cross-datacenter replication happens **asynchronously**.

---

## 🧠 Q&A Bank — Section 9

**Q1. What is the leader's role in single-leader replication?**

> **A:** The leader accepts all write requests, writes to its local storage, and sends a replication log to all followers. It also accepts read requests.

**Q2. What is the difference between synchronous and asynchronous replication?**

> **A:** In synchronous replication, the leader waits for the follower to acknowledge receipt before reporting success to the client — guarantees the follower has the latest data but blocks on slow followers. In asynchronous replication, the leader reports success immediately and sends the log later — better performance but risk of data loss if leader fails before replication.

**Q3. What is the advantage of logical log (CDC) replication over WAL shipping?**

> **A:** Logical logs describe row-level changes rather than storage engine internals, making them decoupled from the storage engine. This allows replicas to run different versions or storage engines, and makes the log easier to parse for external tools.

**Q4. In automatic failover, how is a new leader chosen?**

> **A:** The follower with the **most up-to-date data** (closest to the leader at the time of failure, i.e., fewest missing writes) is typically chosen. This requires a consensus mechanism among the remaining nodes.

**Q5. What is "split brain" in leader-based replication?**

> **A:** Split brain occurs when the original leader recovers after failover and two nodes simultaneously believe they are the leader. Both may accept writes, leading to conflicting data and potential corruption.

**Q6. What is eventual consistency in the context of replication lag?**

> **A:** If writes stop, followers will eventually catch up to the leader, making all replicas consistent. The system is "eventually consistent" — it doesn't guarantee instantaneous consistency, but guarantees eventual convergence.

**Q7. What is the quorum condition for leaderless replication to ensure consistency?**

> **A:** $w + r > n$, where $w$ is the write quorum, $r$ is the read quorum, and $n$ is the total number of replicas. This ensures that every read overlaps with at least one write.

**Q8. What is the "read repair" technique in leaderless replication?**

> **A:** When a client reads from multiple replicas and detects a stale value on one, it writes the latest value back to the stale replica, bringing it up-to-date without background intervention.

**Q9. What is the main advantage of multi-leader replication over single-leader for multi-datacenter setups?**

> **A:** Each datacenter has its own leader, so writes can be processed locally with low latency. Inter-datacenter replication happens asynchronously, so network issues between datacenters don't block writes. Single-leader requires all writes to go to one datacenter, adding latency.

**Q10. Explain the "last write wins" strategy. What is its main risk?**

> **A:** LWW assigns a timestamp to each write and keeps the write with the highest timestamp, discarding concurrent writes with lower timestamps. The main risk is **clock drift** — clocks across nodes are never perfectly synchronized, so the "winner" may not actually be the most recent write, potentially discarding data that should have been kept.

**Q11. What is a version vector and why is it needed?**

> **A:** A version vector maintains a separate version number per replica for each key. When multiple replicas are involved, a single version number cannot capture which replica made which change. A version vector tracks the state from every replica, allowing the system to correctly identify concurrent writes vs. causally ordered writes.

# 10. Consistency Models

## 10.1 What is Consistency?

> **Consistency** refers to the agreement of values in different copies (replicas) of the same data item in a replicated distributed system.

**When is consistency violated?**

- Network partitions delay write propagation.
- Different time taken to write to different replicas.

> **Consistency Model** = A contract between a distributed data store and the processes using it. The processes agree to follow certain rules; in return, the store promises to behave correctly.

**Trade-off:** Strong consistency requires synchronization mechanisms that slow down writes. Weaker consistency models improve performance but allow temporary divergence.

---

## 10.2 Eventual Consistency

> **Eventual Consistency:** The system guarantees that, if no new writes occur, all replicas will **eventually converge to the same value** — but doesn't say when.

- Most replicated databases provide this as the baseline.
- A **very weak guarantee** — says nothing about when convergence happens.
- Edge cases only become apparent under **high concurrency or failures**.
- Writes continue to be accepted regardless of replica convergence state.

---

## 10.3 Sequential Consistency (Lamport)

> **Sequential Consistency:** All processes see the **same global order of all memory access operations**, even if the actual timing of operations differs across processors.

**Key rules:**

- The order of memory operations can be any interleaving of each process's own program order.
- But ALL processes must see the **same interleaving**.
- The actual wall-clock order doesn't matter — only that everyone agrees on one order.

**Formal definition (Lamport):** "A shared-memory system is sequentially consistent if all processes see the same order of all memory access operations."

**Non-deterministic:** Multiple executions of the same distributed program can lead to different valid orderings — it's not deterministic.

**Analogy:** Imagine all processors sharing one logical memory. There's a "switch" that connects processors to memory one at a time. The switch can connect in any order, but all processors see the same sequence of operations.

**Example:** If Process 1 writes A=1 then A=2, and Process 2 writes A=3, a sequentially consistent system ensures all processes see the same order of these three writes, though the order itself (1,2,3 vs. 1,3,2 vs. 3,1,2, etc.) can vary.

---

## 10.4 Causal Consistency

> **Causal Consistency:** All processes see causally related operations in the same (correct) order. Causally unrelated operations can be seen in different orders by different processes.

**Two operations are causally related if one can influence the other.**

- **Relaxation of sequential consistency:** Instead of requiring all operations to be ordered globally, only causally related ones need to be ordered.
- Better concurrency than sequential consistency.

**Example:**

- A: writes post "Hello"
- B: reads A's post, then writes reply "Hi A!"
- B's reply is causally dependent on A's post.
- Any process that sees B's reply must have also seen A's post first.
- But two unrelated posts by C and D can be seen in any order.

---

## 10.5 PRAM Consistency (Pipelined RAM)

> **PRAM Consistency:** All writes performed by a **single process** are seen by all other processes in the **same order they were issued**, as if they were in a pipeline.

- Writes from **different processes** may be seen in **different orders** by different processes.
- Weaker than causal consistency — only imposes ordering per-process, not across causally related operations.

---

## 10.6 Strict Consistency (Linearizability / Strong Consistency)

> **Strict Consistency:** A read operation always returns the value written by the **most recent write to that address**, regardless of where the reading and writing processes are located.

Also called:

- **Atomic consistency**
- **Strong consistency**
- **Linearizability**
- **Immediate consistency**
- **External consistency**

**Key property:** Like sequential consistency, but the execution order must match the **real wall-clock order** in which operations were issued.

**Deterministic:** If each process is deterministic, the entire distributed program is deterministic.

---

## 10.7 Linearizability (Deep Dive)

> **Linearizability** = making a system appear as if there were **only one copy of the data** and all operations on it are **atomic**.

**Core guarantee:** As soon as one client successfully completes a write, **all clients reading from any replica** must be able to see that value.

```
Timeline:
Client A:   ──── Write(x=2) ──────────────────────────────────
Client B:   ──────────────── Read(x) ──── Read(x) ──────────
                             ↑ either 1 or 2      ↑ must be 2

After the write completes, all subsequent reads MUST return 2.
```

**Difference from Sequential Consistency:**

- Sequential: all processes see the same order, but that order might not match wall clock time.
- Linearizable: the order must respect actual real-time ordering of operations.

### Compare-and-Set (CAS) Operation

Linearizability requires an additional operation called CAS:

$$\text{cas}(x, v_{old}, v_{new}) \Rightarrow r$$

|Symbol|Meaning|
|---|---|
|$x$|The register (variable) to operate on|
|$v_{old}$|The expected current value of $x$|
|$v_{new}$|The new value to set $x$ to|
|$r$|Result: OK (if $x = v_{old}$ and set succeeded) or ERROR (if $x \neq v_{old}$)|

**Semantics:** If the current value of $x$ equals $v_{old}$, atomically set it to $v_{new}$. Otherwise, leave $x$ unchanged and return error.

CAS is used for **concurrency control** — e.g., implementing locks, counters, and atomic updates.

### Which Replication Models are Linearizable?

|Replication Model|Linearizable?|
|---|---|
|Single-leader (reads from leader or sync followers)|**Potentially yes**|
|Consensus algorithms (ZooKeeper)|**Yes**|
|Multi-leader|**No** — concurrent writes to different leaders|
|Leaderless|**Probably no** — quorum reads/writes can return stale data|

---

## 10.8 Consistency Models Hierarchy

```
          STRICTEST
               │
    Strict Consistency
    (Linearizability)
               │
    Sequential Consistency
               │
    Causal Consistency
               │
    PRAM Consistency
               │
    Eventual Consistency
               │
          WEAKEST
```

Weaker models = better performance/availability. Stronger models = easier to reason about but slower.

## 🧠 Q&A Bank — Section 10

**Q1. What is the key difference between eventual consistency and strict consistency?**

> **A:** Strict consistency guarantees that every read sees the most recent write, regardless of location. Eventual consistency only guarantees that, if writes stop, replicas will eventually converge — it doesn't guarantee when, and stale reads are possible in the meantime.

**Q2. What does sequential consistency guarantee that eventual consistency doesn't?**

> **A:** Sequential consistency guarantees that all processes see the same global order of all memory operations. Eventual consistency provides no ordering guarantees — different processes can see operations in different orders indefinitely.

**Q3. Give an example where causal consistency is violated.**

> **A:** User A posts "Hello." User B reads A's post and replies "Hi A!" User C sees B's reply but has not yet seen A's original post — this violates causal consistency, since B's reply is causally dependent on A's post (seeing the reply without the cause breaks causal order).

**Q4. What is PRAM consistency and how does it differ from causal consistency?**

> **A:** PRAM ensures all writes by a single process are seen in the same order by all other processes. Causal consistency also enforces ordering for causally related writes from different processes. PRAM is weaker — it doesn't track causal dependencies across processes.

**Q5. Define linearizability in simple terms.**

> **A:** Linearizability makes a distributed system appear as if there is a single copy of data, and all operations are atomic. Once a write is acknowledged, all subsequent reads from anywhere must return the new value.

**Q6. What is the Compare-and-Set (CAS) operation?**

> **A:** CAS atomically compares a register's current value to an expected value, and if they match, sets it to a new value. If they don't match, the register is left unchanged and an error is returned. It is used to implement atomic updates and concurrency control.

**Q7. Why is multi-leader replication not linearizable?**

> **A:** Because two leaders can simultaneously accept conflicting writes. There is no single point of truth — a read from one node may return a different value than a read from another, violating the "single copy" illusion of linearizability.

**Q8. Which consistency model would you use for a bank transfer system? Why?**

> **A:** Linearizability (strict consistency), because financial operations require that the most recent balance is always visible to all clients. Any stale read could allow overdrafts or double-spending.

**Q9. Can a leaderless replication system be linearizable? Under what conditions?**

> **A:** Probably not in general — quorum reads/writes can return stale data. To approach linearizability, read repair and careful quorum configuration (w + r > n) are required, but even then, edge cases with concurrent writes can break the guarantee.

**Q10. In sequential consistency, can two clients observe different orderings of operations?**

> **A:** No. Sequential consistency requires that ALL processes see the same single global order of operations. If two clients see different orderings, the system is not sequentially consistent.

# 11. Distributed Transactions and Two-Phase Commit

## 11.1 What is a Transaction?

> A **transaction** is an operation composed of a number of discrete steps, where **all steps must complete** for the transaction to be committed — otherwise the transaction is aborted and the system state reverts.

**Example:** Buying a house involves multiple steps (offer, contract, payment, deed transfer). If any step fails, everything must be rolled back.

### Transaction Primitives

|Primitive|Action|
|---|---|
|`BEGIN TRANSACTION`|Mark the start|
|`END TRANSACTION`|Mark the end; attempt to commit|
|`ABORT TRANSACTION`|Kill it; restore old values|
|`READ/WRITE`|Access data (must be reversible if aborted)|

---

## 11.2 ACID Properties

Every transaction must satisfy ACID:

### A — Atomic

- The transaction happens as a **single indivisible action**.
- Others see either all changes or none — **no partial results visible**.
- "All or nothing."

### C — Consistent

- If the system has **invariants** (rules that must always hold), they must hold **after the transaction**.
- Example: The total money across all bank accounts must be the same before and after a "transfer funds" transaction.

### I — Isolated (Serializable)

- If transactions run concurrently, the final result must be the same as if they had **executed in some serial order**.
- Transactions don't interfere with each other.

### D — Durable

- Once a transaction commits, results are **permanent**.
- **No failure after a commit** will cause results to revert.
- Typically achieved by writing to non-volatile storage (WAL, disk) before acknowledging commit.

---

## 11.3 Types of Transactions

### Nested Transactions

- A top-level transaction creates **sub-transactions**.
- **Problem:** Sub-transactions may commit (making results durable) but the **parent transaction may abort** — what happens to the sub-transactions?
- **Solution — Private Workspace:** Each sub-transaction works on a private copy of the objects it touches. On commit, the private copy replaces the parent's copy. If parent aborts, private copies are discarded.

### Distributed Transactions

> A **distributed transaction** is a set of operations on data performed across **two or more data repositories** on **different systems**, connected via a network.

**Challenge:** Handle machine, software, and network failures while preserving ACID.

**Two possible outcomes:**

1. All operations successfully complete → **COMMIT**.
2. Any failure → all operations rolled back → **ABORT** (none of the work persists).

This "all or nothing" across distributed nodes is extremely difficult to guarantee.

---

## 11.4 Distributed Transaction System Architecture

The slide shows this architecture:

```
┌──────────────────────────────────────────────────────┐
│                    Application                        │
└────────────────────────┬─────────────────────────────┘
                         │
┌────────────────────────▼─────────────────────────────┐
│             Transaction Coordinator                   │
│  (Orchestrates the entire distributed transaction)    │
└──────────┬───────────────────┬────────────────────────┘
           │                   │
  ┌────────▼──────┐   ┌────────▼──────┐
  │ TR Manager    │   │ TR Manager    │  ...more nodes...
  │   Node 1      │   │   Node 2      │
  └───────────────┘   └───────────────┘
```

**Transaction Manager (per node):**

- Maintains a **local log** for recovery.
- Coordinates concurrent execution of sub-transactions at that node.
- Responsible for sub-transactions on that node.
- Performs `PREPARE`, `COMMIT`, and `ABORT` for sub-transactions.

**Transaction Coordinator:**

- Nominated periodically from local transaction managers.
- **Starts** the execution.
- **Distributes** sub-transactions to appropriate nodes.
- **Coordinates termination** — decides commit or abort at all sites.

---

## 11.5 Concurrency Control

Requirements for concurrency control in distributed transactions:

|Requirement|Description|
|---|---|
|**Resilient to failures**|Must handle site and communication link failures|
|**Allow parallelism**|Parallelism improves performance|
|**Optimal cost**|Minimize communication overhead|
|**Constrain atomic actions**|Atomic actions must behave as if executed serially|

---

## 11.6 Two-Phase Commit Protocol (2PC)

The **2PC protocol** ensures atomicity across multiple sites in a distributed transaction. It is the most widely used commit protocol.

### Core Principle

> 2PC ensures that a transaction is **committed at ALL sites** or **aborted at ALL sites** — never a partial state where some sites committed and others aborted.

### Phase 1: Prepare (Voting)

**Goal:** Ask all nodes if they are ready to commit.

```
                 COORDINATOR
                      │
          ┌───────────────────────┐
          │     PREPARE T         │ → sent to all nodes
          └───────────────────────┘
```

**Coordinator actions:**

1. Write `PREPARE T` to its log.
2. Send `PREPARE T` message to all participant nodes.
3. Wait for responses.

**Each Participant node:**

1. Receives `PREPARE T`.
2. Works on its component of transaction T.
3. **If ready to commit:**
    - Enters **pre-committed** state (cannot abort independently from now on).
    - Writes `READY T` to local log and flushes to disk.
    - Sends `READY T` to coordinator.
    - **Holds all locks** (must not release until phase 2).
4. **If not ready to commit:**
    - Writes `DON'T COMMIT T` to local log.
    - Sends `DON'T COMMIT T` to coordinator.

**Key rule:** Once a node sends `READY T`, it is **locked in** — it cannot abort on its own. It must wait for the coordinator's final decision.

---

### Phase 2: Commit or Abort

**Coordinator decision:**

```
IF all nodes sent READY T:
    Write COMMIT T to log
    Send COMMIT T to all nodes

IF any node sent DON'T COMMIT T:
    Write ABORT T to log
    Send ABORT T to all nodes
```

**Each Participant on receiving COMMIT T:**

1. Commits its component of T.
2. Releases all locks.
3. Writes `COMMIT T` to local log.
4. Sends `DONE` to coordinator.

**Each Participant on receiving ABORT T:**

1. Rolls back all changes.
2. Releases all locks.
3. Writes `ABORT T` to local log.
4. Sends `DONE` to coordinator.

**Coordinator:** Waits for all `DONE` messages, then clears all state.

---

### Visual Summary of 2PC

```
PHASE 1 (PREPARE):
Coordinator ──PREPARE T──→ Node A
Coordinator ──PREPARE T──→ Node B
Coordinator ──PREPARE T──→ Node C

Node A ──READY T──→ Coordinator ✓
Node B ──READY T──→ Coordinator ✓
Node C ──READY T──→ Coordinator ✓

PHASE 2 (COMMIT — all agreed):
Coordinator ──COMMIT T──→ Node A → commits, sends DONE
Coordinator ──COMMIT T──→ Node B → commits, sends DONE
Coordinator ──COMMIT T──→ Node C → commits, sends DONE

─────────────────────────────────

PHASE 2 (ABORT — one disagreed):
Node B sent DON'T COMMIT T in Phase 1
Coordinator ──ABORT T──→ Node A → rolls back, sends DONE
Coordinator ──ABORT T──→ Node B → rolls back, sends DONE
Coordinator ──ABORT T──→ Node C → rolls back, sends DONE
```

---

### 2PC Failures and Problems

|Failure Scenario|Impact|
|---|---|
|**Participant fails before sending READY**|Coordinator doesn't get a response → times out → sends ABORT to all|
|**Participant fails after sending READY**|Participant is in pre-committed state, holds locks, waiting for Phase 2. Coordinator must wait until participant recovers (or times out). **Blocking problem.**|
|**Coordinator fails after Phase 1**|Participants are stuck in pre-committed state, holding locks, unable to proceed. **This is 2PC's main weakness — it is a blocking protocol.**|
|**Network partition**|Same as coordinator failure from participants' perspective|

**2PC is a blocking protocol:** If the coordinator fails at a critical moment, participants can be stuck waiting indefinitely, holding locks.

**3PC (Three-Phase Commit)** was designed to address this by adding a pre-commit phase, making it non-blocking — but it adds complexity and is rarely used in practice.

## 🧠 Q&A Bank — Section 11

**Q1. What are the four ACID properties? Briefly explain each.**

> **A:** **Atomic** — all or nothing; **Consistent** — invariants hold before and after; **Isolated** — concurrent transactions don't interfere (final result is as if serial); **Durable** — committed results survive failures.

**Q2. What is the difference between a nested transaction and a distributed transaction?**

> **A:** A nested transaction involves sub-transactions within a single transaction on one or multiple nodes, where sub-transaction commits are provisional until the parent commits. A distributed transaction involves a set of operations across multiple data repositories on different systems, requiring coordination across nodes.

**Q3. Why is it a problem if a sub-transaction commits but the parent later aborts?**

> **A:** The sub-transaction has made its changes durable, but the parent's abort means those changes should be rolled back too. The private workspace solution addresses this by not making sub-transaction changes permanent until the parent commits.

**Q4. What are the two phases of 2PC?**

> **A:** **Phase 1 (Prepare/Voting):** Coordinator asks all participants if they can commit. Participants respond with READY or DON'T COMMIT. **Phase 2 (Commit/Abort):** Based on responses, coordinator sends COMMIT (if all ready) or ABORT (if any not ready) to all participants.

**Q5. What state does a participant enter after sending READY T? What can it NOT do?**

> **A:** The participant enters the **pre-committed** state. It **cannot unilaterally abort** T — it must hold its locks and wait for the coordinator's final COMMIT or ABORT decision.

**Q6. What happens in 2PC if the coordinator fails after Phase 1 but before Phase 2?**

> **A:** All participants that sent READY T are stuck in the pre-committed state, holding locks, and unable to proceed or release resources. They cannot commit or abort on their own. They must wait for the coordinator to recover or for a timeout mechanism — making 2PC a **blocking protocol**.

**Q7. Why is the coordinator required to write to its log before sending Phase 2 messages?**

> **A:** If the coordinator crashes after deciding but before sending, it must be able to recover and re-send the correct decision. Writing to a log before sending ensures the decision survives a crash and can be replicated on recovery.

**Q8. What does the Transaction Coordinator do in a distributed transaction architecture?**

> **A:** It starts transaction execution, distributes sub-transactions to appropriate nodes, and coordinates termination — deciding whether all nodes commit or all nodes abort, and communicating that decision.

**Q9. What is the role of the local Transaction Manager at each node?**

> **A:** It maintains a local log for recovery, coordinates concurrent execution of sub-transactions at its site, performs prepare/commit/abort for sub-transactions, and communicates with the coordinator.

**Q10. How does 2PC ensure atomicity across distributed nodes?**

> **A:** By gathering a vote from all participants in Phase 1, and only committing if ALL vote yes. Even a single ABORT vote causes the entire transaction to be aborted at all nodes. This guarantees no partial commits — the transaction is either committed everywhere or aborted everywhere.

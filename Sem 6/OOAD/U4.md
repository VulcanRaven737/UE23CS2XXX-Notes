# PART 1: STRUCTURAL DESIGN PATTERNS

---

## 1.1 Introduction to Structural Patterns

Structural Design Patterns focus on **organizing classes and objects** to build larger, efficient, and maintainable software structures. They simplify relationships between entities, support code reuse, and help create scalable architectures.

Two recurring themes in structural patterns:

**Theme 1:** These patterns are particularly useful for making **independently developed class libraries work together**. This means you can plug two libraries that were never designed with each other in mind and still make them cooperate.

**Theme 2:** Structural patterns describe ways to **compose objects to realize new functionality**. The key advantage here is that object composition can be changed at **run-time**, which is impossible with static class composition (inheritance is baked in at compile time).

---

## 1.2 Adapter Pattern

### Definition

The Adapter design pattern is a structural pattern used so that **two unrelated interfaces can work together**. The object that joins these two unrelated interfaces is called an **Adapter**.

Think of the adapter as a translator — it translates one interface's language into another's.

### Real-Life Examples

**Mobile Charger:** A phone battery needs only 3 volts to charge, but the wall socket gives 120V (US) or 240V (India). The mobile charger adapts the wall socket's voltage to what the phone understands. The charger is the adapter.

**Card Reader:** A memory card cannot be directly plugged into a laptop. A card reader sits in between — it acts as an adapter between the memory card interface and the laptop's USB/SD slot interface.

### Two Approaches to Implementing the Adapter Pattern

|Approach|Mechanism Used|How it Works|
|---|---|---|
|Class Adapter|Java Inheritance|Extends the source (adaptee) class|
|Object Adapter|Java Composition|Contains an instance of the source object|

Both approaches produce the **same result**. The Object Adapter approach is generally preferred as it is more flexible (you are not locked into a specific class hierarchy).

**Class Adapter:**

```java
// Uses inheritance — SocketAdapter extends Socket (the source)
class SocketAdapter extends Socket implements SocketAdapterIF {
    // ...
}
```

**Object Adapter:**

```java
// Uses composition — SocketAdapter HOLDS a reference to Socket
class SocketAdapter implements SocketAdapterIF {
    private Socket socket; // composition
    // ...
}
```

### Detailed Example: Audio Player

**Scenario:** An `AudioPlayer` device can only play `.mp3` files. You want it to also play `.vlc` and `.mp4` files using an `AdvancedMediaPlayer`.

**Participants and their roles:**

- `MediaPlayer` (interface) — declares `play(audioType, fileName)` method.
- `AdvancedMediaPlayer` (interface) — declares `playVlc()` and `playMp4()` methods.
- `VlcPlayer`, `Mp4Player` — concrete classes implementing `AdvancedMediaPlayer`.
- `MediaAdapter` — the adapter class. It implements `MediaPlayer` AND internally uses `AdvancedMediaPlayer` objects to handle vlc/mp4 requests.
- `AudioPlayer` — concrete class implementing `MediaPlayer`. Plays mp3 natively. For other formats, it creates a `MediaAdapter` and delegates.
- `AdapterPatternDemo` — the client/demo class.

**Flow of control:**

```
AdapterPatternDemo
    → AudioPlayer.play("vlc", "song.vlc")
        → MediaAdapter.play("vlc", "song.vlc")
            → VlcPlayer.playVlc("song.vlc")
```

### When to Use the Adapter Pattern

- When you want to use an existing class, but its interface is not compatible with the rest of your code.
- When you want to create a reusable class that cooperates with classes that don't necessarily have compatible interfaces.
- When you need to use several existing subclasses, but it's impractical to adapt their interface by subclassing every one.

---

### Q&A — Adapter Pattern

**Q1: What problem does the Adapter pattern solve?** A: It allows two incompatible interfaces to work together by introducing an adapter object that translates one interface into another.

**Q2: What is the difference between a Class Adapter and an Object Adapter?** A: A Class Adapter uses Java inheritance and extends the source class. An Object Adapter uses Java composition and holds a reference to the source object. Both achieve the same result, but Object Adapter is preferred as it is more flexible.

**Q3: Name the key participant classes in the Adapter pattern.** A: Target (the interface expected by the client), Adaptee (the existing class with an incompatible interface), Adapter (bridges Target and Adaptee), and Client (uses the Target interface).

**Q4: Give a real-life example of the Adapter pattern.** A: A mobile charger acts as an adapter between the 240V wall socket and the 3V phone battery. A card reader acts as an adapter between a memory card and a laptop.

**Q5: What does the `MediaAdapter` do in the Audio Player example?** A: `MediaAdapter` implements `MediaPlayer` so the `AudioPlayer` can use it. Internally, `MediaAdapter` creates either a `VlcPlayer` or `Mp4Player` based on the audio type and delegates playback to the appropriate `AdvancedMediaPlayer`.

**Q6: Can you use the Adapter pattern with interfaces you don't own (third-party)?** A: Yes! That is actually one of the main use cases — you write an Adapter that wraps the third-party class and exposes the interface your code expects.

## 1.3 Facade Pattern

### Definition

**Facade** (from French, meaning "face" or "front") — in software, a facade is a class that provides a **simplified, unified interface** to a complex subsystem.

A **Facade Design Pattern** is a structural design pattern that simplifies the interface to a library, framework, or any other complex set of classes. When compared to interacting directly with the subsystem, a facade may have limited capabilities, but it only includes what clients actually care about.

**GoF Definition:** "Provide a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use."

### Real-Life Examples

**Desktop Computer:** When you press the power button, you don't care about BIOS initialization, CPU startup, RAM check, OS boot sequence etc. The power button is your facade — a single interaction that triggers many internal complex operations.

**Database Connection in Java:** When you call `dataSource.getConnection()`, internally many things happen: loading the driver, creating or fetching a connection from the pool, updating stats, etc. But from your perspective as a developer, it is a single method call. This is the Facade pattern.

### Key Characteristics

- A facade is appropriate when you have a **complex system** that should be exposed to clients in a **simplified way**.
- Its purpose is to **hide internal complexity** behind a single interface that appears simple from the outside.
- Facade also **decouples** the client code from the details of the subsystems, making it easier to modify the system later without affecting clients.
- You can create **multiple facades** for the same subsystem — there is no limit of one.

### Facade vs Direct Interaction

Without Facade:

```
Client → SubsystemA
Client → SubsystemB
Client → SubsystemC
Client → SubsystemD
```

With Facade:

```
Client → Facade → SubsystemA
                → SubsystemB
                → SubsystemC
                → SubsystemD
```

The client only talks to the Facade. The Facade talks to all the subsystem classes.

### Example: Mobile Shop

**Scenario:** `IMobileShop` interface is implemented by `iPhone`, `Samsung`, `Nokia`. The `ShopKeeper` (facade) class uses these concrete classes to delegate user requests.

```
FacadePatternClient
    → ShopKeeper (Facade)
        → iPhone / Samsung / Nokia
```

### When to Use Facade

- When you want to provide a simple interface to a complex subsystem.
- When there are many dependencies between clients and implementation classes.
- When you want to layer your subsystems — use a facade to define an entry point to each layer.

---

### Q&A — Facade Pattern

**Q1: What does "Facade" mean and what does it represent in software design?** A: Facade means "a deceptive outward appearance." In software, it is a class that presents a simplified interface to a complex subsystem, hiding the internal complexity from the client.

**Q2: What is the main advantage of the Facade pattern?** A: It decouples the client from the complex subsystem, providing a simple unified interface. This also makes the system easier to modify — changes to subsystem internals don't affect the client.

**Q3: Is it possible to have more than one Facade for a subsystem?** A: Yes! You can create any number of facades for a particular complex subsystem. The idea is simply to make the system easier to use for different types of clients.

**Q4: What is the difference between Adapter and Facade patterns?** A: The Adapter pattern makes two incompatible interfaces work together (interface translation). The Facade pattern creates a new, simpler interface for a complex existing system (interface simplification). Adapter doesn't simplify; it translates. Facade doesn't translate; it simplifies.

**Q5: Give a programming example of the Facade pattern.** A: Calling `dataSource.getConnection()` in Java JDBC. Behind the scenes it loads the driver, checks the pool, updates stats — all hidden from the developer calling that single method.

**Q6: Does the Facade pattern break the Open/Closed Principle?** A: No. The facade adds a new layer on top without modifying the existing subsystem classes.

## 1.4 Proxy Pattern

### Definition

The **Proxy Design Pattern** is a structural design pattern that enables you to create a **stand-in or placeholder** for another object. Using a proxy, you may conduct an action either **before or after** the request reaches the original object, thus controlling access to it.

The proxy has the same interface as the original service, so the client can't tell whether it is using the real object or the proxy.

### The Problem Proxy Solves

Imagine you have a **large object** that consumes significant system resources. You don't always need it, but sometimes you do. The naive solution would be lazy initialization inside each client class — but that causes massive code duplication across all clients. The Proxy pattern centralizes this logic.

### Solution

Create a **Proxy class** with the same interface as the original service object. Update your app to pass the proxy to all clients. The proxy:

1. Receives a request from the client.
2. Performs pre-processing (e.g., access control, caching, logging, lazy initialization).
3. Delegates to the real service object.
4. Optionally does post-processing.

### Types of Proxies

**Virtual Proxy:** Controls access to a resource-intensive object, creating it only when needed (lazy initialization). Example: Loading a high-resolution image only when it is actually displayed.

**Protection Proxy:** Controls access based on authorization/permissions. Example: Employee internet access control.

**Remote Proxy:** Provides a local representative for an object in a different address space (e.g., Remote Method Invocation in Java).

**Caching Proxy:** Caches results of expensive operations and returns the cached result for repeated requests.

### Detailed Example: Employee Internet Access

**Step 1:** Create interface `InternetAccess` with method `grantInternetAccess()`.

**Step 2:** Create `EmployeeInternetAccess` implementing `InternetAccess` — this is the real service. It grants access.

**Step 3:** Create `ProxyInternetAccess` implementing `InternetAccess` — this is the proxy. It checks authorization, then delegates to `EmployeeInternetAccess`.

**Step 4:** Client `ProxyPatternClient` interacts only with `ProxyInternetAccess`.

```java
// Client code only interacts with the proxy interface
InternetAccess access = new ProxyInternetAccess("EmployeeID_123");
access.grantInternetAccess();
// ProxyInternetAccess internally creates EmployeeInternetAccess
// and delegates, after checking authorization
```


_As seen in the slide: `Client` uses the `Service` interface. Both `Proxy` and `RealService` implement `Service`. `Proxy` holds a reference to `RealService` and delegates to it._

### When to Use Proxy

- When you need lazy initialization (virtual proxy).
- When you need access control (protection proxy).
- When you need local execution of a remote service (remote proxy).
- When you want to log requests (logging proxy).
- When you want to cache results (caching proxy).

---

### Q&A — Proxy Pattern

**Q1: What is the core idea of the Proxy pattern?** A: A Proxy is a substitute or placeholder for another object. It controls access to the original object, allowing you to perform something before or after the request reaches the original object.

**Q2: What is lazy initialization in the context of the Proxy pattern?** A: Lazy initialization means the heavy, resource-intensive real object is only created when it is actually needed for the first time — not when the proxy itself is created. This saves memory and startup time.

**Q3: What are the four main types of proxies?** A: Virtual Proxy (lazy initialization), Protection Proxy (access control), Remote Proxy (local representative for remote object), Caching Proxy (cache expensive results).

**Q4: How is Proxy different from Adapter?** A: A Proxy provides the same interface as the service it wraps — it doesn't change the interface. An Adapter provides a different interface to make an incompatible class usable. Proxy controls access; Adapter translates interfaces.

**Q5: How is Proxy different from Facade?** A: Proxy wraps a single object and provides the same interface. Facade wraps an entire subsystem and provides a simplified, different interface.

**Q6: What problem does the Protection Proxy solve?** A: It prevents unauthorized clients from accessing the real service object. The proxy checks credentials or permissions before forwarding the request.

## 1.5 Flyweight Pattern

### Definition

The Flyweight design pattern is used when we need to **create a very large number of objects** of a class, and memory is a constraint. Flyweight achieves efficiency by **sharing objects** — instead of creating many identical objects, shared objects are reused.

**GoF Definition:** "Flyweight design pattern enables sharing of objects to support large numbers of fine-grained objects efficiently. A flyweight is a shared object that can be used in multiple contexts simultaneously. The flyweight acts as an independent object in each context."

### Pre-conditions for Applying Flyweight

Before applying this pattern, consider:

1. The **number of objects** to be created should be **huge**.
2. **Object creation is heavy** on memory and can be time-consuming.
3. Object properties can be divided into **intrinsic** and **extrinsic** properties: the extrinsic properties must be defined by the client program.

### Intrinsic vs Extrinsic State — THE KEY CONCEPT

This is the most important concept in Flyweight:

|Property Type|Definition|Shared?|Stored Where?|
|---|---|---|---|
|Intrinsic|State that is stored/shared in the flyweight object. Independent of context. Should be immutable.|YES — shareable|Inside the Flyweight object|
|Extrinsic|State that varies with context. Cannot be shared.|NO — not shareable|Client passes it to the flyweight|

**Analogy — Pen and Refill:**

- A pen can have different colored refills.
- **Intrinsic attributes:** pen body, pointer, casing — common to all pens, stored in the flyweight.
- **Extrinsic attribute:** refill color — varies per context, passed by the client.
- All modules needing a red pen share the **same red pen instance**. A new flyweight is only created when a different color is needed.

### Real-World Examples

**Java String Pool:** `java.lang.String` constants in Java are flyweight objects. All strings are stored in the string pool. If a string with certain content already exists, the JVM returns a reference to the existing one instead of creating a new object.

**Browser Image Caching:** A browser loads an image only once. For every other occurrence of the same image on a page, it reuses the cached copy.

- **Intrinsic:** URL of the image (fixed and shareable).
- **Extrinsic:** Position coordinates, height, width (vary per occurrence).

### When to Use Flyweight

- When you need a **large number of similar objects** that are unique in terms of only a few parameters.
- When you need to **control memory consumption** of a large number of objects by creating fewer objects and sharing them.

### Benefits of Flyweight

- Reduces memory consumption of heavy objects.
- Reduces the total number of complete but similar objects in the system.
- Provides a centralized mechanism to control the states of many "virtual" objects.

### How Flyweight Works (Flyweight Factory)

A **Flyweight Factory** manages a pool of flyweight objects:

- If the requested flyweight exists in the pool → return the existing instance.
- If it doesn't exist → create a new one, add it to the pool, return it.

```java
// Pseudocode for Flyweight Factory
Map<String, Pen> penPool = new HashMap<>();

public Pen getPen(String color) {
    if (!penPool.containsKey(color)) {
        penPool.put(color, new Pen(color)); // create new flyweight
    }
    return penPool.get(color); // return shared instance
}
```

---

### Q&A — Flyweight Pattern

**Q1: What is the fundamental goal of the Flyweight pattern?** A: To reduce memory usage by sharing as many objects as possible with similar state, instead of creating a new object for every request.

**Q2: What is the difference between intrinsic and extrinsic state?** A: Intrinsic state is independent of the flyweight's context, is shared, and is stored inside the flyweight (it should be immutable). Extrinsic state varies per context, cannot be shared, and is passed by the client to the flyweight at the time of use.

**Q3: What is a Flyweight Factory?** A: It is a class that manages a pool/cache of flyweight objects. When a client requests a flyweight, the factory returns an existing shared instance if one exists, or creates a new one if it doesn't.

**Q4: How are Java Strings an example of Flyweight?** A: Java maintains a String pool. When you create a String literal, Java checks if that string already exists in the pool. If it does, it returns a reference to the existing object instead of creating a new one — this is exactly object sharing, the core of Flyweight.

**Q5: What is the precondition for using Flyweight?** A: The number of objects must be huge, object creation must be expensive, and the objects must have separable intrinsic (shareable) and extrinsic (context-specific) state.

**Q6: Is extrinsic data shareable in the Flyweight pattern?** A: No. Extrinsic data is not shared. The client must pass the extrinsic state to the flyweight each time it needs to use it.

## 2.1 Overview of Behavioral Patterns

**Definition (Wikipedia):** In Software Engineering, behavioral design patterns identify **common communication patterns among objects** and realize these patterns. By doing so, they increase flexibility in carrying out communication between objects.

The key principle: The interaction between objects should be in such a way that they are **loosely coupled** — they talk to each other but remain independent. Loose coupling is the key to n-tier architecture.

### What Do Behavioral Patterns Describe?

- How objects **interact** and how they **divide responsibilities** among themselves.
- How different objects and classes **send messages** to each other to make things happen.
- How the steps of a task are **divided among different objects**.
- A behavioral pattern **abstracts an action** from the object/class that takes the action — by changing the object or class, you can change the algorithm, while retaining the same basic interface.

### Object Composition Over Inheritance

Behavioral object patterns use **object composition** rather than inheritance. A group of peer objects can cooperate to perform a task that no single object can carry out by itself.

**Important problem:** Peer objects need to know about each other. If they maintain explicit references to each other, this increases coupling. Behavioral patterns provide organized solutions to manage this coupling.

---

## 2.2 Chain of Responsibility Pattern

### Definition

Chain of Responsibility is a behavioral pattern used to achieve **loose coupling** in software design where a **request from a client is passed through a chain of objects** to process it. The objects in the chain decide themselves:

1. Whether they will process the request.
2. Whether to pass it to the next object in the chain.

### Real-World Example in JDK

**try-catch blocks** in Java are an example. Each `catch` block is a processor. When an exception occurs:

- The try block sends the exception to the first catch block.
- If that catch block can't process it, it passes to the next.
- If none of the catch blocks can handle it, the exception propagates to the calling program (outside the chain).

### Classic Example: ATM Dispenser

**Scenario:** A user enters an amount to withdraw. The ATM dispenses in denominations of $50, $20, and $10.

**Chain:** $50 Dispenser → $20 Dispenser → $10 Dispenser

If the user enters $80:

- $50 Dispenser: dispenses one $50 bill, remaining = $30 → passes to next
- $20 Dispenser: dispenses one $20 bill, remaining = $10 → passes to next
- $10 Dispenser: dispenses one $10 bill, remaining = $0 → done

Each dispenser in the chain handles part of the request and forwards the rest.

### Implementation Principles

1. Each processor in the chain will have its own implementation for processing a command.
2. Every processor in the chain must have a **reference to the next processor** — beware of dropped commands.
3. Processors should **not form a recursive cycle** — this would cause infinite loops.
4. **Only one processor handles a given command** at any point (or partial handling + forwarding).

### Important Points

- The **client doesn't know** which part of the chain will process the request. It simply sends the request to the first object in the chain.
- Each object in the chain will have its own implementation to process the request — either fully, partially, or just to send it to the next object.
- Every object in the chain holds a reference to the next object — achieved via Java composition.
- Creating the chain carefully is critical. If done poorly, a request may never reach an appropriate processor.
- Chain of Responsibility achieves **loose coupling** but comes with the trade-off of many implementation classes and potential maintenance problems if most code is common across implementations.

### Structure

```
Client → Handler (interface)
              |
              | next
              ↓
    ConcreteHandler1 → ConcreteHandler2 → ConcreteHandler3 → null
    (handles or       (handles or         (handles or
     forwards)         forwards)           throws error)
```

---

### Q&A — Chain of Responsibility

**Q1: What is the main purpose of the Chain of Responsibility pattern?** A: To achieve loose coupling by passing a client's request through a chain of handler objects, where each handler decides to process it or pass it to the next handler.

**Q2: Give an example of Chain of Responsibility in the Java JDK.** A: The try-catch block mechanism. Each catch block is a handler. An exception travels from the first catch block to subsequent ones until one handles it, or it propagates out.

**Q3: What must every processor/handler in the chain have?** A: A reference to the next processor in the chain (achieved via Java composition), so it can forward the request if it cannot handle it.

**Q4: What is the risk if the chain is poorly constructed?** A: Requests may never reach an appropriate processor, or there may be no object capable of handling the request, causing it to be lost or throwing an unhandled exception.

**Q5: Should processors in a chain form a cycle?** A: No. Processors must not form a recursive cycle. A cycle would cause infinite loops where the request bounces around forever without being resolved.

**Q6: What is the key trade-off of using Chain of Responsibility?** A: While it achieves loose coupling, it leads to a large number of implementation classes and can cause maintenance headaches if most of the logic is duplicated across multiple handler implementations.

## 2.3 Command Design Pattern

### Definition

A **Command pattern** is an object behavioral pattern that allows **complete decoupling between the sender and the receiver**.

- **Sender:** An object that invokes an operation.
- **Receiver:** An object that receives the request to execute a certain operation.
- With decoupling, the sender has **no knowledge** of the Receiver's interface.

**GoF Definition:** "Encapsulate a request as an object, thereby letting you parameterize clients with different requests, queue or log requests, and support undoable operations."

Simply put: **Command turns a request into a stand-alone object** that contains all information about the request. This object can be stored, queued, passed as arguments, and supports "undo."

### Real-Life Example: Restaurant Order

A customer orders food at a restaurant:

1. The waiter takes the order (writes it on paper — this is the **Command object**).
2. The waiter goes to the kitchen and posts the order on the wall (the command is **queued**).
3. Eventually, the chef reads the order and cooks — this is the **Receiver executing the command**.
4. The order (command) contains all required information, so the chef doesn't need to run back and clarify.

This decouples the customer (client) from the chef (receiver). The waiter is the invoker.

### When to Use the Command Pattern

- A **history of requests** is needed (audit trails).
- You need **callback functionality**.
- Requests need to be **handled at different times or in different orders** (queuing).
- The invoker should be **decoupled** from the object handling the invocation.
- You need **multiple undo operations** — maintain a stack of executed commands, call `undo()` to reverse.
- Useful for: wizards, progress bars, GUI buttons/menus, transactional behavior.

### Key Participants

|Participant|Role|
|---|---|
|Command (interface)|Declares the `execute()` method — the contract for all commands|
|ConcreteCommand|Implements `execute()`, stores a reference to the Receiver, and defines the receiver-action pair|
|Receiver|Has the knowledge to carry out the actual request (the business logic)|
|Invoker|Asks the command to execute (doesn't know what command does internally)|
|Client|Creates Receiver and ConcreteCommand objects, attaches them, and creates/configures the Invoker|

### How It Works (Flow)

```
Client
  → creates Receiver
  → creates ConcreteCommand(Receiver)
  → creates Invoker
  → sets ConcreteCommand on Invoker
  → calls Invoker.executeAction()
        → ConcreteCommand.execute()
              → Receiver.performAction()
```

### Example: File System Utility

**Scenario:** A file system utility with open, write, close operations on Windows and Unix.

- `FileSystemReceiver` (interface): declares `openFile()`, `writeFile()`, `closeFile()`.
- `WindowsFileSystemReceiver`, `UnixFileSystemReceiver`: concrete receivers.
- `OpenFileCommand`, `WriteFileCommand`, `CloseFileCommand`: concrete commands.
- `FileInvoker`: invoker that stores and executes a command.
- Client creates the appropriate receiver, wraps it in a command, sets it on the invoker.

### Example: Fan and Light Switch

A `Switch` can be connected to either a `Fan` or a `Light`. Both have different interfaces. The Switch is parameterized with a set of commands at construction:

```
Switch flipUp()    → Command.execute() → Fan.startRotation() (or Light.turnOn())
Switch flipDown()  → Command.execute() → Fan.stopRotation()  (or Light.turnOff())
```

The Switch has **no idea** what `execute()` does — it is completely decoupled from the Fan and Light.

### Important Points

- `Command` is the core — defines the contract for implementation.
- Receiver implementation is separate from command implementation.
- Invoker just forwards requests from client to command object.
- The Client is responsible for instantiating both the receiver and the appropriate command and associating them.
- Command pattern is **easily extendible** — add new action methods in receivers and new Command implementations without changing client code.
- **Drawback:** The code gets large and confusing with high numbers of action methods and many associations.

### Advantages

- A command decouples the invoker from the performer.
- A command is a **first-class object** — it can be manipulated and extended like any other object.
- Commands can be assembled into **composite commands** (Macro commands).
- Easy to add new commands without changing existing classes.

---

### Q&A — Command Pattern

**Q1: What is the core concept of the Command pattern?** A: Encapsulating a request as an object, allowing you to parameterize clients with different requests, queue or log requests, and support undoable operations.

**Q2: Who are the five key participants in the Command pattern?** A: Command (interface), ConcreteCommand (implements execute()), Receiver (has the actual business logic), Invoker (calls execute()), and Client (wires everything together).

**Q3: How does the Command pattern support "undo" operations?** A: By maintaining a stack of executed Command objects. When you want to undo, you pop the last command from the stack and call its `undo()` method, which reverses the action.

**Q4: What is the role of the Invoker in the Command pattern?** A: The Invoker stores and triggers commands. It has no knowledge of what the command does — it simply calls `execute()` on the command object it holds.

**Q5: What is the difference between the Invoker and the Receiver?** A: The Invoker is the object that triggers the command (e.g., a button, switch). The Receiver is the object that actually performs the work when the command is executed (e.g., Fan, Light, FileSystem).

**Q6: What is the main drawback of the Command pattern?** A: The code can become very large and confusing when there are many action methods, because each action in each receiver needs a corresponding Command class, leading to a proliferation of small classes.

**Q7: How is the restaurant order analogy related to the Command pattern?** A: The order slip is the Command object. The customer is the Client. The waiter is the Invoker. The chef is the Receiver. The paper order decouples the customer from the chef completely.

## 2.4 Iterator Design Pattern

### Definition

The Iterator pattern provides a **standard way to traverse a group of objects**. It is widely used in Java's Collection Framework.

**GoF Definition:** "Provide a way to access the elements of an aggregate object without exposing its underlying representation."

The pattern doesn't just help you traverse; you can provide **different kinds of iterators** for different requirements. The iterator hides the actual traversal mechanism from the client.

### Design Participants

|Participant|Role|
|---|---|
|Iterator (interface)|Declares methods for traversal — minimum: `next()`, optionally `previous()`, `hasNext()`, `currentPosition()`|
|ConcreteIterator|Implements `Iterator` interface. Keeps track of current traversal position.|
|Aggregate (interface)|Typically a collection interface. Defines a method to create an Iterator object.|
|ConcreteAggregate|Implements Aggregate. Its specific method returns an instance of ConcreteIterator.|

### Example: Radio Channel Iterator

**Scenario:** A list of radio channels. Client programs want to traverse them — some only want English channels.

**Without Iterator:** Client writes custom traversal logic. This is error-prone, and if many clients exist, all need to be updated if the traversal changes.

**With Iterator:** Provide a channel-type-specific iterator. The client accesses channels **only through the iterator**. The iteration logic is centralized and hidden.

```java
// Client usage
ChannelIterator englishIter = channelCollection.getEnglishChannelIterator();
while (englishIter.hasNext()) {
    Channel c = englishIter.next();
    // process English channel
}
```

### How to Implement Iterator (5 Steps)

1. **Declare the Iterator interface** — at minimum, a method to fetch the next element. Optionally: previous, hasNext, currentPosition.
    
2. **Declare the Collection interface** — describe a method to fetch iterators. Return type must match the iterator interface.
    
3. **Implement ConcreteIterator classes** for each traversable collection. An iterator object must be linked with a single collection instance — link established via the iterator's constructor.
    
4. **Implement the Collection interface** in collection classes — provide a shortcut for creating iterators. The collection object passes itself to the iterator's constructor.
    
5. **Replace traversal code in client** — use iterators everywhere. Client fetches a new iterator whenever it needs to traverse.
    

### Consequences

**1. Improved code flexibility:** Clients can iterate over collections in different ways without changing the underlying collection's structure.

**2. Better encapsulation:** Internal details of the collection are hidden. This prevents tight coupling between client and collection internals.

**3. Simplified client code:** A standardized way to access elements reduces client complexity.

**4. Performance overhead:** Involves additional method calls and object creations. However, this is typically negligible for small collections.

### Advantages

- **Single Responsibility Principle:** Traversal algorithms are extracted into separate classes (iterators), keeping collections clean.
- **Open/Closed Principle:** New types of collections and iterators can be added and passed to existing code without breaking anything.
- **Parallel iteration:** Each iterator object has its own state, so you can iterate over the same collection in parallel with multiple iterators.
- **Delayed iteration:** You can pause and resume iteration since state is maintained in the iterator.

### Disadvantages

- May be **overkill** if your app only works with simple collections.
- Using an iterator may be **less efficient** than direct element access in some specialized collections.

---

### Q&A — Iterator Pattern

**Q1: What is the GoF definition of the Iterator pattern?** A: "Provide a way to access the elements of an aggregate object without exposing its underlying representation."

**Q2: What are the four participants in the Iterator pattern?** A: Iterator (interface), ConcreteIterator (implements traversal and tracks position), Aggregate (interface declaring iterator creation method), and ConcreteAggregate (implements aggregate and returns ConcreteIterator).

**Q3: How does Java's Collection Framework use the Iterator pattern?** A: Every Java collection (ArrayList, LinkedList, HashSet, etc.) implements the `Iterable` interface, which provides the `iterator()` method. This returns an `Iterator` object with `hasNext()` and `next()` methods — a direct application of the Iterator pattern.

**Q4: What is the advantage of using separate iterator classes over having the collection traverse itself?** A: It respects the Single Responsibility Principle. The collection manages data; the iterator manages traversal. They can evolve independently. Multiple different iterator types can exist for the same collection.

**Q5: Can you have multiple iterators on the same collection simultaneously?** A: Yes. Each iterator object has its own independent traversal state (current position, etc.), so multiple iterators can traverse the same collection at the same time.

**Q6: When would using the Iterator pattern be an overkill?** A: When the application only uses simple collections (like an array of a few items) that never change structure. The overhead of defining iterator interfaces and classes would exceed the benefit.

## 2.5 Interpreter Design Pattern

### Definition

The **Interpreter pattern** is a behavioral design pattern that defines a way to **interpret a Domain-Specific Language (DSL) or grammar**. It allows you to define the grammar of a language and provide a way to evaluate sentences in that language.

**GoF Definition:** "Given a language, define a representation for its grammar along with an interpreter that uses the representation to interpret sentences in the language."

This pattern is about **interpretation** — just like a human interpreter who translates speeches from one language to another.

This pattern should be used when you have a **simple language** that can be represented as an **Abstract Syntax Tree (AST)**.

### What is an Abstract Syntax Tree (AST)?

In computer science, an **Abstract Syntax Tree (AST)** is a tree representation of the abstract syntactic structure of text (often source code) written in a formal language. Each node of the tree denotes a construct occurring in the text.

Example: The expression `2 + 3` represented as an AST:

```
       +
      / \
     2   3
```

A complex expression like `(2 + 6) + (1 + 9)` becomes:

```
           +
          / \
         +   +
        / \ / \
       2  6 1  9
```

Modeled using the Interpreter pattern:

```
AndExpression(
  AndExpression(ConstantExpression(2), ConstantExpression(6)),
  AndExpression(ConstantExpression(1), ConstantExpression(9))
)
```

### Real-World Examples

**Google Translate:** Input can be in any language; output is in another. Google Translate is an interpreter pattern at scale.

**Java Compiler:** You write Java source code. The compiler interprets the source by following grammar rules and converts it to bytecode. Most compilers are designed using the Interpreter pattern.

### Domain-Specific Languages (DSL) — Common Examples

The Interpreter pattern is particularly suited for building interpreters for DSLs. Common DSLs include:

- **HTML** — defines structure and content of web pages
- **CSS** — defines visual style of HTML documents
- **SQL** — declarative language for managing data in relational databases
- **Regular Expressions** — for pattern matching in strings
- **MATLAB** — for mathematical computations
- **Make** — for build automation

### Intent

- Provide a way to interpret and execute expressions in a language.
- Break down complex expressions into smaller components that can be easily interpreted.
- Create parsers and compilers for a defined language.
- Build scripting languages and domain-specific languages.
- Support the implementation of rules governing behavior.
- Facilitate creation of Abstract Syntax Trees.
- Allow creation of interpreters handling different types of expressions and operators.

### Design Participants

|Participant|Role|
|---|---|
|AbstractExpression|Declares an `interpret()` operation that all nodes (terminal and nonterminal) in the AST override|
|TerminalExpression|Implements `interpret()` for terminal (leaf) expressions — e.g., `NumberExpression`|
|NonterminalExpression|Implements `interpret()` for composite expressions — e.g., `AdditionExpression`, `SubtractionExpression`, `MultiplicationExpression`|
|Context|Contains information global to the interpreter (e.g., the string expression in Postfix notation that needs to be parsed)|
|Client (ExpressionParser)|Builds or is provided the AST from TerminalExpression and NonTerminalExpression objects. Invokes `interpret()`.|

### Motivations for Using Interpreter

**Define a Domain-Specific Language:** Allows developers to define a language tailored to a specific domain or problem, making code more expressive and concise.

**Interpret and execute expressions dynamically:** Handles expressions at runtime — useful for complex expressions that are difficult or impossible to evaluate statically.

**Separate concerns:** Separates the definition of a language from the interpretation of expressions. This makes it easier to modify the language and add new expressions without affecting the interpreter code.

**Build parsers and compilers:** The Interpreter pattern can be used to build parsers and compilers, allowing the language to be used across a wide range of applications.

**Facilitate communication between components:** Defines a common language so different components can exchange messages and cooperate more easily.

### Applications

- **Regular Expressions:** Interpreters parse and execute regex patterns.
- **SQL Queries:** Interpreters parse and execute SQL, abstracting underlying storage.
- **Mathematical Expressions:** Parse and evaluate arithmetic formulas.
- **Programming Languages:** Interpreters execute scripts or compile code on-the-fly.

### Advantages

- **Flexibility:** Define your own language and parse sentences in it.
- **Extensibility:** Easy to add new expressions or modify existing ones without rewriting the interpreter.
- **Separation of Concerns:** Parsing logic and execution logic are separated.

### Disadvantages

- **Complexity:** Can add significant complexity, especially for complex grammars.
- **Performance:** Slower than compiled approaches because expressions are interpreted at runtime.
- **Maintenance:** Difficult to maintain if the language or expressions change frequently.
- **Too complex language structures blow up the number of classes:** Each construct needs its own class.
- **No built-in parsing:** The Interpreter pattern does not do parsing — you need a separate parser for complex language structures.

### Applicability

- When you need to interpret a domain-specific language or grammar (e.g., compiler, scripting language interpreter).
- When implementing rule-based systems where rules are expressed in a DSL.

---

### Q&A — Interpreter Pattern

**Q1: What is the GoF definition of the Interpreter pattern?** A: "Given a language, define a representation for its grammar along with an interpreter that uses the representation to interpret sentences in the language."

**Q2: What is an Abstract Syntax Tree and how does it relate to the Interpreter pattern?** A: An AST is a tree representation of the syntactic structure of a text/program. Each node represents a language construct. The Interpreter pattern uses ASTs to model and interpret expressions — terminal nodes are leaf values, nonterminal nodes are operations.

**Q3: Name the five participants in the Interpreter pattern.** A: AbstractExpression (declares interpret()), TerminalExpression (implements interpret() for leaves), NonterminalExpression (implements interpret() for composite nodes), Context (global information for the interpreter), and Client (builds the AST and invokes interpret()).

**Q4: Give three real-world examples where the Interpreter pattern is applied.** A: Java compiler (source to bytecode), Google Translate (one language to another), SQL engines (interpret SQL queries), Regular Expression engines (interpret regex patterns).

**Q5: What is a Domain-Specific Language (DSL)? Give examples.** A: A DSL is a language designed for a specific domain or problem type. Examples include HTML (web structure), CSS (web styling), SQL (database queries), Regular Expressions (pattern matching), and MATLAB (mathematical computing).

**Q6: What are the main disadvantages of the Interpreter pattern?** A: It can be complex and slow (runtime interpretation), hard to maintain for frequently changing languages, and a complex grammar can cause a large proliferation of classes. Also, it does not include parsing — a separate parser is required.

## 3.1 Introduction to Anti-Patterns

### What is an Anti-Pattern?

An **anti-pattern** in software engineering is a **commonly used but counterproductive solution** to a recurring problem. It appears functional but results in:

- Poor design
- High technical debt
- Unmaintainable code
- Potential project failure

Unlike **design patterns** that offer proven, efficient solutions, **anti-patterns represent pitfalls** — such as hardcoding, lack of documentation, or rigid architecture — that necessitate extensive future fixes.

The term "technical debt" refers to code you have to come back and fix properly later. Anti-patterns are a major source of technical debt.

### Design Pattern vs Anti-Pattern

| Design Pattern                                          | Anti-Pattern                                                                     |
| ------------------------------------------------------- | -------------------------------------------------------------------------------- |
| A repeatable architectural solution for common problems | A widespread but ineffective, risky, or unproductive approach to common problems |
| Proven, documented, and recommended                     | A trap or pattern of mistakes                                                    |
| Results in clean, maintainable code                     | Results in poor design and high maintenance cost                                 |

### Types of Anti-Patterns (Three Categories)

**1. Architectural Anti-Patterns:** Arise as the structure of a system is designed (generally by an architect). Example: Vendor Lock-in.

**2. Management/Organizational Anti-Patterns:** Anti-patterns in project management, encountered by managers or groups of managers. Examples: Death March, Analysis Paralysis, Brooks' Law.

**3. Development Anti-Patterns:** Arise as a system is implemented by programmers. Examples: The Blob (God Class), Spaghetti Code, Golden Hammer, Boat Anchor, Dead Code.

---

## 3.2 Reference Model for Anti-Patterns

### Root Causes — The "Seven Deadly Sins"

Root causes are mistakes in software development that result in failed projects, cost overruns, schedule slips, and unfulfilled business needs. The reference model identifies **seven root causes**, analogized to the "seven deadly sins":

|Root Cause (Sin)|Description|
|---|---|
|**Haste**|Tight deadlines lead to neglecting important activities — taking shortcuts.|
|**Apathy**|The attitude of not caring about solving known problems. Ignoring obvious issues.|
|**Narrow-Mindedness**|Refusal of developers to learn proven solutions. "My way or the highway" thinking.|
|**Sloth**|Choosing the simplest "solution" (laziness) instead of the correct one.|
|**Avarice (Greed)**|Trying to build an overly comprehensive, complex system results in unmaintainable software.|
|**Ignorance**|Lack of motivation to understand things properly. Not seeking knowledge.|
|**Pride**|Failure to reuse existing software packages because they were not invented here ("NIH Syndrome" — Not Invented Here).|

### Primal Forces

Forces are concerns or issues that exist within a decision-making context. Forces that are successfully addressed lead to **benefits**; unresolved forces lead to **consequences**.

The **primal forces** in software development are:

|Force|Description|
|---|---|
|Management of Functionality|Meeting the requirements — doing what the system is supposed to do|
|Management of Performance|Meeting required speed and operational constraints|
|Management of Complexity|Defining proper abstractions to control complexity|
|Management of Change|Controlling the evolution of software over time|
|Management of IT Resources|Managing people and IT artifacts effectively|
|Management of Technology Transfer|Controlling how technology evolves within and across organizations|

### Software Design Level Model (SDLM)

The SDLM is represented as a **pyramid diagram** showing the hierarchy of abstraction in software and system architecture. Levels from top (most detailed) to bottom (broadest):

```
Objects & Classes          ← Individual code entities and behaviors
        ↓
Micro-architectures        ← Small design patterns (MVC, Singleton, Observer)
        ↓
Frameworks                 ← Reusable platforms (Spring, .NET, Django)
        ↓
Application Level          ← Complete software applications
        ↓
System Level               ← Multiple applications working together (ERP, HRMS, CRM)
        ↓
Enterprise Level           ← Organization-wide architecture aligned with business strategy
        ↓
Global / Industry Level    ← Standards, regulations, industry-wide ecosystems
                             (ISO standards, financial regulations, healthcare compliance)
```

**Key Message of the Pyramid:**

- At the top: fine-grained programming constructs.
- As you move downward: scope widens — applications integrate into systems, systems align with enterprises, enterprises fit into global industries.
- Local design decisions (like class structures) eventually influence and must align with global industry requirements.

## 3.3 Management Anti-Patterns

### Anti-Pattern #1 — Cart Before the Horse

**What it is:** Project priorities are the opposite of what they should be. Too many resources are allocated to a project segment that depends on something else that hasn't been done yet.

**Example:** Building a beautiful front-end UI before the back-end API it depends on is designed or built.

**Solution:** Proper research, planning, and consulting. Do due diligence to discover which parts depend on others. Tackle priorities in an order that leads to success. This allows better resource allocation and a more complete product by the deadline.

---

### Anti-Pattern #2 — Death March

**What it is:** A management team denies the obvious failure of a doomed project but still requires developers to work on it until "completion." The team is marching toward certain failure while management pretends success is just around the corner.

**Problem:** No developer wants to work on something with no chance of success. Team morale plummets. Human and capital resources are wasted.

**Solution:** Project managers must be able to recognize failure early, **cut losses, and regroup** before a death march begins. Pivoting is not failure — continuing a doomed project is.

---

### Anti-Pattern #3 — The 99% Rule

**What it is:** The mistaken belief that if 99% of work is done, the final 1% is easy. In reality, the last bit is often the hardest and most time-consuming.

**Example:** "We're almost done — just final testing and integration!" But those final steps reveal bugs, compatibility issues, deployment problems, and missing requirements.

**Solution:** Set and keep regular **daily stand-ups** to ensure timelines remain on track. Spend more time upfront on understanding all the work involved and creating accurate estimates. Spread effort more evenly rather than assuming the end is easy.

---

### Anti-Pattern #4 — Over Engineering

**What it is:** Violating the "Keep It Simple, Stupid" (KISS) principle. Developers build the most complex, feature-rich, technically impressive solution when a simple one would suffice.

**Example:** Using microservices, Kubernetes, event-driven architecture, and distributed caching for a simple internal CRUD application with 10 users.

**Solution:**

- There is no need to reinvent the wheel every time.
- Is there an already-built solution that meets immediate needs?
- Can you build something quickly and iterate later?
- Remember that design overwork or overcoming technical debt could add unplanned complexity or time.

---

### Anti-Pattern #5 — Scope Creep (Feature Creep)

**What it is:** A client adds more and more requirements throughout development until the project far exceeds the original understanding. All these new requirements are somehow "equal priority" and must be built within the original timeline.

**Problem:** This one is particularly hard to handle — you don't want to seem uncooperative, but you also can't keep absorbing endless new work without adjusting timelines or budget.

**Solution:**

- **Proactive and constant communication** throughout the project.
- Scope the project at a **granular level** with buy-in from all stakeholders at the start.
- If new requirements are added, remind stakeholders of the scope and see if something can be removed to compensate.
- If not, consider altering release schedules or adding teams.

---

### Anti-Pattern #6 — Smoke and Mirrors

**What it is:** Presenting incomplete features in such a way as to make them appear finished. "Pay no attention to the man behind the curtain!"

**Problem:** Beyond the ethical issues, this creates an ever-decreasing productivity spiral. Features get continuously deferred, stacked onto future sprints, and eventually the backlog becomes unmanageable. Client trust is destroyed when the deception is discovered.

**Solution:** Project managers must be **upfront and honest** at all demos and deadlines. If communication is effective throughout, there will be no surprises at delivery. Bad news delivered early is far less damaging than bad news delivered at the end.

---

### Anti-Pattern #7 — Brooks' Law

**What it is:** The Catch-22 of project management: you add more people to a late project to increase velocity, but velocity actually **decreases** because of the added overhead of communication and onboarding.

"Adding manpower to a late software project makes it later." — Fred Brooks (_The Mythical Man-Month_)

**Why it happens:** New team members need time to be onboarded. Existing members spend time training them instead of building. Communication overhead grows non-linearly as the team grows (n members → n*(n-1)/2 communication channels).

**Solution:**

- Add **additional teams**, not just more members to existing teams.
- Smaller, focused teams communicate more effectively.
- On large projects, parse the work into smaller units with multiple Scrum teams.
- Don't be afraid of multiple Scrum teams — they can be merged or adjusted as the project evolves.

---

### Anti-Pattern #8 — Copy and Paste

**What it is:** A developer copies and pastes source code without properly reviewing, understanding, or adapting it. Also applies in project management — copying last project's workflow onto a new project without customization.

**Consequences in coding:**

- Less code readability
- Increased maintenance cost (a bug must be fixed in every copy)
- Reduced code reusability

**Consequences in project management:**

- Risk of overlooking important variables unique to the new project
- Wrong timelines, wrong processes, wrong team structure
- Development rework, delays, or project failure

**Solution:** Each project contains a unique combination of requirements, technologies, processes, people, timelines, and deliverables. Workflows should be purpose-built for each project, not templated from the previous one.

## 3.4 Commonly Encountered Anti-Patterns

### Analysis Paralysis (Classic Management Anti-Pattern)

**What it is:** Over-analyzing a situation during the planning phase so that **no decision or action is taken**, essentially paralyzing the development process. The goal of achieving perfection leads to an infinite loop of analysis.

Characterized by: Walking in circles, revising and creating detailed models, continually revisiting decisions — "too much analysis."

### Symptoms and Consequences

- Multiple project restarts and extensive model rework due to personnel changes or direction changes.
- Design and implementation issues are continually reintroduced in the analysis phase.
- **Cost of analysis exceeds expectations** without a predictable endpoint.
- The analysis phase stops involving user interaction — analysis becomes speculative.
- Complexity of analysis models results in intricate implementations.
- Gang of Four design pattern decisions (which should be made in design) are being made during analysis.

### Typical Causes

- Management process assumes a **waterfall progression** — all analysis must complete before design begins.
- Management has more confidence in analysis and decomposition than in design and implementation.
- Goals in the analysis phase are not well-defined.
- Management is unwilling to make firm decisions about when parts of the domain are "sufficiently described."
- Planning lapses when transitioning from analysis to design.
- Project vision and focus on the deliverable is diffused.

### Refactored Solution

The key is **incremental development**:

- A waterfall process assumes you have all knowledge upfront (apriori knowledge). Incremental development acknowledges that details emerge during development.
- Use **internal increments** (infrastructure work: database layer, APIs — builds infrastructure used by many use cases, minimizes rework) and **external increments** (user-visible functionality).
- Don't try to analyze everything before starting — analyze just enough, build, learn, repeat.

## 3.5 Software Architecture Anti-Patterns

### Vendor Lock-In

**What it is:** A software project becomes **completely dependent** on a specific vendor's product technology. When the vendor upgrades, changes, or discontinues features, the application breaks or requires extensive rework. The project's fate is entirely in the vendor's hands.

**Symptoms and Consequences:**

- Commercial product upgrades drive the application's maintenance cycle.
- Promised features are delayed or never delivered, causing the application to fall behind.
- The product varies significantly from advertised open-system standards.
- Missing an upgrade cycle can require a complete product repurchase and re-integration.

**Typical Causes:**

- Product selected entirely based on marketing, not technical inspection.
- No isolation between application software and the vendor's product.
- Application programming requires deep product-specific knowledge.
- The product's complexity exceeds application needs, making direct dependence risky.
- No conformance process for open standards — the vendor deviates from standards.

**Refactored Solution — Isolation Layer:**

Create an **isolation layer** (also called an abstraction layer or anti-corruption layer) that separates your application from the vendor's product. This applies when:

- You need to isolate application software from lower-level infrastructure (middleware, OS, security).
- Changes to underlying infrastructure are anticipated during the software's lifecycle.
- A more convenient programming interface is needed (current abstraction is too primitive or too flexible).
- Consistent handling across many systems is required.
- Multiple infrastructures must be supported simultaneously or over the lifecycle.

The isolation layer means your application code talks to your abstraction, not directly to the vendor's API. Switching vendors only requires changing the isolation layer.

## 3.6 Software Development Anti-Patterns

### The Blob (God Class)

**What it is:** A procedural-style design leads to **one class monopolizing all the processing responsibilities**, while other classes only hold data or execute simple processes.

A "God Class" — one class that knows and does everything. The class diagram shows a single complex controller class surrounded by simple data classes. The majority of responsibilities are allocated to one class.

**Symptoms and Consequences:**

- A single class with **60 or more attributes and operations** — a reliable indicator.
- A disparate collection of unrelated attributes and operations in one class — overall lack of cohesion.
- The Blob class is too complex for reuse and testing.
- Expensive to load into memory even for simple operations — because it carries everything.

**Typical Causes:**

- Lack of object-oriented architecture (procedural thinking applied to OOP code).
- Lack of any architecture at all.
- Lack of architecture enforcement (no code reviews, no design guidelines).
- Too limited intervention (letting it grow unchecked).

**Solution:** Refactor by distributing responsibilities more uniformly across multiple cohesive classes. Apply the Single Responsibility Principle — each class should have one reason to change.

---

### Spaghetti Code

**What it is:** Code with **little to zero structure**. Nothing is modularized. Files are randomly placed in random directories. The flow is difficult to follow — utterly tangled (like spaghetti). Typically a result of someone who didn't think through the program flow before starting to code.

**Consequences:**

- Maintenance nightmare.
- Almost impossible to add new functionality without breaking existing code.
- Impossible to give accurate estimates for changes.
- You constantly break things without understanding the full scope of your changes.

**Solution:** Plan the program's structure before coding. Use proper modularization, follow design principles (SOLID, DRY), and apply appropriate design patterns.

---

### Golden Hammer

**What it is:** Over-reliance on a familiar technology, architecture, or tool — applying it to every problem regardless of whether it fits.

_"I suppose it is tempting, if the only tool you have is a hammer, to treat everything as if it were a nail."_ — Abraham Maslow

**Example:** A team is expert in one architecture. Every new project, regardless of its unique requirements, uses that same architecture. A problem that requires a completely different approach is forced into the "hammer" architecture.

**Consequences:**

- Serious performance hits — forcing an ill-fitting architecture.
- Twice the coding time for the same result.
- Not predictable — rules that worked in one language/domain may fail in another.
- Stagnation — the team never grows beyond their comfort zone.

**Solution:**

- Consistently learning is crucial.
- Pick the right language/architecture for the problem, not the comfortable one.
- Research and investigate new tools.
- Push out of the comfort zone.

---

### Boat Anchor

**What it is:** Programmers leave code in the codebase because they **might need it later**. The code was written slightly off-spec, isn't needed yet, but the developer is "sure" they'll need it next month.

**The Analogy:** A boat anchor is heavy to carry but doesn't do anything. Similarly, this dead-weight code adds technical debt without serving any purpose.

**Consequences:**

- Maintenance nightmare — colleagues can't tell what code is obsolete vs what actively runs.
- During a hot fix, developers waste time reading and debugging obsolete code.
- Longer build times.
- Risk of accidentally "turning on" obsolete code in production.

**Solution:** Delete unused code immediately. Trust your version control system (Git) to preserve the history if you ever need it back. Clean codebases are far easier to work in.

---

### Dead Code

**What it is:** Code that is still present in the codebase but **no longer serves any purpose** in the current system. It may be called from everywhere, but nobody knows exactly what it does or whether it's still needed.

**Characteristics:** The context for why the code exists is missing. You can read and understand the code flow, but "why" is unclear — it looks like it was needed 3 years ago but not now.

**More common in:** Proof-of-concept or research code that accidentally made it into production.

**Solution:**

- Write down everything your module/section is supposed to do.
- Take an iterative approach — remove dead code piece by piece.
- After each removal, verify nothing broke.
- You will either understand why the code is critical (and document it) or successfully delete it.
- Avoid huge rewrites — incremental removal is safer.

---

### Singleton as an Anti-Pattern

**Context:** The Singleton pattern (which ensures a single instance of a class) is listed as both a design pattern and discussed as a potential anti-pattern. The debate is genuine.

**Why Singleton can be an Anti-Pattern:**

**1. Global State Problem:** When you access the singleton instance, you don't know the current state of the object. You don't know who has changed it or when. The correctness of your code depends on the order of access to the singleton — this is unpredictable and hard to reason about.

**2. Violates SOLID — Single Responsibility Principle:** In addition to its direct duties, the singleton class also controls the number of its instances. That's two responsibilities.

**3. Hidden Dependencies:** Dependencies on a singleton are not visible in a class's interface. You have to read the implementation to find them (via `getInstance()` calls buried inside methods).

**4. Reduces Testability:** You cannot replace the singleton with a mock object in unit tests. Tests become dependent on each other because they all share the singleton's global state.

**5. Increases Coupling:** All of the above are consequences of increased coupling introduced by the singleton.

**Alternatives:** Factories, dependency injection (Spring IoC containers), or explicit parameter passing. Spring's IoC containers are called "factories on steroids" — a natural solution to controlling creation of service objects.

---

### Poltergeist

**What it is:** A pointless class that is used only to **call methods of another class** or simply add an unnecessary layer of abstraction. Manifests as short-lived objects that are devoid of state, often used only to initialize other, more permanent objects.

**Consequence:** Adds unnecessary complexity and indirection to the codebase. These "ghost" classes clutter the design without providing any real functionality.

**Solution:** Remove the poltergeist class and either call the other class's methods directly, or properly design the class to have meaningful state and behavior of its own.

### Q&A — Anti-Patterns

**Q1: What is an anti-pattern and how does it differ from a design pattern?** A: An anti-pattern is a commonly used but counterproductive solution to a recurring problem. It appears functional but leads to poor design and high technical debt. A design pattern offers a proven, efficient solution. Anti-patterns are what NOT to do; design patterns are what TO do.

**Q2: What are the three types of anti-patterns?** A: Architectural anti-patterns (arise during system structure design), Management/Organizational anti-patterns (arise in project management), and Development anti-patterns (arise during implementation by programmers).

**Q3: What are the seven "root cause" sins of anti-patterns?** A: Haste (tight deadlines cause shortcuts), Apathy (not caring about known problems), Narrow-Mindedness (refusing to learn proven solutions), Sloth (choosing the easiest path), Avarice (building overly complex systems), Ignorance (lack of motivation to understand things), and Pride (refusing to reuse external solutions — NIH syndrome).

**Q4: What is Analysis Paralysis and what causes it?** A: Analysis Paralysis is over-analyzing a situation during planning so that no decision or action is taken. It is caused by waterfall thinking (all analysis before any design), vague goals, management unwillingness to make firm decisions, and the drive for perfection.

**Q5: What is the Vendor Lock-In anti-pattern and how is it solved?** A: Vendor Lock-In is when a project becomes completely dependent on a specific vendor's technology. It is solved by creating an Isolation Layer — an abstraction that separates application code from the vendor's API, allowing vendor-swapping without rewriting the entire application.

**Q6: What is the Blob (God Class) anti-pattern? What is its main symptom?** A: The Blob is when one class monopolizes all processing responsibility in a system. The main symptom is a single class with 60 or more attributes and operations, containing unrelated functionality. The fix is to refactor and distribute responsibilities across multiple cohesive classes.

**Q7: What is Brooks' Law?** A: "Adding manpower to a late software project makes it later." Adding more people to an existing late team increases communication overhead and onboarding time, actually slowing progress rather than speeding it up.

**Q8: What is the Scope Creep anti-pattern and how do you prevent it?** A: Scope Creep is when a client continuously adds more requirements beyond the original scope. It is prevented through constant proactive communication, granular scoping with stakeholder buy-in at the start, and formal change management processes.

**Q9: Why is Singleton considered both a pattern and an anti-pattern?** A: Singleton is useful for controlling instance creation. However, it introduces global state, violates the Single Responsibility Principle, hides dependencies, reduces testability, and increases coupling — all signs of poor design. Whether it is a pattern or anti-pattern depends on how it is used.

**Q10: What is the Software Design Level Model (SDLM) pyramid?** A: A pyramid showing the hierarchy of software abstraction from most detailed (Objects & Classes at top) through Micro-architectures, Frameworks, Application Level, System Level, Enterprise Level, down to Global/Industry Level at the bottom. It shows how local coding decisions ultimately must align with global industry standards.

**Q11: What is the difference between Boat Anchor and Dead Code anti-patterns?** A: Boat Anchor code was intentionally left in because the developer thinks it might be needed in the future (speculation). Dead Code is code that once served a purpose but no longer does — its context is lost. Both add unnecessary complexity, but their origin differs.

**Q12: What is the Golden Hammer anti-pattern?** A: Over-reliance on a single familiar technology or architecture applied to every problem regardless of fit. Named after Maslow's observation — "if your only tool is a hammer, every problem looks like a nail." The solution is to learn diverse tools and select the right one for each problem.

# 1. GRASP Patterns

## 1.1 What is GRASP?

**GRASP** stands for **General Responsibility Assignment Software Patterns (or Principles)**.

It is a set of **nine fundamental principles** in object-oriented design that guide you in deciding **which class or object should be responsible for which action or role**.

> 📖 _First published by **Craig Larman** in his 1997 book: "Applying UML and Patterns: An Introduction to Object-Oriented Analysis and Design and Iterative Development"_

### Why do we need GRASP?

When you write object-oriented code, the biggest question you face is:

> _"Who should do this job?"_

For example, if you have classes `Order`, `Customer`, `Payment`, and `Invoice`:

- Who creates the `Invoice` object?
- Who calculates the total?
- Who should handle payment?

Assigning responsibilities poorly leads to:

- Classes doing too many things (low cohesion)
- Classes tightly depending on each other (high coupling)
- Code that is hard to maintain, test, and extend

GRASP gives you **a mental toolset** — a set of rules/principles — to make these decisions wisely.

> 💡 According to Larman: _"The critical design tool for software development is a mind well educated in design principles. It is not UML or any other technology."_

### The 9 GRASP Principles at a Glance

|#|Principle|Core Question|
|---|---|---|
|1|Creator|Who should create object A?|
|2|Information Expert|Who should fulfill a responsibility?|
|3|Low Coupling|How to reduce dependencies?|
|4|Controller|Who receives and coordinates system events?|
|5|High Cohesion|How to keep a class focused?|
|6|Polymorphism|How to handle behavior varying by type?|
|7|Indirection|How to avoid direct coupling?|
|8|Pure Fabrication|What if no real domain class fits?|
|9|Protected Variations|How to shield from change/instability?|

### 1. Creator

#### Problem (Question it Answers)

> _"Who should be responsible for creating a new instance of some class?"_

#### Solution

Assign class **B** the responsibility to create an instance of class **A** if **one or more** of these are true (the more the better):

- **B "contains" or compositely aggregates A** — B holds A as a part of itself
- **B closely uses A** — B frequently calls methods on A
- **B records instances of A** — B maintains a collection of A objects
- **B has the initializing data for A** — B already has the information needed to construct A, making it an "expert" about A's creation

#### Deep Explanation

This principle says: _don't randomly assign object creation — give it to the class that has the strongest relationship with the new object._

Think of it this way: If a `Board` contains many `Square` objects, then `Board` is the most natural creator of `Square` objects. It already knows about them, holds them, and has the data to initialize them.

#### Example: Monopoly Game

- `Board` creates `Square` objects → because Board **contains/aggregates** Squares.
- `Player` creates `Token` objects → because Player closely **uses** Tokens.

#### Example: Order System

- `Order` creates `OrderLineItem` objects → because Order **contains** and **records** its line items. Order also has the initializing data (product, quantity, price) for each line item.

#### Java Illustration

```java
public class Order {
    private List<OrderLineItem> lineItems = new ArrayList<>();

    // Order is the Creator of OrderLineItem — it aggregates them
    public void addItem(Product product, int quantity) {
        OrderLineItem item = new OrderLineItem(product, quantity); // Creator in action
        lineItems.add(item);
    }
}
```

### 2. Information Expert

#### Problem (Question it Answers)

> _"What is a general principle of assigning responsibilities to objects?"_ _"Who should fulfill this responsibility?"_

#### Solution

Assign a responsibility to the **Information Expert** — the class that **has the information necessary to fulfill the responsibility**.

#### Deep Explanation

This is the most fundamental GRASP principle and is the basis for most object-oriented design decisions.

The idea is simple: **the class that knows the most about what is needed to do a job should be the one to do that job.**

If you need to calculate an order total, which class has the line items and their prices? → `Order`. So `Order` should calculate its own total. Not `Customer`, not `ShoppingCart`.

#### Why it matters

- It leads to objects that **encapsulate their own data and behavior** — the very essence of OO design.
- It avoids classes asking other classes for all their data and then computing something — that's a sign of a missing responsibility assignment.

#### Example: Online Store

- `SalesLineItem` calculates the subtotal (qty × price) → because it **knows** quantity and price.
- `Sale` calculates the total → because it **knows** all the `SalesLineItem` objects.
- `ProductCatalog` searches for a product → because it **knows** all products.

#### Java Illustration

```java
public class SalesLineItem {
    private int quantity;
    private Product product;

    // This class is the Information Expert for calculating its own subtotal
    public double getSubtotal() {
        return quantity * product.getPrice();
    }
}

public class Sale {
    private List<SalesLineItem> lineItems;

    // Sale is the Information Expert for the overall total
    public double getTotal() {
        return lineItems.stream()
                        .mapToDouble(SalesLineItem::getSubtotal)
                        .sum();
    }
}
```

### 3. Low Coupling

#### Problem (Question it Answers)

> _"How to support low dependency, low change impact, and increased reuse?"_

#### Solution

Assign responsibilities so that **coupling remains low**.

#### What is Coupling?

**Coupling** is the degree of interdependence between software modules — a measure of how closely two classes are connected.

- **High coupling** = Class A directly depends on many other classes. A change in any of them breaks A.
- **Low coupling** = Class A depends on as few other classes as possible. Changes elsewhere have minimal impact on A.

#### Types of Coupling (from worst to best)

|Type|Description|
|---|---|
|Content coupling|One class modifies internals of another (worst)|
|Common coupling|Multiple classes share global data|
|Control coupling|One class controls the logic of another|
|Stamp coupling|Passing unnecessary data structures|
|Data coupling|Only passing necessary simple data (best acceptable)|

#### Why it matters

- Low coupling promotes **reuse** — a self-contained class is easy to drop into another project.
- Low coupling **reduces the ripple effect** — changing one class doesn't cascade into many changes elsewhere.
- Low coupling makes code **easier to test** — each class can be tested in isolation.

#### Example: Monopoly Game

- `Dice` only knows how to roll (returns a number). It doesn't know about `Board`, `Player`, or `Game`. → **Low coupling** — Dice is completely independent and reusable.

#### Anti-Example (High Coupling)

```java
// BAD — Dice is tightly coupled to Board and Player
public class Dice {
    private Board board;
    private Player player;

    public int roll() {
        int result = (int)(Math.random() * 6) + 1;
        player.move(result, board); // Dice should NOT be doing this!
        return result;
    }
}
```

#### Good Example (Low Coupling)

```java
// GOOD — Dice knows nothing about Board or Player
public class Dice {
    public int roll() {
        return (int)(Math.random() * 6) + 1; // Just rolls. Nothing else.
    }
}
```

### 4. Controller

#### Problem (Question it Answers)

> _"What first object beyond the UI layer receives and coordinates ('controls') a system operation?"_

#### Solution

Assign the responsibility to a class that represents one of these:

1. **Facade Controller** — represents the overall "system", a "root object", a device, or a major subsystem (e.g., `System`, `Device`, `Subsystem`).
2. **Use Case Controller** — represents a use case scenario, often named `<UseCaseName>Handler`, `<UseCaseName>Coordinator`, or `<UseCaseName>Session` (e.g., `ProcessSaleHandler`, `LoginSession`).

#### Deep Explanation

The Controller sits **between the UI (View) layer and the domain (Model) layer**.

The key idea is: **the UI should not contain business logic**. When a button is clicked on a UI, the UI should call a Controller method. The Controller then coordinates the business objects (domain objects) to carry out the system operation.

Think of it like a **traffic director**: it doesn't do the actual work (move cars), but it coordinates who does what and when.

#### Why it matters

- Separates the presentation layer from business logic
- Keeps domain objects UI-independent (you could switch from a GUI to a command line without changing domain objects)
- Makes the system easier to test (test business logic without a UI)

#### Example: Monopoly Game

- `MonopolyGame` is the Controller — it orchestrates the flow (start game, play turn, end game). The UI simply calls `monopolyGame.startTurn()`.

#### Example: Hospital System

- `HospitalSystem` manages login, appointment booking, billing. It coordinates between `Patient`, `Doctor`, `Appointment`, and `Billing` objects.

#### Java Illustration

```java
// The Controller — sits between UI and domain objects
public class OrderController {
    private Order currentOrder;
    private PaymentProcessor paymentProcessor;
    private ReceiptPrinter receiptPrinter;

    // UI calls this — Controller coordinates everything
    public void placeOrder(List<Product> products, String paymentMethod) {
        currentOrder = new Order(products);           // delegates to Creator
        double total = currentOrder.getTotal();       // delegates to Expert
        paymentProcessor.process(total, paymentMethod); // delegates to PaymentProcessor
        receiptPrinter.print(currentOrder);           // delegates to Printer
    }
}
```

### 5. High Cohesion

#### Problem (Question it Answers)

> _"How to keep objects focused, understandable, and manageable, while also supporting Low Coupling?"_

#### Solution

Assign responsibilities so that **cohesion remains high**.

#### What is Cohesion?

**Cohesion** is the degree to which the elements inside a module belong together — how closely related and focused the responsibilities of a class are.

- **High cohesion** = A class does one thing and does it well. All its methods and data serve a single, unified purpose.
- **Low cohesion** = A class does many unrelated things. It's a "god class" — it knows too much and does too much.

#### Why it matters

- High cohesion makes classes **easier to understand** — you can read a class and immediately know what it does.
- High cohesion makes classes **easier to maintain** — changes to one responsibility don't accidentally affect another.
- High cohesion and Low Coupling go hand-in-hand: a class with high cohesion naturally has fewer reasons to depend on other classes.

#### Signs of Low Cohesion

- The class has many unrelated methods
- The class name is vague ("Manager", "Helper", "Utility", "Controller" doing everything)
- It's hard to describe what the class does without using "and"

#### Example: Monopoly Game

- `Player` keeps all player-related responsibilities: money, properties, movement position. High cohesion.
- But if `Player` also contained dice-rolling logic, bank transaction logic, and board rendering — that's low cohesion.

#### Java Illustration

```java
// LOW COHESION — Patient does too many unrelated things
public class Patient {  // BAD
    void storePatientDetails() { ... }
    void scheduleAppointment() { ... }  // Should be in AppointmentService
    void generateBill() { ... }         // Should be in BillingService
    void saveMedicalRecord() { ... }    // Should be in MedicalRecordService
}

// HIGH COHESION — Patient only stores patient details
public class Patient {  // GOOD
    private String name;
    private String id;
    private int age;

    // Only patient-related behavior
    public int calculateAge() { return age; }
    public String getFullDetails() { return id + ": " + name; }
}
```

### 6. Polymorphism

#### Problem (Question it Answers)

> _"How to handle alternatives based on type? How to create pluggable software components?"_

#### Solution

When related alternatives or behaviors vary by type (class), assign responsibility for the behavior — using **polymorphic operations** — to the types for which the behavior varies.

#### Deep Explanation

Instead of writing `if (type == "cat") { ... } else if (type == "dog") { ... }` — use polymorphism. Define a common interface/abstract method, and let each subclass define its own behavior.

This is the classic principle: **"Program to an interface, not an implementation."**

#### Why it matters

- Adding a new type (e.g., a new payment method) doesn't require modifying existing code — just add a new class.
- Makes the system **open for extension, closed for modification** (ties directly to OCP in SOLID).
- Makes behavior **predictable** — callers don't need to know the specific type; they just call the method.

#### Example: Monopoly Game

- `SquareAction` interface allows different square types (`PropertySquare`, `ChanceSquare`, `JailSquare`) to define their own `landOn(player)` behavior.
- When a player lands on a square: `square.landOn(player)` — the right behavior fires automatically.

#### Java Illustration

```java
// Polymorphism via interface
public interface SquareAction {
    void landOn(Player player);
}

public class PropertySquare implements SquareAction {
    public void landOn(Player player) {
        player.payRent(calculateRent()); // Collect rent
    }
}

public class ChanceSquare implements SquareAction {
    public void landOn(Player player) {
        drawChanceCard(player); // Draw a card
    }
}

public class JailSquare implements SquareAction {
    public void landOn(Player player) {
        player.sendToJail(); // Go to jail!
    }
}

// Client code — no if-else needed!
public void processLanding(Player player, SquareAction square) {
    square.landOn(player); // Polymorphism handles the rest
}
```

### 7. Indirection

#### Problem (Question it Answers)

> _"Where to assign a responsibility to avoid direct coupling between two or more things? How to de-couple objects so that low coupling is supported and reuse potential remains higher?"_

#### Solution

Assign the responsibility to an **intermediate object** to mediate between other components or services so that they are not directly coupled.

#### Deep Explanation

Indirection introduces a "middleman" — an intermediary class that sits between two classes that would otherwise need to know about each other directly.

This is not the same as Polymorphism (which focuses on type-based behavior variation) or Protected Variations (which focuses on shielding from instability). Indirection is purely about **reducing direct coupling** through a mediator.

#### Real-World Analogy

A **travel agent** is an intermediary between you and airlines, hotels, and car rentals. You don't call each airline directly — you go through the agent. If the airline changes its booking process, you don't need to change how you book through the agent.

#### Example: Monopoly Game

- `TradeManager` mediates trades between players. `Player1` doesn't talk to `Player2` directly — both communicate through `TradeManager`.

#### Example: Hospital System

- `AppointmentManager` mediates between `Patient` and `Doctor`. Patient doesn't directly call Doctor's methods.

#### Java Illustration

```java
// WITHOUT Indirection — Patient and Doctor directly coupled
public class Patient {
    private Doctor doctor;
    public void bookAppointment(String time) {
        doctor.addAppointment(this, time); // Direct coupling!
    }
}

// WITH Indirection — AppointmentManager as intermediary
public class AppointmentManager {
    public Appointment book(Patient patient, Doctor doctor, String time) {
        if (doctor.isAvailable(time)) {
            return new Appointment(patient, doctor, time);
        }
        return null;
    }
}
// Now Patient and Doctor don't know about each other's internals
```

### 8. Pure Fabrication

#### Problem (Question it Answers)

> _"What object should have the responsibility, when you do not want to violate High Cohesion and Low Coupling, but solutions offered by Information Expert are not appropriate?"_

#### Solution

Assign a highly cohesive set of responsibilities to an **artificial or convenience class** that does not represent a problem domain concept — something **made up** to support high cohesion, low coupling, and reuse.

#### Deep Explanation

Sometimes, if you strictly follow the Information Expert principle, you'd be forced to give a domain class (like `Customer`, `Order`, `Patient`) responsibilities that don't naturally belong to it — like saving itself to a database, generating reports, or sending emails.

Giving `Customer` a `saveToDatabase()` method violates High Cohesion (Customer is a business concept, not a persistence concept) and High Coupling (Customer now depends on a database layer).

The solution: **invent a new class that doesn't exist in the real world but serves a design purpose.** This class is the Pure Fabrication.

Examples of Pure Fabrications:

- `DataPersistence` / `Repository` — handles saving objects to database
- `ReportGenerator` — handles report creation
- `NotificationService` — handles sending notifications
- `Bank` in Monopoly — handles financial transactions (there's no physical Bank class in the real Monopoly box)

#### Example: Monopoly Game

- `Bank` handles all financial transactions, keeping monetary logic separate from `Player`. Bank doesn't represent a game piece — it's a fabricated convenience class.

#### Java Illustration

```java
// WITHOUT Pure Fabrication — Customer is overloaded
public class Customer {  // BAD
    String name;
    void placeOrder() { ... }
    void saveToDatabase() { ... }  // Persistence — not Customer's job!
    void sendWelcomeEmail() { ... } // Notification — not Customer's job!
}

// WITH Pure Fabrication — clean separation
public class Customer {  // Domain class stays clean
    String name;
    String email;
    // Only customer-relevant behavior
}

public class DataPersistence {  // Pure Fabrication
    public void save(Customer customer) {
        // Handle all DB logic here
    }
}

public class NotificationService {  // Pure Fabrication
    public void sendWelcomeEmail(Customer customer) {
        // Handle email logic here
    }
}
```

### 9. Protected Variations

#### Problem (Question it Answers)

> _"How to design objects, subsystems, and systems so that variations or instability in these elements does not have an undesirable impact on other elements?"_

#### Solution

**Identify points of predicted variation or instability; assign responsibilities to create a stable interface around them.**

#### Deep Explanation

Protected Variations is arguably the most powerful GRASP principle. The idea:

1. Find the things that are likely to **change** in your system (requirements change, algorithms change, databases change, APIs change).
2. Wrap those unstable points in a **stable interface/abstraction**.
3. The rest of the system talks to the stable interface — it never directly touches the unstable thing.

This is also described as: _"Encapsulate the concept that varies."_

When the unstable thing changes (e.g., you switch from MySQL to MongoDB), only the class behind the interface changes. Everything that uses the interface is **protected** from that variation.

#### Mechanisms used to achieve Protected Variations

- **Interfaces and Abstract Classes** — the primary tools
- **Polymorphism** — lets different implementations hide behind the same interface
- **Indirection** — puts a layer between client and volatile component

#### Difference between Indirection and Protected Variations

||Indirection|Protected Variations|
|---|---|---|
|Focus|How two objects communicate (reduce coupling)|Where the system is likely to change|
|Goal|Remove direct dependency|Shield from instability/change|
|Mechanism|Intermediary object|Stable interface/abstraction around volatile point|

#### Example: Monopoly Game

- `JailSquare` encapsulates all jail logic (can enter jail, must roll doubles, pay $50, etc.). If the jail rules change, only `JailSquare` changes — other classes are **protected**.

#### Java Illustration

```java
// Protected Variations — payment method is likely to change
public interface PaymentGateway {  // Stable interface
    void processPayment(double amount);
}

public class StripeGateway implements PaymentGateway {
    public void processPayment(double amount) { /* Stripe-specific code */ }
}

public class PayPalGateway implements PaymentGateway {
    public void processPayment(double amount) { /* PayPal-specific code */ }
}

// OrderService is PROTECTED from payment gateway variations
public class OrderService {
    private PaymentGateway gateway; // Talks to interface, not concrete class

    public void checkout(Order order) {
        gateway.processPayment(order.getTotal()); // Doesn't care which gateway
    }
}
```

## 1.5 Bad Code Examples & GRASP Fixes

### Example 1: Order Class (Low Cohesion)

**Problem:**

```java
// BAD — Order class is overloaded
public class Order {
    void addItem(Product p) { ... }
    double calculateTotal() { ... }
    void processPayment(String method) { ... }  // ← Should be PaymentProcessor
    void printReceipt() { ... }                 // ← Should be ReceiptPrinter
}
```

**Issues:**

- Low Cohesion: Order manages items, calculates totals, processes payments, AND prints receipts
- Violation of Controller Principle: Payment logic should be in a dedicated service
- Poor Separation of Concerns: Business logic mixed with infrastructure

**GRASP Fix:**

```java
public class Order { /* Only tracks items and totals */ }
public class PaymentProcessor { /* Handles payment logic */ }
public class ReceiptPrinter { /* Handles printing */ }
public class OrderController { /* Coordinates all of the above */ }
```

---

### Example 2: Library Class (High Coupling)

**Problem:** `Library` class mixes book storage, authentication, notifications, and search.

**GRASP Fix:**

```java
public class Library { /* Manages books only */ }
public class AuthService { /* Login/authentication */ }
public class NotificationService { /* Sends notifications */ }
public class SearchService { /* Encapsulates search logic */ }
public class LibraryController { /* Coordinates between all services */ }
```

---

### Example 3: BankAccount Class (Overloaded)

**Problem:** `BankAccount` stores account info, handles login, approves loans, and sends notifications.

**GRASP Fix:**

```java
public class BankAccount { /* Only stores account details and balance */ }
public class AuthService { /* Login/authentication */ }
public class LoanService { /* Loan approval logic */ }
public class NotificationService { /* Sends notifications */ }
public class BankController { /* Coordinates between services */ }
```

## ✅ Q&A Bank: GRASP

**Q1. What does GRASP stand for and who introduced it?**

> GRASP stands for **General Responsibility Assignment Software Patterns (Principles)**. It was introduced by **Craig Larman** in his 1997 book _"Applying UML and Patterns."_

**Q2. How many GRASP principles are there? Name them all.**

> There are **9** GRASP principles: Creator, Information Expert, Low Coupling, Controller, High Cohesion, Polymorphism, Indirection, Pure Fabrication, and Protected Variations.

**Q3. What is the Creator principle? When should B create A?**

> Creator says: assign B the responsibility to create A if: B contains/aggregates A, B closely uses A, B records A objects, or B has the data needed to initialize A. The more of these that apply, the stronger the case.

**Q4. What is the Information Expert principle? Give an example.**

> Assign the responsibility to the class that has the information necessary to fulfill it. Example: `Sale` computes the total because it has all `SalesLineItem` objects. `SalesLineItem` computes its subtotal because it has quantity and price.

**Q5. Distinguish between Coupling and Cohesion.**

> **Coupling** measures how much one class depends on another (we want LOW coupling). **Cohesion** measures how focused/unified the responsibilities within one class are (we want HIGH cohesion). Both are design quality metrics — high cohesion usually leads to low coupling naturally.

**Q6. What is the role of a Controller in GRASP?**

> A Controller is the first object beyond the UI layer that receives and coordinates system operations. It acts as a traffic director — delegating work to domain objects. It can be a facade controller (representing the whole system) or a use-case controller (for a specific use case).

**Q7. What is the difference between Indirection and Protected Variations?**

> **Indirection** focuses on HOW two objects communicate — it introduces a middleman to avoid direct coupling. **Protected Variations** focuses on WHERE the system is likely to change — it builds a stable interface/abstraction around unstable/volatile points to shield other classes from change.

**Q8. What is Pure Fabrication? Why is it used?**

> Pure Fabrication is an invented class that has no counterpart in the real-world problem domain. It is used when giving a responsibility to a domain class would violate High Cohesion or Low Coupling. Example: `DataPersistence` class that saves objects to DB — there's no "persistence" in the real business domain.

**Q9. In the Monopoly game, which GRASP principle does the Bank class illustrate?**

> **Pure Fabrication** — `Bank` handles financial transactions but doesn't represent an actual game piece. It is invented to separate financial logic from `Player`.

**Q10. How does Polymorphism relate to the Open/Closed Principle?**

> Polymorphism allows adding new types/behaviors without modifying existing code. When you add a new `SquareAction` subclass, the rest of the code (`processLanding()`) doesn't change. This directly satisfies OCP (Open for extension, Closed for modification).

**Q11. What is the problem with the following code, and how would you fix it using GRASP?**

```java
public class Patient {
    void storeDetails() { ... }
    void scheduleAppointment() { ... }
    void generateBill() { ... }
    void saveMedicalRecord() { ... }
}
```

> **Problem:** Low Cohesion — Patient class has too many responsibilities. Violation of Information Expert and Controller principles. **Fix:** Split into `Patient` (details only), `AppointmentService`, `BillingService`, `MedicalRecordService`, and a `HealthcareController` to coordinate.

**Q12. Why is it important for Dice to have Low Coupling in the Monopoly game?**

> Because `Dice` only rolls and returns a number. It doesn't know about `Board`, `Player`, or `Game`. This makes `Dice` independently reusable and testable. Changes to other classes don't affect `Dice`, and vice versa.

**Q13. GRASP is called a "mental toolset." What does this mean?**

> It means GRASP is not a tool you install or a library you use. It's a way of thinking — a set of guiding principles you internalize and apply mentally when making OO design decisions. The principles help you ask the right questions: "Who should do this?", "Who knows this?", "How dependent should these be?"

# 2. Features of Bad Design

Before learning how to write good design (SOLID), it's crucial to recognize **bad design**. There are four main symptoms of bad design:

---

### 1. Rigidity

- The impact of a change is **unpredictable**
- Every change causes a **cascade of changes** in dependent modules
- Costs become unpredictable (you can't estimate how long a change will take)

_Example:_ Changing the email format requires modifying 10 other classes that somehow all depend on email formatting.

---

### 2. Fragility

- The software tends to **break in many places** on every change
- The breakage occurs in areas with **no conceptual relationship** to what you changed
- On every fix, the software breaks in unexpected ways

_Example:_ You fix the login bug, and suddenly the invoice generation breaks — even though they seem unrelated.

---

### 3. Immobility

- It's almost impossible to **reuse** interesting parts of the software
- Useful modules have **too many dependencies** — you can't extract them without dragging along 20 other classes
- The cost of **rewriting is less** than the risk of separating those parts

_Example:_ You want to reuse the `Report` class in another project, but it depends on `Database`, `UserSession`, `Config`, `Logger`, and five other classes.

---

### 4. Viscosity

- A **hack is cheaper to implement** than the correct solution within the design
- Preserving design moves are difficult to think about and implement
- It's much easier to do the **wrong thing rather than the right one**

_Example:_ To add a new feature properly would require refactoring 5 classes, but a quick-and-dirty if-else in one file gets it working. Developers inevitably choose the hack.

---

> 💡 **The solution to all four problems:** Follow the **SOLID principles**.

# 3. SOLID Principles

**SOLID** is an acronym for five design principles introduced by **Robert C. Martin** (Uncle Bob) in his 2000 paper _"Design Principles and Design Patterns."_ The acronym was later formalized by **Michael Feathers**.

|Letter|Principle|One-line Summary|
|---|---|---|
|**S**|Single Responsibility Principle (SRP)|One class = one reason to change|
|**O**|Open/Closed Principle (OCP)|Open for extension, closed for modification|
|**L**|Liskov Substitution Principle (LSP)|Subclasses must be substitutable for superclasses|
|**I**|Interface Segregation Principle (ISP)|Don't force classes to implement methods they don't use|
|**D**|Dependency Inversion Principle (DIP)|Depend on abstractions, not concretions|

---

## 3.1 Single Responsibility Principle (SRP)

> **"A class should have one, and only one, reason to change."**

### What it Means

Every class should have **exactly one responsibility** — one cohesive area of functionality. If a class has more than one reason to change (more than one responsibility), it should be split.

### The Simple Test

> _What is the responsibility of your class?_ If your answer includes the word **"and"**, you're probably violating SRP.

- `Order manages items AND processes payments` → Violation!
- `Order manages order line items` → ✅ Good.

### Benefits

- **Fewer test cases** — a class with one responsibility is simpler to test
- **Lower coupling** — less functionality means fewer dependencies
- **Better organization** — small, well-named classes are easy to find and understand
- **Reduced side effects** — changing one responsibility doesn't accidentally affect others

### Example: BankService (SRP Violation)

```java
// BAD — BankService has multiple responsibilities
public class BankService {
    void depositMoney() { ... }
    void withdrawMoney() { ... }
    void printPassbook() { ... }   // ← Printing responsibility
    void getLoanInterestInfo() { ... } // ← Loan responsibility
    void sendOTP() { ... }         // ← Notification responsibility
}
```

This class has at least 3 reasons to change:

1. The bank wants to support other kinds of loans
2. Sending OTP could change to WhatsApp
3. Passbook printing format could change

### SRP Fix: Split into focused classes

```java
public class BankService {      // Core banking operations only
    void depositMoney() { ... }
    void withdrawMoney() { ... }
}

public class PrinterService {   // Single responsibility: printing
    void printPassbook(Account account) { ... }
}

public class LoanService {      // Single responsibility: loan operations
    double getLoanInterestInfo(String loanType) { ... }
}

public class NotificationService { // Single responsibility: notifications
    void sendOTP(String phone) { ... }
    void sendEmail(String email, String message) { ... }
}
```

## 3.2 Open/Closed Principle (OCP)

> **"Classes should be open for extension but closed for modification."**

### What it Means

You should be able to **add new functionality** to a class/module without changing its existing source code.

- **Open for extension** → You can extend behavior through subclassing or implementing interfaces
- **Closed for modification** → You shouldn't need to touch existing, working code to add new features

### Why it Matters

Every time you modify existing code, you risk introducing new bugs. OCP says: _"Leave working code alone — extend it instead."_

### Exception

The one exception: **bug fixes** in existing code are allowed (you're fixing broken behavior, not adding new behavior).

### Example: Notification (OCP Applied)

```java
// BAD — Adding WhatsApp requires modifying NotificationService
public class NotificationService {
    void send(String type, String message) {
        if (type.equals("email")) { sendEmail(message); }
        else if (type.equals("sms")) { sendSMS(message); }
        // To add WhatsApp, we must MODIFY this class — violates OCP!
        else if (type.equals("whatsapp")) { sendWhatsApp(message); }
    }
}
```

```java
// GOOD — Open for extension (new channels), closed for modification
public interface NotificationChannel {   // Abstraction — stable
    void send(String message);
}

public class EmailNotification implements NotificationChannel {
    public void send(String message) { /* Email logic */ }
}

public class SMSNotification implements NotificationChannel {
    public void send(String message) { /* SMS logic */ }
}

public class WhatsAppNotification implements NotificationChannel {  // NEW — no existing code touched!
    public void send(String message) { /* WhatsApp logic */ }
}

public class NotificationService {
    void notify(NotificationChannel channel, String message) {
        channel.send(message); // Works with any current or future channel
    }
}
```

## 3.3 Liskov Substitution Principle (LSP)

> **"Objects of a superclass should be replaceable with objects of its subclasses without breaking the application."**

### What it Means

If class `B` is a subclass of class `A`, you should be able to use a `B` object wherever an `A` object is expected — and the program should still work correctly.

In other words: **subclasses must honor the contract of the parent class.**

### Rules for LSP Compliance

1. Don't implement **stricter validation rules** on input parameters than the parent class
2. Apply at **least the same rules** to all output parameters as the parent class
3. Don't **throw new exceptions** that the parent doesn't throw
4. Don't **override** a method in a way that breaks parent class behavior

### Classic Violation: Penguin extends Bird

```java
// VIOLATION — Penguin can't fly, but Bird.fly() assumes all birds can
public class Bird {
    public void fly() { System.out.println("Flying!"); }
    public void eat() { System.out.println("Eating!"); }
}

public class Penguin extends Bird {
    @Override
    public void fly() {
        throw new UnsupportedOperationException("Penguins can't fly!"); // BREAKS LSP!
    }
}

// Client code breaks when a Penguin is substituted:
List<Bird> birds = Arrays.asList(new Eagle(), new Penguin());
for (Bird b : birds) {
    b.fly(); // CRASH! When Penguin is encountered
}
```

### LSP Fix: Use proper abstraction

```java
// All birds eat, but only some fly → separate the interface
public abstract class Bird {
    public abstract void eat();
}

public interface Flyable {
    void fly();
}

public class Eagle extends Bird implements Flyable {
    public void eat() { System.out.println("Eagle eating"); }
    public void fly() { System.out.println("Eagle flying"); }
}

public class Penguin extends Bird {
    public void eat() { System.out.println("Penguin eating"); }
    // No fly() — Penguins simply don't implement Flyable. No violation!
}

// Client code is now safe:
List<Bird> birds = Arrays.asList(new Eagle(), new Penguin());
for (Bird b : birds) {
    b.eat(); // Works for all birds ✅
}
```

### Social Media Example from Notes

```java
// Facebook can substitute SocialMedia ✅
// WhatsApp cannot substitute SocialMedia ❌ (doesn't have all social media methods)
// Instagram cannot substitute SocialMedia ❌

// Fix: Define separate interfaces for what each supports
public interface Messaging { void sendMessage(); }
public interface PhotoSharing { void sharePhoto(); }
public interface StatusUpdate { void postStatus(); }

public class Facebook implements Messaging, PhotoSharing, StatusUpdate { ... }
public class WhatsApp implements Messaging { ... }  // Only messaging
public class Instagram implements PhotoSharing, StatusUpdate { ... }  // No direct messaging
```

## 3.4 Interface Segregation Principle (ISP)

> **"Clients should not be forced to depend upon interfaces that they do not use."**

### What it Means

Don't create **fat interfaces** — interfaces with too many methods. Instead, break them into **smaller, focused interfaces** so that implementing classes only need to implement the methods relevant to them.

### Why it Matters

When a class is forced to implement an interface method it doesn't need, it typically either:

- Leaves the method empty (bad)
- Throws `UnsupportedOperationException` (violates LSP too)
- Puts dummy code (misleading)

### Similarity to SRP

Both ISP and SRP aim for **small, focused, highly cohesive components**:

- SRP → applies to **classes**
- ISP → applies to **interfaces**

### Example of ISP Violation

```java
// FAT interface — not all devices can do all these things
public interface MultifunctionDevice {
    void print(Document d);
    void scan(Document d);
    void fax(Document d);
    void staple(Document d);
}

// OldPrinter only prints — but forced to implement ALL methods
public class OldPrinter implements MultifunctionDevice {
    public void print(Document d) { /* real implementation */ }
    public void scan(Document d) { throw new UnsupportedOperationException(); }   // Forced!
    public void fax(Document d) { throw new UnsupportedOperationException(); }    // Forced!
    public void staple(Document d) { throw new UnsupportedOperationException(); } // Forced!
}
```

### ISP Fix: Split into smaller interfaces

```java
public interface Printable { void print(Document d); }
public interface Scannable { void scan(Document d); }
public interface Faxable   { void fax(Document d); }

// OldPrinter only implements what it supports
public class OldPrinter implements Printable {
    public void print(Document d) { /* implementation */ }
}

// Modern all-in-one machine implements all
public class AllInOnePrinter implements Printable, Scannable, Faxable {
    public void print(Document d) { ... }
    public void scan(Document d) { ... }
    public void fax(Document d) { ... }
}
```

## 3.5 Dependency Inversion Principle (DIP)

> **"High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend upon abstractions."**

### What it Means

Two key inversions:

1. High-level modules (business logic) should NOT directly use low-level modules (database, email, file system). Both should use an abstraction (interface).
2. Abstractions should be designed independently — the concrete details conform to the abstraction, not the other way around.

### Why "Inversion"?

In traditional procedural design, high-level modules directly call low-level modules. DIP **inverts** this dependency: both layers point to an abstraction in the middle. The concrete (low-level) details depend on the abstract, not the abstract on the concrete.

### The Bookshelf Example

```java
// VIOLATION — Shelf directly depends on Book (concrete, low-level)
public class Book {
    void seeReviews() { ... }
    void readSample() { ... }
}

public class Shelf {
    private List<Book> books = new ArrayList<>(); // Depends on Book — DIP violation!

    void addBook(Book book) { books.add(book); }
}

// Problem: When DVDs are added, Shelf must be modified — also breaks OCP!
public class DVD {
    void seeReviews() { ... }
    void watchSample() { ... }
}
// Now Shelf needs: List<Book> + List<DVD> — getting messy
```

### DIP Fix: Introduce Abstraction (Interface)

```java
// Abstraction — stable, won't change
public interface Product {
    void seeReviews();
}

// Low-level details DEPEND ON the abstraction
public class Book implements Product {
    public void seeReviews() { ... }
    public void readSample() { ... }
}

public class DVD implements Product {
    public void seeReviews() { ... }
    public void watchSample() { ... }
}

public class Magazine implements Product {  // Adding Magazine requires ZERO changes to Shelf
    public void seeReviews() { ... }
}

// High-level module DEPENDS ON abstraction — not on concrete classes
public class Shelf {
    private List<Product> products = new ArrayList<>(); // Depends on interface ✅

    void addProduct(Product p) { products.add(p); }
}
```

### Where is the "Inversion"?

- **Before DIP:** `Shelf` → `Book` (concrete)
- **After DIP:** `Shelf` → `Product` (interface) ← `Book` (concrete)

Both `Shelf` and `Book` now depend on `Product`. The direction of the arrow for `Book` is **inverted** — instead of Shelf depending on Book, Book depends on the interface that Shelf also depends on.

### Real-World DIP Example

```java
// Without DIP — OrderService depends directly on MySQL
public class OrderService {
    private MySQLDatabase db = new MySQLDatabase(); // Tightly coupled!
    public void saveOrder(Order o) { db.save(o); }
}

// With DIP
public interface Database { void save(Object o); }
public class MySQLDatabase implements Database { public void save(Object o) { ... } }
public class MongoDatabase implements Database { public void save(Object o) { ... } }

public class OrderService {
    private Database db; // Depends on abstraction!
    public OrderService(Database db) { this.db = db; } // Injected
    public void saveOrder(Order o) { db.save(o); }
}
// Now you can switch from MySQL to MongoDB without touching OrderService ✅
```

## ✅ Q&A Bank: SOLID

**Q1. What does SOLID stand for?**

> **S**ingle Responsibility, **O**pen/Closed, **L**iskov Substitution, **I**nterface Segregation, **D**ependency Inversion.

**Q2. Who introduced SOLID and when?**

> **Robert C. Martin** in his 2000 paper _"Design Principles and Design Patterns."_ The SOLID acronym was coined by **Michael Feathers**.

**Q3. What is SRP and how do you test if your class violates it?**

> SRP states a class should have only one reason to change. Test: describe your class's responsibility. If the description contains the word **"and"**, the class likely violates SRP.

**Q4. What is the exception to OCP?**

> **Bug fixes** in existing code are the one exception to the rule that classes should be closed for modification. You are fixing broken behavior, not adding new behavior.

**Q5. Explain LSP with the Penguin-Bird example.**

> `Bird` has a `fly()` method. `Penguin` extends `Bird` but can't fly, so it throws `UnsupportedOperationException`. When a `List<Bird>` iterates and calls `.fly()`, it crashes on `Penguin`. This violates LSP because `Penguin` cannot substitute for `Bird`. Fix: separate `Flyable` as an interface; only flying birds implement it.

**Q6. What is the difference between SRP and ISP?**

> Both aim for focused, cohesive components. **SRP** applies to **classes** (one responsibility per class). **ISP** applies to **interfaces** (don't force clients to implement methods they don't use — split fat interfaces into smaller ones).

**Q7. What problem does DIP solve?**

> DIP solves tight coupling between high-level and low-level modules. Without DIP, changing a low-level module (e.g., switching databases) breaks the high-level module. DIP inserts an abstraction (interface) between them so both can change independently.

**Q8. In the Shelf/Book/DVD example, how does DIP also help OCP?**

> By introducing the `Product` interface, `Shelf` no longer depends on concrete classes. When a new product type (like `Magazine`) is added, `Shelf` doesn't need to change — the new class just implements `Product`. This means the system is Open for extension (new Product types) and Closed for modification (Shelf is unchanged). DIP and OCP often work together.

**Q9. What are the four symptoms of bad design?**

> **Rigidity** (unpredictable change cascade), **Fragility** (breaks in unrelated areas), **Immobility** (parts can't be reused), and **Viscosity** (hacks are easier than proper solutions).

**Q10. How does LSP relate to OCP?**

> LSP extends OCP. OCP says extend without modifying; LSP says your extensions (subclasses) must honor the contract of the base class. Together they ensure you can add new types safely without breaking existing code that depends on the base type.

**Q11. Give a real-world scenario that demonstrates DIP.**

> A `NotificationService` (high-level) should not directly instantiate `EmailSender` (low-level). Instead, define a `MessageChannel` interface. Both `EmailSender` and `SMSSender` implement it. `NotificationService` is injected with a `MessageChannel`. You can switch from email to SMS without changing `NotificationService`.

# 4. Design Patterns — Introduction

## 4.1 History

The concept of design patterns in software was inspired by architect **Christopher Alexander**, who wrote in his 1977 book _A Pattern Language: Towns, Buildings, Construction_:

> _"Each pattern describes a problem which occurs over and over again in our environment, and then describes the core of the solution to that problem, in a way that you can use this solution a million times over, without ever doing it the same way twice."_

This idea was adapted to software by the **Gang of Four (GoF)** — four authors:

- **Erich Gamma**
- **Richard Helm**
- **Ralph Johnson**
- **John Vlissides**

Their 1994 book **"Design Patterns: Elements of Reusable Object-Oriented Software"** catalogued **23 design patterns** that became the canonical reference.

---

## 4.2 What is a Design Pattern?

A design pattern is:

1. **A reusable solution** to a commonly occurring problem in software design
2. **Not finished code** — it's a template or blueprint you customize for your context
3. **Language-independent** — the same pattern applies in Java, C++, Python, etc.
4. **Named** — giving it a name creates shared vocabulary ("Let's use a Singleton here")

> A design pattern shows the relationships and interactions between classes or objects, without specifying the final application classes.

---

## 4.3 The Gang of Four (GoF) — 23 Patterns in 3 Categories

### Creational Patterns (Object Creation)

Provide various object creation mechanisms that increase flexibility and reuse.

|Pattern|Purpose|
|---|---|
|**Abstract Factory**|Create families of related objects without specifying concrete classes|
|**Builder**|Construct complex objects step-by-step|
|**Factory Method**|Create objects without specifying the exact class|
|**Prototype**|Clone an existing object|
|**Singleton**|Ensure only one instance of a class exists|

### Structural Patterns (Object Relationships)

Provide ways to assemble objects and classes into larger structures.

|Pattern|Purpose|
|---|---|
|**Adapter**|Make incompatible interfaces work together|
|**Bridge**|Decouple abstraction from implementation|
|**Composite**|Treat a group of objects as a single object|
|**Decorator**|Extend behavior at runtime|
|**Facade**|Provide a simple interface to a complex subsystem|
|**Flyweight**|Reduce cost of many similar objects|
|**Proxy**|Provide a placeholder/surrogate for another object|

### Behavioral Patterns (Object Communication)

Handle algorithms and assignment of responsibilities between objects.

|Pattern|Purpose|
|---|---|
|**Chain of Responsibility**|Pass request along a chain of handlers|
|**Command**|Encapsulate an action as an object|
|**Interpreter**|Implement a specialized language|
|**Iterator**|Sequentially access elements of a collection|
|**Mediator**|Centralize complex communications between objects|
|**Memento**|Capture and restore an object's state|
|**Observer**|Notify subscribers of state changes (publish/subscribe)|
|**State**|Alter behavior when internal state changes|
|**Strategy**|Select an algorithm at runtime|
|**Template Method**|Define the skeleton of an algorithm in a base class|
|**Visitor**|Separate algorithm from the object structure it operates on|

---

## 4.4 Two Core Design Principles Behind GoF Patterns

The GoF patterns are primarily based on two OO design principles:

1. **Program to an interface, not an implementation**
    
    - Depend on abstract types, not concrete classes
    - Allows substitution of implementations
2. **Favor object composition over inheritance**
    
    - Combine simple objects to build complex behavior
    - More flexible than inheritance hierarchies

---

## 4.5 Architecture vs Design Patterns

||Architectural Patterns|Design Patterns|
|---|---|---|
|**Scope**|High-level, universal|Lower-level, specific|
|**Focus**|How components are organized and assembled|How individual components are built|
|**Examples**|MVC, Microservices, Layered, Event-Driven|Singleton, Observer, Factory, Strategy|
|**Granularity**|Large (defines system structure)|Small (defines class/object interactions)|

> **Granularity** = how small or big the components should be.

Architectural patterns define the granularity of the entire system's components. Design patterns operate within those components.

---

## 4.6 How is a Design Pattern Specified (Documentation)?

Every GoF design pattern is documented with these elements:

|Element|Description|
|---|---|
|**Pattern name & classification**|Concise name that becomes design vocabulary; classification (Creational/Structural/Behavioral)|
|**Intent**|Short statement of what the pattern does and which problem it solves|
|**Also known as**|Other names for the same pattern|
|**Motivation**|A scenario/story illustrating the problem and how the pattern solves it|
|**Applicability**|Situations where the pattern can be applied|
|**Structure**|UML diagram of the classes/objects in the pattern|
|**Participants**|Classes/objects involved and their responsibilities|
|**Collaborations**|How participants work together|
|**Consequences**|Costs and benefits of using the pattern|
|**Implementation**|Techniques and language-specific issues|
|**Sample code**|Code fragments showing the pattern in a specific language|
|**Known uses**|Real-world examples of the pattern in practice|
|**Related patterns**|Similar patterns and how they differ|

---

## 4.7 Pros and Cons of Design Patterns

### Pros

- **Consistency** — solving similar problems the same way across teams and languages
- **Clarity** — enables a common design vocabulary ("It's a Strategy pattern")
- **Faster development** — patterns are proven templates; no need to reinvent
- **Reuse** — patterns promote composition and reuse

### Cons

- **Object proliferation** — some patterns create many small classes/objects
- **Performance overhead** — extra layers of abstraction can slow things down
- **Misuse** — patterns can be overused or applied to the wrong problem
- **Subjectivity** — consequences vary depending on the specific situation
- **Anti-Patterns** — misapplied or overused patterns become anti-patterns (solutions that cause more problems than they solve)

---

## 4.8 How to Select a Design Pattern

When you face a design problem and need to pick a pattern:

1. **Consider how patterns solve design problems** — think about object granularity, interfaces, relationships
2. **Scan intent sections** — read the one-line "Intent" of each pattern to find candidates
3. **Study how patterns interrelate** — related patterns often work together
4. **Study patterns of like purpose** — compare similar patterns (e.g., Factory vs Abstract Factory vs Builder)
5. **Examine causes of redesign** — identify your pain points and find patterns that address them
6. **Consider what should be variable** — find what needs to change, and look for patterns that encapsulate that variation

---

## 4.9 How to Use a Design Pattern (Steps)

1. **Read the pattern overview** — focus on Applicability and Consequences to confirm it fits
2. **Study Structure, Participants, Collaborations** — understand the class relationships
3. **Look at sample code** — see a concrete implementation
4. **Choose meaningful names** — rename pattern participants to fit your context (e.g., instead of `ConcreteStrategy`, use `GridLayoutStrategy`)
5. **Define the classes** — declare interfaces, inheritance, instance variables
6. **Define operation names** — use responsibilities and collaborations as naming guides
7. **Implement the operations** — follow implementation hints from the pattern documentation

---

# 5. Creational Design Patterns

> Creational design patterns provide various **object creation mechanisms** that increase flexibility and reuse of existing code.

The five creational patterns are:

1. **Singleton** — one instance only
2. **Factory Method** — let subclasses decide what to create
3. **Abstract Factory** — factory of factories (create families of objects)
4. **Builder** — construct complex objects step by step
5. **Prototype** — clone existing objects

---

## 5.1 Singleton Pattern

### Intent

Ensure that a class has **only one instance** and provide a **global access point** to it.

### When to Use

- You need exactly one object to coordinate actions across the system
- Examples: **logging**, **driver objects**, **caching**, **thread pools**, **configuration**
- Used in core Java classes: `java.lang.Runtime`, `java.awt.Desktop`
- Also used internally by: Abstract Factory, Builder, Prototype, Facade patterns

### Implementation (Three Required Components)

1. **Private constructor** — prevents instantiation from outside the class
2. **Private static instance variable** — holds the single instance
3. **Public static `getInstance()` method** — the global access point

### Basic Implementation

```java
public class Singleton {
    // 2. Private static variable — the single instance
    private static Singleton instance;

    // 1. Private constructor — no one else can create an instance
    private Singleton() {
        System.out.println("Singleton instance created.");
    }

    // 3. Public static method — global access point
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton(); // Created only once
        }
        return instance;
    }

    public void doSomething() {
        System.out.println("Singleton doing something...");
    }
}

// Usage:
public class Main {
    public static void main(String[] args) {
        Singleton s1 = Singleton.getInstance();
        Singleton s2 = Singleton.getInstance();

        System.out.println(s1 == s2); // true — same object!
        s1.doSomething();
    }
}
```

### Thread-Safe Singleton (Double-Checked Locking)

```java
public class ThreadSafeSingleton {
    private static volatile ThreadSafeSingleton instance;

    private ThreadSafeSingleton() {}

    public static ThreadSafeSingleton getInstance() {
        if (instance == null) {
            synchronized (ThreadSafeSingleton.class) {
                if (instance == null) {  // Double check inside sync block
                    instance = new ThreadSafeSingleton();
                }
            }
        }
        return instance;
    }
}
```

### Pros of Singleton

- Controlled access to the single instance
- Reduced namespace (no global variables)
- Allows refinement of operations and representations through subclassing
- Permits variable number of instances (can be extended)

### Cons of Singleton

- Difficult to unit test (global state)
- Can violate SRP (the class manages its own lifecycle)
- Can cause issues in multithreaded environments if not implemented carefully

---

## 5.2 Factory Method Pattern

### Intent

Define an interface for creating an object, but **let the subclasses decide which class to instantiate**.

> "Just define an interface or abstract class for creating an object but let the subclasses decide which class to instantiate. Subclasses are responsible for creating the instance of the class."

### Problem it Solves

When you have a superclass with multiple subclasses and need to return one subclass based on input — instead of using `new SubclassA()` scattered throughout your code, you centralize creation in a **factory class**.

The responsibility of object instantiation is moved from the **client** to the **factory**.

### Key Benefit

- Adding new product types doesn't require modifying client code
- Follows OCP: Open for extension (new products), Closed for modification (factory interface unchanged)
- Loose coupling: client only knows about the product interface

### Example: Notification System

```java
// Step 1: Common interface (Product)
public interface Notification {
    void send(String message);
}

// Step 2: Concrete implementations (Concrete Products)
public class EmailNotification implements Notification {
    public void send(String message) {
        System.out.println("Sending EMAIL: " + message);
    }
}

public class SMSNotification implements Notification {
    public void send(String message) {
        System.out.println("Sending SMS: " + message);
    }
}

public class PushNotification implements Notification {
    public void send(String message) {
        System.out.println("Sending PUSH: " + message);
    }
}

// Step 3: Factory class (Creator)
public class NotificationFactory {
    public static Notification createNotification(String type) {
        switch (type.toLowerCase()) {
            case "email": return new EmailNotification();
            case "sms":   return new SMSNotification();
            case "push":  return new PushNotification();
            default: throw new IllegalArgumentException("Unknown type: " + type);
        }
    }
}

// Step 4: Client code — doesn't know or care about concrete classes
public class Main {
    public static void main(String[] args) {
        Notification n = NotificationFactory.createNotification("email");
        n.send("Your order has been placed!"); // Sends EMAIL
    }
}
```

### Real-World Use Cases

- **Database Access:** `DatabaseConnectionFactory` creates MySQL, PostgreSQL, or Oracle connection objects based on configuration
- **Cross-Platform UI:** Render Windows buttons vs macOS buttons using platform-specific factory
- **File I/O:** `FileReaderFactory` returns readers for CSV, JSON, or plain text

### Pros of Factory Method

- Encapsulates object creation (hides it from client)
- Easy to add new product types without changing client code (OCP)
- Centralizes creation — reduces redundancy
- Consistent object creation through a common interface
- Important in dependency injection frameworks

### Cons of Factory Method

- Increases overall code complexity
- Adds an extra abstraction layer
- Client code is still coupled to the concrete factory
- Overkill for simple, non-variable creation scenarios
- Can be overused

## 5.3 Abstract Factory Pattern

### Intent

Provide an interface for **creating families of related or dependent objects** without specifying their concrete classes.

> Think of it as: **"A factory of factories."**

### How it Differs from Factory Method

||Factory Method|Abstract Factory|
|---|---|---|
|Creates|One product|A family of related products|
|Tool|Inheritance (derived class implements factory method)|Delegation (client uses a factory object)|
|Scope|Over a method|Over a class (the factory itself is an object)|
|Use case|One variable product type|Multiple product types that must match/belong together|

Both follow OCP and SRP, and both produce loosely coupled code.

### Real-Life Example: Furniture Factory

You're designing furniture for two styles: **Victorian** and **Modern**. You need both a Chair and a Table for each style. You must ensure a Modern Chair isn't placed in a Victorian room.

**Solution:** Create one factory per style — each factory creates all the products in that style.

```
FurnitureFactory (Abstract Factory — interface)
├── ModernFurnitureFactory
│   ├── creates ModernChair
│   └── creates ModernTable
└── VictorianFurnitureFactory
    ├── creates VictorianChair
    └── creates VictorianTable
```

### Components of Abstract Factory

|Component|Role|
|---|---|
|**Abstract Factory**|Interface declaring `createChair()` and `createTable()`|
|**Concrete Factories**|`ModernFurnitureFactory`, `VictorianFurnitureFactory`|
|**Abstract Products**|`Chair` (with `sitOn()`), `Table` (with `use()`)|
|**Concrete Products**|`ModernChair`, `ModernTable`, `VictorianChair`, `VictorianTable`|
|**Client**|Uses only the abstract factory and abstract products — no concrete class names in client code|

### Java Implementation: GUI Toolkit (Windows vs macOS)

```java
// Abstract Products
public interface Button   { void render(); }
public interface Checkbox { void render(); }

// Concrete Products for Windows
public class WindowsButton implements Button {
    public void render() { System.out.println("Rendering Windows Button"); }
}
public class WindowsCheckbox implements Checkbox {
    public void render() { System.out.println("Rendering Windows Checkbox"); }
}

// Concrete Products for macOS
public class MacButton implements Button {
    public void render() { System.out.println("Rendering Mac Button"); }
}
public class MacCheckbox implements Checkbox {
    public void render() { System.out.println("Rendering Mac Checkbox"); }
}

// Abstract Factory
public interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

// Concrete Factories
public class WindowsFactory implements GUIFactory {
    public Button createButton()     { return new WindowsButton(); }
    public Checkbox createCheckbox() { return new WindowsCheckbox(); }
}

public class MacFactory implements GUIFactory {
    public Button createButton()     { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}

// Client — only uses GUIFactory, Button, Checkbox interfaces
public class Application {
    private Button button;
    private Checkbox checkbox;

    public Application(GUIFactory factory) {
        button   = factory.createButton();
        checkbox = factory.createCheckbox();
    }

    public void renderUI() {
        button.render();
        checkbox.render();
    }
}

// Main
public class Main {
    public static void main(String[] args) {
        GUIFactory factory;
        String os = System.getProperty("os.name").toLowerCase();
        if (os.contains("win")) {
            factory = new WindowsFactory();
        } else {
            factory = new MacFactory();
        }
        Application app = new Application(factory);
        app.renderUI(); // Renders the right UI for the OS
    }
}
```

### Pros of Abstract Factory

- Ensures compatibility between products of the same family
- Avoids coupling between client and concrete product classes
- Supports OCP — add new family (factory) without modifying client

### Cons of Abstract Factory

- Adding new product types to the interface requires updating ALL concrete factories (can be disruptive)
- Results in many classes

## 5.4 Builder Pattern

### Intent

**Separate the construction of a complex object from its representation** so that the same construction process can create different representations.

Build objects **step by step**, calling only what you need.

### Problem it Solves

Consider a `Computer` object with 10+ attributes (some required, some optional). Using a constructor leads to:

1. **Too many arguments** — easy to mix up the order (e.g., `new Computer(8, 256, "Intel", true, false, "blue", 2, 3, ...)`)
2. **Forced null for optional params** — `new Computer(8, 256, null, null, true, null, ...)` — unreadable
3. **Complex construction embedded in factory** — factory classes become huge

### Solution: The Builder

Extract the construction logic into a separate **Builder** class. The client calls setter-like methods on the Builder, each returning the Builder itself (for chaining), and then calls `build()` to get the final object.

### Implementation Steps

1. Create a **static nested class** `Builder` inside the outer class (or a separate class)
2. Builder has a **public constructor** with required (mandatory) attributes
3. Builder has **setter methods** for optional attributes (each returns `this` for chaining)
4. Builder has a `build()` method that validates and returns the final object
5. The outer class has a **private constructor** that takes a Builder

### Java Implementation: Computer Builder

```java
public class Computer {
    // Required
    private final String ram;
    private final String hdd;
    // Optional
    private final boolean isGraphicsCard;
    private final boolean isBluetooth;

    // Private constructor — only Builder can create Computer
    private Computer(ComputerBuilder builder) {
        this.ram             = builder.ram;
        this.hdd             = builder.hdd;
        this.isGraphicsCard  = builder.isGraphicsCard;
        this.isBluetooth     = builder.isBluetooth;
    }

    @Override
    public String toString() {
        return "Computer [RAM=" + ram + ", HDD=" + hdd +
               ", GPU=" + isGraphicsCard + ", Bluetooth=" + isBluetooth + "]";
    }

    // Static nested Builder class
    public static class ComputerBuilder {
        private final String ram;  // Required
        private final String hdd;  // Required
        private boolean isGraphicsCard; // Optional, default false
        private boolean isBluetooth;    // Optional, default false

        public ComputerBuilder(String ram, String hdd) {
            this.ram = ram;
            this.hdd = hdd;
        }

        public ComputerBuilder setGraphicsCard(boolean val) {
            this.isGraphicsCard = val;
            return this; // Returns builder for chaining
        }

        public ComputerBuilder setBluetooth(boolean val) {
            this.isBluetooth = val;
            return this; // Returns builder for chaining
        }

        public Computer build() {
            return new Computer(this);
        }
    }
}

// Client usage — clean, readable, no null arguments
public class Main {
    public static void main(String[] args) {
        Computer gamingPC = new Computer.ComputerBuilder("16GB", "1TB SSD")
                                .setGraphicsCard(true)
                                .setBluetooth(true)
                                .build();

        Computer officePC = new Computer.ComputerBuilder("8GB", "500GB HDD")
                                .build(); // Optional fields default to false

        System.out.println(gamingPC);
        System.out.println(officePC);
    }
}
```

### Car Builder Example (from notes)

```java
Car car = new Car.Builder("BMW", "X5")  // Required: make and model
    .setColor("Blue")
    .setYear(2023)
    .build(); // Returns Car object
```

Instead of: `new Car("BMW", "X5", "Blue", 2023, 300.0)` — which is cryptic.

### Advantages of Builder

1. Parameters to the constructor are reduced and provided via **readable method calls**
2. No need to pass `null` for optional parameters
3. Object is always instantiated in a **complete state**
4. Supports creating **immutable objects** without complex logic

### Disadvantages of Builder

1. **More code** — at least double the lines (but worth it for readability and flexibility)
2. Requires a **separate ConcreteBuilder** for each different type of Product

## 5.5 Prototype Pattern

### Intent

Create new objects by **cloning an existing object** (the prototype), rather than constructing from scratch.

### Problem it Solves

Sometimes creating an object is expensive:

- Loads lots of data from a database
- Makes network calls
- Requires heavy computation

If you need multiple slightly-different copies of this object, recreating it from scratch each time is wasteful.

**Solution:** Clone the expensive object once and modify the clone.

### Key Rule

> The object being cloned must **provide its own cloning feature**. The prototype itself knows how to copy itself (not some external class doing the copying).

### Shallow Copy vs Deep Copy

- **Shallow copy** — copies primitive fields by value, but object references are shared (both original and clone point to the same nested objects)
- **Deep copy** — recursively copies all nested objects too (completely independent clone)

Choice depends on requirements — both are valid design decisions.

### Java Implementation: Sheep Farm

```java
// Step 1: Prototype Interface
public interface Sheep {
    Sheep clone();
    String getName();
    void setName(String name);
}

// Step 2: Concrete Prototypes
public class BlackSheep implements Sheep {
    private String name;

    public BlackSheep(String name) { this.name = name; }

    public Sheep clone() { return new BlackSheep(this.name); } // Clone itself

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}

public class WhiteSheep implements Sheep {
    private String name;

    public WhiteSheep(String name) { this.name = name; }

    public Sheep clone() { return new WhiteSheep(this.name); }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}

// Step 3: Client code
public class SheepFarm {
    public static void main(String[] args) {
        // Create prototype sheep (expensive initialization)
        Sheep blackPrototype = new BlackSheep("Baa Baa");
        Sheep whitePrototype = new WhiteSheep("Fleecy");

        // Clone cheaply
        Sheep clonedBlack = blackPrototype.clone();
        Sheep clonedWhite = whitePrototype.clone();

        // Customize clones
        clonedBlack.setName("Midnight");
        clonedWhite.setName("Snowball");

        System.out.println("Black sheep: " + clonedBlack.getName()); // Midnight
        System.out.println("White sheep: " + clonedWhite.getName()); // Snowball

        // Original prototypes unchanged
        System.out.println("Original: " + blackPrototype.getName()); // Baa Baa
    }
}
```

### Java's Built-in Cloning (`Cloneable` interface)

```java
public class Employee implements Cloneable {
    private String name;
    private int salary;

    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone(); // Uses Object.clone() — shallow copy
    }
}
```

### When to Use Prototype

- Object creation is expensive (DB calls, network, heavy computation)
- You need multiple copies with slight variations
- You want to avoid subclasses just for different initial states
- The system needs to be independent of how products are created

### Advantages

- Avoids expensive re-initialization
- Can eliminate subclasses that differ only in initial state
- Reduces coupling to concrete classes

### Disadvantages

- Can get complex when objects have **circular references** (A references B, B references A — cloning is tricky)
- Deep cloning requires careful implementation
- With many classes, keeping track of all cloneable prototypes can become messy

## ✅ Q&A Bank: Creational Design Patterns

**Q1. What are the five creational design patterns?**

> Singleton, Factory Method, Abstract Factory, Builder, and Prototype.

**Q2. What three things must a Singleton class have?**

> (1) A **private constructor** (prevent external instantiation), (2) a **private static instance variable** (the single instance), (3) a **public static `getInstance()` method** (global access point).

**Q3. Where is Singleton used in core Java?**

> `java.lang.Runtime` and `java.awt.Desktop` are examples of Singleton in the Java standard library.

**Q4. What is the Factory Method pattern? How does it follow OCP?**

> Factory Method defines an interface for creating an object but lets subclasses decide which class to instantiate. It follows OCP because you can add new product types (new subclasses) without modifying the factory interface or client code.

**Q5. What is the difference between Factory Method and Abstract Factory?**

> Factory Method creates **one product** using inheritance (subclass decides). Abstract Factory creates **a family of related products** using delegation (a factory object creates all related products). Abstract Factory is essentially "a factory of factories."

**Q6. What problem does the Builder pattern solve that Factory cannot?**

> Builder solves three problems with Factory for complex objects: (1) too many arguments in the constructor (hard to maintain order), (2) forced null for optional parameters, (3) complex construction logic polluting factory classes. Builder allows step-by-step construction with readable method chaining.

**Q7. What is method chaining in Builder? Give an example.**

> Each setter method in the Builder returns `this` (the Builder object itself), allowing calls to be chained: `new Car.Builder("BMW", "X5").setColor("Blue").setYear(2023).build()`. This is fluent interface style — very readable.

**Q8. What is the Prototype pattern? When should it be used?**

> Prototype creates new objects by cloning an existing object. Use it when object creation is expensive (DB loads, network calls) and you need multiple slightly-different copies. Clone the expensive object once and customize each clone.

**Q9. Who is responsible for cloning in the Prototype pattern?**

> The object being cloned (the prototype itself) must provide the cloning feature through a `clone()` method. An external class should NOT perform the copying.

**Q10. What is the difference between shallow copy and deep copy in Prototype?**

> **Shallow copy** duplicates primitive fields by value but copied object shares references to the same nested objects as the original. **Deep copy** recursively copies all nested objects too — fully independent clone. The choice depends on requirements.

**Q11. Why is Singleton difficult to unit test?**

> Because it introduces **global state** — the single instance persists across all tests. One test can affect another through the shared singleton, making tests order-dependent and non-isolated.

**Q12. In the Abstract Factory furniture example, why is the pattern needed?**

> To ensure **product family consistency** — you can't accidentally pair a `VictorianChair` with a `ModernTable`. Each factory (`VictorianFurnitureFactory`, `ModernFurnitureFactory`) only creates matching furniture. The client simply picks a factory and gets a consistent set of products.

**Q13. What are the four advantages of the Builder pattern?**

> (1) Fewer, more readable constructor parameters through method calls. (2) No need to pass null for optional parameters. (3) Objects always instantiated in a complete state. (4) Easy to build immutable objects.

**Q14. What is a major disadvantage of the Abstract Factory when you need to add a new product type?**

> Adding a new product type (e.g., adding `Sofa` to the furniture family) requires updating the **Abstract Factory interface** AND all concrete factory classes. This can be disruptive and violates OCP from the perspective of the factory interface itself.

**Q15. Give a real-world use case where Prototype is the best choice.**

> Loading a complex configuration object from a database once, then cloning it multiple times to create slightly-different configurations for different environments (dev, staging, prod). Cloning avoids repeated expensive DB operations.

# Summary Cheat Sheet

## GRASP at a Glance

|Principle|Question|Answer|
|---|---|---|
|**Creator**|Who creates A?|B, if B contains/uses/records/initializes A|
|**Information Expert**|Who fulfills this?|The class with the needed information|
|**Low Coupling**|How to reduce dependency?|Assign so coupling stays low|
|**Controller**|Who receives system events?|A facade or use-case controller|
|**High Cohesion**|How to keep focused?|Assign so cohesion stays high|
|**Polymorphism**|How to handle type-based variation?|Use interfaces/polymorphic methods|
|**Indirection**|How to avoid direct coupling?|Add an intermediate mediator class|
|**Pure Fabrication**|No domain class fits — what now?|Invent a convenience class|
|**Protected Variations**|How to shield from change?|Create stable interface around volatile point|

## SOLID at a Glance

|Principle|Rule|Violation Smell|
|---|---|---|
|**SRP**|One reason to change|Class name contains "and"|
|**OCP**|Extend, don't modify|Adding new feature requires modifying existing class|
|**LSP**|Subclass substitutable|Subclass throws "unsupported" for parent's methods|
|**ISP**|No unused interface methods|Classes have empty/stub method implementations|
|**DIP**|Depend on abstractions|`new ConcreteClass()` in high-level modules|

## Creational Patterns at a Glance

|Pattern|Use when...|Key mechanism|
|---|---|---|
|**Singleton**|Need exactly one instance|Private constructor + static instance|
|**Factory Method**|Subclass decides what to create|Create by calling a factory method|
|**Abstract Factory**|Need families of related objects|Factory object creates all related products|
|**Builder**|Complex object with many optional params|Step-by-step construction with chaining|
|**Prototype**|Object creation is expensive|Clone an existing instance|

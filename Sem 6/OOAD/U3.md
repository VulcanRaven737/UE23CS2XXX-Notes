# PART 1 — GRASP PATTERNS

---

## 1.1 Introduction to GRASP

**GRASP** stands for **General Responsibility Assignment Software Patterns** (also called Principles).

The central question that GRASP answers is:

> **Who should be responsible for doing what in an object-oriented system?**

### Why GRASP matters

When programmers start out, the only goal is to make code work. But over time, this leads to code that is hard to maintain, extend, or understand. GRASP gives us a way to think carefully about _which object should own which responsibility_, producing code that is:

- Clean and readable
- Loosely coupled
- Highly cohesive
- Easily maintainable

### Origin

GRASP was first published by **Craig Larman** in his 1997 book _"Applying UML and Patterns"_. It is a set of **nine fundamental principles** in object design and responsibility assignment.

### The 9 GRASP Patterns

|#|Pattern|Core Question It Answers|
|---|---|---|
|1|Creator|Who should create an instance of a class?|
|2|Information Expert|Who should handle a responsibility?|
|3|Low Coupling|How to reduce dependency between classes?|
|4|Controller|Who handles a system event from the UI layer?|
|5|High Cohesion|How to keep objects focused and manageable?|
|6|Polymorphism|How to handle type-based alternatives?|
|7|Indirection|How to avoid direct coupling between two things?|
|8|Pure Fabrication|Where to put responsibility when no domain class fits?|
|9|Protected Variations|How to shield the system from instability/change?|

> **Larman's Key Insight:** _"The critical design tool for software development is a mind well-educated in design principles. It is not UML or any other technology."_

**Core Equation:** `GRASP = Object-Oriented Design with Responsibilities`

---

## 1.2 The Nine GRASP Patterns (In Depth)

### 1. Creator

**Problem:** Who should be responsible for creating a new instance of class A?

**Solution:** Assign class **B** the responsibility to create an instance of class **A** if one or more of the following is true _(the more conditions that apply, the stronger the justification)_:

- **B "contains" or compositely aggregates A** — B holds A as a part
- **B closely uses A** — B has a strong usage relationship with A
- **B records instances of A** — B keeps a log or list of A objects
- **B has the initializing data for A** — B is an _Expert_ regarding A's creation

**Analogy (Monopoly):** `Board` creates `Square` objects because Board contains/aggregates Squares.

**Analogy (Healthcare):** `HospitalSystem` creates `Appointment` objects.

---

### 2. Information Expert

**Problem:** What is the general principle for assigning responsibilities to objects?

**Solution:** Assign a responsibility to the **information expert** — the class that _has the information necessary_ to fulfill that responsibility.

**Key Idea:** Don't scatter the logic everywhere. The object that owns the data should own the behavior that operates on that data.

**Analogy (Monopoly):** `PropertySquare` calculates rent because it owns the rent amount and owner information.

**Analogy (Healthcare):** `Patient` calculates their own age; `Doctor` checks their own availability.

---

### 3. Low Coupling

**Problem:** How do we support low dependency, low change impact, and increased reuse?

**Solution:** Assign responsibilities so that **coupling remains low**. Use this principle to evaluate design alternatives.

**What is Coupling?** Coupling is the **degree of interdependence** between software modules — a measure of how closely connected two routines or modules are; the strength of the relationships between modules.

**Why Low Coupling Matters:**

- A change in one module should minimally affect others
- Classes become more reusable in isolation
- The system becomes easier to test, maintain, and understand

**Analogy (Monopoly):** `Dice` only knows how to roll — it is independent of all other classes.

---

### 4. Controller

**Problem:** What first object _beyond the UI layer_ receives and coordinates ("controls") a system operation?

**Solution:** Assign the responsibility to a class representing one of these choices:

1. **Facade Controller** — Represents the overall "system", a root object, a device, or a major subsystem (e.g., `HospitalSystem`, `MonopolyGame`).
2. **Use Case Controller** — Represents a specific use case scenario, often named: `<UseCaseName>Handler`, `<UseCaseName>Coordinator`, or `<UseCaseName>Session`.

**Key Idea:** The Controller is NOT a UI element. It sits between the UI and the domain model. It delegates work — it does not do the work itself.

**Analogy (Monopoly):** `MonopolyGame` orchestrates the game flow (start, play turn, end).

**Analogy (Healthcare):** `HospitalSystem` manages login, appointment booking, billing.

---

### 5. High Cohesion

**Problem:** How do we keep objects focused, understandable, and manageable? (Also supports Low Coupling as a side effect.)

**Solution:** Assign responsibilities so that **cohesion remains high**.

**What is Cohesion?** Cohesion refers to the **degree to which elements inside a module belong together**. It measures:

- The strength of the relationship between a class's methods and data
- How well the class is focused on a single, unified purpose

**A highly cohesive class:** does one thing and does it well. **A low cohesion class:** is a "God class" that does everything.

**Analogy (Monopoly):** `Player` keeps all player-related responsibilities: money, properties, movement.

**Analogy (Healthcare):** `Appointment` handles only scheduling, rescheduling, cancellation.

---

### 6. Polymorphism

**Problem:** How do we handle alternatives based on type? How do we create pluggable software components?

**Solution:** When related alternatives or behaviors **vary by type (class)**, assign responsibility for the behavior using **polymorphic operations** to the types for which the behavior varies.

**Key Idea:** Instead of using if-else or switch statements to handle different types, use polymorphism so each type handles its own behavior.

**Analogy (Monopoly):** A `SquareAction` interface allows `PropertySquare`, `ChanceSquare`, `JailSquare` to each define their own `land()` behavior.

**Analogy (Healthcare):** A `Notification` interface supports multiple alert types (Email, SMS, Push).

---

### 7. Indirection

**Problem:** Where should responsibility be assigned to avoid direct coupling between two or more things? How can objects be decoupled so that low coupling is supported and reuse potential remains higher?

**Solution:** Assign responsibility to an **intermediate object** that mediates between the components so they are not directly coupled.

**Key Idea:** Instead of A talking directly to B, introduce a mediator M. A talks to M, and M talks to B. A and B don't know about each other.

**Analogy (Monopoly):** `TradeManager` mediates trades between players — players don't interact directly.

**Analogy (Healthcare):** `AppointmentManager` mediates between `Patient` and `Doctor`.

---

### 8. Pure Fabrication

**Problem:** What object should have the responsibility when solutions offered by Information Expert are not appropriate (e.g., assigning it would violate High Cohesion and Low Coupling)?

**Solution:** Assign a highly cohesive set of responsibilities to an **artificial or convenience class** that does **not** represent a problem domain concept — something made up to support high cohesion, low coupling, and reuse.

**Key Idea:** Sometimes no real-world domain class should own a certain responsibility (e.g., database saving, report generation). In such cases, invent a class for it. This class is called a "Pure Fabrication."

**Analogy (Monopoly):** `Bank` handles financial transactions — there is no "real" bank piece in Monopoly's domain, but we invent one to keep financial logic separate.

**Analogy (Healthcare):** `ReportGenerator` creates medical reports — not a domain concept, but a fabricated helper.

---

### 9. Protected Variations

**Problem:** How do we design objects, subsystems, and systems so that variations or instability in these elements does not have an undesirable impact on other elements?

**Solution:** **Identify points of predicted variation or instability**; assign responsibilities to create a **stable interface** (or abstraction) around them.

**Key Idea:** Wrap whatever is likely to change behind a stable interface. Internal changes stay internal.

**Analogy (Monopoly):** `JailSquare` encapsulates jail logic, so if jail rules change, no other class is affected.

**Analogy (Healthcare):** `InsuranceProcessor` encapsulates insurance logic, shielding other classes from regulatory changes.

**Relationship to other patterns:**

- **Polymorphism** is often the _mechanism_ used to implement Protected Variations.
- **Indirection** also reduces coupling, but focuses on _how_ two objects communicate.
- **Protected Variations** focuses on _where_ the system is likely to change and builds a shield around that instability.

## 1.5 GRASP Violation Analysis (Bad Code Examples)

Understanding violations helps you spot and fix them in exams.

### Example 1: Order Class Violations

**Problems Identified:**

- **Low Cohesion Violation:** The `Order` class manages items, calculates totals, processes payments, AND prints receipts — too many responsibilities.
- **Controller Principle Violation:** Payment logic is embedded in the entity, not a dedicated controller/service.
- **Poor Separation of Concerns:** Business logic mixes with infrastructure concerns (payment gateway, printing).
- **Hard to Maintain/Extend:** Any change to payment provider or receipt format forces changes in `Order`.

**GRASP-Friendly Fix:**

- `Order` → only tracks items and totals
- `PaymentProcessor` → handles payment logic
- `ReceiptPrinter` → handles receipt formatting and printing
- `OrderController` → coordinates all the above

---

### Example 2: Library Class Violations

**Problems Identified:**

- **Information Expert Violation:** `Library` does login and notification tasks that belong to other classes.
- **Low Cohesion:** Mixes authentication, messaging, and search logic with book storage.
- **High Coupling:** Any change in login or notification forces changes in `Library`.
- **Controller Principle Ignored:** No controller to coordinate responsibilities.

**GRASP-Friendly Fix:**

- `Library` → manages books only
- `AuthService` → handles login/authentication
- `NotificationService` → sends notifications
- `SearchService` → encapsulates search logic
- `LibraryController` → coordinates between services

---

### Example 3: Patient Class Violations

**Problems Identified:**

- **Low Cohesion:** `Patient` stores info, schedules appointments, generates bills, saves records.
- **Information Expert Violation:** Billing should be in `BillingService`, not the patient entity.
- **Controller Principle Ignored:** Appointment scheduling embedded in entity instead of service.
- **High Coupling:** Changes in billing or appointment logic force changes in `Patient`.

**GRASP-Friendly Fix:**

- `Patient` → stores patient details only
- `AppointmentService` → manages scheduling
- `BillingService` → handles billing and invoices
- `MedicalRecordService` → manages record storage
- `HealthcareController` → coordinates these services

---

### Example 4: BankAccount Class Violations

**Problems Identified:**

- **Low Cohesion:** `BankAccount` stores info, handles login, approves loans, AND sends notifications.
- **Information Expert Violation:** Loan approval should be in `LoanService`.
- **Controller Principle Ignored:** Authentication and notifications need dedicated services.
- **High Coupling:** Changes in login, loan rules, or notifications force changes in `BankAccount`.

**GRASP-Friendly Fix:**

- `BankAccount` → stores account details and balance only
- `AuthService` → handles login
- `LoanService` → manages loan approval
- `NotificationService` → sends notifications
- `BankController` → coordinates everything

## GRASP — Question & Answer Bank

**Q1. What does GRASP stand for and who introduced it?** GRASP stands for **General Responsibility Assignment Software Patterns (Principles)**. It was introduced by **Craig Larman** in his 1997 book _"Applying UML and Patterns."_

**Q2. What is the core purpose of GRASP?** GRASP helps guide object-oriented design by clearly outlining **who does what** — which object or class is responsible for which action or role. Its key goal is efficient, clean, understandable code.

**Q3. What is the Creator pattern and when is class B a good creator of class A?** Creator assigns class B the responsibility to create instances of A if: B _contains_ or _aggregates_ A, B _closely uses_ A, B _records_ instances of A, or B _has the initializing data_ for A.

**Q4. What is the difference between Low Coupling and High Cohesion?**

- **Low Coupling** minimizes dependencies _between_ classes — how much one class depends on another.
- **High Cohesion** maximizes focus _within_ a class — how related all the responsibilities of a single class are. Both principles support each other: assigning responsibilities to the right class keeps both coupling low and cohesion high.

**Q5. What is the difference between Indirection and Protected Variations?** Both use intermediaries/interfaces. **Indirection** focuses on _how_ two objects communicate to reduce coupling (introducing a mediator). **Protected Variations** focuses on _where_ the system is likely to change and builds a stable interface (shield) around that instability.

**Q6. What is the Controller pattern and what types of controllers exist?** Controller is the first object beyond the UI layer that handles a system operation. Types: (1) **Facade Controller** — represents the whole system or a major subsystem; (2) **Use Case Controller** — represents a specific use case scenario, named like `<UseCaseName>Handler`.

**Q7. What is Pure Fabrication and when should it be used?** Pure Fabrication is an invented class that has no counterpart in the real-world problem domain. It is used when no domain class is an appropriate owner of certain responsibilities without violating High Cohesion and Low Coupling (e.g., `ReportGenerator`, `DataPersistence`).

**Q8. Why can't the polymorphism principle simply be replaced by if-else chains?** If-else chains create high coupling and are hard to extend. When a new type is added, the if-else must be modified (violating OCP). Polymorphism allows each type to handle its own behavior, making the system open for extension without modifying existing code.

**Q9. What is the role of the Information Expert principle?** Information Expert states that the class that **has the information** necessary to fulfill a responsibility should be assigned that responsibility. This avoids scattering behavior away from the data it operates on, leading to a more natural and cohesive design.

**Q10. In the Monopoly example, why is Dice a good example of Low Coupling?** `Dice` only knows how to roll (generate a random number). It has no dependency on `Player`, `Board`, `Square`, or any other game class. This means Dice can be changed, tested, or reused without affecting any other class.

# PART 2 — FEATURES OF BAD DESIGN & SOLID PRINCIPLES

---

## 2.1 Features of Bad Design

A bad design manifests in four characteristic "smells":

### 1. Rigidity

- The impact of a change is **unpredictable**.
- Every change causes a **cascade of changes** in dependent modules.
- Costs become unpredictable.

### 2. Fragility

- The software tends to **break in many places** on every change.
- The breakage occurs in areas with **no conceptual relationship** to the change.
- On every fix, the software breaks in **unexpected ways**.

### 3. Immobility

- It is almost **impossible to reuse** interesting parts of the software.
- Useful modules have **too many dependencies**.
- The cost of rewriting is less than the risk of separating those parts.

### 4. Viscosity

- A **hack is cheaper** to implement than the solution within the design.
- Design-preserving moves are difficult to think about and implement.
- It is much easier to do the **wrong thing** rather than the right one.

> **The Solution to Bad Design:** Follow the **SOLID Principles**.

---

## 2.2 Introduction to SOLID

**SOLID** was introduced by **Robert C. Martin** in his 2000 paper _"Design Principles and Design Patterns."_ The acronym was later formalized by **Michael Feathers**.

Over 20+ years, these five principles have revolutionized object-oriented programming by encouraging us to create **more maintainable, understandable, and flexible software**.

|Letter|Principle|One-line Summary|
|---|---|---|
|S|Single Responsibility Principle (SRP)|A class should have only one reason to change|
|O|Open/Closed Principle (OCP)|Open for extension, closed for modification|
|L|Liskov Substitution Principle (LSP)|Subclasses should be replaceable for their superclasses|
|I|Interface Segregation Principle (ISP)|No client should depend on methods it does not use|
|D|Dependency Inversion Principle (DIP)|Depend on abstractions, not on concretions|

---

## 2.3 Single Responsibility Principle (SRP)

### Definition

> "A class should have one, and only one, reason to change."

A class should have **only one responsibility** and therefore only one reason to be modified.

### Benefits

**Testing:** A class with one responsibility has far fewer test cases, making unit testing simpler.

**Lower Coupling:** Less functionality in a single class means fewer dependencies on other classes.

**Organization:** Smaller, well-organized classes are easier to locate and understand than monolithic ones.

### Why Ignoring SRP Is Dangerous

- Requirements change over time. The more responsibilities a class has, the more often it must change.
- If a class implements multiple responsibilities, they are **no longer independent** of each other.
- Changing one responsibility may force you to update all classes that depend on the changed class — even if those dependent classes only use _one_ of the other responsibilities.

### The "And" Test

> **Quick validation:** If your description of a class's responsibility includes the word **"and"**, you are most likely violating SRP.

Example: "This class manages Orders **and** processes payments **and** prints receipts." → Violates SRP. Split it up.

### BankService Example (SRP Violation → Fix)

**Violation:** A `BankService` class that handles loans, sends OTPs, and prints passbooks. It has multiple reasons to change:

- The bank might support different kinds of loans → changes loan code
- OTP delivery might switch from SMS to WhatsApp → changes notification code
- Passbook format changes → changes print code

**SRP-Compliant Fix:**

- `BankService` → core banking functionality only
- `PrinterService` → handles all passbook/statement printing
- `LoanService` → handles all loan-related functionality
- `NotificationService` → handles OTPs and alerts

Now **each class has a single responsibility** and a single reason to change.

---

## 2.4 Open/Closed Principle (OCP)

### Definition

> "Classes should be open for extension but closed for modification."

You should be able to add new functionality by **adding new code**, not by changing existing code that already works.

### Why This Matters

- Modifying existing code risks introducing bugs in working parts of the application.
- New functionality should come via extension (inheritance, composition, interfaces), not modification.

### Exception to the Rule

The only acceptable exception to OCP is **fixing bugs in existing code**.

### OCP Example (from SRP BankService)

Suppose we need to add WhatsApp notifications. Rather than modifying the existing `NotificationService` class:

**Wrong Approach (OCP Violation):**

```java
class NotificationService {
    void sendOTP(String type) {
        if (type.equals("SMS")) { /* SMS logic */ }
        else if (type.equals("Email")) { /* Email logic */ }
        else if (type.equals("WhatsApp")) { /* add new branch → modifies class */ }
    }
}
```

**Correct Approach (OCP-Compliant):**

```java
interface NotificationService {
    void sendOTP(String message);
}

class SMSNotification implements NotificationService { ... }
class EmailNotification implements NotificationService { ... }
class WhatsAppNotification implements NotificationService { ... }  // NEW — no existing code modified
```

Define an **interface** which can be implemented to handle different media. Each new notification type is a new class that _extends_ behavior — existing code is untouched.

---

## 2.5 Liskov Substitution Principle (LSP)

### Definition

> "Objects of a superclass should be replaceable with objects of its subclasses without breaking the application."

Subclass objects must behave **the same way** as superclass objects from the client's perspective.

### LSP Extends OCP

LSP extends the Open/Closed principle. It ensures that when you substitute a subclass for a superclass, the program still functions correctly.

### Rules for Subclasses to Follow LSP

1. **Do not implement stricter validation** rules on input parameters than the parent class does.
2. **Apply at least the same rules** to all output parameters as the parent class applies.

### Classic LSP Violation: The Bird–Penguin Problem

**Violation:**

```java
class Bird {
    void fly() { /* all birds fly? */ }
}

class Penguin extends Bird {
    void fly() {
        throw new UnsupportedOperationException("Penguins can't fly!");
    }
}
```

If a method iterates through a `List<Bird>` and calls `.fly()`, it will **crash** when it encounters a `Penguin`. This violates LSP.

**Correct Approach:**

```java
interface Flyable {
    void fly();
}

class Bird {
    void eat() { /* all birds eat */ }
}

class Sparrow extends Bird implements Flyable {
    public void fly() { /* sparrow flies */ }
}

class Penguin extends Bird {
    // Does NOT implement Flyable — no fly() method
}
```

Now a client can safely use any `Bird` for eating, and specifically use `Flyable` objects when flight is required — no unexpected runtime exceptions.

### Social Media Example

- `Facebook` implements posting, messaging, chat — fully substitutable for `SocialMedia`.
- `WhatsApp` does not support all SocialMedia features → cannot be a substitute for `SocialMedia`.
- `Instagram` similarly cannot fully substitute for `SocialMedia`.

**Fix:** Define separate **interfaces** with the methods supported by each platform. Each platform implements only the interfaces relevant to it.

---

## 2.6 Interface Segregation Principle (ISP)

### Definition

> "Clients should not be forced to depend upon interfaces that they do not use."

(First defined by Robert C. Martin)

### Goal

Reduce side effects of large, bloated interfaces by **breaking them into smaller, more specific ones**.

### Relationship to SRP

ISP and SRP have approximately the same goal — ensuring small, focused, highly cohesive software components:

- **SRP** is concerned with **classes**.
- **ISP** is concerned with **interfaces**.

### Why ISP Is Important

- A large interface forces implementing classes to provide empty or dummy implementations for methods they don't need.
- This creates unnecessary dependencies and potential for bugs.
- Smaller, focused interfaces give implementors freedom to implement only what they need.

### Example

**ISP Violation — Fat Interface:**

```java
interface Worker {
    void work();
    void eat();
    void sleep();
}

class Robot implements Worker {
    public void work() { /* ok */ }
    public void eat() { /* robots don't eat! */ }    // forced to implement
    public void sleep() { /* robots don't sleep! */ } // forced to implement
}
```

**ISP-Compliant Fix:**

```java
interface Workable { void work(); }
interface Feedable { void eat(); }
interface Sleepable { void sleep(); }

class Human implements Workable, Feedable, Sleepable { ... }
class Robot implements Workable { ... }  // only implements what it needs
```

### Key Insight

Identifying distinct interfaces can sometimes be a challenge. Careful consideration is needed to avoid "interface explosion." When writing an interface, consider whether all possible implementation classes would realistically implement every method. If not, segregate.

---

## 2.7 Dependency Inversion Principle (DIP)

### Definition

> "High-level modules should not depend on low-level modules; both should depend on abstractions. Abstractions should not depend on details. Details should depend upon abstractions."

### The Problem Without DIP

When high-level modules directly reference low-level modules, every change in a low-level module forces changes in high-level modules. The system becomes tightly coupled and brittle.

### The "Inversion" Explained

Normally, dependencies flow **downward**: high-level → low-level.

DIP _inverts_ this: **both** high-level and low-level modules depend on an **abstraction (interface)**. The high-level module defines the interface it needs; the low-level module implements it.

Instead of:

```
Shelf (high-level) ──depends on──> Book (low-level)
```

DIP gives us:

```
Shelf (high-level) ──depends on──> Product (interface)
                                        ^
                              Book ─────┘ DVD ────┘
```

### Bookstore Example

**DIP Violation:**

```java
class Book {
    void seeReviews() { ... }
    void readSample() { ... }
}

class Shelf {
    Book book;           // HIGH-level directly depends on LOW-level
    void addBook(Book b) { this.book = b; }
}
```

When the store wants to add DVDs to shelves, you must modify `Shelf` — violating OCP and DIP.

**DIP-Compliant Fix — Introduce an Abstraction:**

```java
interface Product {
    void seeReviews();
    void watchOrReadSample();
}

class Book implements Product { ... }
class DVD implements Product { ... }
class Magazine implements Product { ... }  // easy to add later

class Shelf {
    Product product;           // depends on ABSTRACTION, not concrete class
    void addProduct(Product p) { this.product = p; }
}
```

Now `Shelf` works with any `Product` type — Book, DVD, Magazine — without modification.

### Why This Also Satisfies LSP

`Product` can be substituted with both its subtypes (`Book` and `DVD`) without breaking the program — satisfying LSP simultaneously.

## SOLID Principles — Question & Answer Bank

**Q1. What are the four features of bad design? Briefly explain each.** Rigidity (unpredictable cascade of changes), Fragility (software breaks in many unrelated places), Immobility (useful parts can't be reused due to dependencies), Viscosity (hacks are cheaper than doing it right).

**Q2. State and explain the Single Responsibility Principle.** "A class should have one, and only one, reason to change." A class should handle only one responsibility. This makes the class easier to test, less coupled, and simpler to understand. The "and" test: if your class description uses "and", you're likely violating SRP.

**Q3. What is the Open/Closed Principle? Give an example of how to follow it.** "Open for extension, closed for modification." New functionality is added via new classes/implementations rather than modifying existing code. Example: Adding a WhatsApp notification by creating a new `WhatsAppNotification` class implementing a `NotificationService` interface, instead of adding an if-else in the existing class.

**Q4. Explain the Liskov Substitution Principle with the classic Bird–Penguin example.** LSP states that objects of a subclass should be replaceable for the superclass without breaking the application. Classic violation: `Penguin` extends `Bird` but can't fly — calling `.fly()` on a Penguin throws an exception. Fix: Separate the `Flyable` interface so only birds that can fly implement it.

**Q5. What is the difference between ISP and SRP?** Both aim for small, focused, cohesive components. SRP is about **classes** having one reason to change. ISP is about **interfaces** — clients should not be forced to depend on interface methods they don't use. SRP applies at the class design level; ISP applies at the interface design level.

**Q6. What does Dependency Inversion Principle mean by "inversion"?** Normally, high-level modules depend on low-level modules (top-down). DIP _inverts_ this by making both high-level and low-level modules depend on an **abstraction (interface)**. The dependency direction shifts from concrete implementations to abstract interfaces — hence "inversion."

**Q7. How do OCP and DIP work together in the Bookstore example?** The `Shelf` class uses a `Product` interface instead of concrete `Book` or `DVD` classes (DIP). When a new product type like `Magazine` is added, `Shelf` needs no modification (OCP) — just a new class implementing `Product`.

**Q8. Why is the "and" word a red flag in class design?** The presence of "and" in a class's responsibility description suggests multiple responsibilities, which violates SRP. For example, "manages orders AND processes payments AND prints receipts" indicates the class should be split into Order, PaymentProcessor, and ReceiptPrinter.

**Q9. Who introduced SOLID principles and when?** **Robert C. Martin** introduced the principles in his 2000 paper _"Design Principles and Design Patterns."_ The **SOLID** acronym was later formalized by **Michael Feathers**.

**Q10. In DIP, what does "abstractions should not depend on details" mean?** Interfaces (abstractions) should be defined based on what the high-level client needs, not based on the specific implementation details of the low-level classes. If the abstraction is tailored to one concrete implementation, it's essentially just a thin wrapper over the detail — not a true abstraction.

## 3.1 Origin of Design Patterns

The concept of patterns in design was introduced by **architect Christopher Alexander** in his 1977 book _"A Pattern Language: Towns, Buildings, Construction"_:

> "Each pattern describes a problem which occurs over and over again in our environment, and then describes the core of the solution to that problem, in a way that you can use this solution a million times over, without ever doing it the same way twice."

This concept was later adapted to software engineering.

---

## 3.2 Gang of Four (GoF) Design Patterns

The **Gang of Four (GoF)** refers to the four authors of the seminal book: _"Design Patterns: Elements of Reusable Object-Oriented Software"_ — Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides.

This book catalogs **23 design patterns** broken into three categories.

### Two Foundational Principles Behind GoF Patterns

1. **Program to an interface, not an implementation**
2. **Favor object composition over inheritance**

---

## 3.3 The Three Categories of GoF Patterns

### Creational Design Patterns

_Purpose: Provide various object creation mechanisms, increasing flexibility and reuse of existing code._

|Pattern|What it Does|
|---|---|
|**Abstract Factory**|Creates families of related objects without specifying concrete types|
|**Builder**|Constructs complex objects step by step|
|**Factory Method**|Creates objects without specifying the exact class — subclasses decide|
|**Prototype**|Creates a new object by copying (cloning) an existing object|
|**Singleton**|Ensures only one instance of a class exists|

### Structural Design Patterns

_Purpose: Describe how classes and objects can be combined to form larger structures._

|Pattern|What it Does|
|---|---|
|**Adapter**|Makes two incompatible interfaces work together by wrapping one|
|**Bridge**|Decouples abstraction from implementation so both can vary independently|
|**Composite**|Treats a group of objects as a single object|
|**Decorator**|Dynamically extends an object's behavior at runtime|
|**Facade**|Provides a simple interface to a complex underlying system|
|**Flyweight**|Reduces the cost of creating large numbers of similar objects|
|**Proxy**|Provides a placeholder for another object to control access|

### Behavioral Design Patterns

_Purpose: Define how objects interact and distribute responsibility._

|Pattern|What it Does|
|---|---|
|**Chain of Responsibility**|Passes a request along a chain of handlers|
|**Command**|Encapsulates an action and its parameters as an object|
|**Interpreter**|Implements a specialized language|
|**Iterator**|Accesses elements of a collection sequentially without exposing internals|
|**Mediator**|Allows loose coupling between classes via a mediator|
|**Memento**|Saves and restores an object's previous state|
|**Observer**|Publish/subscribe — observers are notified of events|
|**State**|Changes an object's behavior when its internal state changes|
|**Strategy**|Selects an algorithm from a family at runtime|
|**Template Method**|Defines a skeleton algorithm; subclasses fill in the details|
|**Visitor**|Separates an algorithm from the object structure it operates on|

---

## 3.4 Architecture vs. Design Patterns

### What is Software Architecture?

Software architecture is how the **defining components** of a software system are organized and assembled, how they communicate with each other, and the constraints the whole system is governed by.

### Comparison Table

|Architectural Patterns|Design Patterns|
|---|---|
|High-level, universal scope|Lower-level scope|
|Define _how components are organized and assembled_|Define _how components are built_|
|Examples: MVC, Microservices, Layered, Event-Driven|Examples: Singleton, Factory, Observer|

### Granularity

Architectural patterns define the **granularity** of components — how small or big the components should be. Design patterns operate _within_ those components.

---

## 3.5 What is a Design Pattern?

Design patterns are:

- **Reusable solutions to problems:** A general, reusable solution to a commonly occurring problem in software design.
- **Interaction templates:** Typically show relationships and interactions between classes or objects, without specifying final classes.
- **Templates, not solutions:** Not finished code that can be directly transferred — they are blueprints to be adapted.
- **Language independent:** Applicable in any OOP language.

---

## 3.6 How a Design Pattern is Formally Specified

Every design pattern is described with four basic elements:

|Element|Purpose|
|---|---|
|**Name**|Describes the pattern; gives us a design vocabulary|
|**Problem**|Identifies the design problem and the context it occurs in|
|**Solution**|Identifies classes, objects, relationships, and other factors|
|**Consequences**|Explains costs and benefits of using the pattern|

---

## 3.7 Full Documentation Structure of a Design Pattern

When documented thoroughly, a design pattern includes:

1. **Pattern name and classification** — essence of the pattern; type (Creational/Structural/Behavioral)
2. **Intent** — short statement of what the pattern does and which problem it solves
3. **Also known as** — other names for the pattern
4. **Motivation** — a scenario illustrating the design problem and solution
5. **Applicability** — situations where the pattern applies; examples of poor designs it addresses
6. **Structure** — graphical representation (class diagrams, interaction diagrams)
7. **Participants** — classes and objects in the pattern and their responsibilities
8. **Collaborations** — how classes/objects collaborate to carry out responsibilities
9. **Consequences** — costs and benefits; what can be changed independently
10. **Implementation** — techniques and language-specific issues
11. **Sample code** — code fragments in Java showing how to implement
12. **Known uses** — real-world examples
13. **Related patterns** — which other patterns are related and how they differ

---

## 3.8 Why Learn Design Patterns?

### Patterns Are a Common Design Vocabulary

- Allow engineers to abstract a problem and discuss it in isolation from implementation
- Embody a culture; domain-specific patterns increase design speed

### Patterns Capture and Communicate Design Expertise

- Promote design reuse and help avoid repeated mistakes
- Expert knowledge is encoded in the pattern and transferred to new developers

### Documentation and Understandability

- Patterns are described well once and understood universally
- Less documentation is needed because patterns are self-explanatory to those familiar with them

---

## 3.9 Pros and Cons of Design Patterns

### Pros

- Adds **consistency** to designs — similar problems solved the same way, regardless of language
- Adds **clarity** — enables a common vocabulary for design communication
- Improves **time to solution** — templates serve as foundations for good design
- Improves **reuse** through composition

### Cons

- Some patterns come with **negative consequences** (object proliferation, performance hits, additional layers)
- Consequences are **subjective** depending on concrete scenarios
- Patterns are subject to different **interpretations, misinterpretations, and philosophies**
- Patterns can be **overused and abused** → leads to **Anti-Patterns**

---

## 3.10 How to Select a Design Pattern

With 23 patterns in the catalog, selection can be challenging. Use these strategies:

1. **Consider how patterns solve design problems** — Think about how each pattern helps find appropriate objects, determines object granularity, and specifies object interfaces.
    
2. **Scan intent sections** — The intent section gives a one-sentence summary of what the pattern does. Find the intent that matches your need.
    
3. **Study how patterns interrelate** — Relationships between patterns will direct you to the right choice or group of patterns.
    
4. **Study patterns of like purpose** — Each pattern's documentation concludes with a comparison to related patterns. Understand similarities and differences.
    
5. **Examine a cause of redesign** — Look at your problem and identify causes of redesign. Find patterns that prevent those redesign causes.
    
6. **Consider what should be variable** — Consider what you want to be able to change without redesign. Focus on **encapsulating the concept that varies**.
    

---

## 3.11 How to Use a Design Pattern

After selecting a pattern, follow these steps:

1. **Read the pattern once through for an overview** — Pay attention to applicability and consequences.
2. **Study structure, participants, and collaborations** — Understand the classes and their relationships.
3. **Look at the sample code** — Studying code helps understand how to implement it.
4. **Choose meaningful names for participants** — Names in patterns are abstract. Incorporate the participant's name into application names (e.g., `UIStrategy`, `GridStrategy` for a Strategy pattern in UI).
5. **Define the classes** — Declare interfaces, inheritance relationships, instance variables.
6. **Define application-specific operation names** — Use responsibilities and collaborations as a guide. Example: prefix factory methods with `Create-`.
7. **Implement the operations** — Follow hints from the implementation section and sample code.

## 4.1 Overview of Creational Patterns

Creational design patterns **provide various object creation mechanisms** that increase flexibility and reuse of existing code. Instead of calling `new ClassName()` directly in client code, creational patterns abstract the object creation process.

The 5 creational patterns are: **Singleton, Factory Method, Abstract Factory, Builder, Prototype**.

---

## 4.2 Singleton Pattern

### Principle

> Singleton pattern **restricts the instantiation of a class** and ensures that **only one instance** of the class exists in the JVM. It provides a **global access point** to get that instance.

### When Is Singleton Used?

- **Logging** — one shared log manager
- **Driver objects** — e.g., database connection drivers
- **Caching** — one shared cache manager
- **Thread pool** — one shared thread pool
- Other patterns — used internally in **Abstract Factory, Builder, Prototype, Facade**
- Core Java classes — `java.lang.Runtime`, `java.awt.Desktop`

### Three Implementation Concepts

All Singleton implementations share these three elements:

1. **Private constructor** — Prevents other classes from instantiating.
2. **Private static variable** of the same type — The only instance.
3. **Public static method** — The global access point that returns the instance.

### Implementation

```java
public class Singleton {
    // 2. Private static variable — the single instance
    private static Singleton instance;

    // 1. Private constructor — no external instantiation
    private Singleton() {}

    // 3. Public static method — global access point
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();   // lazy initialization
        }
        return instance;
    }
}
```

**Usage:**

```java
Singleton s1 = Singleton.getInstance();
Singleton s2 = Singleton.getInstance();
// s1 == s2 → true (same object)
```

## 4.3 Factory Method Pattern

### Definition

> A Factory Pattern says: "Define an interface or abstract class for creating an object, but let the subclasses decide which class to instantiate."

Subclasses are responsible to create the instance. The factory **moves the responsibility of instantiation** from the client to the factory class.

### When to Use Factory Pattern

- When a superclass has multiple sub-classes and based on input, one of the sub-classes needs to be returned.
- When you want to decouple object creation code from the code that uses the object.

### Key Benefits

- **Loosens coupling** by separating the Product's construction code from code that uses the product
- Easy to **introduce new products** without breaking existing code
- Centralizes object creation → **reuse** of the same factory logic

### Notification System Example

**Step 1: Common Interface**

```java
interface Notification {
    void notifyUser();
}
```

**Step 2: Concrete Classes**

```java
class EmailNotification implements Notification {
    public void notifyUser() { System.out.println("Sending Email Notification"); }
}

class SMSNotification implements Notification {
    public void notifyUser() { System.out.println("Sending SMS Notification"); }
}

class PushNotification implements Notification {
    public void notifyUser() { System.out.println("Sending Push Notification"); }
}
```

**Step 3: Factory Class**

```java
class NotificationFactory {
    public static Notification createNotification(String channel) {
        switch (channel) {
            case "EMAIL": return new EmailNotification();
            case "SMS":   return new SMSNotification();
            case "PUSH":  return new PushNotification();
            default: throw new IllegalArgumentException("Unknown channel: " + channel);
        }
    }
}
```

**Step 4: Client Code**

```java
Notification n = NotificationFactory.createNotification("SMS");
n.notifyUser();   // Sending SMS Notification
```

### Real-World Use Cases

- **Database Access:** `DatabaseConnectionFactory` returns MySQL, PostgreSQL, or Oracle connections based on config — all via a common interface.
- **Cross-Platform UI:** A UI factory renders platform-specific components (Windows button vs. macOS button) while core logic stays consistent.
- **File I/O:** `FileReaderFactory` returns CSV, JSON, or text reader objects through a single interface.

### Pros of Factory Pattern

- **Encapsulates object creation logic** — hidden from client code; cleaner, more maintainable code
- **Open for extension** — add new types (new Concrete Product) without altering existing code; just add a new Concrete Factory
- **Centralizes creation** — reduces redundancy across the application
- **Enforces consistency** — all objects created via a common interface
- **Supports dependency injection** — allows creation and management of dependencies

### Cons of Factory Pattern

- **Increases complexity** — extra layer of classes
- **Additional abstraction** — may confuse developers unfamiliar with the pattern; needs documentation/training
- **Client tightly coupled to Concrete Factories** — even though decoupled from products
- **Unnecessary in simple cases** — when object creation is straightforward, Factory Pattern adds overhead
- **Risk of overuse** — apply only when it genuinely solves a problem

---

## 4.4 Abstract Factory Pattern

### Definition

> "An interface for creating **families of related or dependent objects** without specifying their concrete classes."

Think of it as: **"A factory of factories."**

Instead of creating individual objects directly, the client uses a factory that produces _other factories_, each responsible for creating a **set of related objects**.

### Real-Life Example: Furniture Factory

**Scenario:** Designing furniture for Victorian and Modern style houses. Each style needs a `Chair` and a `Table`. You don't want to accidentally mix styles (no Modern Chair in a Victorian room).

**Solution:** Create a factory for each style:

- `ModernFurnitureFactory` → creates `ModernChair` and `ModernTable`
- `VictorianFurnitureFactory` → creates `VictorianChair` and `VictorianTable`

### Components and Their Relationships

**Abstract Factory (`FurnitureFactory`):**

- Declares methods: `createChair()` and `createTable()`
- Acts as the blueprint for all concrete factories

**Concrete Factories:**

- `ModernFurnitureFactory` — implements `FurnitureFactory` to create Modern furniture
- `VictorianFurnitureFactory` — implements `FurnitureFactory` to create Victorian furniture

**Abstract Products:**

- `Chair` — interface with `sitOn()` method
- `Table` — interface with `use()` method

**Concrete Products:**

- `ModernChair` implements `Chair`
- `ModernTable` implements `Table`
- `VictorianChair` implements `Chair`
- `VictorianTable` implements `Table`

**Client (`Main`):**

- Uses `FurnitureFactory` interface
- Delegates creation to specific factories
- Interacts with products only through abstract interfaces

### Flow of Operations

```
Main
 └── selects FurnitureFactory (e.g., ModernFurnitureFactory)
       ├── createChair() → ModernChair
       └── createTable() → ModernTable
       
Main interacts with Chair via sitOn()
Main interacts with Table via use()
```

### Java Structure (GUI Toolkit Example)

```java
// Abstract Factory
interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

// Concrete Factories
class WindowsFactory implements GUIFactory {
    public Button createButton() { return new WindowsButton(); }
    public Checkbox createCheckbox() { return new WindowsCheckbox(); }
}

class MacOSFactory implements GUIFactory {
    public Button createButton() { return new MacOSButton(); }
    public Checkbox createCheckbox() { return new MacOSCheckbox(); }
}
```

### Factory Method vs. Abstract Factory

|Aspect|Factory Method|Abstract Factory|
|---|---|---|
|Design Tool|Uses **inheritance**|Uses **delegation**|
|Scope|Over a **method**|Over a **class**|
|Creates|**One product**|**Families of products**|
|Implementation|Relies on derived class|Relies on composition|
|Both follow|OCP and SRP|OCP and SRP|

Both patterns result in loosely coupled code and creation code in one place.

## 4.5 Builder Pattern

### Problem It Solves

Builder pattern was introduced to solve problems with Factory and Abstract Factory when the **Object contains many attributes**. Three major issues with factories for complex objects:

1. **Too many arguments** — Error-prone because most arguments are the same type, and the order is hard to maintain on the client side.
2. **Optional parameters forced as NULL** — Factory pattern forces sending all parameters; optional ones must be sent as `null`.
3. **Complex creation** — If the object is heavy and creation is complex, all complexity ends up in the Factory class, causing confusion.

### Definition

> "Separate the construction of a complex object from its representation so that the same construction process can create different representations."

The Builder constructs a complex object **step by step**, and the final step returns the completed object.

### Implementation Steps in Java

1. **Create a static nested class** (named `<OuterClass>Builder`, e.g., `ComputerBuilder`).
2. **Public constructor** in the Builder class with all **required** attributes as parameters.
3. **Methods to set optional parameters** — each returns `this` (the same Builder object) for chaining.
4. **`build()` method** — returns the final object. The outer class has a **private constructor** taking the Builder as argument.

### Car Builder Example

```java
public class Car {
    // Required
    private final String make;
    private final String model;
    // Optional
    private final String color;
    private final int year;
    private final double maxSpeed;

    // Private constructor — only Builder can call this
    private Car(Builder builder) {
        this.make = builder.make;
        this.model = builder.model;
        this.color = builder.color;
        this.year = builder.year;
        this.maxSpeed = builder.maxSpeed;
    }

    // Static nested Builder class
    public static class Builder {
        private final String make;   // required
        private final String model;  // required
        private String color;        // optional
        private int year;            // optional
        private double maxSpeed;     // optional

        public Builder(String make, String model) {
            this.make = make;
            this.model = model;
        }

        public Builder setColor(String color) {
            this.color = color;
            return this;  // returns Builder for chaining
        }

        public Builder setYear(int year) {
            this.year = year;
            return this;
        }

        public Car build() {
            return new Car(this);  // final step — creates the Car
        }
    }
}

// Client usage — fluent/chained style
Car car = new Car.Builder("BMW", "X5")
                 .setColor("Blue")
                 .setYear(2023)
                 .build();
```

**What the Code Does:**

- `new Car.Builder("BMW", "X5")` — initializes builder with mandatory fields
- `.setColor("Blue")` / `.setYear(2023)` — sets optional attributes; each returns the builder for chaining
- `.build()` — validates data and returns the actual `Car` object

### Advantages of Builder Pattern

1. **Readable constructor** — parameters passed via clearly named method calls, not positional arguments
2. **No null for optional parameters** — optional fields simply aren't set
3. **Object always fully initialized** — built in a complete state
4. **Immutable objects** — can create immutable objects without complex logic

### Disadvantages of Builder Pattern

1. **More lines of code** — at least doubles the code, but pays off in flexibility and readability
2. **Separate ConcreteBuilder for each Product type** — increases class count

## 4.6 Prototype Pattern

### Definition

> Prototype design pattern is used when **object creation is costly** (time and resources) and a **similar object already exists**. It provides a mechanism to **copy the original object** to a new object and then modify it as needed.

Uses **Java cloning** to copy the object.

### When to Use Prototype Pattern

- When creating a new object requires heavy database queries, network calls, or complex initialization.
- When you need multiple objects that are only slightly different from an existing object.
- To avoid subclasses that differ only in their initial state — use prototypes with different initial states instead.

### Key Rule

**The object itself must provide the copying feature** — not an external class. The prototype class implements the `clone()` method.

### Shallow vs. Deep Copy

- **Shallow copy:** Copies primitive fields by value; copies object references (both original and clone point to same sub-objects).
- **Deep copy:** Copies everything recursively — the clone is completely independent of the original. Whether to use shallow or deep copy depends on requirements and is a design decision.

### Sheep Farm Example

**Step 1: Prototype Interface**

```java
public interface Sheep {
    Sheep clone();
    String getName();
    void setName(String name);
}
```

**Step 2: Concrete Prototype — BlackSheep**

```java
public class BlackSheep implements Sheep {
    private String name;

    public BlackSheep(String name) { this.name = name; }

    public Sheep clone() { return new BlackSheep(this.name); }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}
```

**Step 3: Concrete Prototype — WhiteSheep**

```java
public class WhiteSheep implements Sheep {
    private String name;

    public WhiteSheep(String name) { this.name = name; }

    public Sheep clone() { return new WhiteSheep(this.name); }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}
```

**Step 4: Client Code**

```java
public class SheepFarm {
    public static void main(String[] args) {
        // Create prototype sheep
        Sheep blackPrototype = new BlackSheep("Baa Baa");
        Sheep whitePrototype = new WhiteSheep("Fleecy");

        // Clone sheep as needed
        Sheep clonedBlack = blackPrototype.clone();
        Sheep clonedWhite = whitePrototype.clone();

        // Customize the clones
        clonedBlack.setName("Midnight");
        clonedWhite.setName("Snowball");

        System.out.println("Black sheep: " + clonedBlack.getName());  // Midnight
        System.out.println("White sheep: " + clonedWhite.getName());  // Snowball
    }
}
```

### Database Example

Suppose an object loads data from a database. You need to modify this data in the program multiple times. Rather than:

- Creating a new object with `new` keyword (costly — reloads data from DB)

You should:

- Clone the existing loaded object and then do data manipulation on the clone

### Advantages of Prototype Pattern

- Avoids costly re-creation when new object is only slightly different from existing one
- Allows creation of objects in appropriate states beforehand, then cloning as needed
- Eliminates subclasses that differ only in initial state — use prototypes with different states instead

### Disadvantages of Prototype Pattern

- When there are **many classes**, the cloning process can get complex and messy
- **Circular references** make cloning difficult (A contains B, B contains A)
- Pattern should only be used when appropriate — not a universal solution

## Design Patterns — Question & Answer Bank

**Q1. Who wrote the GoF book and what does GoF stand for?** GoF stands for **Gang of Four** — the four authors of _"Design Patterns: Elements of Reusable Object-Oriented Software"_: **Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides**.

**Q2. What are the three categories of GoF design patterns? Name all 23 patterns.** Creational (5): Abstract Factory, Builder, Factory Method, Prototype, Singleton. Structural (7): Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy. Behavioral (11): Chain of Responsibility, Command, Interpreter, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method, Visitor.

**Q3. What are the four basic elements in a design pattern specification?** Name (design vocabulary), Problem (the design problem and context), Solution (classes, objects, and their relationships), and Consequences (costs and benefits of using the pattern).

**Q4. What are the three components needed to implement a Singleton pattern?** (1) Private constructor — prevents external instantiation. (2) Private static variable of the same type — holds the single instance. (3) Public static method — the global access point that returns the single instance.

**Q5. What is the difference between Factory Method and Abstract Factory patterns?** Factory Method uses **inheritance** (a derived class creates one product); Abstract Factory uses **delegation** (a factory creates families of related products). Factory Method operates over a method; Abstract Factory operates over a class. Abstract Factory is essentially a "factory of factories."

**Q6. What are the three main problems with Factory/Abstract Factory that the Builder Pattern solves?** (1) Too many arguments to the constructor (error-prone, hard to maintain order). (2) Optional parameters must be passed as null in factories. (3) Complex creation logic clutters the Factory class.

**Q7. Explain the Prototype Pattern with a real-world scenario.** Prototype is used when object creation is expensive. Instead of creating new objects from scratch, you clone an existing prototype. Example: A Sheep farm creates `BlackSheep` and `WhiteSheep` prototypes, then clones them to create multiple sheep — each clone can be customized independently. The clone method is implemented by the object itself.

**Q8. What are the two foundational OOP principles that GoF patterns are based on?** (1) _Program to an interface, not an implementation._ (2) _Favor object composition over inheritance._

**Q9. What is the difference between Architecture Patterns and Design Patterns?** Architecture patterns are high-level and define how components of an entire system are organized and assembled. Design patterns are lower-level and define how individual components are built. Architecture patterns define granularity; design patterns operate within that granularity.

**Q10. What is an Anti-Pattern and why does it arise?** An Anti-Pattern is a pattern that is overused or abused — a commonly occurring solution that seems helpful but is counterproductive in practice. It arises when design patterns are applied without understanding the problem or context. Over-application of patterns is one of the key cons of design patterns.

**Q11. What are the advantages and disadvantages of the Builder Pattern?** Advantages: Readable method-chained construction, no null for optional parameters, always creates fully initialized objects, enables immutable objects. Disadvantages: At least doubles lines of code, requires a separate ConcreteBuilder for each product type.

**Q12. What is the practical use case of the Factory Pattern in database access?** A `DatabaseConnectionFactory` creates the appropriate connection object (MySQL, PostgreSQL, Oracle) based on configuration settings. The client uses a common `DatabaseConnection` interface — it doesn't need to know which database engine is behind it.

**Q13. Why does the Prototype Pattern require the prototype object itself to provide the cloning feature?** Because only the object itself knows its internal structure and can correctly copy all its fields (including private ones). An external class wouldn't have access to private fields and couldn't perform a complete, accurate copy.

**Q14. In the Abstract Factory furniture example, what are the abstract products and what are the concrete products?** Abstract Products: `Chair` (with `sitOn()`) and `Table` (with `use()`). Concrete Products: `ModernChair`, `ModernTable`, `VictorianChair`, `VictorianTable`. Concrete Factories: `ModernFurnitureFactory` and `VictorianFurnitureFactory`.

## 1. The Role of the Lexical Analyzer

### What is a Lexical Analyzer?

The **lexical analyzer** (also called the **lexer** or **scanner**) is the first phase of a compiler. It sits between the raw source code and the parser.

Its two primary jobs are:

1. **Read** the input characters of the source program one by one.
2. **Group** those characters into meaningful units called **lexemes**, and for each lexeme, **produce a token** — a symbolic representation — to be passed to the parser.

Think of it this way: if a compiler is like reading a book, the lexical analyzer is the part of your brain that groups letters into words before you can understand sentences.

### The Big Picture — Where Does the Lexer Fit?

```
Source Program
      |
      v
+------------------+
|  Lexical Analyzer|  <---> Symbol Table
+------------------+
      | (stream of tokens)
      v
+------------------+
|     Parser       |  <---> Symbol Table
+------------------+
      | (parse tree / AST)
      v
  ... rest of compiler
```

The lexer **interacts with the symbol table**: when it finds an identifier (e.g., a variable name), it enters that identifier into the symbol table (or looks it up if it already exists).

### Secondary Tasks of the Lexical Analyzer

Beyond just tokenizing, the lexer often:

- **Strips comments** and **whitespace** (blanks, tabs, newlines) — these separate tokens but carry no meaning to the parser.
- **Tracks line numbers** — so that error messages produced later can point to the exact line in the source code.
- **Expands macros** (in some compilers).
- **Inserts error messages** at appropriate positions in a copy of the source.

### Scanning vs. Lexical Analysis

Sometimes the lexer is split into two sub-phases:

|Sub-phase|What it does|
|---|---|
|**Scanning**|Simple tasks: deletes comments, compacts consecutive whitespace into one|
|**Lexical Analysis**|The complex part: produces the sequence of tokens|

---

### 🔎 Q&A — The Role of the Lexical Analyzer

**Q1. What are the two primary tasks of a lexical analyzer?**

> **A:** (1) Read the input characters and group them into lexemes. (2) Produce a sequence of tokens for each lexeme, which is sent to the parser.

**Q2. Why does the lexical analyzer interact with the symbol table?**

> **A:** When the lexer discovers a lexeme that is an identifier, it needs to store (or retrieve) information about that identifier — such as its name, type, and declaration location — in the symbol table.

**Q3. What is the difference between a scanner and a lexical analyzer?**

> **A:** A scanner performs simple preprocessing — deleting comments and compacting whitespace. The lexical analyzer does the heavier work of actually producing tokens from the cleaned input. They are two sub-phases of the overall lexing process.

**Q4. Give two secondary tasks the lexical analyzer performs besides tokenizing.**

> **A:** Any two of: (1) stripping comments and whitespace, (2) tracking line numbers for error messages, (3) expanding macros, (4) inserting error messages into a source copy.

**Q5. Where in the compiler pipeline does the lexical analyzer sit, and what does it output?**

> **A:** It is the first phase, directly after the raw source program. It outputs a stream of tokens, which it sends to the parser (the second phase).

## 2. Tokens, Patterns, and Lexemes

These three terms are tightly related but distinct. Confusing them is one of the most common mistakes beginners make.

### Token

A **token** is a pair:

```
<token-name, attribute-value>
```

- The **token name** is an abstract symbol representing a _category_ of lexical unit (e.g., `id`, `number`, `if`, `relop`). This is what the **parser** works with.
- The **attribute value** is optional extra information (e.g., a pointer to the symbol table entry, or the actual numeric value).

**Examples of tokens:**

- `<id, pointer to "score">` — an identifier
- `<number, 42>` — a numeric literal
- `<if>` — the keyword IF (no attribute needed, since the token name itself is enough)
- `<relop, LT>` — relational operator "less than"

### Pattern

A **pattern** is the rule (description) that defines what lexemes can belong to a token. It is typically written as a **regular expression**.

- For a keyword like `if`, the pattern is simply the string `if`.
- For identifiers, the pattern is more general: `letter_ (letter_ | digit)*` — any string starting with a letter/underscore followed by more letters, digits, or underscores.

### Lexeme

A **lexeme** is an **actual sequence of characters** in the source program that matches a pattern. It is the raw text.

> 🎯 **Memory trick:**
> 
> - **Pattern** = the rule (like a regex / template)
> - **Lexeme** = the actual text found in the source code
> - **Token** = the labeled representation of that lexeme, passed to the parser

### Concrete Example

Consider the C statement:

```c
printf("Total = %d\n", score);
```

|Lexeme|Token Name|Attribute|
|---|---|---|
|`printf`|`id`|pointer to symbol table entry for `printf`|
|`(`|`(`|—|
|`"Total = %d\n"`|`literal`|pointer to literal value|
|`,`|`,`|—|
|`score`|`id`|pointer to symbol table entry for `score`|
|`)`|`)`|—|
|`;`|`;`|—|

**Total: 7 tokens/lexemes**

### Token Classes in Programming Languages

Most languages use these token categories:

1. **One token per keyword** — pattern = the keyword string itself (`if`, `while`, `for`, etc.)
2. **Operator tokens** — either individually (`+`, `-`) or grouped (e.g., `relop` for all comparison operators)
3. **One token for all identifiers** — the pattern matches all valid identifiers
4. **Constant tokens** — for integer literals, float literals, string literals
5. **Punctuation tokens** — `(`, `)`, `[`, `]`, `,`, `;`, etc.

## 3. Attributes for Tokens

When multiple lexemes can match the same pattern, the lexer must pass additional information to downstream phases. This is what the **attribute value** carries.

### Key Cases

**Identifiers:** The attribute value is a **pointer to the symbol table entry** for that identifier.

```
E    →  <id, ptr→E>
M    →  <id, ptr→M>
C    →  <id, ptr→C>
```

**Numbers:** The attribute is the **actual value** (or a pointer to a table of constants).

```
10   →  <number, 10>
3.14 →  <number, 3.14>
```

**Relational Operators:** A symbolic constant is used to indicate _which_ operator.

```
<    →  <relop, LT>
<=   →  <relop, LE>
=    →  <relop, EQ>
<>   →  <relop, NE>
>    →  <relop, GT>
>=   →  <relop, GE>
```

**Keywords and Punctuation:** Usually **no attribute** is needed.

```
if   →  <if>    (or <IF>)
;    →  <;>
```

### Why Attributes Matter

The **token name influences parsing decisions** (the parser uses grammar rules based on token names), while the **attribute value influences code generation** (the code generator needs to know if the number was `10` or `11` to emit the right machine code).

---

### 🔎 Q&A — Attributes for Tokens

**Q1. For the token `relop`, why is an attribute value necessary?**

> **A:** Because `relop` covers six different operators (`<`, `<=`, `=`, `<>`, `>`, `>=`). The attribute (e.g., `LT`, `LE`, `EQ`) tells the code generator exactly which comparison to perform.

**Q2. What kind of attribute value is associated with an identifier token, and why?**

> **A:** A pointer to the symbol table entry. This allows all compiler phases to access the identifier's full information (name, type, scope, etc.) efficiently without duplicating data.

**Q3. Why do keyword tokens like `if` typically not carry an attribute value?**

> **A:** The token name itself is sufficient — `<if>` unambiguously tells the parser what was found. There is no variation within a keyword.

**Q4. What is the token representation for the statement `E = M * C ** 2`?**

> **A:**
> 
> ```
> <id, ptr→E>  <assign_op>  <id, ptr→M>  <mult_op>  <id, ptr→C>  <exp_op>  <number, 2>
> ```

## 4. Why Separate Lexical and Syntax Analysis?

Many students wonder: why not do it all in one phase? There are three strong reasons.

### 1. Simplicity of Design

Separating concerns makes each task simpler. A parser that also had to handle comments, whitespace, and character-level decisions would be much more complex. By stripping those out first, the parser sees a clean stream of tokens.

> If we are designing a new language, separating lexical and syntactic concerns also leads to a cleaner overall language design.

### 2. Improved Compiler Efficiency

The lexical analyzer can use highly optimized buffering and scanning techniques tailored to character-by-character reading. These techniques would be overkill (and out of place) inside a parser.

Specialized lexer generators (like Lex/Flex) produce extremely fast scanners using DFA table lookups, which would be harder to integrate into a general parser.

### 3. Enhanced Portability

Machine-specific quirks — such as different end-of-line conventions (`\n` vs `\r\n`), different encodings, or unusual whitespace characters — can be handled entirely within the lexer. The parser above it remains portable.

---

### 🔎 Q&A — Why Separate?

**Q1. Name the three reasons for separating lexical analysis from parsing.**

> **A:** (1) Simplicity of design, (2) Improved compiler efficiency, (3) Enhanced portability.

**Q2. How does separating lexical analysis improve parser simplicity?**

> **A:** The parser receives a clean stream of tokens without needing to handle comments, whitespace, or character-level decisions, making the grammar rules far simpler.

**Q3. What kind of efficiency gains come from a separate lexer?**

> **A:** Specialized input buffering techniques (like two-pointer/two-buffer schemes) can be used just for character reading. DFA-based scanners produced by tools like Lex are extremely fast at token recognition.

## 5. Input Buffering

Reading source code one character at a time would be too slow. The lexical analyzer uses a **two-buffer scheme** to read input efficiently.

### The Two-Buffer Scheme

Two buffers of size `N` (typically 4096 characters, matching a disk block) are used alternately.

```
+---------------------+---------------------+
|   Buffer 1          |   Buffer 2          |
|  E = M * ...  eof   |  C ** 2   ...  eof  |
+---------------------+---------------------+
        ^                      ^
   lexemeBegin              forward
```

- **`lexemeBegin`** pointer: marks the start of the current lexeme being scanned.
- **`forward`** pointer: scans ahead until a pattern match is found.

Once `forward` hits `eof` at the end of buffer 1, buffer 2 is loaded. This alternation continues until the end of the source file.

### The Sentinel — Using `eof` as a Guard

To avoid checking for buffer boundaries on _every_ character read, a special sentinel character `eof` is placed at the **end of each buffer half**. This means:

- Most characters are processed without any boundary check.
- Only when `eof` is seen do we check: is this a real end-of-file, or just the buffer end?

This optimization makes the inner loop of the scanner as fast as possible.

### Why Two Pointers?

The `forward` pointer can look several characters ahead (lookahead) before we commit to a lexeme. Once a lexeme is found:

- The text from `lexemeBegin` to `forward` (or just before it) is the lexeme.
- `lexemeBegin` is advanced to just after the found lexeme.
- `forward` continues scanning from there.

---

### 🔎 Q&A — Input Buffering

**Q1. Why does the lexical analyzer use a two-buffer scheme rather than reading one character at a time?**

> **A:** Reading one character at a time would require a system call for every character, which is extremely slow. Buffering reads large blocks (e.g., 4096 bytes) at once, amortizing the I/O cost.

**Q2. What are the two pointers used in input buffering, and what does each track?**

> **A:** `lexemeBegin` marks the start of the current lexeme being recognized. `forward` scans ahead looking for the end of the lexeme.

**Q3. What is a sentinel character, and why is it used?**

> **A:** A sentinel is a special character (`eof`) placed at the end of each buffer half. It eliminates the need to check for buffer boundaries on every character read — the boundary check only happens when `eof` is encountered, reducing overhead in the inner scan loop.

**Q4. What happens when the `forward` pointer reaches the end of buffer 1?**

> **A:** The second buffer is loaded with the next block of input characters, and scanning continues from the start of buffer 2.

## 6. Specification of Tokens — Regular Expressions

The standard formalism for describing token patterns is **regular expressions (regex)**.

### Alphabet and Strings

- An **alphabet** Σ is any finite set of symbols (e.g., ASCII characters).
- A **string** over Σ is a finite sequence of symbols from Σ.
- The **empty string** is denoted ε.
- `|s|` denotes the length of string `s`.

### Operations on Languages

If L and M are languages (sets of strings):

|Operation|Notation|Meaning|
|---|---|---|
|Union|L ∪ M|Strings in L **or** M|
|Concatenation|LM|All strings formed by a string from L followed by one from M|
|Kleene closure|L*|Zero or more concatenations of strings from L|
|Positive closure|L+|One or more concatenations of strings from L|

### Regular Expressions — Full Definition

A **regular expression** over alphabet Σ is defined inductively:

**Basis:**

- `ε` is a regex denoting `{ε}` (the set containing only the empty string).
- `a` (for any `a` ∈ Σ) is a regex denoting `{a}`.

**Inductive rules** (if `r` and `s` are regular expressions over L(r) and L(s)):

|Regex|Meaning|Notation|
|---|---|---|
|`(r)|(s)`|Union: L(r) ∪ L(s)|
|`(r)(s)`|Concatenation: L(r)L(s)|sequence|
|`(r)*`|Kleene closure: (L(r))*|zero or more|
|`(r)`|Grouping: L(r)|just L(r)|

### Operator Precedence (highest to lowest)

1. `*` (unary, highest precedence, left-associative)
2. Concatenation (left-associative)
3. `|` (lowest precedence, left-associative)

So `ab|c` means `(ab)|c`, and `ab*` means `a(b*)`.

### Algebraic Laws for Regular Expressions

These allow simplification:

|Law|Expression|
|---|---|
|Commutativity of `\|`|`r\|s = s\|r`|
|Associativity of `\|`|`r\|(s\|t) = (r\|s)\|t`|
|Associativity of concat|`r(st) = (rs)t`|
|Identity for concat|`εr = rε = r`|
|Annihilator|`∅r = r∅ = ∅`|
|Distributivity|`r(s\|t) = rs\|rt`|
|Idempotency of `\|`|`r\|r = r`|
|Closure identities|`(r*)* = r*`, `r*\|ε = r*`, `(r\|ε)* = r*`|

### 🔎 Q&A — Regular Expressions

**Q1. What does the regular expression `a(b|c)*d` denote? Give three example strings it matches.**

> **A:** It matches strings starting with `a`, followed by zero or more `b`s or `c`s (in any mix), followed by `d`. Examples: `ad`, `abd`, `abcd`, `acbd`, `abbcbd`.

**Q2. What is the difference between `r*` and `r+`?**

> **A:** `r*` means zero or more repetitions of `r` (includes the empty string ε). `r+` means one or more repetitions (does NOT include ε). Formally, `r+ = rr*`.

**Q3. In the regex `ab|c`, what strings are matched?**

> **A:** The strings `ab` and `c`. The `|` has lower precedence than concatenation, so this is `(ab)|c`, not `a(b|c)`.

**Q4. Write a regular expression for unsigned integers (no leading zeros, except for the number 0 itself).**

> **A:** `0 | [1-9][0-9]*`

**Q5. What algebraic law shows that `r|r = r`?**

> **A:** Idempotency of union/alternation.

**Q6. What does ε (epsilon) represent as a regular expression?**

> **A:** It represents the language containing only the empty string: `{ε}`. A regex matches ε when it can match a string of zero length.

**Q7. What is the Kleene closure of `{a, b}`?**

> **A:** `{ε, a, b, aa, ab, ba, bb, aaa, ...}` — all strings (including empty) made of `a`s and `b`s.

## 7. Regular Definitions & Extensions

### Regular Definitions

A **regular definition** gives names to regular expressions, allowing reuse:

```
d1 → r1
d2 → r2 (may use d1)
...
dn → rn
```

Each `di` is a distinct name, and each `ri` is a regex over Σ ∪ {d1, d2, ..., d(i-1)}.

**Example — Identifiers in C:**

```
letter_  →  A | B | ... | Z | a | b | ... | z | _
digit    →  0 | 1 | ... | 9
id       →  letter_ ( letter_ | digit )*
```

**Example — Unsigned numbers:**

```
digit    →  [0-9]
digits   →  digit+
number   →  digits (\. digits)? (E [+-]? digits)?
```

This definition captures: integers (`1234`), floats (`55.99`), and scientific notation (`5.105E255`, `6.22E-88`).

### Extensions / Shorthand

These are notational conveniences (they don't add expressive power beyond basic regex):

**1. `r+` — One or more:** `r+ = rr*`

> "One or more occurrences of r" Example: `digit+` = one or more digits

**2. `r?` — Zero or one:** `r? = r | ε`

> "Optional r" Example: `[+-]?` = optional sign

**3. `[a1a2...an]` — Character class:** Shorthand for `a1 | a2 | ... | an`

> `[abc]` = `a|b|c` `[a-z]` = `a|b|c|...|z` (using hyphen for range) `[A-Za-z_]` = any letter or underscore

**4. `.` — Any character except newline** (in most implementations)

**5. `[^...]` — Negated character class:**

> `[^0-9]` = any character that is NOT a digit

---

> **A:** `[A-Za-z_][A-Za-z0-9_]*`

**Q2. What does `digits (\. digits)? (E [+-]? digits)?` match? Give two examples.**

> **A:** It matches unsigned numbers in integer or floating-point (with optional exponent) form. Examples: `123`, `55.99`, `6.22E-88`, `5.105E255`.

**Q3. Is the `+` operator in regex strictly necessary, or is it just shorthand?**

> **A:** It is shorthand — `r+` is equivalent to `rr*`. Regular expressions without `+` have exactly the same expressive power. The `+` just makes patterns more readable.

**Q4. Write a regular definition for a C floating-point constant (e.g., `3.14`, `.5`, `3.`).**

> **A:**
> 
> ```
> digit    → [0-9]
> digits   → digit+
> float    → digits . digits? | . digits
> ```
> 
> (simplified version)

**Q5. What does `[^aeiou]` match?**

> **A:** Any single character that is NOT a vowel (i.e., any consonant, digit, whitespace, punctuation, etc.).

## 8. Recognition of Tokens — Transition Diagrams

### What is a Transition Diagram?

A **transition diagram** is a stylized flowchart used as an intermediate step between a regular expression pattern and actual lexer code. It is essentially a **visual deterministic finite automaton (DFA)**.

Components:

- **States** (circles/nodes) — each represents a situation during scanning.
- **Edges** (arrows) — labeled with a character or set of characters; represent transitions.
- **Start state** — one state designated as the initial state (marked with an arrow labeled "start" entering from nowhere).
- **Accepting/Final states** — double circles; indicate a lexeme has been found.
- **`*` on accepting state** — indicates the `forward` pointer must be **retracted** one position (the character that caused acceptance is not part of the lexeme).

### How Scanning Works

1. Begin in the start state.
2. Read the next input character; follow the edge labeled with that character.
3. Repeat until reaching an accepting state (lexeme found) or no valid edge (error or restart).
4. If an accepting state is marked `*`, retract `forward` by one.

### Example — Relational Operators (`relop`)

For the token `relop` (which recognizes `<`, `<=`, `<>`, `=`, `>`, `>=`):

```
Start →[<]→ State1 →[=]→ State2* (return LE)
              |----[>]→ State3* (return NE)
              |----[other]→ State4* (return LT, retract)
       →[=]→ State5* (return EQ)
       →[>]→ State6 →[=]→ State7* (return GE)
              |----[other]→ State8* (return GT, retract)
```

### Example — Identifiers and Keywords

```
Start →[letter_]→ State9 →[letter_ or digit]→ State9 (loop)
                   |----[other]→ State10* (return getToken(), retract)
```

At state 10, we call `getToken()` which checks the symbol table — if the lexeme is a keyword, return the keyword token; otherwise return `id`.

### Recognition of Reserved Words — Two Strategies

**Strategy 1: Pre-install keywords in the symbol table**

- Before scanning begins, insert `if → IF`, `then → THEN`, `else → ELSE`, etc., into the symbol table.
- Use a single transition diagram for identifiers.
- `installID()` places a found identifier in the table if not already there; `getToken()` returns whatever the table says the token is.
- Any identifier not already in the table is an `id` (not a keyword).

**Strategy 2: Separate transition diagrams per keyword**

- Build a distinct diagram for each keyword (e.g., `t`→`h`→`e`→`n`→ check for non-letter-digit).
- Must check that the keyword isn't a prefix of a longer identifier (hence the non-letter-digit check).
- Keywords must be prioritized over `id` patterns.

Strategy 1 is more common in practice.

### Unsigned Numbers — Transition Diagram

Recognizing numbers like `15567`, `55.99`, `5.105E255`, `6.22E-88`:

The diagram has branches for:

- Integer: just digits
- Float: digits `.` digits
- Scientific: float `E` optional `[+-]` digits

### 🔎 Q&A — Transition Diagrams

**Q1. What does an accepting state with a `*` mean?**

> **A:** It means the `forward` pointer must be retracted by one position — the last character read (which caused the transition to the accepting state) is NOT part of the current lexeme. It will be re-read as the first character of the next lexeme.

**Q2. In the relop transition diagram, why does recognizing `<` require a lookahead?**

> **A:** Because `<` alone might be the start of `<=` or `<>`. The lexer reads one more character to decide: if it sees `=`, the token is `LE`; if `>`, it's `NE`; otherwise it's `LT` and the lookahead character is put back.

**Q3. Why must the transition diagram for keyword `then` check for a non-letter-or-digit after reading the keyword?**

> **A:** To distinguish the keyword `then` from an identifier like `thennextvalue`. The check ensures that the next character cannot continue an identifier, confirming the keyword has truly ended.

**Q4. Describe the two strategies for recognizing reserved words. Which is more commonly used?**

> **A:** (1) Pre-install keywords in the symbol table — use a single identifier diagram and let the symbol table distinguish keywords from identifiers. (2) Create separate diagrams for each keyword with priority over the general identifier diagram. Strategy 1 is more common.

**Q5. How is the architecture of a transition-diagram-based lexer typically implemented in code?**

> **A:** Each state is represented by a piece of code. A variable `state` holds the current state number. A `switch` statement on `state` dispatches to the code for each state, which in turn reads the next character and determines the next state (often via another `switch` or multi-way branch).

## 9. Finite Automata (NFA and DFA)

### What is a Finite Automaton?

A **finite automaton (FA)** is a formal model of computation used to recognize regular languages. It has:

- A finite set of **states** Q
- An **input alphabet** Σ
- A **transition function** δ
- A **start state** q₀ ∈ Q
- A set of **accepting/final states** F ⊆ Q

### NFA — Nondeterministic Finite Automaton

In an **NFA**, from a given state on a given input symbol, there may be:

- **Zero** transitions (no move — dead end)
- **One** transition
- **Multiple** transitions (nondeterminism — the NFA "guesses" which to take)
- **ε-transitions** — transitions that consume no input symbol

Formally: δ : Q × (Σ ∪ {ε}) → 2^Q

An NFA **accepts** a string if _at least one_ path through the automaton from start to an accepting state exists.

### DFA — Deterministic Finite Automaton

In a **DFA**, from any state on any input symbol, there is **exactly one** transition (no ε-transitions, no ambiguity).

Formally: δ : Q × Σ → Q

A DFA **accepts** a string if and only if the unique path from the start state leads to an accepting state after consuming the entire input.

### Key Differences

|Feature|NFA|DFA|
|---|---|---|
|Transitions per state/symbol|0, 1, or many|Exactly 1|
|ε-transitions|Yes|No|
|Acceptance|Any accepting path exists|Unique path reaches accepting state|
|Implementation complexity|Higher (simulate all paths)|Lower (simple table lookup)|
|Number of states|Often fewer|Can be exponential vs. NFA|
|Expressive power|Equal (same languages recognized)|Equal|

> 🎯 **Key insight:** NFAs and DFAs recognize exactly the same class of languages — the **regular languages**. But DFAs are more efficient to simulate. This is why we convert NFA → DFA.

### ε-closure

The **ε-closure** of a state s (written `ε-closure(s)`) is the set of all states reachable from s using only ε-transitions (including s itself).

**ε-closure(T)** for a set T = union of ε-closure(s) for all s in T.

This is needed for subset construction (converting NFA → DFA).

### Simulating an NFA

To simulate an NFA on input `a1a2...an`:

1. Start with `S = ε-closure({q₀})`
2. For each input symbol `aᵢ`: `S = ε-closure(move(S, aᵢ))`
    - `move(S, a)` = all states reachable from any state in S via `a`
3. Accept if `S ∩ F ≠ ∅`

---

### 🔎 Q&A — Finite Automata

**Q1. What is the key difference between an NFA and a DFA?**

> **A:** In a DFA, each state has exactly one transition for each input symbol, and there are no ε-transitions. In an NFA, a state can have zero, one, or multiple transitions on the same symbol, and ε-transitions (consuming no input) are allowed.

**Q2. Do NFAs recognize a larger class of languages than DFAs? Explain.**

> **A:** No. Both recognize exactly the class of regular languages. Every NFA can be converted to an equivalent DFA (though the DFA may have exponentially more states).

**Q3. What is ε-closure(s)?**

> **A:** The set of all states reachable from state s by following zero or more ε-transitions, including s itself.

**Q4. Why is an NFA preferred as an intermediate representation when compiling regular expressions?**

> **A:** NFAs are easy to construct from regular expressions (using Thompson's construction), and they tend to have fewer states than DFAs. They are then converted to DFAs for efficient simulation.

**Q5. An NFA accepts a string if ____. A DFA accepts a string if ____.**

> **A:** NFA: if _at least one_ computation path from the start state leads to an accepting state after reading the full input. DFA: if the _unique_ path from the start state leads to an accepting state after reading the full input.

**Q6. What formal property of δ distinguishes an NFA from a DFA?**

> **A:** NFA: δ : Q × (Σ ∪ {ε}) → **2^Q** (maps to a _set_ of states). DFA: δ : Q × Σ → **Q** (maps to a _single_ state).

## 10. Thompson's Construction (RE → NFA)

**Thompson's Construction** is a systematic algorithm to convert any regular expression into an NFA. It is the standard method used by Lex internally.

### Base Cases

**For ε:**

```
→ (i) --ε--> (f)
```

Start state i, accepting state f, single ε-transition.

**For a single symbol `a`:**

```
→ (i) --a--> (f)
```

### Inductive Cases

**For `r|s` (union):**

Create a new start state and accepting state; add ε-transitions to the sub-NFAs for r and s.

```
        ε → NFA(r) → ε
→ (i) ↗              ↘ (f)
       ↘              ↗
        ε → NFA(s) → ε
```

**For `rs` (concatenation):**

Merge the accepting state of NFA(r) with the start state of NFA(s).

```
→ NFA(r) → NFA(s) →
```

**For `r*` (Kleene closure):**

Add a new start and accepting state; add ε-loops.

```
         ε (loop back)
         ↑___________|
→ (i) --ε--> NFA(r) --ε--> (f)
       ↘________________________↗
                   ε (skip)
```

### Properties of Thompson NFAs

- Exactly **one start state** and **one accepting state**.
- The accepting state has no outgoing transitions.
- Each state has at most **two ε-transitions** OR **one transition on a symbol** (never both).
- If NFA for r has n states, the combined NFA has at most 2n states.

---

### 🔎 Q&A — Thompson's Construction

**Q1. What algorithm is used to convert a regular expression to an NFA?**

> **A:** Thompson's Construction.

**Q2. How does Thompson's construction handle `r|s`?**

> **A:** It creates a new start state and a new accepting state. It adds ε-transitions from the new start state to the start states of both NFA(r) and NFA(s), and ε-transitions from the accepting states of both sub-NFAs to the new accepting state.

**Q3. How does Thompson's construction handle concatenation `rs`?**

> **A:** The accepting state of NFA(r) is merged with (or connected via ε to) the start state of NFA(s). The result starts at NFA(r)'s start state and ends at NFA(s)'s accepting state.

**Q4. What is a key structural property of any NFA built by Thompson's construction?**

> **A:** It has exactly one start state and one accepting state, and no transitions go into the start state or out of the accepting state.

**Q5. Construct the NFA for `ab` using Thompson's construction.**

> **A:**
> 
> ```
> → (0) --a--> (1) --b--> (2)
> ```
> 
> State 0 is start, state 2 is accepting.

## 11. Subset Construction (NFA → DFA)

The **subset construction** algorithm converts an NFA to a DFA. The key idea: each **DFA state** corresponds to a **set of NFA states** — all the NFA states the NFA could possibly be in simultaneously.

### Algorithm

**Input:** NFA N = (Q, Σ, δ_N, q₀, F) **Output:** DFA D = (Q_D, Σ, δ_D, q_{D0}, F_D)

```
1. q_{D0} = ε-closure({q₀})   // initial DFA state
2. Worklist = {q_{D0}}
3. While worklist is not empty:
     Take T from worklist
     For each input symbol a ∈ Σ:
         U = ε-closure(move(T, a))
         If U is not in Q_D:
             Add U to Q_D and worklist
         δ_D(T, a) = U
4. F_D = {T ∈ Q_D | T ∩ F ≠ ∅}
```

### Key Functions

- **`move(T, a)`** = { s | ∃ t ∈ T such that t →(a) s } — all states reachable from T on input `a`
- **`ε-closure(T)`** = all states reachable from T using only ε-transitions

### Example (Illustrated)

For the NFA recognizing `a*b+`:

- Start: ε-closure({0}) = {0, 1, 3, 7} (example)
- On `a`: compute ε-closure(move({0,1,3,7}, a)) → new DFA state
- On `b`: compute ε-closure(move({0,1,3,7}, b)) → new DFA state
- Continue until no new states emerge.

### Accepting States in the DFA

A DFA state T is an **accepting state** if it contains at least one NFA accepting state.

### Worst Case

In the worst case, a DFA constructed from an NFA with n states may have **2^n states**. However, in practice (especially for lexer patterns), the number is much smaller.

---

### 🔎 Q&A — Subset Construction

**Q1. What is the core idea behind subset construction?**

> **A:** Each DFA state represents a _set_ of NFA states — the set of all NFA states that could possibly be active simultaneously for a given input prefix. This eliminates nondeterminism.

**Q2. When is a DFA state constructed by subset construction an accepting state?**

> **A:** When the set of NFA states it represents contains at least one NFA accepting state.

**Q3. What is the worst-case number of DFA states when converting an NFA with n states?**

> **A:** 2^n (exponential). In practice, far fewer states are typically needed.

**Q4. Define `move(T, a)` as used in subset construction.**

> **A:** `move(T, a)` is the set of all NFA states s such that there is a transition from some state t ∈ T to s on input symbol a (i.e., t →(a) s ∈ δ_N).

**Q5. In what order are the two steps applied? NFA input → ε-closure, or ε-closure → NFA input?**

> **A:** For each symbol a: first apply `move(T, a)` (follow a-transitions), THEN apply `ε-closure` to the result. So: `ε-closure(move(T, a))`.

**Q6. Why is subset construction needed at all? Why not just simulate the NFA?**

> **A:** You can simulate an NFA, but it requires maintaining a set of active states and is slower. A DFA has a single active state at all times and can be simulated in O(|input|) time with a simple table lookup — ideal for a fast lexer.

## 12. Minimizing DFAs

After subset construction, the DFA may have redundant states (states that behave identically for all inputs). **DFA minimization** reduces it to the smallest equivalent DFA.

### Algorithm — Table-Filling / Partition Refinement

1. **Start:** Partition states into two groups: accepting states (A) and non-accepting states (NA).
2. **Refine:** Repeatedly split groups: two states p, q are in the same group only if for _every_ input symbol a, δ(p,a) and δ(q,a) are in the **same group**.
3. **Stop:** When no further splitting is possible.
4. **Build:** One DFA state per group; transitions determined by the transition of any representative.

### Dead States

A **dead state** (or error state) is a non-accepting state with transitions only to itself (on all inputs). It represents failure to match any pattern. It can often be omitted from the minimized DFA representation (transitions to it are implicit).

---

### 🔎 Q&A — DFA Minimization

**Q1. What is the goal of DFA minimization?**

> **A:** To produce the smallest DFA (fewest states) that accepts exactly the same language as the original DFA.

**Q2. What is the starting partition in the table-filling minimization algorithm?**

> **A:** Two groups: all accepting states in one group, all non-accepting states in another.

**Q3. When are two states p and q distinguishable?**

> **A:** When there exists some string w such that exactly one of δ*(p,w) and δ*(q,w) is an accepting state. Equivalently, on some input symbol a, δ(p,a) and δ(q,a) land in different groups of the current partition.

**Q4. What is a dead state in a DFA?**

> **A:** A non-accepting state from which no accepting state can ever be reached — all transitions from it loop back to itself. It represents a permanent failure to match any pattern.

## 13. The Lex/Flex Tool

### What is Lex?

**Lex** is a tool (program) that **automatically generates a lexical analyzer** (lexer) from a specification file. You describe token patterns using regular expressions, and Lex produces C code that performs the lexical analysis.

**Flex** (Fast Lex) is the modern, improved version — faster, more efficient, but backward-compatible with Lex.

### Internal Pipeline

```
Regular Expressions
        |
        v (Thompson's Construction)
       NFA
        |
        v (Subset Construction)
       DFA
        |
        v (Minimization)
  Minimized DFA
        |
        v (Code Generation)
   lex.yy.c  (C code with DFA tables)
        |
        v (compiled with C compiler)
  Lexical Analyzer executable
```

### Structure of a Lex Program

A Lex specification file has three sections separated by `%%`:

```
declarations
%%
translation rules
%%
auxiliary functions
```

#### Section 1: Declarations (Optional)

- **C code in `%{ ... %}`** — header files, `#define`s, variable declarations; copied verbatim to `lex.yy.c`.
- **Regular definitions** — named abbreviations for patterns, e.g.:
    
    ```
    digit   [0-9]letter  [A-Za-z_]id      {letter}({letter}|{digit})*
    ```
    

Note: Regular definitions used in patterns are enclosed in `{}`.

#### Section 2: Translation Rules (Must be present)

Format: `<pattern> <action>`

```lex
if          { return IF; }
[A-Za-z_][A-Za-z0-9_]*   { yylval = yytext; return ID; }
[0-9]+      { yylval = atoi(yytext); return NUMBER; }
[ \t\n]+    { /* ignore whitespace */ }
.           { printf("Unknown character: %c\n", yytext[0]); }
```

- Pattern starts at beginning of line; action follows.
- Action is a C statement (or multiple statements in `{}`).
- Unmatched characters are copied to stdout by default.

#### Section 3: Auxiliary Functions (Optional)

Any C code — typically includes `main()`:

```c
%%
int main() {
    yylex();
    return 0;
}
```

### Predefined Variables in Lex

|Variable|Meaning|
|---|---|
|`yytext`|Pointer to the string of the matched lexeme|
|`yyleng`|Length of the matched lexeme|
|`yyin`|Input file pointer (default: `stdin`)|
|`yyout`|Output file pointer (default: `stdout`)|

### Lex Library Functions

|Function|Purpose|
|---|---|
|`yylex()`|The main lexing function; scans input and returns token values|
|`yymore()`|Appends next matched string to current `yytext` (instead of replacing)|
|`yyless(n)`|Keeps only first n chars in `yytext`; returns rest to input|
|`yywrap()`|Called when input is exhausted; default returns 1 (stop)|

### Conflict Resolution in Lex — Maximum Munch Rule

When multiple patterns could match, Lex uses two rules:

1. **Longest match (Maximum Munch):** Always prefer the longest possible prefix that matches any pattern.
    
    > Example: `<=` is preferred over `<` followed by `=`.
    
2. **First listed:** If two patterns match the same longest prefix, prefer the pattern listed **first** in the Lex program.
    
    > This is why keywords are listed before the general `id` pattern — `then` will be recognized as keyword `THEN` rather than identifier `then`.
    

### Execution Steps

```bash
lex program.l          # Produces lex.yy.c
cc lex.yy.c -ll -o lexer   # Compile with Lex library
./lexer < input.txt    # Run on input
```

(With Flex: replace `lex` with `flex` and `-ll` with `-lfl`)

### 🔎 Q&A — The Lex/Flex Tool

**Q1. What does Lex take as input, and what does it produce as output?**

> **A:** Lex takes a specification file (`.l` extension) containing regular expression patterns and associated actions. It produces `lex.yy.c`, a C file containing a DFA-based lexer.

**Q2. What is the Maximum Munch rule in Lex?**

> **A:** Lex always chooses the longest possible prefix of the remaining input that matches any pattern. Among equally long matches, the pattern listed first in the specification wins.

**Q3. What does `yytext` contain?**

> **A:** A null-terminated string containing the text of the most recently matched lexeme.

**Q4. Why are keywords listed before the general identifier pattern in a Lex program?**

> **A:** To give keywords priority over identifiers when both patterns match the same string. Since `then` also matches the identifier pattern, listing the keyword pattern first ensures it is recognized as `THEN` (keyword), not as `ID`.

**Q5. What happens to input characters that don't match any pattern in Lex?**

> **A:** By default, they are copied verbatim to `stdout`. This default behavior can be overridden by adding a catch-all rule: `. { /* action */ }`.

**Q6. What is `yywrap()`, and when is it called?**

> **A:** `yywrap()` is called by Lex when the end of the input is reached. If it returns 1, scanning stops. It can be overridden to switch to a new input file, allowing Lex to process multiple files in sequence.

**Q7. What is the difference between Lex and Flex?**

> **A:** Flex (Fast Lex) is a faster, modern replacement for Lex. It is backward compatible (most Lex programs work with Flex), more efficient, and supports additional features. Both serve the same purpose — generating C-based lexers from regex specifications.

**Q8. List the three sections of a Lex program and what each contains.**

> **A:** (1) Declarations — C includes, `#define`s in `%{...%}`, and regular definitions. (2) Translation rules — pattern-action pairs. (3) Auxiliary functions — any legal C code, typically `main()`.

**Q9. Trace the output of this Lex spec on input `aab`:**

> ```
> aa  printf("1");
> a   printf("2");
> ab  printf("3");
> ```
> 
> **A:** Lex applies Maximum Munch. The longest match at position 0 is `aa` (matches rule 1). Output: `1`. Remaining input: `b`. No rule matches `b`, so `b` is echoed to stdout. Final output: `1b`. Tokens: `aa`, then `b` (unmatched).

**Q10. What does `yymore()` do?**

> **A:** Instead of replacing `yytext` with the new match, `yymore()` appends the next matched string to the current value of `yytext`, allowing multi-rule accumulation of a lexeme.

## 14. Design of a Lexical-Analyzer Generator

This section describes how a lexer generator (like Lex) actually works internally — connecting the theory to the implementation.

### The Big Picture

Given a set of patterns (regular expressions) with associated actions:

```
p1   { action A1 }
p2   { action A2 }
p3   { action A3 }
```

The generator:

1. Converts each `pᵢ` to an NFA using Thompson's construction.
2. Combines all NFAs into one: a new start state with ε-transitions to each sub-NFA.
3. Simulates the combined NFA (or converts to DFA first) on the input.

### Pattern Priority in Combined NFA

When building the combined NFA, each pattern's accepting state is **tagged** with its action. When the combined NFA/DFA reaches an accepting state that corresponds to multiple patterns, the **pattern listed first** wins (first-listed priority rule).

### Recognizing the Longest Match

The lexer must find the **longest prefix** of the remaining input that matches any pattern. The algorithm:

1. Begin simulation at the start state of the combined NFA/DFA.
2. Keep reading input and advancing the `forward` pointer.
3. Track the **last accepting state** seen and the position of `forward` at that point.
4. When the simulation gets stuck (dead state or no valid transition), **retract** `forward` to the position of the last accepting state.
5. Execute the action associated with that accepting state.
6. Set `lexemeBegin` to just after the found lexeme and repeat.

### Example (from the book)

Patterns:

```
a       { action A1 }
abb     { action A2 }
a*b+    { action A3 }
```

Input: `aaba`

Trace through the NFA states:

- After `a`: in set containing state for pattern `a` (A1 could fire)
- After `aa`: still in states
- After `aab`: in accepting state for `a*b+` (A3 can fire); this is the **longest match** so far
- After `aaba`: dead (no transitions from accepting state on `a`)
- **Retract** to position after `aab` — lexeme is `aab`, fire action A3.

### DFA States for Multiple Patterns

When DFA state {s1, s2, ...} contains two accepting states (corresponding to patterns A and B), the pattern listed **first** in the specification wins.

### The Lookahead Operator `/`

The **lookahead operator** `r1/r2` means: match `r1` only if it is followed by `r2`, but the lexeme is only `r1` (r2 is not consumed).

Implementation in NFA:

- Treat `/` as ε in the NFA.
- When the combined NFA enters its accepting state (after reading `r1 r2`), the actual lexeme end is the position after reading `r1` (where the ε-transition for `/` occurred).

Fortran IF example: `IF / ( [a..z]* )` — recognize `IF` only if followed by `(`.

---

### 🔎 Q&A — Lexical-Analyzer Generator Design

**Q1. How does a lexer generator combine multiple NFA patterns into one?**

> **A:** A new start state is created with ε-transitions to the start states of each individual NFA. The accepting states of each sub-NFA are tagged with their respective patterns/actions.

**Q2. How does the "longest match" strategy work when simulating the combined NFA?**

> **A:** The simulator reads input and tracks the last position at which the NFA was in an accepting state. When no further transitions are possible, it retracts `forward` to that last accepting position and fires the corresponding action.

**Q3. When a DFA state corresponds to two different accepting NFA states (for two different patterns), which action fires?**

> **A:** The action for the pattern listed **first** in the Lex specification.

**Q4. What does the lookahead operator `/` do, and how is it implemented in an NFA?**

> **A:** `r1/r2` matches `r1` only when followed by `r2`, but only `r1` is consumed as the lexeme. In the NFA, `/` is treated as an ε-transition; the actual lexeme end is the position where this ε-transition occurred.

**Q5. In the NFA simulation for `a`, `abb`, `a*b+` on input `aaba`, what is the recognized lexeme and why?**

> **A:** The lexeme is `aab` (action A3 for `a*b+`). After reading `aab`, the NFA reaches an accepting state for `a*b+`. Reading the fourth character `a` leads to a dead state, so we retract to after `aab` — the longest match.

## 15. Tricky Problems & Special Languages

Lexical analysis is not always straightforward. Some languages present unique challenges.

### Fortran — Unbounded Lookahead

In fixed-format Fortran, **blanks are ignored** in statements. This creates ambiguity:

```fortran
DO 5 I = 1.25    →  DO5I = 1.25  (assignment, identifier DO5I)
DO 5 I = 1,25    →  DO 5 I = 1,25  (loop: keyword DO, label 5, variable I)
```

The lexer cannot decide whether `DO` is a keyword or the start of an identifier until it sees a **comma or a dot** much later in the statement. This requires potentially unlimited lookahead — early lexers handled this by reading the entire statement before tokenizing.

### C++ — Context-Dependent Tokens

In C++, the characters `<`, `>`, `<<`, `>>` are **overloaded**:

|Context|Meaning|
|---|---|
|`a < b`|relational operator|
|`vector<int>`|template bracket|
|`a >> 4`|bitwise right shift|
|`A<B<C>>D`|nested template (two `>`s, not a shift)|

The lexer alone cannot resolve these — it needs **parser feedback** to determine context. This is why C++ lexing is hard, and why C++ compilers often combine lexer and parser phases more tightly.

### PL/I — Unreserved Keywords

In **PL/I**, keywords are **not reserved** — they can also be used as variable names!

```pli
DECLARE IF FIXED BIN;   -- IF is used as a variable name
IF IF > 0 THEN ...      -- IF is used as both variable and keyword!
```

The lexer cannot classify `IF` as a keyword or identifier in isolation — only the parser, seeing the full syntactic context, can decide.

### Python — Indentation-Sensitive Lexing

Python uses **indentation** to define block structure (instead of `{` and `}`). The lexer must handle this.

The Python lexer uses a **stack of indentation levels**:

- A single `0` is pushed initially.
- For each logical line, the indentation level is compared to the stack top.
    - **Equal:** nothing happens.
    - **Larger:** push the new level; emit an `INDENT` token.
    - **Smaller:** pop levels off the stack until equal; emit a `DEDENT` token for each popped level.
- At end-of-file, emit `DEDENT` for each level > 0 remaining on the stack.

The parser sees `INDENT` as start-of-block and `DEDENT` as end-of-block.

---

### 🔎 Q&A — Tricky Problems & Special Languages

**Q1. Why does Fortran present such a challenge for lexical analysis?**

> **A:** Because blanks are ignored in Fortran statements, the lexer cannot immediately tell whether `DO5I` is an identifier or the keyword `DO` followed by a loop variable. It may need to read far ahead (past a comma or dot) before deciding — requiring potentially unbounded lookahead.

**Q2. Why is C++ lexical analysis particularly hard?**

> **A:** Because the same symbols (`<`, `>`, `>>`, `<<`) have different meanings depending on syntactic context (relational operators, template brackets, bitwise shifts, stream operators). The lexer needs information from the parser to resolve these ambiguities.

**Q3. What is an "unreserved keyword" in PL/I, and why does it complicate lexing?**

> **A:** In PL/I, keywords like `IF` can also be used as programmer-defined identifiers. The lexer cannot classify a token as keyword vs. identifier on its own — it needs the syntactic context from the parser. This breaks the clean separation between lexing and parsing.

**Q4. What special tokens does Python's lexer emit to handle indentation?**

> **A:** Three special tokens: `NEWLINE` (end of a logical line), `INDENT` (beginning of a new block, when indentation increases), and `DEDENT` (end of a block, when indentation decreases).

**Q5. In Python's indentation stack, what happens if a DEDENT brings the indentation to a level that is NOT on the stack?**

> **A:** It is a syntax error — the indentation level must exactly match one of the levels already on the stack. Python enforces consistent indentation.

**Q6. What does "syntactic sugar" mean in the context of lexical analysis?**

> **A:** Syntactic sugar is syntax that makes code easier to read/write but that the compiler expands into a more fundamental form ("desugaring"). Examples: `b[j]` → `*(b+j)` in C; `x += y` → `x = x + y`. The lexer/parser transforms sugar into canonical forms.

> **A:** The token name is the abstract category (e.g., `relop`, `id`, `number`). The attribute is extra information about the specific lexeme (e.g., `LT` for `<`, a pointer to the symbol table for an identifier, or `42` for a number literal).

**Q2. Explain why, during NFA simulation on the input `aaba` with patterns `a`, `abb`, `a*b+`, the lexeme `aab` is chosen.**

> **A:** The NFA simulation reads `a` (accepting for pattern `a`), `aa` (still active), `aab` (accepting for `a*b+` — longest match so far), `aaba` (no transition possible — dead). Since the last accepting state was at position 3 (after `aab`), we retract and choose `aab` as the lexeme with action A3 (`a*b+`).

**Q3. What is the purpose of the `ε-closure` operation in NFA → DFA conversion?**

> **A:** After following input transitions, there may be ε-transitions available. `ε-closure` computes all states reachable by ε-transitions from a given set of states. It ensures the DFA state captures all NFA states that could be active "for free" (without consuming input).

**Q4. Given this Lex specification, what does it print for input `abcaa cacaaabbaaabcaaca`?**

> ```
> a*b       printf("1");
> ca        printf("2");
> a*ca*     printf("3");
> ```
> 
> **A:** Output: `13 2311132`. Tokens: `ab`, `caa`, `ca`, `caaa`, `b`, `b`, `aaab`, `caa`, `ca`. (Maximum Munch + first-listed priority.)

**Q5. A student says "every DFA is also an NFA." Is this correct? Explain.**

> **A:** Yes, this is correct. A DFA is a special case of an NFA where: (a) there are no ε-transitions, and (b) for each state and input symbol, there is exactly one transition (i.e., δ maps to a singleton set). Every DFA trivially satisfies the NFA definition.

**Q6. Why is the two-buffer input scheme important for the lexical analyzer's performance?**

> **A:** It minimizes I/O overhead by reading large blocks at once, allows efficient lookahead without re-reading input, and the sentinel character eliminates per-character boundary checks, making the inner scan loop as fast as possible.

**Q7. What is "conflict resolution" in Lex, and what two rules govern it?**

> **A:** Conflict resolution handles cases where multiple patterns could match. Rule 1: prefer the **longest match** (maximum munch). Rule 2: if two patterns match the same longest prefix, prefer the **first-listed** pattern in the Lex program.

**Q8. Why might a DFA constructed by subset construction have up to 2^n states?**

> **A:** Because each DFA state corresponds to a _subset_ of the n NFA states, and there are 2^n possible subsets of an n-element set. In the worst case, all subsets are reachable.

**Q9. Describe the complete pipeline from a Lex specification to a running lexical analyzer.**

> **A:** (1) Lex reads the `.l` file. (2) It applies Thompson's construction to convert each regex to an NFA. (3) All NFAs are combined with a new start state. (4) Subset construction converts the combined NFA to a DFA. (5) The DFA is minimized. (6) Lex generates `lex.yy.c` — C code with DFA transition tables. (7) A C compiler compiles `lex.yy.c` to produce the final lexer executable.

**Q10. In Python's indentation processing, what is on the stack initially, and why is it never popped off?**

> **A:** A single `0` is pushed before the first line. It is never popped because DEDENT only occurs when the new indentation level is _smaller_ than the stack top — but a level of 0 means we're at the outermost scope, and code cannot legally be indented _less_ than 0.

## 📌 Quick Reference Summary

|Concept|Definition|
|---|---|
|**Lexeme**|Actual sequence of characters in the source matching a pattern|
|**Token**|Abstract `<name, attribute>` pair passed to the parser|
|**Pattern**|Regular expression describing valid lexemes for a token|
|**NFA**|Automaton with ε-transitions and multiple possible transitions per symbol|
|**DFA**|Automaton with exactly one transition per (state, symbol) pair|
|**ε-closure(S)**|All states reachable from S via ε-transitions|
|**Thompson's Construction**|Algorithm to convert a regex to an NFA|
|**Subset Construction**|Algorithm to convert an NFA to a DFA|
|**Maximum Munch**|Always match the longest possible prefix|
|**First-listed priority**|Among equal-length matches, the first rule in the Lex program wins|
|**yytext**|Lex variable holding the current matched lexeme string|
|**yyleng**|Lex variable holding the length of the current matched lexeme|
|**INDENT/DEDENT**|Special Python tokens for block structure from indentation|
|**Lookahead `/`**|Match `r1` only when followed by `r2`; lexeme is only `r1`|

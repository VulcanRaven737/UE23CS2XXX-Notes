# Compiler Design – Chapter 1: Complete Study Notes

---

## 1. Why Do We Need a Compiler?

Programming languages serve as a bridge between human thinking and machine execution. High-level languages like C, Java, and Python are designed to be readable and writable by humans. However, machines operate exclusively on binary (machine code) and cannot directly understand these high-level constructs.

A **compiler** is the software system that bridges this gap — it reads a program written in a **source language** and translates it into an equivalent program in a **target language** (typically machine code), while also reporting any errors it detects during translation.

---

## 2. History of Compilers

|Year|Milestone|
|---|---|
|1952|Grace Hopper wrote the first implemented compiler for the **A-0 System** language. She also coined the term _compiler_. The A-0 functioned more like a loader/linker than a modern compiler.|
|1957|The **FORTRAN** compiler, led by John W. Backus at IBM, became the first commercially available compiler. It took **18 person-years** to build.|
|1960|**COBOL** became the first language compiled on **multiple platforms**.|

### What Does Compiler Design Touch Upon?

Compiler design is deeply interdisciplinary. It draws from:

- **Programming Languages** — compiler writers must track new language features
- **Machine Architecture** — compilers must exploit new hardware features
- **Language Theory** — uses finite automata (lexing) and push-down automata (parsing)
- **Algorithms** — greedy algorithms (register allocation), graph algorithms (dead-code elimination), dynamic programming (instruction selection)
- **Software Engineering** — working on a compiler is one of the most complex software engineering experiences

---

### 🧠 Q&A — History & Motivation

**Q1: Why can't machines directly execute high-level language programs?**

> Machines operate on binary instructions specific to their architecture. High-level languages use abstract constructs (variables, loops, functions) that have no direct hardware equivalent and must be translated.

**Q2: Who coined the term "compiler" and when?**

> Grace Hopper in 1952, for the A-0 System language.

**Q3: What was significant about the FORTRAN compiler?**

> It was the first _commercially available_ compiler (1957), took 18 person-years to create, and demonstrated that automatic translation from high-level language to machine code was feasible.

**Q4: Why is compiler design considered an interdisciplinary subject?**

> Because it requires knowledge of programming languages, machine architecture, formal language theory (automata), algorithms, and software engineering principles all working together.

**Q5: What is the difference between a compiler and a loader/linker?**

> A compiler translates source code to target code. A loader places the executable into memory for execution. A linker combines separately compiled modules. Grace Hopper's A-0 was more of a loader/linker than a true compiler.

## 3. Types of Compilers

|Type|Description|
|---|---|
|**One-pass compiler**|Makes a single pass over the source code; fast but limited in optimization|
|**Multi-pass compiler**|Makes multiple passes; allows more thorough analysis and optimization|
|**Cross compiler**|Runs on one platform but generates code for a _different_ target platform (e.g., compiling ARM code on an x86 machine)|
|**Native compiler**|Generates code for the same platform it runs on|
|**Decompiler**|Translates from low-level (machine code) back to high-level source code|
|**Source-to-source compiler (Transpiler)**|Translates from one high-level language to another (e.g., TypeScript → JavaScript)|
|**Self-hosting compiler (Bootstrapping)**|A compiler written in the same language it compiles|
|**Compiler-compiler**|A tool that takes a grammar specification and generates a compiler/parser (e.g., YACC, Bison)|

## 4. Language Processors

### 4.1 What is a Compiler?

A **compiler** is a program that:

1. Reads a program in the **source language**
2. Translates it into an equivalent program in the **target language**
3. Reports any errors detected during translation

```
Source Program → [Compiler] → Target Program
```

If the target is executable machine code:

```
Input → [Target Program] → Output
```

---

### 4.2 What is an Interpreter?

An **interpreter** does **not** produce a target program as output. Instead, it **directly executes** the operations specified in the source program, statement by statement, using the user's input.

```
Source Program + Input → [Interpreter] → Output
```

**Key differences:**

|Feature|Compiler|Interpreter|
|---|---|---|
|Output|Target program (machine code)|No separate output program|
|Execution speed|Faster (pre-compiled)|Slower (interprets on the fly)|
|Error diagnostics|Detects errors before execution|Better diagnostics (statement-by-statement)|
|Examples|C, C++, Fortran|Python, Ruby, PHP, MATLAB, Lisp|

---

### 4.3 Hybrid Compilers

Java uses a **hybrid** approach:

1. Java source code is compiled into **bytecode** (intermediate form)
2. Bytecode is interpreted by the **Java Virtual Machine (JVM)**
3. **Just-In-Time (JIT) compilers** translate bytecode to machine code immediately before execution for speed

```
Java Source → [Translator] → Bytecode → [Virtual Machine] → Output
                                         ↑
                                       Input
```

**Benefit:** Bytecode compiled on one machine can run on any machine with a JVM — _write once, run anywhere_.

---

### 🧠 Q&A — Language Processors

**Q1: What is the fundamental difference between a compiler and an interpreter?**

> A compiler translates the entire source program into a target program first, then the target program is executed. An interpreter executes the source program directly, statement by statement, without producing a separate target program.

**Q2: Why are compiled programs generally faster than interpreted ones?**

> The compiler does all the translation work upfront, and the resulting machine code runs directly on the hardware. An interpreter must translate and execute each statement at runtime, adding overhead.

**Q3: Why do interpreters give better error diagnostics?**

> Because they execute the program statement by statement. When an error occurs, they can immediately point to the exact statement that caused it, with the exact runtime context.

**Q4: What is a JIT compiler and why is it used?**

> A Just-In-Time compiler translates bytecode into native machine code just before execution. It provides the portability benefit of interpretation (bytecode runs anywhere) with performance closer to native compilation.

**Q5: What type of language processor does Python use?**

> Python primarily uses an interpreter, though CPython (the standard implementation) does compile Python source to bytecode first, then interprets the bytecode.

## 5. The Language Processing System (Compiler Is Not Alone!)

A compiler doesn't work in isolation. Creating an executable from source code involves several tools:

```
Source Program
      ↓
 [Preprocessor]  ← Inclusion of header files, macro expansion, conditional compilation
      ↓
Modified Source Program
      ↓
  [Compiler]
      ↓
Target Assembly Program
      ↓
  [Assembler]
      ↓
Relocatable Machine Code
      ↓
[Linker & Loader] ← Library files, relocatable object files
      ↓
Target Machine Code
```

### 5.1 Preprocessor

- Collects the source program from separate modules/files
- Handles **inclusion of header files** (e.g., `#include <stdio.h>`)
- Expands **macros** (`#define`)
- Handles **conditional compilation** (`#ifdef`, `#endif`)

In GCC: `$ gcc -E hello.c > hello.i`

### 5.2 Compiler

Translates preprocessed source into **assembly language**. Assembly is chosen as output because:

1. It is easier to produce as output
2. It is easier to debug
3. It makes code relocation much easier

In GCC: `$ gcc -S hello.i` then `$ cat hello.s`

### 5.3 Assembler

Converts assembly language into **relocatable machine code** (object files).

In GCC: `$ as hello.s -o hello.o`

### 5.4 Linker and Loader

**Linker:** Resolves external memory addresses — code in one file may refer to a location in another file. Combines relocatable object files and library files.

**Loader:** Puts all executable object files into memory for execution.

In GCC: `$ gcc hello.o` then `$ ./a.out`

### 5.5 Static vs. Shared Libraries

|Feature|Static Library (.a / .lib)|Shared Library (.so / .dll)|
|---|---|---|
|Linking time|At compile time|At runtime (dynamic linking)|
|Code in executable|Machine code copied into executable|Only a reference table is stored|
|File size|Larger executable|Smaller executable|
|Memory usage|Each program has its own copy|One copy shared among all programs|
|Updatability|Must recompile to update|Library can be updated independently|

---

### 🧠 Q&A — Language Processing System

**Q1: What is the role of the preprocessor?**

> The preprocessor handles tasks before actual compilation: it includes header files, expands macros, and processes conditional compilation directives. It transforms the source code into a "modified source program" ready for the compiler.

**Q2: Why does the compiler produce assembly code rather than machine code directly?**

> Assembly is easier to produce as output, easier to debug (humans can inspect it), and makes code relocation easier. An assembler then converts it to machine code.

**Q3: What is relocatable machine code?**

> Machine code that can be placed at any memory address. The actual addresses are determined later by the linker/loader, which is why it's called "relocatable."

**Q4: What is the difference between a linker and a loader?**

> A linker resolves external references between object files and libraries, combining them into an executable. A loader places the completed executable into memory for actual execution.

**Q5: What is the key advantage of shared/dynamic libraries over static libraries?**

> Shared libraries save disk space and memory because a single copy is shared among multiple programs. They can also be updated without recompiling the programs that use them.

**Q6: What does the GCC command `gcc -E hello.c > hello.i` do?**

> It runs only the preprocessor stage, producing the preprocessed output in `hello.i` (`.i` for C, `.ii` for C++).

## 6. The Structure of a Compiler

### 6.1 Front End and Back End

A compiler has two major parts:

```
Source Program → [Front End] → IR → [Back End] → Target Program
```

|Part|Also Called|Role|
|---|---|---|
|**Front End**|Analysis Part|Understands the source program|
|**Back End**|Synthesis Part|Maps program to the target machine|

### 6.2 Analysis vs. Synthesis

**Analysis (Front End):**

- Breaks the source program into constituent pieces
- Imposes grammatical structure
- Creates an **Intermediate Representation (IR)**
- Collects information into the **Symbol Table**

**Synthesis (Back End):**

- Constructs the target program from the IR and Symbol Table

This is also called a **Two-Pass Compiler:**

```
Source Program → [Front-end / Pass-1] → Intermediate Code → [Back-end / Pass-2] → Target Program
                                    ↕
                              Symbol Table
```

## 7. The Six Phases of a Compiler

```
Character Stream
      ↓
[1. Lexical Analyzer]
      ↓
Token Stream
      ↓
[2. Syntax Analyzer]
      ↓
Syntax Tree
      ↓
[3. Semantic Analyzer]
      ↓
Syntax Tree (annotated)
      ↓
[4. Intermediate Code Generator]
      ↓
Intermediate Representation
      ↓
[5. Machine-Independent Code Optimizer]
      ↓
Intermediate Representation (optimized)
      ↓
[6. Code Generator]
      ↓
Target Machine Code
```

_(The Symbol Table interacts with all six phases)_

---

### Phase 1: Lexical Analysis (Scanning)

**What it does:**

- Reads the stream of characters making up the source program
- Groups characters into meaningful sequences called **lexemes**
- Produces **tokens** of the form `<token-name, attribute-value>`

**Types of Lexemes:**

1. Keywords (`while`, `int`, `return`)
2. Identifiers (`position`, `rate`)
3. Literals/Constants (`60`, `3.14`, `"hello"`)
4. Operators (`+`, `*`, `=`)
5. Punctuators (`;`, `{`, `}`)

**Example:** For the statement `position = initial + rate * 60`

|Lexeme|Token (Method 1)|Token (Method 2)|
|---|---|---|
|`position`|`<id, 1>`|`<id, 1>`|
|`=`|`<=>`|`<assign_op, =>`|
|`initial`|`<id, 2>`|`<id, 2>`|
|`+`|`<+>`|`<arith_op, +>`|
|`rate`|`<id, 3>`|`<id, 3>`|
|`*`|`<*>`|`<arith_op, *>`|
|`60`|`<60>`|`<num, 60>`|

**Symbol Table (built during lexical analysis):**

|Index|Name|Type|Size|Scope|
|---|---|---|---|---|
|1|position||||
|2|initial||||
|3|rate||||

**Important notes:**

- When an identifier is found, it's entered into the symbol table
- For keywords, operators, and punctuators, the attribute value is optional
- The lexer can theoretically be modeled as a large **Deterministic Finite Automaton (DFA)**
- Tool used: **LEX/FLEX** — reads regular expressions, generates a C scanner

---

### 🧠 Q&A — Lexical Analysis

**Q1: What is a lexeme? How is it different from a token?**

> A **lexeme** is the actual character sequence from the source program (e.g., `position`, `60`). A **token** is a structured pair `<token-name, attribute-value>` that represents the lexeme in a form useful for subsequent phases (e.g., `<id, 1>`, `<num, 60>`).

**Q2: What is the attribute value in a token, and when is it omitted?**

> For identifiers, the attribute value is an index/pointer to the symbol table entry. For numbers, it may be the value itself. It is omitted (optional) for keywords, operators, and punctuation symbols because the token-name alone is sufficient.

**Q3: Why is the lexical analyzer modeled as a DFA?**

> Regular expressions (used to define token patterns) are equivalent in power to Finite Automata. A DFA can efficiently recognize all token classes in a single pass through the input.

**Q4: What tool is used to automatically generate a lexical analyzer?**

> LEX (or its GNU version FLEX). It reads a specification file containing regular expressions and generates a C routine that performs lexical analysis.

**Q5: What is the role of the symbol table during lexical analysis?**

> When the lexer encounters an identifier for the first time, it creates an entry in the symbol table. Subsequent phases then fill in more attributes (type, scope, size) for that entry.

**Q6: Can the lexical analyzer detect all errors in a program?**

> No. It can only detect errors at the level of individual tokens — for example, an illegal character or a malformed number. It cannot detect structural or type errors, which require syntax and semantic analysis.

---

### Phase 2: Syntax Analysis (Parsing)

**What it does:**

- Takes the token stream from the lexer
- Uses the **first component** (token-name) of each token
- Builds a **tree-like intermediate representation (syntax tree / parse tree)**
- The tree represents the **grammatical structure** of the token stream

**Why not just use regular expressions?**

- Regular expressions cannot check **balanced tokens** (e.g., matching parentheses)
- Syntax analysis uses **Context-Free Grammars (CFG)**, which are recognized by **Push-Down Automata**

**Example CFG:**

```
AssignExpr → id = E
E → E + T | T
T → T * F | F
F → id | num | (E)
```

**For `position = initial + rate * 60`**, the syntax tree is:

```
        =
       / \
   <id,1>  +
           / \
       <id,2>  *
              / \
          <id,3>  60
```

**Why syntax trees over parse trees?**

- Closer to the **semantic meaning** of constructs
- More **compact** than a full parse tree

---

### 🧠 Q&A — Syntax Analysis

**Q1: What is parsing?**

> Parsing is the process of determining how a string of terminal symbols (tokens) can be generated by a grammar. It verifies that the token sequence conforms to the grammatical rules of the language.

**Q2: Why is a CFG used for syntax analysis rather than regular expressions?**

> Regular expressions cannot handle nested or recursive structures like balanced parentheses or nested if-else statements. CFGs can express these through recursive productions and are recognized by push-down automata.

**Q3: What is the difference between a parse tree and a syntax tree?**

> A **parse tree** shows every step of the grammar derivation, including all intermediate non-terminals. A **syntax tree** (abstract syntax tree / AST) is a condensed version that captures only the essential hierarchical structure, omitting unnecessary grammar details. Syntax trees are more compact and semantically meaningful.

**Q4: What does each node in a syntax tree represent?**

> Each **interior node** represents an operation (e.g., `+`, `*`, `=`). The **children** of that node represent the arguments/operands of the operation.

**Q5: What is an example of an error a syntax analyzer can catch but a lexical analyzer cannot?**

> A missing semicolon, mismatched parentheses, or an expression like `int x = ;` (missing right-hand side). These are structurally invalid but contain only valid individual tokens.

---

### Phase 3: Semantic Analysis

**What it does:**

- Uses the **syntax tree** and **symbol table** to check semantic consistency
- Gathers **type information** and saves it in the syntax tree or symbol table
- Performs **type checking** — each operator must have matching operands
- Performs **coercions** when allowed (e.g., converting `int` to `float`)

**Two kinds of semantics:**

#### Static Semantics (Compile-Time Checks)

Rules checked at compile time without running the program:

- Type mismatch in computation (`int a = "value"` — error!)
- Undeclared variable
- Reserved identifier misuse
- Multiple declaration of a variable in the same scope
- Accessing out-of-scope variable
- Actual vs. formal parameter mismatch (wrong number/type of arguments)

#### Dynamic Semantics (Runtime Behavior)

Defines what happens when the program runs:

- Whether arithmetic overflow occurs
- Whether array bounds are exceeded
- Whether recursion exceeds stack limits
- Whether heap memory is insufficient

**Example of coercion:** For `position = initial + rate * 60`, since all variables are `FLOAT` but `60` is an integer, the semantic analyzer inserts an **`inttofloat`** conversion node:

```
        =
       / \
   <id,1>  +
           / \
       <id,2>  *
              / \
          <id,3>  inttofloat
                      |
                      60
```

---

### 🧠 Q&A — Semantic Analysis

**Q1: What is the difference between static and dynamic semantics?**

> **Static semantics** are rules checked at compile time (type checking, variable declaration). **Dynamic semantics** describe the runtime behavior of the program (overflow, array bounds violations).

**Q2: Give an example that passes syntax analysis but fails semantic analysis.**

> `int a = "value";` — this is syntactically correct (it matches the grammar for a declaration with assignment), but semantically wrong because you're assigning a string to an integer variable (type mismatch).

**Q3: What is type coercion in the context of semantic analysis?**

> When an operator receives operands of compatible but different types, the compiler automatically converts (coerces) the lower type to the higher type. For example, in `rate * 60` where `rate` is float and `60` is int, the integer `60` is coerced to `60.0` (float).

**Q4: What is a Syntax-Directed Definition (SDD)?**

> A Syntax-Directed Definition is a CFG augmented with semantic rules attached to grammar productions. When the parser applies a production, the corresponding semantic rules are evaluated. It's the formal mechanism used to implement semantic analysis.

**Q5: What does CFG + semantic rules equal?**

> Syntax Directed Definitions (SDDs).

**Q6: What information does the semantic analyzer store in the symbol table?**

> Type information for each identifier (e.g., FLOAT, INT), which is then used during intermediate code generation and code generation.

---

### Phase 4: Intermediate Code Generation

**What it does:**

- After semantic analysis, generates an explicit low-level, machine-like **Intermediate Representation (IR)**
- The IR is for an **abstract machine** — not tied to any specific hardware

**Two critical properties of IR:**

1. Easy to produce (from the syntax tree)
2. Easy to translate into the target machine code

**Types of IR:**

```
                    IR
                   /  \
            Graphical   Linear
            /      \    /    \
      Syntax      DAG  3-addr  SSA
      Tree              code
```

#### Three-Address Code (TAC)

- Each instruction has **at most one operator** on the right side
- Each instruction has **at most three operands** (addresses)
- Compiler-generated **temporary variables** hold intermediate values

**Example:** For `position = initial + rate * 60`

After semantic analysis (with coercion):

```
t1 = inttofloat(60)
t2 = id3 * t1
t3 = id2 + t2
id1 = t3
```

**Three key observations about TAC:**

1. Each instruction has at most one operator on the right — this **fixes the evaluation order**
2. The compiler must generate temporary names for intermediate values
3. Some instructions have fewer than three operands (e.g., `t1 = inttofloat(60)`, `id1 = t3`)

#### Directed Acyclic Graph (DAG)

- Identifies **common subexpressions** (subexpressions that appear more than once)
- In a syntax tree, a common subexpression is **replicated** as many times as it appears
- In a DAG, a **node has more than one parent** if it represents a common subexpression — saving space and computation

**Example:** For `a + a*(b-c) + (b-c)*d`

- The syntax tree would have `(b-c)` appearing twice
- The DAG has a single node for `(b-c)` with two parents

#### Why Intermediate Code?

IR enables **retargeting** — you can:

- Have multiple front ends (C, C++, Java, Fortran) all producing the same IR
- Have multiple back ends (x86, ARM, MIPS, SPARC) all consuming the same IR
- This means M front ends + N back ends only needs M+N components instead of M×N compilers

---

### 🧠 Q&A — Intermediate Code Generation

**Q1: What are the two most important properties an IR must have?**

> (1) It must be easy to produce from the syntax/semantic tree. (2) It must be easy to translate into the target machine code.

**Q2: What is Three-Address Code (TAC)?**

> TAC is a linear IR where each instruction has at most one operator on the right side and at most three operands (addresses). Temporary variables are generated by the compiler to hold intermediate results.

**Q3: What is the purpose of compiler-generated temporary variables in TAC?**

> They hold intermediate values computed during evaluation of complex expressions. Using temporaries allows TAC to be easily rearranged and optimized.

**Q4: What is a DAG and how does it differ from a syntax tree?**

> A Directed Acyclic Graph (DAG) is a graphical IR that identifies common subexpressions. Unlike a syntax tree (where common subexpressions are duplicated), in a DAG a node representing a common subexpression has multiple parents, avoiding redundant computation.

**Q5: Why is intermediate code beneficial for compiler portability?**

> With IR, you only need M front ends and N back ends (M+N total) instead of M×N separate compilers for every language-architecture combination. The front ends all generate the same IR; the back ends all consume it.

### Phase 5: Code Optimization

**What it does:**

- An **optional** phase
- Transforms the IR to produce **faster code** that uses **fewer resources**
- Maintains **functional correctness** — the output must be the same

**Common optimization techniques:**

|Technique|Description|Example|
|---|---|---|
|**Constant Propagation**|Substitutes a variable with its assigned constant|`a=10; t1=5*a` → `t1=5*10`|
|**Constant Folding**|Computes constant expressions at compile time|`t1=5*10` → `t1=50`|
|**Common Subexpression Elimination**|Eliminates repeated computation of same expression|Compute `b-c` once, reuse|
|**Dead Code Elimination**|Removes variables/statements whose values are never used|Remove `t3 = id2 + t2; id1 = t3` → `id1 = id2 + t2`|
|**Unreachable Code Elimination**|Removes code that can never be executed|Code after `return`|

**Example from our running example:**

Before optimization (TAC 1.3):

```
t1 = inttofloat(60)
t2 = id3 * t1
t3 = id2 + t2
id1 = t3
```

After optimization (TAC 1.4):

```
t1 = id3 * 60.0
id1 = id2 + t1
```

**Two optimizations applied:**

1. `inttofloat(60)` computed at **compile time** → replaced with `60.0` (constant folding)
2. `t3` is used only once → eliminated, `id1` assigned directly (dead code / copy propagation)

---

### 🧠 Q&A — Code Optimization

**Q1: Is code optimization mandatory in a compiler? Why or why not?**

> No, it is an optional phase. A compiler can produce correct code without optimization. However, optimization is important for performance-critical applications, and most modern compilers include it (e.g., GCC's `-O1`, `-O2`, `-O3` flags).

**Q2: What is constant folding?**

> Constant folding evaluates constant expressions at compile time rather than at runtime. For example, `t1 = 5 * 10` becomes `t1 = 50` during compilation, so no multiplication is needed at runtime.

**Q3: What is dead code elimination?**

> It removes statements that compute values that are never subsequently used, or variables that are never referenced after their last assignment. This reduces the size and execution time of the program.

**Q4: In our example, why could `inttofloat(60)` be eliminated?**

> Because `60` is a compile-time constant. Its conversion to `60.0` can be done once during compilation, so there's no need to perform the conversion at runtime every time the code executes.

**Q5: What is the key constraint on all code optimizations?**

> Optimizations must preserve **semantic correctness** — the optimized program must produce exactly the same outputs as the original program for all valid inputs.

---

### Phase 6: Code Generation

**What it does:**

- Takes the optimized IR as input
- Maps it to the **target language** (usually machine code or assembly)
- Selects **registers or memory locations** for each variable
- Translates intermediate instructions into sequences of machine instructions

**Key challenge: Register Allocation** — deciding which variables live in which registers at any given point.

**Example:** For optimized TAC (1.4):

```
t1 = id3 * 60.0
id1 = id2 + t1
```

Generated machine code (using registers R1, R2):

```
LDF   R2, id3        ; Load float id3 into R2
MULF  R2, R2, #60.0  ; Multiply R2 by 60.0, store in R2
LDF   R1, id2        ; Load float id2 into R1
ADDF  R1, R1, R2     ; Add R1 and R2, store in R1
STF   id1, R1        ; Store R1 into id1
```

**Notes on the machine code:**

- First operand specifies the destination
- `F` in opcode means it operates on floating-point numbers
- `#` means the operand is an immediate (constant) value

---

### 🧠 Q&A — Code Generation

**Q1: What are the two main tasks of the code generator?**

> (1) **Instruction selection** — choosing the right machine instructions to implement the IR operations. (2) **Register allocation** — deciding which variables to keep in registers versus memory.

**Q2: Why is register allocation a crucial aspect of code generation?**

> Registers are the fastest storage on a machine, but there are a very limited number of them. Good register allocation minimizes slow memory accesses, significantly improving program performance.

**Q3: What does the `#` symbol typically signify in machine code?**

> It denotes an **immediate constant** — a value that is encoded directly in the instruction rather than being loaded from a register or memory location.

**Q4: What does the `F` suffix in instructions like `LDF`, `MULF`, `ADDF` indicate?**

> It indicates the instruction operates on **floating-point** numbers, as opposed to integer operations.

## 8. Symbol Table Management

### What is a Symbol Table?

A **symbol table** is a data structure that stores a **record for each identifier** in the source program, along with its **attributes**.

It acts as a **central repository** used by **all six phases** of the compiler.

### Attributes Stored

**For variable names:**

- Name (actual characters)
- Type (int, float, etc.)
- Storage size (in bytes)
- Scope (local/global, scope level)
- Line declared, lines referenced

**For procedure/function names:**

- Number and types of arguments
- Method of passing each argument (by value, by reference)
- Return type

**For array names:**

- Type of elements
- Number of dimensions
- Array bounds

**For file names:**

- Record size, record type

### Role in Each Phase

|Phase|Symbol Table Role|
|---|---|
|Lexical Analysis|Creates initial entries when identifiers are first encountered|
|Syntax Analysis|Adds attribute info: data type, scope, dimension, location|
|Semantic Analysis|Used heavily for type-checking, scope resolution, verifying declarations|
|Intermediate Code Generation|Stores info for temporary variables; consulted for memory requirements|
|Code Generation|Final addresses and storage details are retrieved|

### Five Core Symbol Table Operations

|Operation|Function|Description|
|---|---|---|
|1|`lookup(s)`|Returns index of entry for name `s`, or 0 if not found|
|2|`insert(s, t)`|Adds new entry for name `s` of token type `t`|
|3|`delete(s)`|Deletes/hides `s` when it goes out of scope|
|4|`set_attribute(s, a)`|Associates attribute `a` with entry `s`|
|5|`get_attribute(s)`|Retrieves attribute associated with entry `s`|

### Implementation Mechanisms

|Method|Complexity|Notes|
|---|---|---|
|**Linear list** (sorted/unsorted)|O(n) lookup|Simplest to implement; poor for large programs|
|**Hash table**|O(1) average|Best performance; most commonly used|
|**Binary search tree**|O(log n)|Good balance between simplicity and performance|

### Scoping and Multiple Symbol Tables

- Programs can have multiple declarations of the **same identifier** in different scopes
- A **separate symbol table is created for each scope**
- Scopes are numbered (level-0 = global, level-1, level-2, ...)
- **Scope:** The portion of the program to which a declaration applies

**Example — scope representation:**

```
Global symtab: {x: int, f: fun(int→void), g: fun(int→int)}
    ↓
func f symtab: {m: arg int, x: var float, y: var float}
    ↓
    inner block symtabs: {i,j: var int}, {x,l: var int}

func g symtab: {n: arg int, t: var bool}
```

### 🧠 Q&A — Symbol Table Management

**Q1: What is a symbol table and why is it needed?**

> A symbol table is a data structure that records information (attributes) about every identifier in the source program. It's needed because compilers must track properties like type, scope, and size of every variable, function, and other named entity throughout all phases of compilation.

**Q2: Why is a hash table the most preferred symbol table implementation?**

> Hash tables provide O(1) average-case lookup, insertion, and deletion, making them highly efficient even for large programs with thousands of identifiers. Linear lists have O(n) lookup which is too slow for large programs.

**Q3: Why are multiple symbol tables needed (one per scope)?**

> Because the same identifier name can be declared in multiple scopes with different meanings (e.g., a local variable `x` in a function vs. a global variable `x`). Separate tables per scope allow proper scope resolution — looking up `x` in the current scope first, then enclosing scopes.

**Q4: What is the `lookup(s)` operation used for?**

> It checks whether an identifier `s` has already been recorded in the symbol table. It returns the index of the entry if found, or 0 if not. This is crucial during semantic analysis to catch use-before-declaration errors.

**Q5: What information does the lexical analyzer store in the symbol table?**

> The lexer creates initial entries for newly encountered identifiers, storing the identifier's name. Subsequent phases fill in attributes like type, scope, size, etc.

**Q6: At what phase is the symbol table most heavily used?**

> Semantic analysis uses it most heavily — for type checking (verifying operator/operand compatibility), scope resolution (finding which declaration an identifier refers to), and verifying declarations (checking that variables are declared before use).

**Q7: What is the scope of a declaration?**

> The scope of a declaration is the portion of the program to which the declaration applies — i.e., the region where the declared name is visible and can be referenced.

## 10. Final Comprehensive Q&A

**Q1: What are the six phases of a compiler in order?**

> Lexical Analysis → Syntax Analysis → Semantic Analysis → Intermediate Code Generation → Code Optimization → Code Generation.

**Q2: Which phases belong to the front end and which to the back end?**

> **Front end (Analysis):** Lexical Analysis, Syntax Analysis, Semantic Analysis, and Intermediate Code Generation. **Back end (Synthesis):** Code Optimization and Code Generation.

**Q3: What is the role of the symbol table relative to the six phases?**

> The symbol table is a shared data structure used by all six phases. It's built during the analysis phases and consulted during the synthesis phases. It stores all identifier attributes needed throughout compilation.

**Q4: Trace through what happens to the token `<id, 3>` across all six phases.**

> (1) **Lexical:** `rate` is recognized as an identifier, entered into symbol table at index 3, token `<id,3>` produced. (2) **Syntax:** The token is used as a leaf node in the syntax tree. (3) **Semantic:** The type of `rate` (FLOAT) is looked up and verified to be compatible with `*`. (4) **IR Gen:** Referenced as `id3` in TAC (`t2 = id3 * t1`). (5) **Optimization:** Referenced as `id3` in optimized TAC (`t1 = id3 * 60.0`). (6) **Code Gen:** Loaded from memory into register R2 (`LDF R2, id3`).

**Q5: What is the key difference between one-pass and multi-pass compilers?**

> A one-pass compiler processes the source code once from start to finish, producing output as it goes. A multi-pass compiler makes multiple passes over the code (or IR), allowing it to gather more information and perform more thorough analysis and optimization. Multi-pass compilers generally produce better code.

**Q6: What errors can each phase detect?**

> **Lexical:** Illegal characters, malformed tokens. **Syntax:** Grammatical errors (missing semicolons, unbalanced parentheses, invalid expression structure). **Semantic:** Type mismatches, undeclared variables, wrong number/type of function arguments. **Optimization/Code Gen:** Typically no new errors; they assume the previous phases produced correct output.

**Q7: Why is it better to compile to assembly rather than directly to machine code?**

> Assembly is easier to produce, easier for developers to inspect and debug, and makes code relocation easier. The assembler (a simpler, well-understood tool) then handles the mechanical translation to machine code.

**Q8: Summarize what an IR must satisfy and why TAC is a popular choice.**

> An IR must be easy to produce from semantic analysis output and easy to translate to target machine code. TAC satisfies both: it closely mirrors what a compiler can derive from expressions (one operation per instruction) and each TAC instruction maps almost directly to one or a few machine instructions. It also allows easy manipulation for optimization.
